# Task Breakdown — Remove Gismeteo and Simplify Weather Config

## Overview

Gismeteo was a comparison-only, plain-HTTP scrape of `www.gismeteo.kz` that fed a
side-by-side block into the morning weather summary. Open-Meteo is and remains the sole
source of truth. This plan removes the entire gismeteo footprint **and** the surrounding
provider-config mechanism:

1. Delete the gismeteo HTTP client, health inspector, coverage repository, and their tests.
2. Delete the `weather_sources` table and its whole "provider config" apparatus. Open-Meteo
   becomes hardcoded, always-on (no `active` toggle, no per-provider row). Behaviour barely
   changes: Open-Meteo's `weather_sources` row was already inert (only `active` mattered) and
   a missing row defaulted to active, so "always-on" is what production already does.
3. Delete the deprecated, always-`NULL` `weather_user_cities.gismeteo_city_id` column via a
   SQLite table rebuild that carries every live row forward (including the just-landed
   forced-thaw rows and their `alert_latched` / `last_notified_at` / `condition_value` state).
4. Reduce the morning summary and the collector to Open-Meteo only.
5. Add four forward migrations (022–025) to drop the two tables, delete gismeteo observation
   rows, and rebuild `weather_user_cities`.
6. Rewrite the CLAUDE.md weather documentation.

The removal is **more entangled than a single delete**: the `provider` column on the largest
weather table, a cross-layer `RenderMorningSummary` signature, a `NewWeatherAgent` signature,
two `cmd/` composition roots, and eight test files are all in the blast radius. Every touch
point is enumerated below.

### Two decisions surfaced for the orchestrator

- **`weather_observations.provider` column — RECOMMEND KEEP (option A).** See
  [Decision D1](#decision-d1--weather_observationsprovider-column). Investigated: `provider`
  is **not** in the PRIMARY KEY (`id` is) and **not** in any UNIQUE constraint. It appears
  only in two composite indexes: `idx_weather_observations_loc_prov_captured
  (location_id, provider, captured_at DESC)` and `idx_weather_observations_loc_prov_date
  (location_id, provider, forecast_date)` (migration 012). Recommendation: keep the column as
  a vestigial constant `'open-meteo'`. Rationale in D1.
- **Does the `provider` concept collapse? — NO, it goes single-valued.** With option A the DB
  column, the `domain.WeatherObservation.Provider` field, and `domain.ProviderOpenMeteo` all
  stay. Only `domain.ProviderGismeteo` and the multi-provider *rendering* are removed. See
  [Decision D2](#decision-d2--does-the-provider-concept-survive).

## Assumptions

- LOCKED scope per the request: remove all gismeteo, drop `weather_sources` and the whole
  provider-config mechanism, drop `gismeteo_city_id`. These are not re-litigated here.
- Latest applied migration is `202607.021`; migrations 011/012/016/017 are already applied to
  production and are **immutable**. New migrations start at `202607.022`.
- No env var, systemd unit, nginx config, or `.env` key references gismeteo or
  `weather_sources` (verified by grep across `deploy/`, `configs/`, `cmd/*/README.md`,
  `Makefile`, `*.service`, `*.conf`). Only `CLAUDE.md` and Go source/doc-comments do.
- The DTO wire contract and the WASM frontend contain **zero** gismeteo references (verified:
  `grep -rni gismeteo internal/dto/ cmd/wasm/` is empty). The morning-summary comparison was
  server-rendered Telegram HTML, never a DTO shape, so the UI is untouched.
- No table has a `FOREIGN KEY ... REFERENCES` into `weather_user_cities`, `weather_sources`,
  or `weather_gismeteo_cities`; there are no triggers or views anywhere (verified by grep).
  The rebuild is therefore FK-safe and needs no `PRAGMA foreign_keys` toggling.
- **Migrator transaction model (corrects the brief):** `sqlitedb.Migrator.Run`
  (`internal/infrastructure/sqlitedb/migrator.go`) applies **all pending migrations in ONE
  shared transaction**, not one-transaction-per-file. Consequence: the four new migrations are
  **atomic as a group** — if any fails, all roll back and the DB stays at its pre-batch state.
  A multi-statement file (the rebuild) is fine: `tx.ExecContext` runs the whole file and the
  `modernc.org/sqlite` driver executes `;`-separated statements (existing multi-statement
  migrations 011/016/017 already rely on this).
- The deploy applies migrations (root one-shot `beacon-migrate`) **before** restarting
  services. The new binaries only ever run against the post-migration schema. A short
  cross-version window exists where the **old** binary runs against the reduced schema; see
  [Risks](#risks).

## Tasks

Tasks are ordered "callers before callees" so the tree stays compilable at every step, with
the schema migrations last (Task 7) so the test schema loses tables/columns only after all
code stopped referencing them. See [Execution Order](#execution-order) for the dependency
graph and what can run in parallel.

## Decision D1 — `weather_observations.provider` column

**Recommendation: KEEP the column (option A). Orchestrator to confirm.**

Investigation (migration 012):
- PRIMARY KEY is `id` alone. `provider` is **not** in the PK and **not** in any UNIQUE
  constraint.
- `provider` appears only in two composite indexes: `(location_id, provider, captured_at DESC)`
  and `(location_id, provider, forecast_date)`.
- The repository reads it via `ObtainLatestObservation(ctx, locationID, provider string)` and
  writes `obs.Provider`.

| | Option A — KEEP (recommended) | Option B — DROP |
|---|---|---|
| Migration cost | Zero schema change on the largest weather table (only the row DELETE in 022) | A second full rebuild of `weather_observations` — the biggest weather table — plus redefining two indexes |
| Repository surface | `ObtainLatestObservation(..., provider)` and both indexes stay optimal with a constant provider | Method loses a parameter → ripples into the collection agent, the check agent, and all their tests |
| Query correctness | `WHERE location_id=? AND provider=?` keeps using the composite index | Indexes shrink to `(location_id, captured_at DESC)` etc. — must be recreated |
| Future re-add of a 2nd provider | Free (column already there) | Needs another migration |
| Cost of keeping | One vestigial column holding a constant `'open-meteo'` | — |

The only downside of A is a vestigial constant column, which is cheap and de-risks the change.
B buys nothing functional and adds a rebuild on the highest-churn weather table plus a
cross-layer signature change. **Recommend A.** If the orchestrator picks B, it becomes an
additional Hard task (rebuild `weather_observations`, drop both `_prov_` indexes, add
`(location_id, captured_at DESC)` / `(location_id, forecast_date)` indexes, and drop the
`provider` parameter across the repository, collection agent, check agent, and tests).

## Decision D2 — Does the `provider` concept survive?

- **Retained:** the `weather_observations.provider` column, `domain.WeatherObservation.Provider`,

**Deploy reality (single release):** code + migrations ship in one `r_*` tag. At deploy the
`beacon-migrate` one-shot applies 022–025 atomically, then services restart onto the new
binaries. The task ordering above is an *implementation* sequence, not a deploy sequence — the
release is atomic at the artifact level.

## Risks

- **Highest risk — `weather_user_cities` rebuild loses live data (Task 7).** A wrong or missing
  column in the `INSERT ... SELECT` silently drops user subscriptions, including the forced-thaw
  rows and their `alert_latched` / `last_notified_at` state. Mitigation: explicit column names,
  a before/after row-count and field-level assertion on a **production-shaped** fixture, and the
  existing `weatherusercity_backfill_test.go` must pass against the full chain.
- **Cross-version window at deploy.** Migrations commit atomically *before* the service restart,
  so for a few seconds the **old** binary runs against the reduced schema. The old code names
  `gismeteo_city_id` in its core `weatherUserCitySQLSelect` and `ObtainDistinctWeatherLocations`
  (`MIN(gismeteo_city_id)`), so old web/notifier/collector weather reads error ("no such
  column") until restart completes. FX functionality is unaffected (separate code path); weather
  self-heals on restart. Because the migrator batch is atomic, there is no partial-schema state
  and a failed migration rolls the DB fully back (old binary keeps working). Accepted for a
  single-prod, low-traffic bot. **Alternative if zero transient errors are required:** a two-
  release expand/contract — release 1 removes all code references (Tasks 1–6) but keeps the
  tables/column; release 2 adds the migrations (Task 7). Doubles the release effort; flagged for
  the orchestrator to choose. Default: single release.
- **Placeholder miscount in `RetainWeatherUserCity` (Task 5).** 18→17 `?`. A miss is a runtime
  `sql: expected N arguments` error, not a compile error. Covered by the repo retain test.
- **Migration immutability.** 022–025 filenames/spelling are permanent once applied. A typo
  needs yet another migration to correct.
- **Backfill-test coupling (Task 7).** Migration 021 inserts thaw rows referencing
  `gismeteo_city_id`; 025 then drops the column in the same in-test transaction. Sequential DDL
  in one SQLite transaction handles this, but the backfill test must be re-run to confirm.

## Trade-offs

- **Keep `weather_observations.provider` (D1-A) over dropping it (D1-B):** chose the vestigial
  constant column to avoid a rebuild of the largest weather table and a cross-layer repository
  signature change, for zero functional gain from dropping. Leaves the door open to a future
  second provider without a migration.
- **Drop the whole `weather_sources` mechanism, not just gismeteo’s row:** chose full removal
  (hardcoded always-on Open-Meteo) because Open-Meteo’s row was already inert (only `active`
  mattered) and a missing row already defaulted to active — the config table bought nothing once
  gismeteo is gone, and deleting it removes a whole load-config-and-degrade code path.
- **Collapse `RenderMorningSummary` to a single observation** rather than keeping the variadic
  signature "just in case": chose the simpler signature because there is exactly one provider and
  one caller; a speculative multi-obs API would be dead surface. Re-adding a provider later would
  reintroduce it deliberately.
- **Explicit DELETE of gismeteo observation rows (022)** even though the collector’s 48 h vacuum
  (`RemoveWeatherObservationsOlderThan`) would purge them within two days: chose the explicit
  DELETE so the removal is immediate, complete, and auditable rather than relying on a
  time-delayed side effect.
- **Single atomic release over expand/contract:** chose one release (accepting the seconds-long
  old-binary error window) because this is a single-prod, low-traffic FX/weather bot where a
  brief weather-read blip is immaterial; expand/contract’s zero-downtime guarantee is not worth
  doubling the release ceremony. Flagged as a reversible choice.
