# Task Breakdown

## Overview

Expand the equity (`kind=LAST`) watchlist from the single seeded Yahoo source
(`US_YAHOO_LAST_AAPL_USD`) to 19 total by adding 18 new Yahoo Finance v8 chart
sources: 11 US equities, 2 US ETFs, and 5 crypto pairs. All rows are additive
`INSERT OR IGNORE` into `rate_sources` via one new seed migration. No schema
change.

**Correction to the task premise.** The claim "no new Go code expected — the
Mini App equity chart already handles `kind=LAST` sources generically" is only
half true and must not be taken at face value:

- The **series colour** (amber `ratepair.ColorLast`) *is* derived generically
  from `kind=LAST` in `chart/service.go:buildSeriesRow` (line 454). This part
  needs nothing.
- The **category grouping** ("equity" vs "fiat") is *not* generic. It is
  derived from a hardcoded symbol map: `buildPairRow` (chart/service.go:408)
  calls `ratepair.CategoryOf(g.labelBase)`, which reads
  `ratepair.equitySymbols` (`internal/domain/ratepair/ratepair.go`), a set that
  currently contains only `AAPL` and `CCBN`. The package carries an explicit
  DUPLICATION NOTE: *"a ticker present only in the migration silently classifies
  as fiat."*

So without a Go change, all 18 new tickers would render with the correct amber
LAST colour but be grouped under the **fiat** category header — an amber line
sitting in the wrong bucket, contradicting the documented contract in CLAUDE.md
("`kind=LAST` (equity) sources render under the equity category"). This plan
therefore includes a small, mandatory Go data change (extend the map + update
its tests) alongside the migration. It remains a data task, not a feature.

## Assumptions

- **Migration number `018`.** The weather-providers plan shipped first and
  claimed `202607.016`/`202607.017` (both applied); the next free global
  counter value is `018`, current month prefix is `202607`. Final filename:
  `migrations/202607.018.rate_sources.seed_yahoo_watchlist.sql`. Do **not**
  use `010`: its lexicographic position sorts *before* the already-applied
  `011`–`015`, so it is a stale reserved slot, not a valid next number.
- **`quote_currency = 'USD'` for all 18.** Verified against the live Yahoo probe
  (`chart.result[0].meta.currency == "USD"` for every ticker). The stored quote
  is static seed data; if Yahoo ever returned a non-USD `meta.currency` for a
  ticker the stored quote would drift, but for this fixed, verified list it is
  USD. See Risks.
- The universal JSON rule `chart.result[0].meta.regularMarketPrice` and the
  `interval='6h'`, `fetcher_kind='plain'`, `active=1` shape are copied verbatim
  from the AAPL row.
- The `User-Agent` header is a **data token**: copied byte-for-byte from the
  AAPL `options` column, not paraphrased.

## Source naming decision (design question 2)

| Group | `name` prefix | `base_currency` | url ticker | Rationale |
|-------|---------------|-----------------|-----------|-----------|
| US equities / ETFs | `US_YAHOO_LAST_<TICKER>_USD` | `<TICKER>` (no dash) | `<TICKER>` | Matches the existing AAPL convention exactly; SPY/QQQ are US-listed. |
| Crypto | `CRYPTO_YAHOO_LAST_<BASE>_USD` | `<BASE>` (dashless, e.g. `BTC`) | `<BASE>-USD` (dash, byte-for-byte as Yahoo's path expects) | Crypto is not a US market, so `US_` would be a lie. The dash lives **only** in the URL path segment; `name` and `base_currency` stay dashless so the PK is clean and the canonical pair label renders as `BTC/USD`. |

**Canonical-label safety (verified).**
`ratepair.canonicalPair` (ratepair.go:165) simply uppercases both codes and
joins with `/` (`min/max`); `labelfmt.GroupThousands` is pure numeric
formatting. Neither chokes on `BTC/USD`, `SPY/USD`, `DOGE/USD`, etc. — they are
ordinary 3–5 char alnum codes. The only requirement is that `base_currency`
carry no dash (handled above), otherwise the label would read `BTC-USD/USD`.

## `active` on seed (design question 3)

**Recommendation: seed all 18 as `active=1`** (matching AAPL). The rate-limit
budget is not a real concern:

- The collector scrapes sequentially per invocation; 18 extra Yahoo v8 chart
  requests per 6h tick is a lightweight JSON GET each, spread across a single
  serial pass — nowhere near Yahoo's unauthenticated throttle for that cadence.
- Egress honours `BEACON_PROXY_URL`, so if throttling ever appears it is a
  config/proxy lever, not a reason to ship rows disabled.
- Shipping rows `active=0` would mean the watchlist silently collects nothing
  until an operator flips each via `PATCH /api/sources/{name}/active` — worse UX
  for no benefit. Any single misbehaving ticker can be disabled individually
  later.

## Tasks

### Task 1: Create the seed migration file

- **Description:** Add `migrations/202607.018.rate_sources.seed_yahoo_watchlist.sql`
  containing 18 `INSERT OR IGNORE INTO rate_sources (...) VALUES(...)` rows, one
  per the table above. Copy the column-list form and every non-varying value
  (title, interval, kind, active, options UA header, rules, rule_metadata,
  fetcher_kind) verbatim from the AAPL row in
  `202606.009.rate_sources.seed_stocks.sql`. Only `name`, `base_currency`, and
  the ticker in `url` change per row. Add a short header comment mirroring the
  009 file's style (immutability note, UA-rotation-via-new-migration note).
- **Acceptance Criteria:**
   - [ ] File named exactly `202607.018.rate_sources.seed_yahoo_watchlist.sql`.
   - [ ] Exactly 18 `INSERT OR IGNORE` rows, matching the table byte-for-byte on
         `name` / `base_currency` / url ticker.
   - [ ] `quote_currency='USD'`, `kind='LAST'`, `interval='6h'`, `active=1`,
         `fetcher_kind='plain'` on every row.
   - [ ] The `options` UA string is identical (diff-clean) to the AAPL row's UA.
   - [ ] `rules` is `[{"method":"json","pattern":"chart.result[0].meta.regularMarketPrice"}]`
         on every row.
   - [ ] `make migrate` on a fresh DB applies cleanly and records the file in
         `__schema_migrations`; re-running is a no-op (idempotent).
- **Pitfalls & edge cases:**
   - Crypto url ticker keeps the dash (`BTC-USD`); `base_currency` must not.
   - `INSERT OR IGNORE` means a name collision is silently skipped — a typo that
     duplicates an existing `name` would vanish without error. Cross-check all
     19 names (18 new + AAPL) are unique.
   - `name` is the PK and is FK-referenced by subscriptions; it is effectively
     immutable once any user subscribes. Get the naming right now.
   - Filename is immutable once applied — do not rename after deploy.
- **Complexity:** Easy

1. **Task 1** (migration file) — foundation; Tasks 3 and 4 depend on its exact row count.
2. **Task 2** (equitySymbols map) — independent of Task 1; can run in parallel.
3. **Task 3** (tests) — after Tasks 1 and 2 (needs the 56 count and the map change).

## Risks

- **Yahoo `meta.currency` drift.** The stored `quote_currency='USD'` is static.
  If Yahoo ever reports a non-USD currency for a ticker (unlikely for this list),
  the stored quote would be wrong while the price stays correct. Mitigation: the
  list is verified USD today; `doctor audit` surfaces extraction health but not
  currency mismatch — a future enhancement could assert `meta.currency` against
  the seeded quote.
- **Silent `INSERT OR IGNORE` skip.** A duplicate `name` (typo) is swallowed
  with no error and the ticker simply never collects. Mitigation: the 56-count
  test catches a wrong total, and `doctor audit --all` surfaces a missing source.
- **Rate budget under a faster cron.** At `6h` the 18 extra requests are
  trivial; if the collector cadence is ever lowered, revisit. Not a concern now.

## Trade-offs

- **Extend `equitySymbols` vs. make `CategoryOf` derive equity from
  `kind=LAST`.** Chosen: extend the map (minimal, matches the pattern the
  codebase already committed to). Rejected for now: refactoring `CategoryOf` to
  take the source kind and classify any `LAST` source as equity — that would
  kill the dual-maintenance foot-gun the DUPLICATION NOTE warns about, but it
  changes a shared package signature and its callers/tests, which is a separate
  concern and disproportionate to a seed task. Flagged as a worthwhile follow-up.
- **Crypto under `equity` category vs. a new `crypto` category.** Chosen: fold
  crypto into `equity` so all `LAST` assets share one amber-coloured group.
  Rejected: adding `CategoryCrypto` — it touches `categoryOrder`, its
  completeness test, the DTO wire contract, and frontend category-header
  rendering. Semantically "equity" is loose for BTC, but the alternative is
  scope creep for a data task.
- **`CRYPTO_YAHOO_` prefix vs. reusing `US_YAHOO_`.** Chosen: a distinct
  `CRYPTO_` prefix so the PK does not misrepresent crypto as a US market.
  Rejected: one uniform prefix — simpler but dishonest, and the prefix is
  permanent once subscribed.
- **All `active=1` vs. staggering crypto/trending inactive.** Chosen: all
  active, matching AAPL — the rate budget is a non-issue and inactive rows
  collect nothing until manually enabled. Any single bad ticker is disabled
  individually via the existing PATCH endpoint.
