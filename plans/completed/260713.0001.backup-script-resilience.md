# Task Breakdown

## Overview

Harden `configs/sqlite_dump.sh` (deployed as `/opt/beacon/backups/sqlite_dump.sh`, run
by root cron at 00:00 UTC) so a Google Drive failure can no longer silently disable
local backup hygiene, and so any failure is surfaced immediately via Telegram instead
of rotting unseen in `/opt/beacon/logs/backup.log`.

Context: the Drive mirror was broken for ~2 weeks (service-account zero-quota → HTTP 403).
`set -euo pipefail` meant `rclone copy` failing aborted the script *before* the
local-retention prune ran, and nothing alerted. OAuth is now fixed and uploads work;
this change removes the two latent footguns the incident exposed (remote failure kills
local hygiene; failures are silent).

## Assumptions

- Script runs as root and can read `/opt/beacon/.env` (mode 0600 root) for bot creds.
- Bot creds live in `BEACON_TELEGRAMBOT_DSN` (`<chatID>:<botToken>@<host>`), the same var
  the Go services use. The exact on-disk DSN shape is verified on the host via a `selftest`
  subcommand before trusting the parse (the dev box must not read `.env`).
- `curl` and `hostname` exist on the host (standard Debian; verified `rclone`+`sqlite3` present).
- Host script == repo script confirmed identical before this change (no divergence).
- Folder decision is settled: remote stays `gdrive:backups/dev_seilbekskindirov/beacon`.

## Tasks

### Task 1: Isolate remote failures from local hygiene
- Description: Drop blanket `set -e`; keep `set -uo pipefail`. Guard the sqlite backup,
  `rclone copy`, `rclone` remote-prune, and the local prune each with an explicit success
  check. The local snapshot and local retention prune ALWAYS run regardless of remote outcome.
- Acceptance Criteria: with rclone forced to fail, the local snapshot is still written AND
  the local prune still runs; the script exits non-zero.
- Pitfalls: `set -u` needs `${VAR:-}` guards on optional env; `grep | head` under `pipefail`
  returns non-zero on no-match (fine — no `-e`); `find` returns non-zero only on real errors,
  not on empty matches.
- Complexity: Medium

### Task 2: Fail-loud Telegram alert
- Description: Accumulate failure messages; on any failure send ONE Telegram message to the
  admin chat (🔴 + host + summary). Creds resolution order: (1) `ALERT_TELEGRAM_TOKEN` +
  `ALERT_TELEGRAM_CHAT_ID` from the backup `.env`; else (2) parse `BEACON_TELEGRAMBOT_DSN`
  from `$MAIN_ENV` (default `/opt/beacon/.env`); else (3) log "alerting disabled" and continue.
  Alerting is never fatal. `curl -m 15` bounds the call. Mirrors the CI `sendMessage` pattern.
- Acceptance Criteria: a forced failure produces exactly one Telegram message; missing creds
  or missing curl degrade to a log line, never crash the backup.
- Pitfalls: one alert per run (not per failure); never log the token; the DSN token itself
  contains a colon — chat is the first `:`-field, token is the rest up to `@host`; strip an
  optional `scheme://` prefix and surrounding quotes.
- Complexity: Medium

### Task 3: `selftest` subcommand for safe on-host verification
- Description: `sqlite_dump.sh selftest` resolves creds, prints `chat=<id> token_len=<n>`
  (never the token), sends a test alert, and exits. Confirms the parse + send path on the
  host without triggering a real failure and without leaking the secret.
- Acceptance Criteria: `selftest` on the host delivers a Telegram message and prints a
  non-zero `token_len`.
- Complexity: Easy

### Task 4: Docs + env example
- Description: Document the new optional vars in `configs/sqlite_dump.env.example`
  (`ALERT_TELEGRAM_TOKEN`, `ALERT_TELEGRAM_CHAT_ID`, `MAIN_ENV`) and the failure-alert
  behavior in the `README.md` Backups section.
- Complexity: Easy

### Task 5 (manual, operator): purge stale pre-rename snapshots
- Description: One-time `sudo rm /opt/beacon/backups/stage_monitor.*.sqlite` (~20 MB,
  Jun 21–28). Not script logic — the `beacon.*` glob correctly ignores them.
- Complexity: Easy

### Task 6: success summary with backup inventory
- Description: On a clean run send a `#BEACON backup OK` Telegram message reporting the
  new snapshot name+size, the retained inventory (count, total, per-file sizes), and any
  pruned files (captured with sizes before deletion). Gated by `NOTIFY_ON_SUCCESS`
  (default on). The remote target shown is `${GDRIVE_REMOTE}` — the same value that
  drives the upload, so it can never drift from reality. Sizes render via a portable
  `human_size` (numfmt with an awk fallback), byte counts via a GNU/BSD-portable stat.
- Acceptance Criteria: a clean run emits `#BEACON backup OK` with the new-snapshot line,
  a kept-inventory list, and a pruned list; `NOTIFY_ON_SUCCESS=0` suppresses it.
- Pitfalls: capture pruned name+size BEFORE `rm`; keep stat/size portable; never hardcode
  the remote path.
- Complexity: Medium

## Execution Order

1 → 2 → 3 → 4 land together as one atomic change (code + docs). Task 5 is an independent
manual op on the host. Deploy: scp the new script over `/opt/beacon/backups/sqlite_dump.sh`,
`selftest` to confirm the alert path, then a normal run.

## Risks

- DSN parse mismatch vs the real `.env` shape → mitigated by `selftest` + the explicit-var
  fallback.
- One-alert-per-run bundles all failures of a run into a single message; distinct failures in
  the same run share one alert (acceptable — the summary lists them all).

## Trade-offs

- Explicit `ALERT_TELEGRAM_*` duplicates the bot token into a second root-only file if the
  operator chooses it; the default path reuses `BEACON_TELEGRAMBOT_DSN` with no duplication.
- No "recovery" (back-to-green) alert in v1 — failure-only, to keep state out of the script.
