# Task Breakdown

## Overview

Three weather notification kinds were explicitly deferred by the shipped weather
work (`plans/completed/260630.0001.weather-monitoring.md` "Deferred" section and
`260630.0002.weather-gismeteo.md`). This plan designs all three, grounded in the
code that actually shipped:

1. **On-demand current-weather screen** — a read-only `GET /api/me/weather/current`
   returning the latest stored observation for each of the caller's cities, plus a
   new WASM screen. Independent of the alert kinds; lowest risk; no new domain, no
   migration, no anti-spam.
2. **Threshold alerts** (heat / frost / high-wind / thunderstorm) — fire when a
   stored current/daily field crosses a bound. Reuses fields **already stored** by
   the Open-Meteo MVP (no new fetch, no new observation storage). Needs a
   subscription condition value, a per-kind evaluator, anti-spam, and UX.
3. **Rain-incoming alert** — fire when hourly `precipitation_probability` crosses a
   threshold within the next N hours. The MVP requests the Open-Meteo `hourly` block
   but **does not store it** (`decodeOpenMeteoForecast` ignores it and the call uses
   `forecast_days=1`). This kind therefore needs hourly storage + a forecast-window
   change. Biggest slice.

**Grounding facts that shape the design** (verified by reading the code):

- `domain.WeatherUserCity` already carries `NotifyKind WeatherNotifyKind`,
  `NotifyHour`, `LastNotifiedAt`, coordinates, and `Timezone`. The table
  `weather_user_cities` (migration `202606.011`) has
  `UNIQUE(user_type, user_id, location_id, notify_kind)` and
  `RetainWeatherUserCity` upserts on that key with `RETURNING id`. **The schema
  already supports multiple notify kinds per city as separate rows** — this is the
  spine of the recommended subscription model. There is currently **no** condition
  /threshold column.
- `domain.WeatherNotifyKind` has exactly one value: `WeatherNotifyMorningSummary`.
- `domain.WeatherObservation` stores daily (`TempMax`, `TempMin`, `PrecipSum`,
  `PrecipProbMax`, `WeatherCode`, `Sunrise`, `Sunset`) and current
  (`TempCurrent`, `TempFeels`, `Humidity`, `WindSpeed`, `WindDir`, `Precip`,
  `CloudCover`) fields, all nullable pointers. **There is no daily wind-max field
  and no hourly data stored.**
- `internal/infrastructure/weather/openmeteo.go` `Forecast` requests
  `hourly=precipitation_probability,temperature_2m` and `forecast_days=1`, but
  `decodeOpenMeteoForecast` decodes **only** `current` and `daily[0]`. The hourly
  arrays are fetched and thrown away, and `forecast_days=1` means the hourly window
  stops at local midnight.
- `internal/application/notification/weathercheckagent.go` loads
  `ObtainDueWeatherUserCities(morning_summary)`, evaluates `IsMorningDue`, loads the
  latest Open-Meteo observation, optionally folds in gismeteo, renders, queues a
  `domain.RateUserEvent{SourceName: ""}` (NULL → no `rate_sources` FK), and advances
  `last_notified_at`. **It reuses the FX `rate_user_events` queue + `RateDispatchAgent`
  transport** — exactly what this plan continues to reuse for all alert kinds.
- The gismeteo provider populates only `TempMax`, `TempMin`, `PrecipSum`,
  `WeatherCode` (verified in `gismeteo.go`). It has no wind, humidity, or hourly
  data. **All alerts must therefore evaluate against Open-Meteo only**; gismeteo
  stays comparison-only, matching the shipped check-agent design.

**Recommended subscription model (resolved below in detail).** Add a single
`condition_value TEXT` column to `weather_user_cities` and make each alert a row of
its own kind on the city, exactly mirroring `RateUserSubscription.ConditionValue`
parsed per `ConditionType`. Rationale: the shipped schema already keys rows by
`(user, location, notify_kind)`, the collector already dedups fetches by
`location_id` regardless of kind, and the user already knows the FX condition-value
pattern. This avoids a separate `weather_alert_subscriptions` table (which would
duplicate the coordinate columns or add a join + a new repo) and avoids a second
migration when rain lands (rain reuses the same column).

**Recommended hourly storage for rain (resolved below).** Add a nullable
`hourly_json TEXT` column to `weather_observations` and a typed `Hourly` slice on
the domain observation, rather than a normalized `weather_hourly` table. The hourly
block is a child of one observation snapshot, is always read whole for one location,
and is never aggregated across rows — so a table buys nothing and costs a repo +
vacuum + a new `identity.Kind`.

**Recommended anti-spam (resolved below).** A per-row cooldown keyed on the
existing `last_notified_at` column (advanced by the existing `AdvanceLastNotifiedAt`),
with a per-kind cooldown constant. Edge-triggering (a stored "last condition state"
column) is the documented alternative, deferred unless cooldown proves noisy.

**Recommended ship sequence:** Slice A (on-demand) → Slice B (threshold) →
Slice C (rain). Each slice is independently shippable and reviewable.

## Assumptions

- All three kinds reuse the FX `rate_user_events` queue and `RateDispatchAgent`
  (empty `SourceName` → NULL). No new transport, no new binary. (Verified: the
  shipped morning-summary path already does this.)
- Both providers stay keyless; no new `dsninjector` DSN / env var. Alerts and rain
  evaluate against Open-Meteo only; gismeteo remains comparison-only.
- Threshold + rain evaluation runs inside the existing `WeatherCheckAgent` in
  `cmd/notifier`, on the same one-shot cron cadence as the morning summary. The
  on-demand endpoint runs inside `cmd/web`.
- Migration counter: weather owns 011–016. **011 and 012 are used.** This plan uses
  **013** and **014**, assigned in *ship order* because filenames are immutable once
  applied. Recommended order makes `202606.013` = the `condition_value` column
  (Slice B) and `202606.014` = the `hourly_json` column (Slice C). Slice A needs no
  migration. If Slice C somehow ships before Slice B, the numbers swap — assign at
  ship time.
- Open-Meteo default units are stored verbatim by the decoder: temperature in **°C**,
  wind speed in **km/h**, precipitation in **mm**, probability in **%**. Threshold
  values and the UI labels are in these units. (No `wind_speed_unit` override is set
  in the shipped query; do not silently change it — it would reinterpret stored data.)
- The on-demand screen reads **stored** observations (collector-produced); it does
  not trigger a live fetch. It therefore shows data only for cities the user already
  subscribed to (any kind), which the collector has fetched. A city with no stored
  observation yet renders a "no data yet" state.
- No new off-limits PII. Alert condition values (a number) and the hourly block (a
  forecast array) are not personal data. The existing chosen-city-coordinate stance
  (documented in CLAUDE.md "Data & Privacy") is unchanged.
- Telegram is the only delivery channel (`domain.UserTypeTelegram`).

### Task 4: [Slice B] Domain: alert kinds, condition value, evaluator

- **Description:** Extend `internal/domain/weatheruser.go` to model threshold alerts,
  mirroring the FX `RateUserSubscription` condition pattern
  (`internal/domain/rateusersubscription.go`).
  - Add `WeatherNotifyKind` values (data tokens, persisted — never rename):
    `WeatherNotifyAlertHeat = "alert_heat"`, `WeatherNotifyAlertFrost = "alert_frost"`,
    `WeatherNotifyAlertWind = "alert_wind"`,
    `WeatherNotifyAlertThunderstorm = "alert_thunderstorm"`.
  - Add `ConditionValue string` to `WeatherUserCity` (analogous to
    `RateUserSubscription.ConditionValue`; empty for `morning_summary` and
    thunderstorm).
  - Add `Validate() error` validating `ConditionValue` per `NotifyKind`:
    heat/frost/wind parse as a float64 (`AlertThreshold()`), thunderstorm requires no
    value, morning_summary ignores it.
  - Add `AlertThreshold() (float64, error)` parsing `ConditionValue` for the numeric
    kinds (heat/frost/wind), erroring for kinds that take no numeric value.
  - Add the pure evaluator
    `EvaluateAlert(obs WeatherObservation) (fired bool, reason string, err error)`
    on `WeatherUserCity`. The **metric and operator are implied by the kind** (a
    fixed mapping — this is the key simplification over FX's free-form value):
    - `alert_heat`: `obs.TempMax != nil && *obs.TempMax >= threshold` (today's
      forecast high). Reason e.g. `"High +36.0°C ≥ +35.0°C"`.
    - `alert_frost`: `obs.TempMin != nil && *obs.TempMin <= threshold` (default
      threshold typically 0). Reason e.g. `"Low −7.0°C ≤ 0.0°C"`.
    - `alert_wind`: `obs.WindSpeed != nil && *obs.WindSpeed >= threshold` — note
      this is the **current** snapshot wind (the only wind field stored; there is no
      daily wind-max). Reason includes the km/h value.
    - `alert_thunderstorm`: `obs.WeatherCode != nil && *obs.WeatherCode >= 95`
      (WMO thunderstorm band 95/96/99). No threshold. Reason from
      `domain.WMOWeatherCode`.
    A nil required field → `fired=false` (absence is not a crossing). An unknown kind
    → error.
- **Acceptance Criteria:**
   - [ ] New `WeatherNotifyKind` constants + `ConditionValue` field compile with
         godoc; values are the literal tokens above (never translated).
   - [ ] `TestWeatherUserCity_Validate` (subtests): valid heat/frost/wind values
         pass; non-numeric value for a numeric kind errors; thunderstorm with empty
         value passes; unknown kind errors.
   - [ ] `TestWeatherUserCity_EvaluateAlert` (subtests, one top-level test per the
         one-Test-per-method rule): each kind fires above/at its bound and does not
         fire below; nil metric → not fired, no error; thunderstorm fires for
         code 95/96/99 and not for 3 (overcast); unknown kind → error. Uses a fixed
         observation; asserts the human `reason` string contents.
   - [ ] Metric→kind mapping is documented in the `EvaluateAlert` godoc so the
         implied semantics (forecast high/low vs current wind vs daily-dominant code)
         are unambiguous.
- **Pitfalls & edge cases:** Stored wind is **current**, not a daily max — frame the
  wind alert as "it is windy now," distinct from the forecast-based heat/frost. The
  decoder overwrites `WeatherCode` with the daily-dominant code, so thunderstorm =
  "today is forecast stormy," not "storm at this instant"; document that. Units are
  Open-Meteo defaults (°C, km/h) — the reason string and the UI must label them.
  Use `formatWeatherTemp` (U+2212 minus) for temperatures in the reason for
  consistency with the FX/morning-summary style.
- **Complexity:** Medium
- **Code Example:**
  ```go
  // EvaluateAlert reports whether this city's alert condition is currently met by
  // obs and a short human reason. The metric and comparison are implied by NotifyKind
  // (heat→TempMax≥t, frost→TempMin≤t, wind→WindSpeed≥t [current snapshot, km/h],
  // thunderstorm→WeatherCode≥95). A nil required field means "no crossing", not an
  // error. Anti-spam (cooldown) is the caller's concern, not this pure evaluator.
  func (c *WeatherUserCity) EvaluateAlert(obs WeatherObservation) (bool, string, error) {
      switch c.NotifyKind {
      case WeatherNotifyAlertHeat:
          t, err := c.AlertThreshold()
          if err != nil { return false, "", err }
          if obs.TempMax == nil || *obs.TempMax < t { return false, "", nil }
          return true, fmt.Sprintf("High %s ≥ %s", formatWeatherTemp(*obs.TempMax), formatWeatherTemp(t)), nil
      // ... frost, wind, thunderstorm ...
      default:
          return false, "", fmt.Errorf("weather city %s: not an alert kind: %q", c.ID, c.NotifyKind)
      }
  }
  ```

## Risks

- **Wind/thunderstorm semantics vs stored data (medium).** Stored wind is the
  *current* snapshot (no daily max), and `WeatherCode` is overwritten with the
  daily-dominant code. So "high-wind" is "windy now" and "thunderstorm" is "today is
  forecast stormy" — not instantaneous. Mitigation: document the implied metric per
  kind in `EvaluateAlert` and label it in the UI; a forecast wind-max would require
  extending the daily request + a column (deferred).
- **Cooldown noise vs missed events (medium).** A pure cooldown re-fires a sustained
  condition each cooldown window and can miss a second short event inside the window
  (e.g. a second thunderstorm). Mitigation: per-kind cooldown tuning; edge-triggering
  (a stored last-state column) is the documented upgrade if users report noise or
  misses.
- **Forecast-window change for rain (medium).** Bumping `forecast_days` must not shift
  `daily[0]` off today or the morning summary breaks. Mitigation: assert the existing
  morning-summary tests stay green and verify `daily[0]==today` against the live API
  before merging Task 10.
- **Per-(city,kind) row duplication (low/medium).** Each kind is its own row, so the
  coordinate/tz/display fields repeat across a city's kinds and the GET list returns
  multiple rows per city. Mitigation: the collector already dedups fetches by
  `location_id`; the WASM groups the list by `location_id`; the duplication is
  consistent with the shipped denormalized model.
- **On-demand reads stale data (low).** The endpoint reads stored observations
  (throttled to ~1 h by the collector), so "current" can be up to ~1 h old, and a
  city with no fetch yet shows "no data." Mitigation: documented; a live fetch from
  cmd/web is the alternative (rejected for egress/latency reasons).
- **Migration immutability (low).** 013/014 are frozen once applied. Mitigation:
  additive `ADD COLUMN` only; assign the number at ship time in ship order.
- **Reusing `rate_user_events` for alerts (low).** Alert rows carry NULL
  `source_name` and appear in `/api/notifications`, same as the morning summary.
  Accepted (the dispatcher is transport-only); split later if semantics diverge.

## Trade-offs

- **`condition_value TEXT` on `weather_user_cities`, each kind its own row — chosen
  over a separate `weather_alert_subscriptions` table and over a `threshold REAL`
  column.** The shipped schema already keys rows by `(user, location, notify_kind)`
  with an upsert and dedups collection by `location_id`, so per-kind rows need zero
  new structure. A `TEXT` value (parsed per kind, exactly like
  `RateUserSubscription.ConditionValue`) is a pattern the user already knows and lets
  rain (prob, and later prob+window) reuse the **same** column with no second
  migration — which a single `REAL threshold` could not. A separate table would
  duplicate the coordinate columns or add a join + a new repo + its own collection-
  dedup integration for no benefit. Rejected both alternatives.
- **Metric + operator implied by the kind, not stored.** Unlike FX (where
  `condition_value` carries the full rule), weather kinds are a closed enum with a
  fixed metric/operator mapping (heat→TempMax≥, frost→TempMin≤, wind→WindSpeed≥,
  thunderstorm→code≥95, rain→max hourly prob≥). Storing only the numeric bound keeps
  the value column simple and the evaluator pure and testable.
- **Cooldown anti-spam keyed on the existing `last_notified_at` — chosen over
  edge-triggering.** It reuses the shipped column + `AdvanceLastNotifiedAt`, needs no
  extra state, and makes the first alert slice atomic. Edge-triggering (store the
  last boolean condition state, fire only on false→true) is more precise but adds a
  column and must handle flapping with hysteresis. Deferred until cooldown proves
  noisy. Per-kind cooldown durations bridge the gap (long for daily-forecast kinds,
  short for current-condition kinds).
- **`hourly_json TEXT` column on `weather_observations` — chosen over a
  `weather_hourly` table.** The hourly block is a child of one snapshot, always read
  whole for one location, never aggregated across rows — so a normalized table buys
  no query power and costs a repo, a vacuum path, and a new `identity.Kind`. The JSON
  column is one additive migration with the existing repo. Rejected the table.
- **On-demand reads stored observations — chosen over a live cmd/web fetch.** Keeps
  cmd/web egress-free and the response fast, and matches the "collector fetches, web
  reads" separation. A live fetch (fresher, but adds web→Open-Meteo egress + latency
  per request, and cmd/web has no proxy) is the documented alternative.
- **Introduce `WeatherGatewayDeps` now (Slice A).** Slice A adds the second gateway-
  side weather dependency (the obs repo), which is exactly the trigger the MVP review
  named for collapsing the positional constructor params. Doing it as a standalone
  no-behaviour-change refactor first de-risks the endpoint task.
- **Extend the existing create/list/delete surface — chosen over a new
  `/api/me/weather/alerts` endpoint.** Each alert is a city row of a different kind,
  so the existing per-row create + per-id delete already fit; only the create body
  (kind + value) and the row DTO grow. A parallel endpoint would duplicate auth,
  validation, and ownership logic for no gain.
