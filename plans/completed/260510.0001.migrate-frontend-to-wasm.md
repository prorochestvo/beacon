# Task Breakdown

## Overview

Today the project ships two parallel frontends and only one of them runs:
`cmd/web/static/index.html` (vanilla JS, all screens, ~16 KB) and
`cmd/web/static/app.wasm` (Go-WASM, two screens, ~3.3 MB) which is built and
embedded but never loaded — no `<script src="wasm_exec.js">` tag exists in any
served HTML. ADR-001 (`docs/decisions/260505.0001.adopt-wasm-frontend.md`) and
its addendum lock the direction: WASM becomes the **sole** frontend, raw
`syscall/js` (no framework, no new `go.mod` entries), standard Go compiler
(TinyGo deferred), with the package layout
`pkg/api/` (shared DTOs) + `internal/wasm/{application,ui,apiclient,dom}/` +
`cmd/wasm/main.go` (composition root).

The migration touches three feature areas — sources list, source detail,
errors page (all in `index.html`), plus the Telegram Mini App
(`cmd/web/static/app/subscriptions.html`) — and one structural change
(extract `pkg/api/`). It must remain incrementally shippable: after every task
in the execution order `make build` succeeds, `make test` is green, and a
served URL works end-to-end (either old JS or new WASM, never broken).

The phasing therefore is:

1. **Structural prep** — extract DTOs to `pkg/api/`, point server handlers at
   them. Pure refactor, no behaviour change. Lands before any WASM work.
2. **WASM foundation** — `internal/wasm/dom/` helpers and
   `internal/wasm/apiclient/` typed client, both fully unit-tested.
3. **First end-to-end screen** — sources list ported and switched on. WASM is
   booted from `index.html`; the JS route for the same screen is removed once
   parity is verified. JS still serves the other two screens.
4. **Remaining `index.html` screens** — source detail, then errors page. Each
   is a separate, independently shippable task. After each, the JS for that
   screen is removed.
5. **Mini App port** — `subscriptions.html` ported. Telegram WebApp `initData`
   read inside Go and sent on every `/api/me/...` call.
6. **JS removal & cleanup** — both HTML files reduced to loaders/skeletons;
   embed/static serving verified; `Makefile` and `cmd/web/main.go` confirmed
   to need no further change. The migration is done when no JS-only logic
   remains in either HTML file.

The plan is 13 tasks. Tasks 4–6 (sources list, source detail, errors) and
Task 8 (Mini App) are themselves end-to-end deliverables and are intentionally
scoped so each can ship on its own commit + restart.

## Assumptions

- ADR-001 + addendum are binding. The plan does not relitigate framework
  choice, compiler choice, or package layout.
- "No new direct `go.mod` entries" is binding. Anything WASM needs at runtime
  must be implementable on top of `syscall/js` and the standard library.
- The current API surface (the routes listed in `CLAUDE.md` § "HTTP Routes")
  is stable for the duration of this migration. If an API change is needed
  for a screen, it is added at the start of the corresponding task — but the
  expectation is none will be needed.
- "Incrementally shippable" means: between any two consecutive tasks in the
  execution order, the user-visible UI on every URL is functional. JS and
  WASM coexist during the migration; the JS for screen X is only removed
  inside the same task that ships the WASM port of screen X.
- The Mini App's `initData` is read from `window.Telegram.WebApp.initData`.
  When that string is empty (page opened outside Telegram) the WASM shows the
  same "open from the bot" message the JS version shows today.
- Asset relocation to `./web/` is **out of scope** (separate plan,
  `00X-move-web-to-web-dir.md`). Until that lands, embedded assets stay at
  `cmd/web/static/`.
- `wasm_exec.js` continues to be copied from `$(go env GOROOT)/lib/wasm/`
  during `make build` (see `Makefile:87`). The Go version controls its
  contents; we do not vendor or modify it.
- Tests for plain Go behind `//go:build js && wasm` work under the host
  toolchain when `GOOS=js GOARCH=wasm` is set and a `wasm_exec` runner is
  available (`node` or `go test -exec wasm_exec.sh`). The CI question is
  addressed in Task 13.
- Server-side `internal/gateway/httpV1/dto` is currently the only consumer of
  the DTO types. Migrating to `pkg/api/` is therefore a mechanical move +
  import-path rewrite, with no second consumer to coordinate with.

## Tasks

### Task 1: Extract `pkg/api/` from `internal/gateway/httpV1/dto`
- **Description:** Create the `pkg/api/` package as the single source of
  truth for HTTP DTOs shared by server (`internal/gateway/httpV1/handlers/`)
  and the future WASM client (`internal/wasm/apiclient/`). Move every type
  from `internal/gateway/httpV1/dto/dto.go` into `pkg/api/`, keeping the JSON
  field tags byte-for-byte identical so wire format does not change. Add the
  one type that does not yet exist as a DTO: the `PATCH
  /api/sources/{name}/active` request body (currently an inline anonymous
  struct at `internal/gateway/httpV1/handlers/handlers.go:357`) → expose as
  `pkg/api.SourceActiveRequest`. Update `handlers.go` to import `pkg/api`
  and remove the now-empty `internal/gateway/httpV1/dto` package (or keep it
  as a one-line re-export shim if removal causes wide churn — judgement
  call by the engineer; default is to remove).
- **Acceptance Criteria:**
   - [ ] `pkg/api/` exists with one file per logical group (e.g.
         `sources.go`, `rates.go`, `history.go`, `notifications.go`,
         `subscriptions.go`, `events.go`, `errors.go`, `stats.go`,
         `me.go`) or a single `api.go` — engineer's choice, but no file
         exceeds ~150 lines.
   - [ ] Every DTO that was in `internal/gateway/httpV1/dto/dto.go` is now
         in `pkg/api/` with the same exported name, same fields, same JSON
         tags, same `omitempty` flags.
   - [ ] `pkg/api.SourceActiveRequest` exists with `Active bool
         \`json:"active"\``.
   - [ ] `internal/gateway/httpV1/handlers/handlers.go` no longer references
         `internal/gateway/httpV1/dto` (or, if the shim option is chosen,
         the shim file is the only thing in that package and its godoc
         clearly states it is deprecated).
   - [ ] `internal/gateway/httpV1/handlers/handlers_test.go` still passes.
   - [ ] `make test` green; `make build` green.
   - [ ] A `curl http://.../api/sources` response diffs as zero bytes
         against `main` for the same DB state. Verify by smoke-running the
         server locally with the same SQLite file before and after.
- **Pitfalls:**
   - The `time.Time` field on `NotificationResponse.CreatedAt`/`SentAt` is
     not a string — it relies on Go's default `time.RFC3339Nano` JSON
     encoding. Do not "normalize" it to a string during the move.
   - `omitempty` differences are wire-format-visible. Keep them exactly.
   - `pkg/` is the **public** module surface. Once published, renames are
     painful. Pick names that read well from outside the repo
     (`api.SourceResponse`, not `api.SourceResponseDTO`).
- **Complexity:** Easy
- **Code Example:**
   ```go
   // pkg/api/sources.go
   package api

   // SourceResponse is the JSON representation of a configured rate source,
   // decorated with its most recent execution status.
   type SourceResponse struct {
       Name          string `json:"name"`
       Title         string `json:"title"`
       BaseCurrency  string `json:"base_currency"`
       QuoteCurrency string `json:"quote_currency"`
       Interval      string `json:"interval"`
       Active        bool   `json:"active"`
       LastSuccess   bool   `json:"last_success"`
       LastError     string `json:"last_error,omitempty"`
       LastRunAt     string `json:"last_run_at,omitempty"`
   }

   // SourceActiveRequest is the body of PATCH /api/sources/{name}/active.
   type SourceActiveRequest struct {
       Active bool `json:"active"`
   }
   ```

### Task 2: Add `internal/wasm/dom/` helpers (escape, fetchJSON, on)
- **Description:** Introduce the WASM cross-cutting helper package. Every
  file carries `//go:build js && wasm`. The package owns three concerns
  that every screen needs and that today are scattered in
  `cmd/wasm/main.go`: HTML escaping, a typed `fetchJSON[T]` wrapper around
  `window.fetch`, and an `on(el, event, handler)` event-binder with correct
  `js.Func.Release()` semantics returning a teardown closure. The fetch
  wrapper is the single place where the project will eventually plumb the
  Telegram `X-Telegram-Init-Data` header for `/api/me/...` (see Task 8) —
  design its signature so an optional headers map fits in cleanly.
- **Acceptance Criteria:**
   - [ ] `internal/wasm/dom/escape.go` exposes
         `func Escape(s string) string` matching the JS `esc()` semantics
         in `cmd/web/static/index.html:42-48` (`&`, `<`, `>`, `"`).
   - [ ] `internal/wasm/dom/fetch.go` exposes
         `func FetchJSON[T any](ctx context.Context, method, url string,
         body any, headers map[string]string) (T, error)`. The function
         must release every `js.Func` it creates after the promise chain
         settles (use `defer fn.Release()` for each `js.FuncOf`, exactly
         like the existing `fetchText` at `cmd/wasm/main.go:118`).
   - [ ] `internal/wasm/dom/event.go` exposes
         `func On(target js.Value, event string, handler func(js.Value))
         (release func())`. The returned closure calls `Release()` on the
         underlying `js.Func`. Document that callers MUST invoke it when
         the bound element is destroyed; otherwise the function table
         leaks.
   - [ ] Plain Go unit tests for `Escape` (table-driven, every special
         character exercised). The escape function must be implemented in
         a way that does not reach for `syscall/js`, so its `_test.go`
         file can omit the build tag and run under regular `go test`.
         Either split the package into `escape.go` (no build tag) +
         `event.go`, `fetch.go` (`js && wasm` only), or move escape to a
         tiny sibling package — engineer's choice, but the escape tests
         MUST run under `make test`.
   - [ ] `var _ <iface>` compile-time checks for any interfaces the
         package introduces (e.g. a `Fetcher` interface used elsewhere).
   - [ ] No new entries in `go.mod`.
- **Pitfalls:**
   - `js.FuncOf` allocates an entry in the runtime's function table. Every
     allocation must be matched by a `Release()` or the table grows
     unbounded. The current `cmd/wasm/main.go` documents this for the
     `fetchText` case — the new package must not regress on this.
   - `_loadRates` in `cmd/wasm/main.go:36-44` is intentionally never
     released (it lives forever because it's bound to `window`). The new
     architecture deletes the inline-`onclick` pattern entirely; do not
     port that idiom.
   - When `body` is `nil` for a `GET`, do not pass `null` to `fetch`
     options or older browsers may misbehave — omit the `body` key
     entirely.
   - JSON unmarshalling into a generic `T` with `encoding/json` is fine
     under standard Go but is one of the things TinyGo is bad at. ADR-001
     defers TinyGo, so this is acceptable today; do not engineer around a
     compiler we are not using.
   - Avoid a `nowFn func() time.Time` field on any struct (memory:
     `feedback_no_clock_func_field.md`). If a test needs a deterministic
     clock, take `now time.Time` as a parameter.
- **Complexity:** Medium
- **Code Example:**
   ```go
   //go:build js && wasm

   package dom

   import (
       "context"
       "encoding/json"
       "fmt"
       "syscall/js"
   )

   // FetchJSON issues an HTTP request via window.fetch and decodes the
   // response body as JSON into T.
   //
   // The function MUST be called from a goroutine — the JS event loop is
   // owned by the main goroutine, and a synchronous wait there deadlocks.
   func FetchJSON[T any](ctx context.Context, method, url string,
       body any, headers map[string]string) (T, error) {

       var zero T
       opts := map[string]any{"method": method}
       if len(headers) > 0 {
           h := map[string]any{}
           for k, v := range headers {
               h[k] = v
           }
           opts["headers"] = h
       }
       if body != nil {
           buf, err := json.Marshal(body)
           if err != nil {
               return zero, fmt.Errorf("marshal body: %w", err)
           }
           opts["body"] = string(buf)
       }
       // ... promise → text → json plumbing identical to fetchText, with
       // every js.FuncOf paired to a defer ...Release().
       _ = ctx
       return zero, nil
   }
   ```

### Task 3: Add `internal/wasm/apiclient/` with one method per route
- **Description:** Build a typed HTTP client that wraps `dom.FetchJSON` and
  exposes one method per `/api/...` route the WASM frontend needs. The
  return types are `pkg/api` DTOs from Task 1; query parameters are typed
  Go arguments (no string-mashed URLs at call sites). Construct one
  `Client` per WASM lifetime in `cmd/wasm/main.go` and inject it into the
  `application` layer. Keep the client free of DOM concerns — it is pure
  HTTP. Auth-bearing endpoints (`/api/me/*`) take the init-data string as
  a method parameter rather than reading it from `window` themselves;
  reading `window.Telegram.WebApp.initData` is the responsibility of
  `cmd/wasm/main.go` (see Task 8).
- **Acceptance Criteria:**
   - [ ] `internal/wasm/apiclient/client.go` exists with **no build tag**
         and exports a `Client` struct backed by the `Fetcher` interface.
         (Diverges from the original draft, which placed `//go:build js && wasm`
         on `client.go`. The build tag was removed during implementation so
         the file compiles under the host toolchain, enabling the unit tests
         in `client_test.go` to run under regular `make test`.
         `fetcher_wasm.go` carries the `js && wasm` tag instead, keeping the
         `syscall/js` dependency isolated. Documented for plan honesty per
         `CLAUDE.md` § Planning Workflow.)
   - [ ] Methods (one per route, in the order they appear in
         `CLAUDE.md` § "HTTP Routes" — only the ones the frontend
         consumes):
         - `ListSources(ctx, limit) ([]api.SourceResponse, error)`
         - `ListRates(ctx, name, limit) ([]api.RateResponse, error)`
         - `ListSubscriptions(ctx, name, page) ([]api.SubscriptionDetailResponse, error)`
         - `ListDailyEvents(ctx, name, page) ([]api.DailyEventResponse, error)`
         - `ListExecutionErrors(ctx, page) ([]api.ExecutionErrorResponse, error)`
         - `ListFailedNotifications(ctx, offset, limit) ([]api.NotificationResponse, error)`
         - `Stats(ctx) (api.StatsResponse, error)`
         - `SetSourceActive(ctx, name, active) error`
         - `MeSubscriptions(ctx, initData, page, pageSize, query) (api.MeSubscriptionsResponse, error)`
   - [ ] All other endpoints listed in `CLAUDE.md` (chart data, history,
         events/failed per source, subscriptions summary, events/pending,
         notifications) are **not** in scope for this client because they
         are not consumed by either `index.html` or `subscriptions.html`
         today. Adding them is a follow-up.
   - [ ] `MeSubscriptions` sets `X-Telegram-Init-Data` from the parameter,
         not from `js.Global()`.
   - [ ] Tests under `internal/wasm/apiclient/` use a fake `Fetcher`
         (interface introduced in Task 2 or in this task; reused by the
         `application` layer in later tasks). Each `TestClient_<Method>`
         is a single top-level function with `t.Run("<scenario>", ...)`
         subtests for happy path, server 5xx, malformed JSON, and (for
         `MeSubscriptions`) header propagation.
   - [ ] `var _ Fetcher = (*fakeFetcher)(nil)` compile-time assertion at
         the top of every test file using a fake.
   - [ ] No `_ = err` discards in the client or its tests.
- **Pitfalls:**
   - URL building: `name` may contain `/` or `%`; use `url.PathEscape` for
     path segments and `url.Values.Encode()` for query strings, never
     `fmt.Sprintf` with string concatenation for user-influenced values.
   - The server returns `[]` for empty lists, not `null`. Make sure the
     decoder treats both as empty (default `encoding/json` behaviour does,
     but assert it in a test).
   - `SetSourceActive` returns no body on success; do not try to decode
     into a DTO. Either branch on `Content-Length`/`204`-like behaviour
     or keep an explicit `FetchNoContent` helper in `dom`.
- **Complexity:** Medium
- **Code Example:**
   ```go
   //go:build js && wasm

   package apiclient

   import (
       "context"
       "fmt"
       "net/url"

       "github.com/seilbekskindirov/monitor/internal/wasm/dom"
       "github.com/seilbekskindirov/monitor/pkg/api"
   )

   type Client struct{}

   func New() *Client { return &Client{} }

   func (c *Client) ListSources(ctx context.Context, limit int) ([]api.SourceResponse, error) {
       u := fmt.Sprintf("/api/sources?limit=%d", limit)
       return dom.FetchJSON[[]api.SourceResponse](ctx, "GET", u, nil, nil)
   }

   func (c *Client) MeSubscriptions(ctx context.Context, initData string,
       page, pageSize int, q string) (api.MeSubscriptionsResponse, error) {

       v := url.Values{}
       v.Set("page", fmt.Sprintf("%d", page))
       v.Set("page_size", fmt.Sprintf("%d", pageSize))
       if q != "" {
           v.Set("q", q)
       }
       headers := map[string]string{"X-Telegram-Init-Data": initData}
       return dom.FetchJSON[api.MeSubscriptionsResponse](
           ctx, "GET", "/api/me/subscriptions?"+v.Encode(), nil, headers,
       )
   }
   ```

### Task 4: Port "Sources List" screen end-to-end and switch it on
- **Description:** First screen ported. Implement the page controller in
  `internal/wasm/application/sources.go` (state: source list, filter values,
  sort direction; methods: `OnFilterTitle`, `OnFilterPair`,
  `OnFilterStatus`, `OnFilterActive`, `ToggleSort`, `ToggleActive`,
  `Render() State`) — pure Go, no `syscall/js`. Implement the rendering in
  `internal/wasm/ui/sources.go` (`Render(state) string` returns HTML). Wire
  it through `cmd/wasm/main.go`: on hash `#/` (or empty hash) the WASM
  installs the sources controller, fetches via `apiclient`, applies state
  to the DOM. Update `cmd/web/static/index.html`: add
  `<script src="/wasm_exec.js"></script>` plus the
  `WebAssembly.instantiateStreaming(fetch("/app.wasm"), …)` boot block,
  and remove the `renderSources()` function (lines 78–196) plus its
  dispatch entry in `route()` (line 73). The `route()` function continues
  to handle `#/sources/<name>` and `#/errors` via the JS path. After this
  task, `#/` is rendered by WASM; everything else still by JS.
- **Acceptance Criteria:**
   - [ ] `internal/wasm/application/sources.go` (build-tagged) defines a
         `Page` struct with no DOM dependencies. Its tests
         (`sources_test.go`) cover: filter by title (case-insensitive,
         matches name OR title), filter by pair, status filter, active
         filter, sort toggle. One `TestPage_*` per method, scenarios as
         `t.Run` subtests.
   - [ ] `internal/wasm/ui/sources.go` (build-tagged) exposes
         `Render(state) string`. Its tests (`sources_test.go`) compare
         rendered string output with golden snippets — no DOM. **Every
         user-influenced field** (`name`, `title`, currencies, error)
         passes through `dom.Escape`; tests assert this with a row whose
         name contains `<script>` and one with `&` + `"`.
   - [ ] On `#/`, the page renders the source table with title/pair/
         interval/last-run/status/active columns matching the JS version
         visually (header text, column count, ordering).
   - [ ] Filters and sort toggle work without a network round-trip
         (state-only, just like the JS version).
   - [ ] Active toggle button issues `PATCH /api/sources/{name}/active`
         and re-fetches the list, exactly mirroring `index.html:159-171`.
   - [ ] `index.html` no longer contains `renderSources` or its `window.*`
         filter callbacks. The `route()` function falls back to JS only
         for `#/sources/...` and `#/errors`.
   - [ ] `make build && make test` green. Manual smoke test: load `/`,
         confirm sources render, click a source link → JS detail page
         still works.
- **Pitfalls:**
   - "Boot WASM" plus "JS still owns other routes" is the dangerous
     handoff. Decide upfront: does WASM listen for `hashchange` itself,
     or does the existing JS `route()` decide and call into WASM via a
     `window._renderSources` shim? Recommendation: **JS owns the router
     until the last screen lands.** WASM exports a single
     `window._wasm = { renderSources(): void, renderSourceDetail(name):
     void, renderErrors(): void }` object whose surface grows over the
     migration, and JS `route()` dispatches to whichever exists. This
     means JS removal is incremental and there is one obvious migration
     finish line: when JS `route()` is just three calls into `_wasm`, it
     can be deleted (Task 12).
   - The WASM `instantiateStreaming` must be awaited before
     `window._wasm.renderSources` is called. Boot the WASM, then call
     `route()` — do not let `route()` race the WASM init.
   - The `_loadRates` global pattern from `cmd/wasm/main.go:36-44`
     **must be deleted in this task** (the file is rewritten anyway).
     Inline `onclick="..."` strings in WASM-generated HTML are forbidden
     in the new architecture — use `dom.On` against rendered DOM
     elements found via `document.querySelector`.
   - Escaping is the single biggest XSS risk (review §5). The test that
     renders a source named `<script>alert(1)</script>` must pass.
   - `index.html` currently has cosmetic-only data: `↓`/`↑` arrows in the
     sort header. Keep them.
- **Complexity:** Hard
- **Code Example:**
   ```go
   //go:build js && wasm

   package application

   import "github.com/seilbekskindirov/monitor/pkg/api"

   type SourcesState struct {
       All          []api.SourceResponse
       FilterTitle  string
       FilterPair   string
       FilterStatus string // "all" | "ok" | "error"
       FilterActive string // "all" | "yes" | "no"
       SortDesc     bool
   }

   // Visible returns the slice after filter+sort. Pure: no I/O, no DOM.
   func (s *SourcesState) Visible() []api.SourceResponse { /* ... */ return nil }
   ```

### Task 6: Port "Errors" screen
- **Description:** Third and final `index.html` screen. Implement
  `internal/wasm/application/errors.go` and `internal/wasm/ui/errors.go`.
  Wire into `cmd/wasm/main.go` as `_wasm.renderErrors()`. Remove the JS
  `renderErrors` function from `index.html` (lines 310–364). After this
  task, JS `route()` is three calls into `_wasm` and nothing else — but
  still needs to live until `index.html` is reduced to a loader (Task 12).
- **Acceptance Criteria:**
   - [ ] Two paginated tables: execution errors and event (notification)
         errors, mirroring the JS version's columns and pagination
         semantics exactly.
   - [ ] Execution errors page param goes via `?page=<n>`; event errors
         use `?offset=<(page-1)*50>&limit=50` (exact mirror of
         `index.html:333`). Do not silently "harmonize" the two —
         changing pagination is out of scope.
   - [ ] Tests: pagination boundaries; UI escaping for `error` and
         `last_error` fields (these come from upstream HTTP errors and
         are not sanitized server-side).
   - [ ] `make build && make test` green.
- **Pitfalls:**
   - The two tables share pagination behaviour but use different limits
     (`25` vs `50`). The shared `pagination` UI component from Task 5
     must accept `limit` as a parameter, not bake it in.

## Risks

- **Risk: WASM init race with hash router.** The most likely live-prod
  bug. JS dispatches `route()` on `window.load` synchronously; WASM
  init is async. Mitigation: the JS shim calls `route()` only after
  `wasmReady` resolves, and `route()` fails closed (renders a "Loading…"
  skeleton) if `_wasm.<method>` is not yet defined.
- **Risk: XSS regression.** The WASM port concatenates HTML strings the
  same way the existing `cmd/wasm/main.go` does — and the existing one
  already has the bug (review §5). Mitigation: every UI test asserts
  escaping with payloads containing `<`, `>`, `&`, `"`, `'`. Reviewer
  blocks merge on any UI Render function that produces HTML without
  passing user-influenced fields through `dom.Escape`.
- **Risk: bundle size regression.** The new code is more Go than the
  current `cmd/wasm/main.go` (152 lines → ~1500 lines). Standard Go
  compiler bundles the runtime regardless of code size, so the actual
  bytes-on-the-wire delta should be small (~50–200 KB). Mitigation: log
  `app.wasm` size before and after each task; if any task inflates
  binary by >10% something is wrong (probably `reflect`-heavy code that
  could be replaced).
- **Risk: Telegram theme variables.** The Mini App today reads
  `--tg-theme-*` CSS variables that Telegram injects on page load. If
  the WASM port replaces the styled `<div>`s with Go-rendered HTML, the
  variables still apply (CSS cascades into innerHTML). But if anyone
  later migrates to inline-style strings, they will inherit nothing.
  Mitigation: Task 8 acceptance criteria explicitly requires class-
  based styling, no inline styles for theme-able properties.
- **Risk: stale embed.** `cmd/web/main.go:201` embeds
  `cmd/web/static/`. After `make clean` removes `app.wasm` and
  `wasm_exec.js` (`Makefile:120`), running the binary without a
  rebuild shows a working HTML loader that 404s on `/app.wasm`. The
  user sees a blank page. Task 9 mitigates by adding a loader-side
  fail-loud message, but the deeper fix (CI-level "build before
  test") is out of scope for this plan.
- **Risk: divergence between `pkg/api` and the actual server response.**
  Even with a shared package, a handler can drop a field by accident.
  Mitigation: the existing handler tests in
  `internal/gateway/httpV1/handlers/handlers_test.go` already snapshot
  the JSON output. Task 1 keeps those tests passing unchanged. Going
  forward, any new field added to the wire should be added to
  `pkg/api/` and used at both ends, not to the server alone.
- **Risk: `go.mod` drift.** Someone in a later task adds an `import` for
  a frontend-helper module ("just one CSS-in-Go library, harmless").
  ADR-001 forbids this. Mitigation: `make lint` already runs
  `go vet`; add a forbidden-import check that fails on any new entry
  in `go.mod` whose module path the project doesn't already depend on.
  This is a reviewer-eyeball check at minimum; add the lint rule if
  cheap.
- **Risk: standard-Go WASM `wasm_exec.js` ABI break between Go versions.**
  Bumping the host Go version regenerates `wasm_exec.js` shape; the
  binary expects a matching one. Because the `Makefile` always copies
  the host's `wasm_exec.js` next to a freshly built `app.wasm`, this
  risk is contained — unless someone ships a binary with a stale
  embedded `wasm_exec.js` (pre-built and `make clean` skipped).
  Mitigation: leave the `Makefile` build-order alone (copy first,
  build WASM second); reviewer spot-checks the embedded files are from
  the same build.

## Trade-offs

- **No framework over `gomponents`/`vecty`/etc.** Locked by ADR-001.
  Cost: HTML-string concatenation everywhere in `internal/wasm/ui/`
  with disciplined escaping. Benefit: zero `go.mod` additions, full
  visibility of the JS↔Go boundary (the explicit learning goal). The
  mitigation if pain grows: ADR-003 to adopt `gomponents`.
- **Standard Go compiler over TinyGo.** Locked by ADR-001 addendum.
  Cost: ~3 MB cold load. Benefit: zero compiler-quirk surprises with
  `encoding/json`, `reflect`, goroutine scheduling. Re-evaluate when
  someone complains about size, not before.
- **One `_wasm` global object as the JS↔Go dispatch surface, vs. WASM
  owning its own router.** Chose the global-object approach for the
  duration of the migration so JS removal is incremental (the JS
  router shrinks to a 5-line shim by Task 7). Cost: one global. Benefit:
  every intermediate ship is functional.
- **Pure-Go controller layer (`internal/wasm/application/`) with no
  `syscall/js` calls.** Costs slightly more code (event handlers in
  `cmd/wasm/main.go` translate `js.Value` to Go primitives before
  calling controller methods). Benefit: controllers are testable under
  the host toolchain via `make test`, no `wasm_exec` headless runner
  needed in CI.
- **Skip WASM-headless CI (Task 11 option B).** Cost: `ui` and `dom` are
  not exercised in automated CI runs against PRs. Benefit: no Node
  dependency, no fragile `wasm_exec` flake. The pure-Go layers around
  them (controllers, apiclient with httptest seam) cover the bulk of
  the logic, and HTML-string output of the `ui` layer is reviewable
  on inspection.
- **`pkg/api/` as a public package, not `internal/api/`.** A future
  external Go client (or a sibling `cmd/cli/` admin tool) could import
  the same DTOs without leaking internal types. Cost: the export
  surface is now part of the project's stability promise. Benefit:
  the original WASM rationale ("share types end-to-end") delivers
  value beyond just WASM. If we later change our mind, moving public
  → internal is a one-day chore.
- **Mini App port last, not first.** Telegram WebApp `initData`
  validation is the most exotic auth flow in the project; doing it
  after the dashboard surface settles means the reviewer is fully
  fluent in the new architecture before staring at HMAC details.
  Cost: longer wait before Mini App users see Go-typed code on their
  side. Benefit: lower likelihood of an auth-flow bug being missed
  because reviewer fatigue had already set in on the dashboard.
