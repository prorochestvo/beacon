# Task Breakdown

## Overview

Plan 008 delivered the AI client infrastructure (`internal/infrastructure/artificialintelligence/`). Plan 009 delivered the rule-generator service (`internal/application/rulegen/` and `cmd/rulegen`) with a primary→fallback audit loop. A live smoke-test of branch `llm_v2` against Kazakhstani bank sources exposed four root-cause issues that two stop-gap commits (`24b3478` prompt addendum, `6dcdcf2` 400 KB cap) only papered over. This plan addresses the root causes and bundles three deferred items from the plan-009 review.

Concretely, this plan:

1. Splits the misleading `"invalid regex pattern"` error in `internal/tools/rateextractor/matchers.go` into a distinct compile-error and no-match-error path so the audit loop's transcript can feed a meaningful signal back to the LLM.
2. Adds a `Locate` step to `internal/application/rulegen/sanitizer.go` that scans the stripped body for currency-code anchors and returns a window around the first hit, so the cap can drop from 400 KB to a safer ~80 KB and most HTML sources fit back inside Groq's 8000 TPM free tier.
3. Adds symmetric retry to the fallback path in `Generator.Generate` with a configurable budget (default 2), so a fallback model that emits invalid RE2 syntax on its first shot gets the same transcript-feedback loop the primary already has.
4. Adds a pre-compile validation pass in `Generator.Generate` immediately after `parseRulesResponse` so a malformed regex is caught with a clean, RE2-specific error string before the executor sees the body.
5. Bundles three small deferred items from the plan-009 review: the `sequentialExecutor` compile-time interface assertion, splitting `AIClient.Name()` into `Name()` + `Model()`, and unifying the value-range lower-bound check across `rateextractor`, `sourceaudit`, and `rulegen`.

All work stays in Go, no new dependencies, no migrations, no schema changes. The `domain.RateSourceRuleMetadata` shape already has separate `Provider` and `Model` string fields (`internal/domain/ratesource.go:22-23`), so task 7 only needs the interface split plus persistence call-site update; no struct change.

## Assumptions

1. **Smart-locate is byte-level only.** No goquery, no html parser. Plain `bytes.Index` against the stripped body. This matches the existing pattern in `sanitizer.go` which uses precompiled regex for `<script>` and `<style>` stripping but never parses the DOM.
2. **First-hit-wins, not closest-to-middle.** When scanning for anchors, the first byte offset of the first matching anchor in the priority list becomes the window centre. Bank rate tables in the four seeded HTML samples (BCC, Halyk, Bank Freedom Finance, NationalBank) appear once, so "closest to middle" would be over-engineering.
3. **Anchor priority is currency-pair first, then generic tags.** The priority list is `[BaseCurrency, QuoteCurrency, "<table", "<tbody", "<tr "]`. The pair comes from `domain.RateSource.BaseCurrency` / `.QuoteCurrency` already in scope when `Sanitize`/`Locate` is called from `Generator.Generate`.
4. **Window size of 80 KB centred (±40 KB) by default.** Empirical evidence from `tmp/testdata/sources/`:

   | source | raw size | first USD offset | first KZT offset |
   |--------|----------|------------------|------------------|
   | bcc.html | 766 KB | 543,980 | 543,687 |
   | halykbank.html | 325 KB | 139,693 | 186,798 |
   | bankffin.html | 146 KB | 80,336 | 86,480 |
   | nationalbank.html | 375 KB (JSON-API anyway) | 323,825 | 314,633 |

   These bodies are unstripped raw HTML. Post-`<script>`/`<style>` strip the bodies shrink by roughly 40–60% based on the smoke-test logs the user reported. An 80 KB window (±40 KB around first anchor) captures the rate row in every HTML sample with margin — including the worst-case 543k-byte BCC offset which then sits inside a ~503k–583k slice in the raw body, or roughly the middle of the stripped body. Smaller windows risk amputating the wrapping `<tr>`/`<td>` structure the LLM relies on for anchoring; larger ones erode the Groq TPM headroom we are trying to recover.
5. **Cap is 80 KB after smart-locate, retained as a safety net.** When `Locate` finds no anchor, fall through to the existing truncate-from-start behaviour, capped at the same 80 KB. The hard 5 MB pre-strip reject (`maxRawBodyBytes`) is unchanged.
6. **Fallback retry default of 2.** The smoke-test showed `gpt-5.4-nano` regressing to non-RE2 syntax on its single shot. 2 attempts is the smallest value that lets the second call see a transcript pointing out the first failure; 3+ would erode the cost difference between primary and fallback. Configurable via `--max-fallback-attempts` for the operator override.
7. **The pre-compile error wording fed back to the LLM is exactly:** `"regex did not compile: <go regex error>; revise the pattern to comply with RE2 syntax (no lookarounds, no backreferences, no \u escapes, no \\/, no possessive quantifiers)"`. This mirrors the bullet list already in the system prompt and refreshes the constraint at the moment the LLM is about to retry.
8. **The user-facing error wording change in `ApplyRegex` is non-breaking for `sourceaudit`.** The audit's `probeSource` (`internal/application/sourceaudit/audit.go:99-104`) catches the error, truncates it to 80 chars, and sets `StatusRegexNoMatch`. With the new wording (`"pattern %q produced no match in body (len=%d)"`) the status enum and `Detail` shape stay valid. The `report_test.go` cases use hand-crafted Detail strings (`"no match found"`) — those tests are not invalidated. Test updates required only in `matchers_test.go` (lines 27 and 36) where the literal `"invalid regex pattern"` is asserted.
9. **`AIClient.Name()` continues to return a human-readable composite like `"Groq[openai/gpt-oss-20b]"`** for the `cmd/web/main.go:135-136` startup log lines. The new `Model()` returns just the model id (e.g. `"openai/gpt-oss-20b"`). The `Generator.persist` (`internal/application/rulegen/generator.go:208-216`) is the only call site that needs to consume both: `Provider = client.Name()` for the log-friendly composite, `Model = client.Model()` for the bare id.
10. **Value-range lower bound becomes `<= 0` everywhere.** The `0.0001` floor in `rulegen/executor.go:91` is stricter but no production currency pair trades below 1 unit of any other (USD/BTC at ~$0.00001 in 2009 is not in scope here — we monitor fiat pairs and gold). A uniform `<= 0` is simpler, matches the existing rate-extractor and audit code paths, and removes a "why does this constant differ" footgun. The trade-off: an LLM-generated rule that latches onto a stray `0.0` in some decoration would have been caught by the `0.0001` floor; with `<= 0` it gets through. The audit loop will catch it on the next live extraction when the value diverges from market — acceptable.

   - [ ] `ApplyRegex` returns an error containing `"produced no match in body (len="` when the regex compiles but `FindSubmatch` returns nothing.

- **Pitfalls:** Resist the temptation to also rename `StatusRegexNoMatch` — the enum string is persisted to log files and grepped by ops. The wording change applies only to the Go-side `error` value. - **Complexity:** Easy
- **Code Example:**

  ```go
  // internal/tools/rateextractor/matchers.go
  func ApplyRegex(pattern string, payload []byte) ([]byte, error) {
      re, err := regexp.Compile(pattern)
      if err != nil {
          err = fmt.Errorf("compile pattern %q: %w", pattern, err)
          return nil, errors.Join(err, internal.NewTraceError())
      }

      matches := re.FindSubmatch(payload)
      if len(matches) < 2 {
          err = fmt.Errorf("pattern %q produced no match in body (len=%d)", pattern, len(payload))
          return nil, errors.Join(err, internal.NewTraceError())
      }

      return matches[1], nil
  }
  ```

### Task 2: Add `Locate` helper to `sanitizer.go` and integrate into `Sanitize`

- **Description:** Add a new exported function `Locate(body []byte, anchors []string, windowBytes int) ([]byte, bool)` to `internal/application/rulegen/sanitizer.go`. It walks `anchors` in order, calls `bytes.Index(body, []byte(anchor))` for each, picks the smallest non-negative offset across the list, and returns `body[max(0, hit-windowBytes/2) : min(len(body), hit+windowBytes/2)], true`. When no anchor is found it returns `body, false` (caller decides what to do).

  Modify `Sanitize` to accept the anchors slice and the window size, so the composition is: hard-reject → strip scripts/styles → `Locate` → cap. Signature changes from `func Sanitize(body []byte) ([]byte, int, error)` to `func Sanitize(body []byte, anchors []string) ([]byte, int, error)`. The `windowBytes` is a package-level constant `locateWindowBytes = 80 * 1024`; not exposed via the function signature because there is a single call site and the constant doubles as documentation.

  Drop `maxBodyBytesForLLM` from 400 KB to 80 KB. This becomes the hard ceiling applied after `Locate`: if `Locate` returns a 60 KB slice it stays 60 KB; if `Locate` returns `false` (no anchor found) the original truncate-from-start kicks in capped at 80 KB.

  Update `internal/application/rulegen/generator.go:113` to pass anchors derived from the source row:

  ```go
  anchors := []string{src.BaseCurrency, src.QuoteCurrency, "<table", "<tbody", "<tr "}
  body, originalSize, err := Sanitize(rawBody, anchors)
  ```

  Update the hardcoded `"BODY (truncated to 200 KB, ..."` string at `generator.go:250` to read `"BODY (sectioned to %d KB around anchor, scripts/styles stripped, original was %d bytes)"` with `locateWindowBytes/1024` and `originalSize` interpolated. The "200 KB" literal is already stale post-`6dcdcf2` — this fix is overdue.

  Update `sanitizer_test.go`: the existing six subtests still apply but the `TestSanitize_truncates_oversized_body_to_the_configured_cap` subtest must be adjusted because `maxBodyBytesForLLM` is now 80 KB and the cap applies *after* `Locate` returns no-anchor-found. Add three new subtests for `TestLocate`: anchor-found-at-start (window clamps to `[0, windowBytes/2]`), anchor-found-mid-body (window is centred), anchor-not-found (returns body unchanged with `false`).

- **Acceptance Criteria:**
   - [ ] `Locate` is exported, has a doc comment, and a single test function `TestLocate` with at least three subtests (`anchor at start`, `anchor mid-body`, `no anchor returns false`).
   - [ ] `Sanitize` signature takes `anchors []string` as a second argument.
   - [ ] `maxBodyBytesForLLM` constant is `80 * 1024`.
   - [ ] When the bcc.html sample (raw 766 KB) is fed through the new pipeline with anchors `["USD","KZT","<table","<tbody","<tr "]`, the returned slice has length `locateWindowBytes` (80 KB) and contains the substring `"USD"`. (Add this as a fixture-based subtest reading `tmp/testdata/sources/bcc.html` via `os.ReadFile`; skip with `t.Skip` if the fixture is missing so the test still runs on a clean checkout — `tmp/` is not always present.)
   - [ ] The "BODY (truncated to 200 KB ..." literal is gone from `generator.go`.
   - [ ] All existing `sanitizer_test.go` subtests still pass with the new signature.
   - [ ] `make test` passes.
- **Pitfalls:**
  1. `Locate` operates on the *stripped* body, not raw. The 5 MB hard-reject and the script/style strip happen first; only then do we have the bytes that the LLM will see and that the anchor scan should run against.
  2. Off-by-one at the body ends: `hit - windowBytes/2` can be negative; `hit + windowBytes/2` can exceed `len(body)`. Clamp both before slicing.
  3. Anchor priority matters when both `USD` and `KZT` are present at very different offsets. The smallest offset across the list wins, *not* the first list element that matches. Iterate every anchor, collect all valid offsets, return the slice around `min(offsets)`. This means a `<table` at byte 50,000 wins over `USD` at byte 500,000 — correct behaviour because the table tag is the structural anchor and the LLM benefits from seeing the table header rows.
  4. When `BaseCurrency == QuoteCurrency` (shouldn't happen but guard anyway) deduplicate the anchors slice to avoid scanning the body twice for the same literal.
  5. The `<tr ` anchor has a trailing space on purpose — bare `<tr` would match `<tr_` or `<truncated`. The space ensures it's a real tag-open. Same logic for keeping `<table` and `<tbody` without trailing space because the next char is always `>` or whitespace; both already exclude false positives like `<tablename`.
- **Complexity:** Medium
- **Code Example:**

  ```go
  // Locate returns the smallest window of body centred on the earliest match of
  // any anchor in anchors. When no anchor is found, it returns body unchanged
  // and found=false; callers fall back to truncate-from-start.
  func Locate(body []byte, anchors []string, windowBytes int) ([]byte, bool) {
      best := -1
      for _, a := range anchors {
          if a == "" {
              continue
          }
          i := bytes.Index(body, []byte(a))
          if i < 0 {
              continue
          }
          if best < 0 || i < best {
              best = i
          }
      }
      if best < 0 {
          return body, false
      }
      half := windowBytes / 2
      start := best - half
      if start < 0 {
          start = 0
      }
      end := best + half
      if end > len(body) {
          end = len(body)
      }
      return body[start:end], true
  }
  ```

### Task 3: Add pre-compile validation pass to `Generator.Generate`

- **Description:** Today, `parseRulesResponse` (in `internal/application/rulegen/generator.go:270-312`) only validates the JSON shape and method enum; it does not check that a `MethodRegex` rule's `Pattern` is a syntactically valid RE2 expression. The first time we discover a malformed pattern is inside `executor.Execute` → `ApplyRegex` → `regexp.Compile`, deep in the call stack, with the error message wrapped through three layers (`"rulegen: rule %d (regex): compile pattern \"...\": error parsing regexp: ..."`).

  Add a helper `func validateRulePatterns(rules []domain.RateSourceRule) error` that loops `rules`, and for every rule with `Method == MethodRegex`, runs `_, err := regexp.Compile(rule.Pattern)`. On the first failure return `fmt.Errorf("regex did not compile: %v; revise the pattern to comply with RE2 syntax (no lookarounds, no backreferences, no \\u escapes, no \\/, no possessive quantifiers)", err)`. The exact wording matters: the resulting string is what gets appended as `outcome` in the next prompt's `PREVIOUS ATTEMPTS` section, so it has to be both human-meaningful and a hint the LLM can use.

  Call `validateRulePatterns(rules)` in `Generator.Generate` immediately after `parseRulesResponse` returns success and before `g.executor.Execute(rules, body)`. On error, treat it exactly like the existing parse-error and exec-error paths: append to `transcript`, log, `continue` (in the primary loop) or fall through to the fallback path. Apply the same treatment in the fallback path (after task 4 adds the fallback retry loop).

  Add subtests to `generator_test.go`: `TestGenerator_Generate/primary_attempt_with_invalid_regex_retries_with_transcript_feedback` exercising a primary that returns a rule with `(?<=USD)` (lookbehind, RE2-illegal) and verifying that (a) the executor is never called for that attempt, (b) the transcript entry's outcome contains `"regex did not compile"`, (c) attempt 2 of the primary fires.

- **Acceptance Criteria:**
   - [ ] `validateRulePatterns` is a package-level function in `generator.go` (or a new file `validate.go` in the same package — engineer's call, but keep it package-private).
   - [ ] Called from `Generator.Generate` immediately after `parseRulesResponse`, before `executor.Execute`, in both primary and fallback paths.
   - [ ] Failure produces a transcript entry whose `Outcome` field starts with `"regex did not compile:"`.
   - [ ] The original Go regexp error message is included verbatim after the colon (via `%v` of the returned error) so the LLM gets the specific RE2 violation.
   - [ ] When a regex compiles fine but produces no match at execute time, the path remains unchanged (task 1's new no-match error wording still flows through).
   - [ ] At least one new subtest in `TestGenerator_Generate` verifies the invalid-regex-retries-with-feedback behaviour using `(?<=USD)` or `\1` as the malformed input.
   - [ ] `make test` passes.
- **Pitfalls:**
  1. Do not pre-compile the regex once and reuse — `executor.Execute` re-compiles internally, and trying to cache a `*regexp.Regexp` across layers would couple the validator to the executor's internals. Validation is a side effect; throw away the compiled regex.
  2. `MethodJSONPath` rules do not need pre-validation here — the JSON path parser already runs at execute time and produces a descriptive error. Adding a duplicate pass would only add code surface.
  3. The error wording is part of the prompt contract; if you change it later you must also update the system prompt's `RE2 CONSTRAINTS` section in `prompt.go` so the two stay aligned.
- **Complexity:** Easy
- **Code Example:**

  ```go
  // validateRulePatterns runs regexp.Compile on every MethodRegex rule. It
  // returns a descriptive error on the first compile failure so the audit
  // loop can feed an RE2-specific hint back to the LLM without exercising
  // the executor.
  func validateRulePatterns(rules []domain.RateSourceRule) error {
      for i, r := range rules {
          if r.Method != domain.MethodRegex {
              continue
          }
          if _, err := regexp.Compile(r.Pattern); err != nil {
              return fmt.Errorf(
                  "rule %d: regex did not compile: %v; revise the pattern to comply with RE2 syntax "+
                      "(no lookarounds, no backreferences, no \\u escapes, no \\/, no possessive quantifiers)",
                  i, err,
              )
          }
      }
      return nil
  }
  ```

## Risks

1. **Smart-locate window too narrow for a site we haven't seen.** The 80 KB default is calibrated against the four seeded sources. A future bank with a deeply nested page where the rate table is announced by `<table>` 10 KB before the actual `USD` row, and the row itself spans 30 KB of decorative markup, could spill past the window. *Mitigation:* engineer adds a `--locate-window-kb` CLI flag in a follow-up only if observed; not part of this plan. Operators today can bypass by setting `Options.Reserve` hints; that escape valve remains untouched.
2. **Fallback retry budget burning cost.** Each fallback attempt is a paid API call (OpenAI, OpenRouter). Default of 2 doubles worst-case fallback cost per `cmd/rulegen` invocation from `1×fallback_call_cost` to `2×fallback_call_cost`. With `gpt-5.4-nano` at smoke-test prices this is negligible; with a future fallback set to `gpt-4o` (the openai.go default) the doubling matters. *Mitigation:* the CLI flag exists; ops can `--max-fallback-attempts 1` to opt out per-invocation, and the README guides the trade-off.
3. **Error-wording change in `ApplyRegex` may break out-of-tree log greps.** Operators or downstream tools that grep service logs for the literal string `"invalid regex pattern"` will silently miss matches after this change. *Mitigation:* the in-repo callers (sourceaudit's `StatusRegexNoMatch`, executor's wrapping) are stable. Any external log grep belongs to ops tooling that lives outside this repo and is on the operator to update; the new wording is more searchable (`"produced no match in body"` is unambiguous), so the migration cost is one-time.
4. **`Model()` returning `"stub"` from the stub client could leak into the persisted DB.** Anyone running `cmd/rulegen` without a real `AI_FALLBACK_DSN` and hitting the stub on fallback will record `metadata.Model = "stub"` against a source. This is a feature, not a bug — it makes "this row was generated against the stub, treat with suspicion" trivially greppable — but document it in `cmd/rulegen/README.md` so a future operator doesn't file it as a bug.
5. **Task 7's lower-bound relaxation (`0.0001` → `0`) admits a class of bad rules.** The previous floor rejected an LLM that latched onto a `0.0001` literal somewhere in page decoration; the new floor only rejects exact zero. *Mitigation:* the audit loop catches divergent values on the next live extraction when the rate visibly disagrees with reality; this is monitoring's job, not the validator's. The trade-off is documented in Assumption 10.

## Trade-offs

- **First-anchor-hit over closest-to-middle.** Simpler. The four seeded HTML samples have a single rate-table region with one `USD` literal in or near it. Choosing the offset closest to the body's middle would require an extra pass and a tunable parameter, with no observed benefit. If a future source has multiple `USD` mentions (e.g. a marketing banner at byte 5,000 and the real table at byte 500,000), the smart-locate picks the wrong one; but the audit loop will catch the wrong-value extraction on the next attempt, so the cost is one extra LLM call, not a corrupted DB. Cheap fix later if needed.
- **Symmetric retry on fallback, not retry with a different model.** Plan 008 supports multiple AI client types; one could imagine a 3-tier escalation `Groq → OpenAI → OpenRouter`. Rejected here because: (a) it adds a third DSN env var, multiplying cost surface; (b) the plan-009 single-fallback design has worked for JSON-API sources; (c) the smoke-test failure mode was "fallback regressed on its single shot", not "fallback model is fundamentally wrong" — retry with feedback solves the actual observed problem. If a future smoke-test shows the fallback model is structurally incapable, that's a different plan.
- **Pre-compile validation as a separate pass, not folded into `parseRulesResponse`.** Keeping `parseRulesResponse` purely about JSON shape preserves its single responsibility and keeps its tests focused. The validator is a small dedicated function that can grow new checks (e.g. JSON-path syntax validation, anchor sanity) without distending the parser.
- **80 KB cap, not 60 KB.** The user's original spec floated ~60 KB. The empirical evidence from `tmp/testdata/sources/` shows the bcc.html `USD` offset at 543,980 of 766,048 stripped raw bytes; a 60 KB window would leave 30 KB to either side of the anchor. That is enough for most rows but tight when the wrapping `<table>` and column headers sit further upstream. 80 KB keeps a 5–10 KB safety margin without meaningfully eroding the Groq 8000 TPM tier (60 KB of text ≈ 15–20k tokens; 80 KB ≈ 20–27k tokens — both are above the 8k cap on a single call but the cap is per-*minute*, so we are throttling the *rate*, not the per-call ceiling; the call itself proceeds, just slower). The 8k TPM concern is a deploy-cadence concern, not a per-call gating concern; reviewer should sanity-check this framing against the smoke-test logs the user has on file.
- **Constants for value bounds instead of a `RateBounds` struct.** The bounds are two scalars used at three call sites; a struct would invite hangers-on (`Currency`, `Side`, `Tolerance`) that don't belong here. Two exported `const`s are the minimum surface area.
