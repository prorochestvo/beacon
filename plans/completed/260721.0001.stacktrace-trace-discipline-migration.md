# Task Breakdown

## Overview

Update beacon from loginjector v1.0.5 to v1.0.7 and adopt the cheap-vs-expensive trace
discipline. beacon consumes loginjector for its **logger** (`internal/logger.go` calls
`loginjector.NewLogger` and `loginjector.CyclicOverwritingFilesHandler` and declares its
own `LogLevel` ladder), both of which changed (breaking) in the v1.0.6 API finalization
that v1.0.7 carries. Separately, beacon ships a hand-rolled `internal.StackTraceError`
plus a package-level gopsutil `init()` (identical to com_lingocrm_api / hive /
neuro_compass) that the library now makes redundant. The real work: beacon calls the
EXPENSIVE full-stack `NewStackTraceError()` at **108** repository/database error sites —
paying the full-stack cost for what a single line already tells you. Route those to the
cheap single-frame trace and keep the full stack only where the failure is genuinely
unexpected (the panic-recovery paths in `cmd/doctor/rulegen.go` and `cmd/collector/main.go`).

## Rationale — which trace for what

- **Cheap — `loginjector.NewTraceError()` (LineTrace, ONE frame via `runtime.Caller`).**
  ~0.5 µs, a single `file.go:line (method)`. Use for EXPECTED, ubiquitous errors —
  database operations, validation, lookups. beacon already uses it 306 times.
- **Expensive — `loginjector.NewStackTraceError()` (full `runtime.Stack` dump).** ~4-12 µs
  in v1.0.7, ALL frames plus goroutine header/state. Use ONLY on panic recovery, where the
  failure is unexpected and you must walk the full call chain.
- Materializing a full stack on 108 expected DB errors pays the expensive cost ~108 times
  for information a single line already provides. v1.0.7 also backs `StackTrace` with a
  bounded 8 KiB `runtime.Stack` buffer (~3× faster on deep stacks, byte-identical output),
  so the handful of legitimately-full-stack recovery sites get faster too.

### Note on binary stripping (does NOT change the above)

Both traces resolve file:line:function from Go's pclntab, which `-ldflags="-s -w"` does
NOT remove (it strips only the gdb symbol table and DWARF). Verified empirically: a
`-s -w` binary still prints full `file:line func` for both. `-trimpath` only shortens
paths. Only obfuscators (`garble -tiny`) strip names/line info — and degrade both traces
equally. The cheap/expensive choice is about cost and detail, never stripping resilience.

## Assumptions

- loginjector is bumped from v1.0.5 to v1.0.7 in one step (v1.0.7 carries both the v1.0.6
  breaking API finalization and the StackTrace perf). In-house and pinned.
- beacon keeps gopsutil (v4.26.2) — the OS-details content is still wanted; only WHERE it
  is gathered moves (into a loginjector provider registered once).
- Full stack stays only on the panic-recovery paths; every expected-error site changes.

## Tasks

### Task 1: Bump loginjector to v1.0.7 and reconcile the two logger breaking changes
- Description: `go get github.com/prorochestvo/loginjector@v1.0.7`, then `go mod tidy`.
  Fix the two v1.0.6-era breaking call sites in `internal/logger.go`:
  - `l, err := loginjector.NewLogger(LogLevelWarning, fHNDL)` → `l := loginjector.NewLogger(LogLevelWarning, fHNDL)`
    (NewLogger dropped its always-nil error return); remove the following `if err != nil`.
  - `loginjector.CyclicOverwritingFilesHandler(folder, name, defaultFileSize, 7)` →
    `loginjector.RotatingFileHandler(folder, name, loginjector.WithMaxFileSize(defaultFileSize), loginjector.WithMaxFiles(7))`.
  See the loginjector CHANGELOG rename table for the full mapping.
- Acceptance Criteria: builds against v1.0.7; `CGO_ENABLED=0 go test ./...` green.
- Pitfalls: `RotatingFileHandler` now resumes at the highest existing index across a
  restart and defaults file lines to a leading timestamp — benign, but note if anything
  parses beacon's log files by column.
- Complexity: Easy

### Task 2: Adopt loginjector's StackTraceError + runtime-details provider
- Description: Delete beacon's hand-rolled `internal.StackTraceError` and the package-level
  `debugOSDetails` `init()` in `internal/errors.go`. Point the retained full-stack sites at
  `loginjector.NewStackTraceError()` (v1.0.7's optimized 8 KiB buffer). Move the gopsutil
  gathering into a provider registered ONCE at startup:
  `loginjector.SetRuntimeDetailsProvider(func() string { /* cpu.Info + mem.VirtualMemory + PID/PPID */ })`;
  loginjector caches it via `sync.Once`.
- Acceptance Criteria: the hand-rolled type and `init()` are gone; OS details still logged
  and computed once.
- Complexity: Medium

### Task 3: Route the 108 expected-error sites to the cheap trace
- Description: At the ~108 repository/infrastructure sites doing
  `errors.Join(err, internal.NewStackTraceError())` on database operation errors, switch to
  `errors.Join(err, loginjector.NewTraceError())`. Leave the full `NewStackTraceError()`
  ONLY on the genuine panic-recovery paths (`cmd/doctor/rulegen.go`, `cmd/collector/main.go`).
  Sweep with `rg 'NewStackTraceError\(\)' internal/ cmd/`; every hit outside a recover path
  is a target.
- Acceptance Criteria: `rg 'NewStackTraceError' internal/ cmd/` returns only the
  panic-recovery paths; every repository/db error join uses `NewTraceError`; tests green.
- Complexity: Medium (mechanical but ~108 sites — verify each is an expected-error path).

### Task 4: Verify behaviour
- Description: `CGO_ENABLED=0 go test ./...`; confirm a forced panic still logs a full
  stack and a forced DB error logs a single trace line.
- Acceptance Criteria: build + tests green; manual spot-check of one panic and one DB error.
- Complexity: Easy

## Execution Order
1. Task 1 — bump + reconcile the two logger breaking changes (unblocks compilation).
2. Task 2 — adopt the library StackTraceError + provider.
3. Task 3 — the 108-site cheap/expensive routing sweep.
4. Task 4 — verify.

## Risks
- v1.0.7's breaking changes ripple into `internal/logger.go` before the rest compiles —
  Task 1 must land first.
- The ~108-site sweep risks flipping a genuine panic-recovery site to the cheap trace;
  grep-verify that only expected-error joins change and the two recover paths keep the full stack.

## Trade-offs
- Keep gopsutil (richer CPU/RAM/PID detail) over loginjector's os/arch/go-only `Runtime()`:
  the content is wanted; only its gathering site moves into the provider.

## Outcome

Shipped in commit `refactor(errors): migrate to loginjector v1.0.7 trace discipline`.
Final state: zero hand-rolled error types; 411 expected-error sites on the cheap
`loginjector.NewTraceError()`; exactly one full-stack site,
`cmd/collector/main.go` (process-level panic recovery). Gate green:
`CGO_ENABLED=0 go build ./...`, `go vet`, `make test` (`-race`), `make lint`.
Three-lens review (correctness, security/ops, performance/architecture) all
returned SHIP with no P0/P1.

Two deviations from the plan as written, both corrections of factual errors in it:

- **Scope widened to full adoption.** The Rationale claimed beacon already used
  `loginjector.NewTraceError()` 306 times. It did not — those sites used beacon's
  own `internal.NewTraceError()`. Routing only the 108 ex-`StackTraceError` sites to
  the library would have left two functionally identical cheap-trace types coexisting
  (sometimes in one function). Resolved by also migrating the ~305 `internal.NewTraceError`
  sites and deleting `internal.TraceError` outright — symmetric with removing the
  hand-rolled `StackTraceError`, and the end-state the Rationale actually implied.
- **`cmd/doctor/rulegen.go` was never a full-stack site.** The plan named it a second
  retained full-stack path; its per-source batch recover only ever printed a terse
  `FAIL source=... reason=panic: %v` line (a cron-greppable summary, not a process
  panic). Left as-is and documented in code with a comment; `cmd/collector` is the
  sole full-stack site.

Post-review additions: `TestRuntimeDetails` (restoring coverage the deleted
`TestDebugOSDetails` provided) and the doctor rationale comment. Library-level
findings loginjector's own consumption surfaced were routed to a separate plan in
the loginjector repository.
