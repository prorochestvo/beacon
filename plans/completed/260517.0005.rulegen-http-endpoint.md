# Task Breakdown

## Overview

Plans 008–014 built and hardened the rule-generator service in
`internal/application/rulegen/`. Today it is exposed only through the
`cmd/rulegen` CLI, so re-generating a rule (e.g. after `cmd/sourceaudit`
reports MISS) requires SSH to the deploy host. Plan 015 makes the same
capability available over HTTP from the existing `cmd/web` service so future
Telegram-bot admin commands and a possible Mini-App button can trigger
re-generation without shell access. The CLI path stays — the HTTP path is
additive.

The endpoint must be admin-only because each call burns LLM tokens
(real money), may launch chromedp (memory/wall-clock), and overwrites the
persisted `rules` / `rule_metadata` columns on `rate_sources`. The bot's
admin chat id is already available in `cmd/web/main.go` (encoded in
`TELEGRAMBOT_DSN.Login()` and consumed by `telegrambot.NewTBotClient`), and
Telegram WebApp initData HMAC validation already lives in
`internal/tools/tgwebapp/initdata.go` and is wired into
`/api/me/subscriptions`. We reuse that infrastructure rather than introducing
a new bearer-token mechanism.

Approach in one paragraph: add `POST /api/sources/{name}/rules/generate`,
authenticated via the existing initData HMAC path but with an additional
"is admin" check against the bot's admin chat id; run `rulegen.Generator.Generate`
synchronously inside the handler with a 120-second hard ceiling; serialise
concurrent calls for the same source name behind a per-source mutex map and
return 409 on contention; respond with a structured JSON describing the
persisted rule plus extracted value, mapping known error kinds to specific
HTTP status codes via `internal.PublicError`. The plan also closes the
plan-009 loose end where `aiPrimary`/`aiFallback` are constructed in
`cmd/web/main.go` but only logged — they will now feed an actual
`rulegen.Generator` injected into the new handler.

## Assumptions

- The admin is exactly one Telegram user, identified by
  `TELEGRAMBOT_DSN.Login()` (an int64 chat id). Multi-admin support is out of
  scope.
- Re-generation is rare: a handful of sources, regenerated at most a few
  times per week. No need for queueing, deduplication beyond a per-source
  mutex, persisted job rows, or audit log table (regular logs are sufficient).
- The `cmd/web` binary in the deploy environment has the same fetcher set as
  `cmd/rulegen` (plain HTTPFetcher + chromedp), because both binaries are
  built from the same Go module with `CGO_ENABLED=0`. `chromedp` requires
  Chromium on the host (the systemd unit must provide
  `CHROMIUM_PATH` for chromedp sources). If the binary is deployed without
  Chromium installed, `ErrUnsupportedFetcherKind` may still be returned at
  request time for `fetcher_kind="chromedp"` sources — we map it explicitly
  to HTTP 503 rather than 500.
- The endpoint is mutating but idempotent in spirit (each call overwrites
  with whatever the LLM produces this round). Callers must understand they
  may get a different rule each time even for the same source.
- The Mini App's auth-cookie / header semantics are unchanged; admin status
  is derived from the Telegram user id encoded in the validated initData.
- Existing `internal.PublicError` is the only mechanism for user-facing
  error text (per CLAUDE.md's Error Handling section). LLM error strings are
  NEVER returned verbatim; the handler maps them to safe public messages.
- The endpoint is invoked from trusted contexts (admin's Telegram, ops
  curling with the admin's initData). No rate limiting in v1 — the per-source
  mutex naturally caps concurrent generations, and the chromedp / OpenAI
  fallback paths self-rate-limit upstream.
- HTTP server timeouts in `cmd/web/main.go` (`HttpTimeOut = 30s`) are too
  tight for this endpoint. We do NOT raise the global server timeout (that
  would change every other endpoint); instead the handler uses its own
  request-scoped `context.WithTimeout(120s)` and the server-level
  ReadTimeout/WriteTimeout for `cmd/web` must be raised to ≥120s for this
  route to actually deliver its response. See Task 6.

## Tasks

### Task 1: Lock down endpoint shape, auth, and status mapping

- **Description:** Document the locked design decisions in the plan and in
  code comments so the engineer does not re-litigate them.

  **Endpoint.** `POST /api/sources/{name}/rules/generate`. POST because
  it mutates `rate_sources.rules` and triggers external calls; the
  `/api/sources/{name}/...` namespace already exists in
  `internal/gateway/httpV1/routes/routes.go` and is read-only except for
  `PATCH /api/sources/{name}/active`, so adding a sub-resource is the
  least-surprising shape.

  **Auth.** Telegram WebApp initData HMAC. The header is
  `X-Telegram-Init-Data` (with `?initData=` query-string fallback, identical
  to `/api/me/subscriptions`). After `tgwebapp.ValidateInitData` returns a
  user id, the handler compares it to the bot's admin chat id
  (currently stored on `telegrambot.TelegramBotClient.adminChatID`, a private
  field). To avoid threading a new dependency, expose a public accessor
  `func (tbot *TelegramBotClient) AdminChatID() int64` returning
  `int64(tbot.adminChatID)`, and pass that int64 down through
  `gateway.NewGateway` and `httpV1.NewRouter` into `handlers.NewHandler` as
  a new field `adminChatID int64`. Mismatch returns HTTP 403 with the public
  message `"This endpoint is admin-only"`.

  **Request body.** JSON, all fields optional. Wire-format defined in a
  new `pkg/api` struct (Task 4 creates the file):

  ```go
  type RulegenRequest struct {
      ForceFallback       bool `json:"force_fallback,omitempty"`
      MaxPrimaryAttempts  int  `json:"max_primary_attempts,omitempty"`
      MaxFallbackAttempts int  `json:"max_fallback_attempts,omitempty"`
  }
  ```

  Empty body (`Content-Length: 0` or `{}`) is valid and uses defaults
  `max_primary_attempts=3`, `max_fallback_attempts=2`, `force_fallback=false`
  — identical to the CLI defaults in `cmd/rulegen/main.go:84-86`. Values
  outside `[1, 10]` for either attempt-count field are rejected as 400
  (`"max_primary_attempts must be between 1 and 10"`). Per CLAUDE.md's
  no-new-fields-without-need rule, do not add other parameters in v1.

  **Important about attempt-count plumbing.** `rulegen.NewGenerator` takes
  `maxPrimaryAttempts` / `maxFallbackAttempts` at construction time
  (`internal/application/rulegen/generator.go:77`) — the per-call `Generate`
  signature only accepts `forceFallback`. To honour per-request overrides,
  the handler must construct a *new* `Generator` per call when overrides are
  supplied, or — preferred — store a `func(maxPrimary, maxFallback int)
  (*rulegen.Generator, error)` factory closure on the handler and resolve it
  per request. The factory closes over the AI clients, fetcher, executor,
  and source repo built once in `cmd/web/main.go`. Defaults are produced by
  the factory when both fields are zero. This keeps the hot path
  (defaults-only) allocation-light — only override paths allocate a fresh
  Generator.

  **Response body on success (HTTP 200).** Mirror `RateSourceRuleMetadata`
  plus the extracted value and the persisted rules:

  ```go
  type RulegenResponse struct {
      Source       string                  `json:"source"`
      Value        float64                 `json:"value"`
      Rules        []domain.RateSourceRule `json:"rules"`
      AttemptsUsed int                     `json:"attempts_used"`
      Escalated    bool                    `json:"escalated"`
      Provider     string                  `json:"provider"`
      Model        string                  `json:"model"`
      GeneratedAt  string                  `json:"generated_at"` // RFC3339 UTC, mirrors metadata
  }
  ```

  **Failure body.** Existing pattern in `handlers.go` uses
  `http.Error(w, '{"error":"..."}', status)`. Stay consistent. The error
  string is `PublicError.Details()` when the source returned a
  `*internal.PublicError`, otherwise a fixed fallback per status code.

  **HTTP status mapping.** Enumerate explicitly so the engineer has nothing
  left to guess:

  | Condition                                                       | Status | Public message                                  |
  |-----------------------------------------------------------------|--------|-------------------------------------------------|
  | Happy path                                                      | 200    | (body)                                          |
  | Malformed JSON body / unparseable override fields               | 400    | `"invalid request body"`                        |
  | Override outside `[1,10]`                                       | 400    | specific (`"max_primary_attempts must be 1..10"`) |
  | Missing `{name}` path segment                                   | 400    | `"missing source name"`                         |
  | Missing/invalid initData                                        | 401    | `"unauthorized"`                                |
  | Authenticated user is not the admin                             | 403    | `"This endpoint is admin-only"`                 |
  | Source name does not exist (`Generate` returns wrapped not-found) | 404  | `"source not found"`                            |
  | Per-source mutex contention                                     | 409    | `"rule generation already in progress for this source"` |
  | Per-call context deadline (120s) exceeded                       | 504    | `"rule generation timed out"`                   |
  | `errors.Is(err, rulegen.ErrUnsupportedFetcherKind)`             | 503    | `"required fetcher is not available in this build"` |
  | "all attempts exhausted" wrapped error                          | 502    | `"rule generation failed for source <name>"`    |
  | Any other error                                                 | 500    | `"internal error"` (existing fallback)          |

  Distinguishing 404 vs 502 requires teaching the handler about the error
  shapes the Generator produces. `Generate` currently emits
  `fmt.Errorf("rulegen: source %q not found", ...)` for missing sources
  (generator.go:128) — match by exact prefix is fragile. Preferred:
  introduce two sentinel errors in `internal/application/rulegen/generator.go`:

  ```go
  var ErrSourceNotFound       = errors.New("rulegen: source not found")
  var ErrAttemptsExhausted    = errors.New("rulegen: all attempts exhausted")
  ```

  Wrap the existing returns with these sentinels (`fmt.Errorf("...: %w",
  ErrSourceNotFound)`), then the handler uses `errors.Is`. This change is
  internal-only and keeps backward compat with `cmd/rulegen`'s stderr text.
  Plan task 2 covers the wrap.

- **Acceptance Criteria:**
   - [ ] The endpoint, method, auth, body schema, response schema, and
         status-code table above are reproduced verbatim in the doc comment
         above the new handler method.
   - [ ] `RulegenRequest` and `RulegenResponse` types live in
         `pkg/api/sources.go` (alongside the existing
         `SourceActiveRequest`).
   - [ ] No alternative design decisions (bearer tokens, async jobs,
         different status codes) sneak back in during implementation.
- **Pitfalls:**
   - Sentinel introduction must keep existing `cmd/rulegen` stderr output
     byte-identical or break a downstream operator habit (`grep "FAIL
     source="`). Wrapping with `fmt.Errorf("...: %w", ErrSourceNotFound)`
     and keeping the original prefix is fine; do NOT swap to `errors.Join`
     because join inserts a newline.
   - The 503 mapping for `ErrUnsupportedFetcherKind` differs from
     `cmd/rulegen`'s exit code 2 (usage error). That's intentional: from
     HTTP's perspective the build configuration is a server-side capability
     gap, not a client error.
- **Complexity:** Easy

### Task 3: Per-source serialization helper

- **Description:** SQLite is a single-writer engine and `rulegen.Generator`
  calls `RetainRateSource` (a `BEGIN IMMEDIATE` write transaction inside
  `repository.RateSourceRepository.RetainRateSource`). Two concurrent
  generations for the same source name would either both succeed and
  silently overwrite each other (the second writer wins, the first's
  expensive LLM round trips are wasted) or hit `database is locked` on
  contention. For the v1 HTTP path we want a clear, in-process serialiser:
  at most one in-flight generation per source name; concurrent generations
  for *different* sources are allowed.

  Implement a small lock manager in
  `internal/gateway/httpV1/handlers/rulegen_locks.go`:

  ```go
  package handlers

  import "sync"

  // rulegenLockManager serialises rule generation per source name.
  // TryAcquire returns (release, true) on success; (nil, false) when another
  // call holds the lock for the same source. The caller MUST invoke release
  // exactly once when finished. The manager retains the per-source mutex
  // even after release so future contention is detected immediately
  // without re-allocating; this trades a small steady-state memory cost
  // (one mutex per ever-seen source name) for branch-free fast path.
  type rulegenLockManager struct {
      mu    sync.Mutex
      locks map[string]*sync.Mutex
  }

  func newRulegenLockManager() *rulegenLockManager {
      return &rulegenLockManager{locks: make(map[string]*sync.Mutex)}
  }

  func (m *rulegenLockManager) TryAcquire(sourceName string) (func(), bool) {
      m.mu.Lock()
      lk, ok := m.locks[sourceName]
      if !ok {
          lk = &sync.Mutex{}
          m.locks[sourceName] = lk
      }
      m.mu.Unlock()

      if !lk.TryLock() {
          return nil, false
      }
      return lk.Unlock, true
  }
  ```

  Add a `lockMgr *rulegenLockManager` field to `Handler` and initialise
  it in `NewHandler`.

- **Acceptance Criteria:**
   - [ ] `TryAcquire("X")` followed by a second `TryAcquire("X")` (before
         the first `release()` is called) returns `(nil, false)` the second
         time.
   - [ ] After the first `release()` is called, a subsequent
         `TryAcquire("X")` returns `(release, true)`.
   - [ ] `TryAcquire("X")` and `TryAcquire("Y")` are independent and both
         succeed concurrently.
   - [ ] Unit test in `rulegen_locks_test.go` with subtests under
         `TestRulegenLockManager_TryAcquire` covers the three cases above,
         plus a goroutine stress check with `-race`.
   - [ ] Compile-time interface assertion is NOT needed here (no
         interface); the file simply exports `*rulegenLockManager`.
- **Pitfalls:**
   - `sync.Mutex.TryLock` exists since Go 1.18 — it is non-blocking and
     matches what we want. Do not use a `select` with a buffered channel
     pattern; it's strictly more code for the same semantics.
   - The manager grows monotonically with the set of source names. That's
     acceptable: source names are a closed set seeded by migrations
     (currently ~6), not user input.
   - Releasing twice would panic. Make absolutely sure the handler defers
     `release()` exactly once and only after `TryAcquire` returned `true`.
- **Complexity:** Easy

### Task 4: Implement the handler

- **Description:** Add `GenerateRules` to `Handler` in
  `internal/gateway/httpV1/handlers/handlers.go` (or a new file

  Add a `Generator` interface and a constructor parameter:

  ```go
  // rulegenGenerator is the narrow interface the handler needs from the
  // application-layer generator. Defined inside the handlers package so
  // tests can substitute a fake without depending on rulegen's full surface.
  type rulegenGenerator interface {
      Generate(ctx context.Context, sourceName string, forceFallback bool) (*rulegen.Result, error)
  }

  // rulegenGeneratorFactory builds a Generator with caller-supplied attempt
  // budgets. The handler invokes this per-request only when overrides are
  // supplied; the default-budget Generator is cached on the Handler.
  type rulegenGeneratorFactory func(maxPrimary, maxFallback int) (rulegenGenerator, error)
  ```

  Extend `NewHandler` to accept `defaultGenerator rulegenGenerator`,
  `generatorFactory rulegenGeneratorFactory`, and `adminChatID int64`.
  Store them on `Handler`. The lock manager is created internally in
  `NewHandler`. Wire the new parameters in `httpV1.NewRouter` and
  `gateway.NewGateway` (Task 5).

  Handler skeleton (drop into `rulegen_handler.go`):

  ```go
  // GenerateRules triggers the rule-generator audit loop for a named source.
  //
  // POST /api/sources/{name}/rules/generate
  // Auth: X-Telegram-Init-Data header (or ?initData= query fallback).
  // Authorisation: the authenticated Telegram user id must equal the
  // configured admin chat id.
  // See plans/completed/.../015-rulegen-http-endpoint.md for full status
  // mapping and request/response shapes.
  func (h *Handler) GenerateRules(w http.ResponseWriter, r *http.Request) {
      name, err := extractName(r)
      if err != nil {
          http.Error(w, `{"error":"missing source name"}`, http.StatusBadRequest)
          return
      }

      // Auth.
      initData := r.Header.Get("X-Telegram-Init-Data")
      if initData == "" {
          initData = r.URL.Query().Get("initData")
      }
      userID, err := h.validateInitData(initData, h.botToken, meSubscriptionsMaxAge, h.nowFn())
      if err != nil {
          http.Error(w, `{"error":"unauthorized"}`, http.StatusUnauthorized)
          return
      }
      if userID != h.adminChatID {
          http.Error(w, `{"error":"This endpoint is admin-only"}`, http.StatusForbidden)
          return
      }

      // Body.
      var req api.RulegenRequest
      if r.ContentLength > 0 {
          if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
              http.Error(w, `{"error":"invalid request body"}`, http.StatusBadRequest)
              return
          }
      }
      if req.MaxPrimaryAttempts < 0 || req.MaxPrimaryAttempts > 10 {
          http.Error(w, `{"error":"max_primary_attempts must be between 1 and 10"}`, http.StatusBadRequest)
          return
      }
      if req.MaxFallbackAttempts < 0 || req.MaxFallbackAttempts > 10 {
          http.Error(w, `{"error":"max_fallback_attempts must be between 1 and 10"}`, http.StatusBadRequest)
          return
      }

      // Lock.
      release, ok := h.lockMgr.TryAcquire(name)
      if !ok {
          http.Error(w, `{"error":"rule generation already in progress for this source"}`, http.StatusConflict)
          return
      }
      defer release()

      // Pick generator: cached default vs override.
      gen := h.defaultGenerator
      if req.MaxPrimaryAttempts > 0 || req.MaxFallbackAttempts > 0 {
          mp := req.MaxPrimaryAttempts
          if mp == 0 {
              mp = defaultMaxPrimary
          }
          mf := req.MaxFallbackAttempts
          if mf == 0 {
              mf = defaultMaxFallback
          }
          custom, buildErr := h.generatorFactory(mp, mf)
          if buildErr != nil {
              h.internalError(w, buildErr)
              return
          }
          gen = custom
      }

      // 120-second hard ceiling.
      ctx, cancel := context.WithTimeout(r.Context(), rulegenRequestTimeout)
      defer cancel()

      result, err := gen.Generate(ctx, name, req.ForceFallback)
      if err != nil {
          h.writeRulegenError(w, name, err, ctx.Err())
          return
      }

      writeJSON(w, api.RulegenResponse{
          Source:       result.Source.Name,
          Value:        result.Value,
          Rules:        result.Rules,
          AttemptsUsed: result.AttemptsUsed,
          Escalated:    result.Escalated,
          Provider:     result.Metadata.Provider,
          Model:        result.Metadata.Model,
          GeneratedAt:  result.Metadata.GeneratedAt,
      })
  }

  const (
      rulegenRequestTimeout = 120 * time.Second
      defaultMaxPrimary     = 3
      defaultMaxFallback    = 2
  )
  ```

  Error classifier helper (same file):

  ```go
  func (h *Handler) writeRulegenError(w http.ResponseWriter, sourceName string, err error, ctxErr error) {
      // Log the full error with trace for ops.
      log.Print(errors.Join(
          fmt.Errorf("generate rules for %q: %w", sourceName, err),
          internal.NewTraceError(),
      ))

      // Order matters: ctx deadline must be checked before generic 500.
      switch {
      case ctxErr != nil && errors.Is(ctxErr, context.DeadlineExceeded):
          http.Error(w, `{"error":"rule generation timed out"}`, http.StatusGatewayTimeout)
      case errors.Is(err, rulegen.ErrSourceNotFound):
          http.Error(w, `{"error":"source not found"}`, http.StatusNotFound)
      case errors.Is(err, rulegen.ErrUnsupportedFetcherKind):
          http.Error(w, `{"error":"required fetcher is not available in this build"}`, http.StatusServiceUnavailable)
      case errors.Is(err, rulegen.ErrAttemptsExhausted):
          msg := fmt.Sprintf(`{"error":"rule generation failed for source %q"}`, sourceName)
          http.Error(w, msg, http.StatusBadGateway)
      default:
          h.internalError(w, err)
      }
  }
  ```

### Task 6: Raise HTTP server timeouts for the long-running endpoint

- **Description:** The current server in `cmd/web/main.go:203-209` uses
  `ReadTimeout` and `WriteTimeout` both = `HttpTimeOut` (30s default). The
  rulegen endpoint may legitimately take up to 120s wall-clock (chromedp
  fetch + OpenAI fallback round trips). At 30s WriteTimeout the server
  cuts the response before the handler finishes and the client sees a
  truncated/empty body.

  Three options, ranked:

  1. **(preferred) Raise the global server WriteTimeout to 130s** (10s
     headroom over the per-request 120s ceiling). Cost: every other
     endpoint can also block for 130s, which is bad for clients hanging
     onto leaked connections but otherwise harmless because handlers
     finish in milliseconds. This is a 1-line change.

  2. Move the rulegen route onto a separate `http.Server` listening on
     `:8081` with its own timeouts. Cost: new port, new systemd unit
     concern, more config surface. Overkill for v1.

  3. Use `http.Server.Handler = http.TimeoutHandler(mux, 120s, ...)`
     selectively. Cost: doesn't actually help because `TimeoutHandler`
     wraps the response writer and still respects the outer
     `WriteTimeout`.

  Take option 1. Concretely: rename the existing knob from `HttpTimeOut`
  to `HttpReadTimeout` (semantically what it is) and add a separate
  `HttpWriteTimeout = 130s`. The `WriteTimeout` field uses the new value;
  `ReadTimeout` and `IdleTimeout` keep using the existing 30s knob.

- **Acceptance Criteria:**
   - [ ] `cmd/web/main.go` declares `HttpWriteTimeout = 130 * time.Second`
         and the `http.Server` literal uses it for `WriteTimeout`.
   - [ ] The `--timeout` CLI flag's doc string is updated to mention that
         it controls ReadTimeout only (the WriteTimeout is fixed to give
         the rulegen endpoint headroom).
   - [ ] Manual smoke: trigger the rulegen endpoint against a slow source
         (use a fake fetcher that sleeps 60s) and observe the response
         arriving intact.
- **Pitfalls:**
   - Don't conflate "request context deadline" (120s, enforced by the
     handler) with "server write timeout" (130s, enforced by net/http).
     The 10s gap is intentional: the handler must always finish before
     the server cuts the connection, otherwise the operator sees no error
     body and has to dig in logs.
   - The `--timeout` flag is documented in
     `cmd/web/main.go:238`. Keep backward-compat: existing operators may
     set `--timeout=45s` for unrelated reasons. Don't break the flag, just
     narrow its scope.
- **Complexity:** Easy

## Risks

- **SQLite write contention beyond the per-source mutex.** Our mutex
  serialises by source name. If a different writer (e.g. the collector
  inserting `rate_values` rows while the generator updates
  `rate_sources`) happens to grab the SQLite write lock first, the
  generator's `RetainRateSource` may briefly block on the busy timeout.
  Worst case: the operator sees a slow response. The default SQLite
  busy_timeout in `modernc.org/sqlite` is generous; no action needed
  unless we observe `database is locked` in logs.

- **Concurrent generations on different sources stress the OpenAI API.**
  The admin firing off 6 parallel generates (one per source) hits the
  fallback provider's rate limits. Mitigation: per-admin global mutex
  too? Not in v1 — the admin can self-pace. Document.

- **Telegram WebApp initData leaks if the admin shares the URL.** The
  `?initData=` query fallback puts the admin's signed token in the URL,
  which lands in logs, browser history, and any HTTP proxy that touches
  the request. Mitigation: prefer the `X-Telegram-Init-Data` header
  inside the Mini App; query fallback exists only for ad-hoc curl
  testing. Document the risk in `cmd/rulegen/README.md`'s See-also note.

- **The endpoint shadows future "list rules" reads.** Adding
  `POST /api/sources/{name}/rules/generate` reserves `/rules/...` for
  rule-related operations. If a future plan wants `GET
  /api/sources/{name}/rules`, that's fine and orthogonal. No conflict.

- **`ErrUnsupportedFetcherKind` at runtime in production.** If the deploy
  host runs `cmd/web` without Chromium installed, every chromedp-bound
  source returns 503 here. The CLI path (`cmd/rulegen` in SSH) hits the
  same gap with exit code 2. Both paths require the same precondition
  (Chromium available on the host). Track this in deploy docs, not this
  plan.

- **Sentinel-error refactor breaks an unforeseen caller.** `rulegen` is
  imported only by `cmd/rulegen` and (after Task 5) by `cmd/web`. A
  global grep before merging Task 2 catches any third caller; nothing
  found at plan-write time.

## Trade-offs

- **Synchronous vs async job model.** Picked sync. Async adds a job
  table, a polling endpoint, and the question of what happens when the
  server restarts mid-job. We have ~6 sources regenerated rarely; the
  90s worst case is acceptable. If usage scales (e.g. UI button that's
  rapidly clicked), revisit.

- **Per-source mutex vs database advisory lock.** Picked in-process
  mutex. SQLite has no advisory locks. We could insert a sentinel row
  into a `rulegen_in_flight` table, but that adds a migration, a
  cleanup story, and a new failure mode (zombie row). For a single
  `cmd/web` process the in-memory mutex is strictly simpler. If we ever
  run two `cmd/web` instances (we don't today), this needs revisiting.

- **Telegram initData + chat-id check vs separate `ADMIN_API_TOKEN`.**
  Picked initData. Reuses existing `tgwebapp.ValidateInitData` machinery
  (zero new code in `internal/tools/`), keeps secrets management out of
  `.env`, and matches the eventual "admin taps button in Mini App" UX.
  Cost: ad-hoc curl from a laptop is awkward (you need to capture a
  fresh initData payload). The CLI path remains for that.

- **Generator construction strategy: cache default + per-call factory.**
  Picked it over "always construct per call" (wastes allocations on
  default path) and "single shared Generator with per-call overrides
  via Generate parameter" (would require widening the
  `rulegen.Generator.Generate` signature, plumbing through Result
  construction, and breaking `cmd/rulegen`). The factory closure keeps
  the rulegen package untouched.

- **`writeRulegenError` helper vs inline `switch`.** Picked helper.
  Keeps the handler body short and makes the status-mapping table
  trivially unit-testable in isolation if we ever want to.

- **Add sentinel errors in rulegen vs string-matching in the handler.**
  Picked sentinels. String-matching error text is brittle and the
  current text is informative enough to be tempting to format-change,
  which would silently break the handler. The sentinels cost ~6 lines
  of code and one round of test updates.
