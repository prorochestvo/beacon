# Task Breakdown

## Overview

Two related defects are masking real failure modes for BCC rate sources
(`KZ_BCC_FX_BID_*`) in production:

1. **Inverted error check in `RateExtractor.Run`.**
   `internal/tools/rateextractor/rateextractor.go` lines 102–111 currently read:

   ```go
   payload, err := extractor.fetchHtmlPage(ctx, source.URL)
   if err != nil || payload == nil {
       if err != nil {                      // condition is inverted
           err = errors.New("page is nil")  // overwrites the real error
       }
       err = fmt.Errorf("could not read html page %v: %w", source.URL, err)
       ...
   }
   ```

   The intent of the inner branch is "if `payload == nil` AND `err == nil`,
   synthesise a `page is nil` sentinel so the outer wrap has something to print."
   The check is inverted: today it overwrites every real `fetchHtmlPage` error
   (HTTP non-2xx, timeout, transport failure, body read failure) with the literal
   string `page is nil`. Every BCC failure surfaces in `execution_history.error`
   as `could not read html page <url>: page is nil`, hiding the actual cause.

2. **Collector dispatcher ignores `fetcher_kind`.**
   The `domain.RateSource.FetcherKind` column was added in migration `202605.010`
   and BCC rows were updated to `chromedp` in `202605.011`, but
   `internal/application/collection/rateagent.go` constructs a single
   `*rateextractor.RateExtractor` (plain HTTP) at startup via
   `collection.NewRateAgent` and dispatches every source through it
   unconditionally (`a.rateExtractor.Run(ctx, &source)` at `rateagent.go:129`).
   There is no branch on `source.FetcherKind`. The only chromedp usages in the
   tree are `cmd/web/main.go` and `cmd/rulegen/main.go`, both wiring
   `rulegen.ChromedpFetcher` into the rule-generator path — not into the
   collection loop. So BCC, despite being marked `fetcher_kind='chromedp'`,
   silently runs through plain HTTP and fails on JS-hydrated content. Combined
   with bug 1, the failure mode is unreadable in operational logs.

Approach: ship the two fixes as independent commits in dependency order.
Task 1 (error-check fix + tests) is fully self-contained and lands first. Task 2
(fetcher_kind dispatch) is structurally larger and benefits from the improved
error messaging produced by Task 1 to verify the BCC sources actually succeed
after the switch. Scope of Task 2 is intentionally narrow: introduce a small
dispatcher abstraction inside the `collection` package that picks an extractor
implementation based on `source.FetcherKind`; do not redesign the extractor
layer, do not add a browser pool, do not modify rule generation.

## Assumptions

- The `chromedp` fetcher implementation in
  `internal/application/rulegen/chromedpfetcher.go` is correct and already in
  production use behind the `/rules/generate` admin endpoint. Reusing it for
  the collection loop is acceptable; if the package boundary is awkward we
  expose it via a thin adapter rather than copy-pasting.
- The only valid `fetcher_kind` values today are `plain` and `chromedp`
  (enforced by `repository.RateSourceRepository.RetainRateSource`). An empty
  string is normalised to `plain` on write; on read it may still be empty for
  legacy rows. The dispatcher must treat `""` as `plain` for symmetry.
- The deploy host has Chromium available (the rulegen endpoint already runs in
  production), so introducing chromedp into the collector adds no new
  operational dependency.
- `cmd/collector` is the only binary that constructs `collection.RateAgent`
  (verified at `cmd/collector/main.go:151`). No other call site needs to be
  reworked.
- `make test` already passes on `main`; the chromedp subtest gate in
  `chromedpfetcher_test.go` (`findChromiumOrSkip`) is the right pattern to copy
  for any new chromedp-backed test in the collection layer.
- The 30-minute response cache in `RateExtractor` is keyed by URL only and
  returns the cached body unconditionally. If a chromedp render and an HTTP
  GET land on the same URL within the cache window, they would share a body
  today. We are not changing cache semantics in this plan; we will document
  the fact that the chromedp path uses an independent fetch (no cache reuse
  between fetchers) and revisit only if it causes a regression.

## Tasks

### Task 1: Fix inverted err-check in RateExtractor.Run and add regression subtests

- **Description:**
  - In `internal/tools/rateextractor/rateextractor.go` (lines 102–111), invert
    the inner condition so the `page is nil` synthesis fires only when the
    underlying fetch returned no error but also returned no payload. Concretely
    the inner guard becomes `if err == nil { err = errors.New("page is nil") }`.
    Keep the outer `fmt.Errorf("could not read html page %v: %w", ...)` wrap and
    the `errors.Join(err, internal.NewTraceError())` join as they are — they are
    correct, only the synthesis branch is wrong.
  - Add three new `t.Run` subtests **inside** the existing
    `TestRateExtractor_Run` function in
    `internal/tools/rateextractor/rateextractor_test.go` (do **not** create new
    top-level `TestRateExtractor_Run_*` functions — per CLAUDE.md, one `Test*`
    per method, scenarios as subtests). The new subtests cover the failure
    branch that today has zero coverage:
    1. `"fetch error wraps underlying response"` — stand up an `httptest`
       server returning HTTP 503 with a recognisable body marker, run the
       extractor, assert the returned error is non-nil, that `err.Error()`
       contains the source URL, contains `"could not read html page"`, contains
       the substring `"unexpected status 503"` (from the inner
       `fetchHtmlPage` wrap), and crucially does **not** contain
       `"page is nil"`. This is the regression test that would have caught
       the bug.
    2. `"nil payload with nil error synthesises page is nil"` — exercise the
       defensive branch by injecting an `http.RoundTripper` (via
       `NewRateExtractorWithHTTPClient`) that returns `&http.Response{StatusCode:
       200, Body: http.NoBody, ContentLength: 0}, nil`. After the body read the
       returned slice from `io.ReadAll` is a non-nil zero-length slice, which
       would bypass the `payload == nil` guard, so this subtest needs a custom
       transport whose response yields a `nil` slice after read. Implement that
       by returning a body whose `Read` returns `(0, io.EOF)` immediately and
       whose `Close` returns nil; assert the wrapped error contains both `"could
       not read html page"` and `"page is nil"`. (See pitfalls — if the
       `io.ReadAll` path makes `payload == nil` unreachable in practice, document
       that and replace this subtest with one that asserts the synthesis path
       via direct invocation of a small private helper extracted from `Run`;
       extraction is optional and must not be done unless required.)
    3. `"fetch timeout error preserves cause"` — stand up an `httptest` server
       whose handler blocks past the client timeout (e.g. `time.Sleep(2 * timeout)`
       with the extractor built using a 50 ms `http.Client.Timeout`). Assert
       the returned error wraps a timeout (`errors.Is(err, context.DeadlineExceeded)`
       OR `strings.Contains(err.Error(), "context deadline exceeded")`, since
       net/http surfaces deadline failures as `*url.Error` wrapping the deadline
       error — pick the assertion that matches the actual stdlib behaviour
       observed when running the test), and that it does **not** contain
       `"page is nil"`.

- **Acceptance Criteria:**
  - [ ] `internal/tools/rateextractor/rateextractor.go:105` reads

        The `errors.New("page is nil")` and outer `fmt.Errorf` wrap remain.
  - [ ] `TestRateExtractor_Run` in

- **Pitfalls & edge cases:**
  - The third subtest depends on net/http surfacing timeouts as a particular
    error type. Newer Go versions wrap the deadline error inside `*url.Error`;
    `errors.Is(err, context.DeadlineExceeded)` works on Go 1.21+ but the
    extractor also applies a `context.WithTimeout` of its own at
    `fetchHtmlPage:189`. Decide which timeout fires first (the
    `http.Client.Timeout` is the outer bound; the per-request context timeout
    equals the same value, so they race). Pick a server delay clearly longer
    than the client timeout (e.g. 200 ms vs 50 ms client timeout) and assert
    on a substring rather than on type identity if `errors.Is` proves flaky.
  - The 30-minute response cache (`extractor.cache`) is keyed by raw URL. Make
    sure each new subtest uses a freshly-constructed `*RateExtractor`
    (`NewRateExtractorWithHTTPClient` per test) so a previous subtest cannot
    cache a body that the next subtest then reads. The existing happy-path
    subtests already follow this pattern.
  - For the second subtest (nil-payload synthesis), `io.ReadAll` on an empty
    body returns `([]byte{}, nil)` — a non-nil, zero-length slice — which would
    bypass the `payload == nil` guard entirely. The transport injection needs
    to produce a response whose body, when fully read, yields a `nil` slice.
    One viable approach: a custom `http.RoundTripper` that returns a `Response`
    whose `Body` is an `io.ReadCloser` returning `(0, io.EOF)` on first `Read`.
    `io.ReadAll` of that yields `([]byte{}, nil)`, not `nil`. If neither path
    can produce `payload == nil`, document it in the subtest comment and
    instead exercise the synthesis by calling a small extracted helper, or
    drop subtest 2 and capture the synthesis case as a unit test on that
    helper. **Do not silently delete the subtest.** Either the path is
    reachable or the unreachable defence is documented and rationalised.
  - Do not change `fetchHtmlPage` — its error wrapping is fine; only `Run`
    is broken.
  - Watch out for double-wrapping: the test asserts the *final* error string
    from `Run`. `fetchHtmlPage` already wraps its own causes (e.g.
    `"fetch %s: unexpected status %d"`), then `Run` wraps again with
    `"could not read html page %v: %w"`. The final string contains both
    fragments; tests should assert against the inner fragment for
    diagnosability, not the outer.
  - `mockRateValueRepository` in the test file is not used by these failure-path
    subtests (the extractor returns before reaching `RetainRateValue`), but
    pass a fresh instance anyway for consistency with the existing happy-path
    setup.

- **Complexity:** Easy

### Task 2: Honor source.FetcherKind in the collection dispatcher

- **Description:**
  - Investigation already performed (recorded here so the engineer does not
    repeat it): the only sites referencing `chromedp` or `fetcher_kind` in the
    repository are
    - `internal/application/rulegen/chromedpfetcher.go` (the implementation),
    - `cmd/rulegen/main.go` and `cmd/web/main.go` (which wire it into the
      rulegen pipeline),
    - `internal/repository/ratesource.go` (which validates and persists the
      column).
    `internal/application/collection/rateagent.go` constructs exactly one
    `*rateextractor.RateExtractor` at startup
    (`rateextractor.NewRateExtractor(rRateValue, proxyURL, time.Minute, logger)`)
    and calls `a.rateExtractor.Run(ctx, &source)` for every source regardless
    of `source.FetcherKind`. The interface `rateExtractor` at `rateagent.go:145`
    is satisfied only by that one implementation. So bug 2 lives entirely in
    the dispatch step of `RateAgent.execution`, not in seeds, migration data,
    or domain conversion. The plan proceeds.
  - Introduce a minimal dispatcher inside the `collection` package. The
    dispatcher is *not* a new package and *not* a registry pattern; it is a
    private method that picks the right extractor for the source it is about
    to run. Concretely:
    1. Extend `RateAgent` with a second extractor field
       `chromedpExtractor rateExtractor` alongside the existing `rateExtractor`.
       Both satisfy the existing `rateExtractor` interface
       (`Run(context.Context, *domain.RateSource) error`).
    2. In `NewRateAgent`, build both: the existing plain HTTP one as today,
       and a new chromedp-backed one. To avoid a circular package import
       (`collection` already imports `rateextractor`; pulling in `rulegen`
       from the collection layer is a layering smell), wrap
       `rulegen.ChromedpFetcher` behind a tiny adapter type that lives in a
       new file `internal/tools/rateextractor/chromedp.go`. The adapter:
       - constructs a `rulegen.ChromedpFetcher` with sensible defaults (30 s
         timeout, `ChromiumPath` from an env var the collector already reads,
         `WaitSelector` taken from `source.Options.WaitSelector`),
       - calls `Fetch(ctx, source.URL)`,
       - feeds the resulting bytes through the **exact same** rule pipeline
         the plain extractor uses (regex → jsonpath → parse_float → store_as_rate
         → final ParseFloat → range check → `RateValueRepository.RetainRateValue`).
       Factoring out a shared `applyRulesAndStore(ctx, source, payload)`
       helper inside the existing `rateextractor.go` is the cleanest way to
       avoid duplicating the rule loop; the helper is the body of the existing
       `Run` from line 113 onwards.
    3. Replace the `a.rateExtractor.Run(ctx, &source)` call at
       `rateagent.go:129` with a switch on `source.FetcherKind`:
       - `"", "plain"` → existing extractor,
       - `"chromedp"` → new chromedp extractor,
       - default → return a synthetic error
         `fmt.Errorf("source %q: unsupported fetcher_kind %q", source.Name, source.FetcherKind)`
         (do NOT panic — `repository.RetainRateSource` already gates writes,
         but a DB row written before that validation existed could still slip
         through). Wrap with `internal.NewTraceError()` per existing pattern.
    4. Update `cmd/collector/main.go:151` `buildRunners` to thread through any
       new construction parameters (chromium path, etc.) without breaking the
       existing CLI / env contract. If the chromedp fetcher requires
       `CHROMIUM_PATH`, the env-var read should mirror what `cmd/rulegen` and
       `cmd/web` already do for that variable.
  - Add an explicit dispatcher unit test
    `TestRateAgent_execution_dispatchesByFetcherKind` in
    `internal/application/collection/rateagent_test.go` that stubs the two
    extractor interfaces and asserts:
    - a source with `FetcherKind = "plain"` (and `FetcherKind = ""`) is routed
      to the plain stub;
    - a source with `FetcherKind = "chromedp"` is routed to the chromedp stub;
    - a source with `FetcherKind = "bogus"` causes a non-nil entry in the
      returned `errs` map whose `Error()` contains `unsupported fetcher_kind`.
    Use scenario subtests inside one top-level test function per CLAUDE.md.
    If `rateagent_test.go` does not yet exist, create it with the standard
    `var _ rateExtractor = &stubExtractor{}` compile-time assertion.
  - Do NOT add an integration test that spins up real Chromium for this task.
    `findChromiumOrSkip(t)` is the right gate, but the chromedp adapter's
    correctness is already exercised by `chromedpfetcher_test.go`; the new
    dispatcher test stubs `rateExtractor`, so Chromium is not required.

- **Pitfalls & edge cases:**
  - **Layering / import cycle.** `internal/application/collection` already
    imports `internal/tools/rateextractor`. Pulling
    `internal/application/rulegen` from inside `collection` (or inside
    `rateextractor`) would put a `tools` package above an `application`
    package, which is backwards relative to the existing layering rules in
    CLAUDE.md. The chromedp adapter in `internal/tools/rateextractor/chromedp.go`
    should depend on `chromedp` directly OR be defined in a new
    `internal/tools/chromedpfetcher` package, with `rulegen` updated to import
    from there. Pick the smallest move that doesn't break `rulegen` callers.
    If neither is clean, file a follow-up and inline a copy of the chromedp
    fetcher into `tools/` rather than introducing the cycle — but flag this
    as tech debt in the plan completion notes.
  - **Cache key collisions.** The plain extractor caches by URL for 30 min.
    If chromedp eventually shares that cache it would return HTML rendered
    by one fetcher to callers using the other. The plain `RateExtractor.cache`
    field is private; the chromedp adapter must own its own cache (or skip
    caching for chromedp — chromedp fetches are expensive but rare). Do not
    paper over this by reusing the plain cache.
  - **Chromium availability on collector host.** The deploy host already runs
    `cmd/web` which uses chromedp; assume Chromium is installed. If the
    collector binary fails to start because Chromium is missing, surface that
    as a `log.Fatalf` at `cmd/collector/main.go` boot time only if there is
    at least one row with `fetcher_kind = "chromedp"`. Otherwise log a warning
    and continue — the collector should not die because of an unused capability.
    A pragmatic alternative: lazy-construct the chromedp adapter on first use
    and let the `Run` error surface in `execution_history` like any other
    fetch failure. Choose lazy construction unless it complicates testing.
  - **Wait-selector source-specific behaviour.** The chromedp fetcher in
    `rulegen` honours `source.Options.WaitSelector`. BCC seeds (migration
    `202605.012`) already set this. The adapter must thread
    `source.Options.WaitSelector` into `ChromedpFetcherOptions.WaitSelector`
    per-call — do NOT bake a constructor-time selector.
  - **Per-call vs per-process browser lifecycle.** `ChromedpFetcher` spawns
    a fresh Chromium subprocess per `Fetch` call (see the doc comment on
    `ChromedpFetcher`). For an N-source collection cycle that includes
    M chromedp sources, that is M Chromium spawns. Acceptable for the current
    BCC volume (handful of sources); flag a follow-up plan for a browser
    pool if collection cycle latency becomes a problem. Do NOT introduce
    the pool in this plan.
  - **`source.Active` filter already applies upstream.** The dispatcher is
    only reached for active sources, but if migration `202605.013` deactivates
    BCC homepage, the dispatcher will not run for that row — which is fine.
    Don't add a redundant `Active` check.
  - **Plain-path regression.** Adding a dispatcher should be a no-op for
    every non-BCC source. Run `make test` *before* and *after* the dispatcher
    change to confirm.

## Execution Order

1. **Task 1 first.** It is self-contained, independently shippable, and
   immediately improves the diagnosability of every Task-2 verification step.
   Land it as a single commit. After this commit, the BCC failures in
   `execution_history` start carrying the real cause instead of `page is nil`,
   which de-risks Task 2.
2. **Task 2 second.** Depends on Task 1 only insofar as the operator
   verification of Task 2 (checking `execution_history.error` for a chromedp
   source after the dispatcher swap) relies on Task 1's improved error wrap to
   diagnose any regression. The code changes do not overlap, so Task 2 could
   technically be developed in parallel on a separate branch — but ship Task 1
   to production first so any incidental Task 2 regression has accurate logs.

No subtasks parallelisable within a single agent session.

## Risks

- **Risk A — Investigation reveals a different root cause for BCC chromedp
  bypass.** Investigation was already performed (see Task 2 description) and
  the dispatcher is confirmed as the only branch missing fetcher_kind
  awareness; the seeds, migrations, repository read path, and domain
  conversion all preserve `FetcherKind` correctly. Should a fresh symptom
  emerge during implementation that contradicts this — e.g. `FetcherKind`
  arrives empty at the dispatcher despite being non-empty in SQL — Task 1
  still ships standalone. Pause Task 2, re-open the investigation, and
  re-scope. Task 1's value is independent and unrelated to Task 2's outcome.
- **Risk B — Chromium not present on collector host.** Mitigated by lazy
  adapter construction (see Pitfalls). The collector continues to serve
  `plain` sources even if Chromium is absent; chromedp sources surface a
  clear adapter-init error in `execution_history.error`.
- **Risk C — Adapter import cycle.** If extracting chromedp into
  `internal/tools/` proves invasive (because `rulegen` callers are tightly
  coupled to the type's exported API), fall back to defining the adapter
  inside the `rateextractor` package with a direct `github.com/chromedp/chromedp`
  import and document the temporary duplication. Open a follow-up plan to
  consolidate. Do not block this plan on the consolidation.
- **Risk D — Cache reuse between fetchers.** The plain extractor's URL cache
  could theoretically return chromedp-rendered HTML if the cache were shared.
  Mitigated by giving the chromedp adapter its own cache (or no cache).
  Reviewer must verify both extractors have independent caches at code review.
- **Risk E — Increased collection cycle latency.** Each chromedp fetch spawns
  a Chromium subprocess (~1–5 s). At current BCC source volume this is fine.
  If volume grows, latency may exceed the source interval. Mitigation lives
  in a future browser-pool plan; out of scope here. Set an alert threshold in
  the production runbook on collector cycle duration.

## Trade-offs

- **Chose: minimal `switch` dispatcher inside `RateAgent.execution`.**
  Rejected alternative: a registry / strategy pattern with a
  `map[string]rateExtractor`. The map adds a dynamic indirection for two
  entries and pretends the system is open for extension when it is not — we
  intentionally limit `fetcher_kind` to a closed enum validated at the
  repository layer. A `switch` makes the closed set obvious at the call
  site and surfaces a compile-time question if someone later adds a third
  value to the repository whitelist without updating the dispatcher.
- **Chose: adapter inside `internal/tools/rateextractor/`.**
  Rejected alternative: importing `internal/application/rulegen` from
  `internal/application/collection`. That would put an `application` package
  consuming another `application` sibling, blurring the architectural
  boundary documented in CLAUDE.md. Putting the chromedp wrapper next to the
  plain HTTP one in `tools/` keeps both extractors at the same architectural
  level and removes the temptation for future tools to reach into
  `application/`.
- **Chose: per-call Chromium subprocess (reuse current `ChromedpFetcher`
  lifecycle).** Rejected alternative: introducing a browser pool now. The pool
  is the right long-term answer but it is a separate plan with its own
  trade-offs (memory, restart strategy, leak detection). Today's volume
  doesn't justify the complexity. Document this decision so future-us
  understands why the pool was deferred.
- **Chose: bundle both fixes in the same plan, ship as two commits.**
  Rejected alternative: two separate plan files. The two bugs are mechanistic
  consequences of one operational story (BCC chromedp sources fail invisibly).
  A single plan keeps the narrative coherent for review; staged commits keep
  the diff reviewable. The completion criterion in CLAUDE.md is per-plan, not
  per-commit, so a single plan still allows the engineer to ship Task 1
  before Task 2 is finished.
- **Chose: keep `errors.New("page is nil")` literal.** Rejected alternative:
  introduce a sentinel `var ErrPageIsNil = errors.New(...)` exported from the
  package so callers can `errors.Is` against it. No call site exists today
  that wants to branch on this condition; introducing a sentinel for one
  defensive guard adds API surface without a consumer. Reconsider only if a
  future caller actually needs the discrimination.
