# Task Breakdown

## Overview

A cold-start validation against `KZ_JUSAN_ASK_USD_KZT` exposed a semantic gap in the rule-generation prompt: gpt-5.4 produced a regex that scanned the Jusan rate table, recognised both BID and ASK columns, embedded the ASK column's CSS module class (`thirdColumn__fJaMI`) as a trailing anchor, but still placed its capture group on the BID column (`secondColumn__bin18`, value `458.8`) when ASK (`478.80`) was requested. The plausibility band for USD/KZT is `[100, 1000]` — wide by design — so the audit loop accepted the wrong number.

The mistake is linguistic, not structural. The user message labels each PAIR with `(BID)` or `(ASK)`, but the system prompt never tells the model what those tokens mean, that BID/ASK map to "Buy"/"Sell" or to localised labels like Russian `Покупка`/`Продажа` and Kazakh `Сатып алу`/`Сату`, or that the ASK number is normally strictly greater than the BID number on the same row. Without that mapping the model defaulted to the first numeric column.

The fix is a tight semantics block added to `systemPrompt` in `internal/application/rulegen/prompt.go`. No code paths change; no plausibility table changes; no new tests beyond one guard subtest that prevents a future refactor from silently deleting the block.

## Assumptions

- The prompt change alone is sufficient to flip the cold-start outcome on Jusan ASK. If the next live run still extracts a BID-column value, the engineer escalates per the Risks section rather than expanding plan scope on the fly.
- `cmd/rulegen` is the canonical cold-start smoke-test entry point and a developer-local re-run against `KZ_JUSAN_ASK_USD_KZT` is acceptable evidence; we do not block on a production retry of `cmd/sourceaudit`.
- The plausibility table in `internal/application/rulegen/plausibility.go` stays keyed on `(Base, Quote)` only. Tightening it to `(Base, Quote, Kind)` is intentionally deferred (see Trade-offs).
- The `systemPrompt` 4 KB ceiling enforced by `TestSystemPrompt` is honored. The new block is ~14 lines of narrow text and stays well inside that budget.
- The prompt remains English-language prose; multilingual label tokens (`Покупка`, `Сату`, etc.) appear only as quoted strings the model should match against page content, not as prose. This satisfies the "English-only persisted docs" rule because the tokens are data, not documentation.
- BCC, Halyk, Forte, Kaspi, and other Kazakh banks use either English-only or Russian/Kazakh labels (sometimes mixed); the prompt addresses both families with a single, compact label list.

## Tasks

### Task 1: Add a BID/ASK semantics block to `systemPrompt`

- **Description:**
  Edit `internal/application/rulegen/prompt.go` and insert a new `BID/ASK SEMANTICS` section into the `systemPrompt` constant. Place it immediately **before** `METHOD SELECTION` and **after** the opening `CONTRACT` block — the model must internalise what (BID)/(ASK) mean before it starts deciding which method and which anchor to use. Putting it after `RE2 CONSTRAINTS` or `POST-PROCESSING` is too late: by then the model has already picked its anchor.

  Use declarative bullet rules, not fenced WRONG/RIGHT examples — declarative rules generalise to JSON bodies and non-Jusan banks, whereas examples risk overfitting the model to one site's DOM. The Kazakh and Russian label tokens are listed inline as quoted strings so the model can match them against page text without needing a separate few-shot example.

  Exact text to insert (final wording, not a sketch):

  ```
  BID/ASK SEMANTICS
  - Each PAIR in the user message ends with (BID) or (ASK). This is mandatory routing — match the side of the spread that the suffix names.
  - BID = the rate the bank PAYS the customer to BUY the base currency. Labels in the wild that map to BID: "Buy", "Buying", "Bid", "We buy", "Покупка", "Покупаем", "Покупка валюты", "Сатып алу", "Сатып аламыз".
  - ASK = the rate the bank CHARGES the customer to SELL the base currency. Labels that map to ASK: "Sell", "Selling", "Ask", "We sell", "Продажа", "Продаём", "Продажа валюты", "Сату", "Сатамыз".
  - On a typical bank rates row both columns are present; the ASK number is strictly greater than the BID number (the spread). When the page shows two adjacent numeric columns for one currency and labels are ambiguous, the LARGER number is ASK and the SMALLER is BID.
  - If the page exposes only ONE numeric column for the pair (e.g. a central-bank official rate), use that value regardless of the requested kind.
  - Do NOT default to the first numeric column or to the column nearest the currency code. Always anchor on the column whose header (or the row's label) maps to the requested kind.
  ```

  Reference files:
  - `internal/application/rulegen/prompt.go` — the `systemPrompt` const (lines 5-32 today).
  - `internal/application/rulegen/generator.go` — read-only confirmation that the user message already emits `PAIR: USD/KZT (ASK)` so the `(BID)/(ASK)` suffix the new block references is real.

- **Acceptance Criteria:**
   - [ ] The string `BID/ASK SEMANTICS` appears exactly once in `systemPrompt`.
   - [ ] The new section is positioned between `CONTRACT` and `METHOD SELECTION` in source order.
   - [ ] `len(systemPrompt) < 4*1024` still holds (verified by the existing `is under 4 KB` subtest passing unchanged).
   - [ ] All existing `TestSystemPrompt` subtests pass without modification.
   - [ ] `make test` is green.
   - [ ] `make lint` and `make format` produce no diff.

- **Pitfalls & edge cases:**
   - Do NOT add fenced code examples — they bloat the prompt and risk overfitting on Jusan's DOM. Keep the block declarative.
   - Do NOT escape the Kazakh/Russian Unicode characters as `\uXXXX` — they go in as literal UTF-8 (mirrors the existing RE2 guidance about literal Unicode in patterns; the same applies to the prompt source for readability and for the model's exact-string matching).
   - Do NOT introduce section-divider comments (`// --- ... ---`) in `prompt.go`; the project forbids them.
   - Resist re-ordering or rewording surrounding sections. The only structural change in this task is one insertion.
   - The raw-string literal uses backticks — verify no stray backtick sneaks into the new lines.
   - Don't append a trailing blank line inside the new block that would push the total over 4 KB; current prompt sits around 1.7 KB so there is headroom, but pre-check with a manual `len` if uncertain.

- **Complexity:** Easy

### Task 2: Lock in the no-change decision for the plausibility table and add a guard subtest

- **Description:**
  Two pieces of work, both small:

  1. Record in this plan's Trade-offs section the explicit decision NOT to extend the plausibility table to `(Base, Quote, Kind) -> range`. The rationale is captured below; the engineer copies the rationale into a one-paragraph comment **only if** it adds value at the `plausibility.go` site. Otherwise the trade-off lives in the plan and is referenced from the commit message. No code change to `plausibility.go` is required.

  2. Add a single new subtest to `internal/application/rulegen/prompt_test.go` under the existing `TestSystemPrompt` function. The subtest is the guard that future prompt edits cannot silently delete the semantics block. It asserts the presence of three load-bearing markers chosen to span all three languages and the spread rule:

     - `BID/ASK SEMANTICS` — section header
     - `Сату` — Kazakh ASK label (proves Unicode is preserved)
     - `the LARGER number is ASK` — the spread heuristic that lets the model disambiguate unlabeled columns

     Three assertions in one subtest, not three subtests — they together prove "the block exists and is intact", which is one logical scenario per the `one Test* per method, scenarios as t.Run` rule. Keep it as a single `t.Run("contains BID/ASK semantics block", ...)`.

  Reference files:
  - `internal/application/rulegen/prompt_test.go` — the `TestSystemPrompt` function (existing subtests at lines 12-46).
  - `internal/application/rulegen/plausibility.go` — read-only; no edits.

- **Acceptance Criteria:**
   - [ ] A new subtest `contains BID/ASK semantics block` exists under `TestSystemPrompt` and asserts the three markers listed above.
   - [ ] No file other than `prompt.go` (Task 1) and `prompt_test.go` (Task 2) is modified by this task.
   - [ ] `CGO_ENABLED=0 go test -race -run 'TestSystemPrompt' ./internal/application/rulegen/` passes.
   - [ ] The new subtest calls `t.Parallel()` to match the surrounding style.
   - [ ] No assertion uses `_ =` to drop a return value (project rule).

- **Pitfalls & edge cases:**
   - Three assertions inside one `t.Run` is intentional — splitting them into three subtests would violate the "scenarios are independent, not facets of one scenario" intent of the testing rule.
   - The Cyrillic `С` (U+0421) is visually identical to Latin `C` (U+0043). When copy-pasting `Сату`, verify the bytes match the prompt's Cyrillic. A failing assertion with two seemingly-equal strings is the smoking gun for this; if it happens, type the Cyrillic via the IDE's character palette, not from a screen capture.
   - Do not assert on the entire block verbatim — long literal strings make the test brittle against minor wording polish. Three load-bearing fragments are the right granularity.

- **Complexity:** Easy

- **Description:**
  After Tasks 1 and 2 land and `make test` is green, re-run the cold-start flow against the Jusan ASK source from the developer's local checkout. The procedure:

  1. Confirm `KZ_JUSAN_ASK_USD_KZT` exists in the local test database with its rule fields cleared (no `rule_method`, no `rule_pattern`) so `cmd/rulegen` is forced through the full LLM path rather than reusing a cached rule.
  2. Run `cmd/rulegen` against that source. Record the returned regex (or json path) and the extracted numeric value.
  3. Open `https://jusan.kz/exchange-rates` in a browser at roughly the same time and read the visible ASK rate for USD/KZT (the "Сату" column).
  4. Confirm the extracted value falls within ±5 KZT of the visible ASK value AND is strictly greater than the BID value (`Сатып алу` column) on the same page.

  If both checks pass: record the regex, the extracted value, and the visible-on-page values in the plan's completion notes; the reviewer moves the plan to `plans/completed/` per the lifecycle rule.

  If either check fails: do NOT silently widen scope. Add a paragraph to the plan's Risks section enumerating the observed failure mode (regex still anchors on the BID column / regex anchors on something unrelated / value parses but outside the spread / etc.) and hand back to the architect for a follow-up plan.

- **Acceptance Criteria:**

   - [ ] The extracted value is strictly greater than the value `cmd/rulegen` extracts for `KZ_JUSAN_BID_USD_KZT` re-run under the same conditions.

   - [ ] No production code is modified during this validation task — it is observation-only.

   - Jusan rates fluctuate intraday; record the wall-clock time alongside the on-page reference values so a later auditor can sanity-check.
   - The audit loop in `cmd/rulegen` may take multiple LLM attempts. The acceptance criterion is the final accepted rule, not the first attempt — but if the first attempt already succeeds, that is the strongest signal the prompt change worked.
   - If the BID re-run regresses (was `458.8`, now something else), the prompt change has overcorrected and the architect needs to revisit wording. Flag this as a P0 finding in the Risks section.
   - Do not run this validation against the production database; use the local test DB. The pre-existing user feedback about preferring offline ops applies — scp the SQLite file rather than poking the prod service.

## Execution Order

Strictly sequential — each task depends on the prior task's output:

1. **Task 1** (prompt edit) lands first. The prompt change has no compile-time dependency on the test, but Task 2's assertions reference the exact strings introduced here, so writing the test before the prompt invites churn.
2. **Task 2** (guard subtest) follows immediately. It is the regression net that prevents silent deletion of the semantics block in any future PR; running `make test` at the end of this task is the gate before live validation.
3. **Task 3** (cold-start re-run) happens last and only after `make test` passes. A failed live run feeds back into a follow-up plan, not into a Task 1 rewrite within this plan.

No parallelism is available or worthwhile — Tasks 1 and 2 each take under 15 minutes, and Task 3 needs the binary built from Tasks 1+2.

## Risks

- **Prompt change is insufficient.** The model may continue to anchor on the BID column even with the new semantics block. If this happens on Task 3's first live run, plausible next steps (in order of escalation):
  1. Add one explicit example to the block (`When PAIR ends with (ASK) and the page has columns labeled "Сатып алу" then "Сату", the capture group must be inside the "Сату" column.`). Trade-off: bloats prompt, risks Jusan overfit.
  2. Move the BID/ASK semantics block AFTER `RE2 CONSTRAINTS` and `POST-PROCESSING` so it is the last thing the model reads before producing its rule (recency bias).
  3. Lower the model temperature for the rule-generation call (out of scope here; would be its own plan).
  4. Add a sanity heuristic in `internal/application/rulegen/executor.go` that, when extracting an `(ASK)` value, refuses to accept a value smaller than the corresponding `(BID)` source's latest value when both exist. This is plan 017+ territory.
- **Plausibility band still admits wrong-column values.** The USD/KZT `[100, 1000]` band passes both BID `458.8` and ASK `478.8` equally — the audit loop is not a backstop for this class of error. The prompt is the only line of defence in this plan; if the prompt fails, the wrong number reaches production. Mitigation lives outside this plan (see Trade-offs item 1).
- **Multilingual label drift.** Banks rebrand and rewrite their UI text. A future Jusan redesign that replaces `Сату` with `Сатамыз` or with English-only `Sell` is still covered by the label list, but a redesign that drops column labels entirely and relies on row order leaves only the "larger number is ASK" heuristic. That heuristic is in the block precisely for this case, but it's untested against real-world headerless layouts in this plan.
- **Cold-start non-determinism.** The OpenAI fallback path is sampling-based. A single cold-start success is weak evidence; a repeat failure is strong evidence. If Task 3 passes on the first attempt, the reviewer should still encourage one additional cold-start re-run from a clean cache before moving the plan to `completed/`.

## Trade-offs

- **Plausibility table stays `(Base, Quote) -> range`, NOT `(Base, Quote, Kind) -> range`.**
  - Considered: split each pair into a tighter BID band (low end) and ASK band (high end) so the plausibility guard rejects a wrong-column pick by value alone.
  - Rejected because:
    1. Real-world bank spreads for USD/KZT range from ~5 KZT (Halyk on a calm day) to ~15 KZT (Forte under volatility). Any range tight enough to reject a wrong-column pick would also reject legitimate values during currency stress events — exactly when monitoring matters most.
    2. The error this plan addresses is semantic (which column?), not numeric (is the number reasonable?). The right layer for semantic disambiguation is the prompt, not a downstream guardrail.
    3. Introducing a Kind axis doubles the table size and forces every future pair addition to specify two bands instead of one. The cost compounds; the benefit is conditional on the prompt failing.
  - Future revisit: if cross-source ASK > BID validation lands (plan 017+), it subsumes most of the value a Kind-aware plausibility table would provide, without the per-pair tuning cost.

- **Declarative rules, no fenced examples in the prompt.**
  - Considered: a `WRONG: ... RIGHT: ...` example pair anchored on Jusan-style markup.
  - Rejected because:
    1. Fenced examples risk model overfit — gpt-5.4 has been observed to mimic example anchors (CSS class names, exact whitespace) even when the target page uses different markup.
    2. The current cold-start failure was at the semantic level (BID vs ASK meaning), not the markup level (how to construct a regex around a CSS class). Examples address the latter, the bullet rules address the former.
    3. Token budget — three or four bullets cost ~150 tokens; an annotated example pair runs 400+.

- **Section placement: before `METHOD SELECTION`, not after `POST-PROCESSING`.**
  - Considered: appending the block at the end so it is the freshest context when the model emits the JSON.
  - Rejected because the model picks the method and anchor by the time it reaches the tail of the prompt — putting BID/ASK semantics late means the model has already chosen its column. Place semantic constraints early, syntactic constraints (RE2 rules) late.

- **One guard subtest, three assertions, not three subtests.**
  - Considered: three separate subtests (`section header present`, `Kazakh label present`, `spread rule present`).
  - Rejected because the three assertions test one scenario ("the BID/ASK semantics block survives in the prompt"), not three independent scenarios. The project's testing rule is one `Test*` per method with `t.Run` for scenarios — splitting one scenario across three subtests inflates the test surface for no signal gain.
