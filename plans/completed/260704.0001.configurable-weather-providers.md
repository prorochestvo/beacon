# Task Breakdown

## Overview

Weather provider behaviour is currently split between two hardcoded spots in Go:

1. **`internal/infrastructure/weather/gismeteo.go`** — the `gismeteoCities`
   package-global `map[string]gismeteoCity` (4 rows: Almaty `1526384` → `{almaty, 5205}`,
   Astana `1526273` → `{astana, 5164}`, Shymkent `1518980` → `{shymkent, 5324}`,
   Moscow `524901` → `{moscow, 4368}`), plus the base URL, User-Agent, and throttle
   interval. Adding a gismeteo city today means: edit the Go map, add a case to
   `TestGismeteo_curatedMap`, rebuild, redeploy. This is the real operator pain.
2. **`cmd/collector/main.go`** — provider on/off is expressed only in code
   (`collection.WithGismeteo(...)` is always wired; disabling gismeteo means editing
   `wireWeather` and rebuilding).

The FX side already solved this: `rate_sources` rows are declarative, seeded by
migration, and toggled at runtime via the `active` column. This plan brings the same
data-driven model to weather, scoped honestly:

- The **coverage map is the primary deliverable** — a new `weather_gismeteo_cities`
  table replaces the Go map so an operator adds a city with an `INSERT`, no rebuild.
- A secondary `weather_sources` table externalises per-provider **deploy config**
  (active flag, throttle interval, base URL, User-Agent) for both providers.

Deliberately **out of scope** (see Trade-offs): a generic "add an arbitrary new weather
provider entirely via SQL" engine. Unlike `rate_sources` (generic regex extraction),
each weather provider has bespoke Go scraping/parsing (gismeteo's Russian-tooltip → WMO
regex logic). Config externalises *which cities* and *deploy knobs*, not *parsing logic*.
Open-Meteo is global, keyless, and coordinate-driven — it needs **no** per-city config;
we only give it a row so its `active` flag is togglable.

## Assumptions

- The plan-file index number `015` is used per the explicit deliverable path, even
  though `015`/`016` already appear as embedded `NNN` prefixes inside `plans/completed/`.
  The **migration** counter is independent and continues at **016** (highest existing
  migration is `202606.015`; there is no `010` — it was squashed).
- Current date is 2026-07, so new migration files carry the `202607` `YYYYMM` prefix.
- The collector process is short-lived / one-shot per cron tick (per `WeatherAgent.Run`
  godoc and CLAUDE.md "scrapes … on every invocation"). Loading the coverage map once at
  process startup therefore costs one query per tick — cheap and acceptable. No
  long-lived cache-invalidation concern.
- The domain provider token constants `domain.ProviderOpenMeteo` (`"open-meteo"`) and
  `domain.ProviderGismeteo` (`"gismeteo"`) remain the canonical enum used by observation
  storage and throttle queries (`ObtainLatestObservation(ctx, loc, provider)`). They are
  **not** removed; the new tables reference these exact tokens.
- `gismeteoConditionRules` (Russian keyword → WMO code) stays in Go — it is parsing
  logic coupled to the DOM regexes, not deployment config (see Task 6 / design Q4).
- No new `go.mod` dependency. Pure-Go `modernc.org/sqlite`. `-race` needs `CGO_ENABLED=1`.
- Data tokens preserved byte-for-byte on migration: the four city slugs/ids, the
  provider tokens, the gismeteo base URL, and the browser User-Agent string. English-only
  applies to prose, never to these.
- Six implementation decisions surfaced during an engineer dry-run (2026-07-04) are
  inlined in Tasks 6, 7, and 8 and marked **Decision (Dn)**. They pin otherwise-ambiguous
  choices — the `NewGismeteo` parameter type (D1), the test-helper signatures (D2), the
  `/health/check` behaviour when a provider is disabled (D3), the `cmd/web` boot ordering
  (D4), the User-Agent fallback read site (D5), and the `buildRunners` injection/fatal
  semantics (D6) — so implementation never has to stop and guess.

## Tasks

### Task 1: Migration 016 — create and seed `weather_sources`

- **Description:** Add `migrations/202607.016.weather_sources.table_initiate.sql`. Create
  a provider-config table mirroring the spirit of `rate_sources` but minimal, and seed
  the two existing providers. This externalises the active flag, throttle interval, base
  URL, and User-Agent that currently live as Go constants in `gismeteo.go` /
  `collection/weatheragent.go`.

  Schema:
  ```sql
  CREATE TABLE IF NOT EXISTS weather_sources (
      provider          TEXT NOT NULL PRIMARY KEY,
      title             TEXT NOT NULL DEFAULT '',
      active            INTEGER NOT NULL DEFAULT 1,
      base_url          TEXT NOT NULL DEFAULT '',
      throttle_interval TEXT NOT NULL DEFAULT '',
      options           TEXT NOT NULL DEFAULT '{}'
  );
  ```
  Seed (data tokens verbatim — the UA and base URL are copied byte-for-byte from
  `gismeteoUserAgent` / `gismeteoBaseURL` in `gismeteo.go`):
  ```sql
  INSERT OR IGNORE INTO weather_sources
    (provider, title, active, base_url, throttle_interval, options)
  VALUES
    ('open-meteo','Open-Meteo',1,'','','{}'),
    ('gismeteo','Gismeteo',1,'https://www.gismeteo.kz','3h',
     '{"user_agent":"Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/124.0.0.0 Safari/537.36"}');
  ```
  `throttle_interval` is a Go duration string (`"3h"` = current
  `DefaultGismeteoThrottleInterval`); empty string means "use the provider's compiled-in
  default". `open-meteo.base_url` is intentionally empty — its client has hardcoded
  geocoding/forecast endpoints and ignores this column (documented in Task 4 godoc).

- **Acceptance Criteria:**
   - [ ] File named exactly `202607.016.weather_sources.table_initiate.sql`.
   - [ ] `make migrate` on a fresh DB creates `weather_sources` with exactly 2 rows.
   - [ ] `SELECT options FROM weather_sources WHERE provider='gismeteo'` returns the UA
         string identical byte-for-byte to `gismeteoUserAgent`.
   - [ ] `SELECT throttle_interval FROM weather_sources WHERE provider='gismeteo'` = `'3h'`.
   - [ ] Migration is idempotent: re-running `make migrate` is a no-op (filename tracked
         in `__schema_migrations`, `INSERT OR IGNORE`).
- **Pitfalls & edge cases:** Filename immutability — once this ships to any environment
  it can never be renamed. Do not translate the UA or base URL. `active` is INTEGER
  (SQLite has no bool) to match the `rate_sources.active` convention. Do not add FKs from
  `weather_observations`/`weather_user_cities` to this table — providers are an enum, not
  a lifecycle-cascading parent, and a stray provider row deletion must not cascade-wipe
  observations.
- **Complexity:** Easy

### Task 2: Migration 017 — create and seed `weather_gismeteo_cities` (the coverage map)

- **Description:** Add `migrations/202607.017.weather_gismeteo_cities.table_initiate.sql`.
  This table is the **single source of truth** for gismeteo coverage, replacing the Go
  `gismeteoCities` map. Keyed by the Open-Meteo geocoding id (== `location_id` on every
  `weather_user_cities` row, == `LocationKey(geo)`), it stores the two components needed
  to build the gismeteo URL (`<base>/weather-<slug>-<id>/`), plus a human label so an
  operator editing by hand can see which city a row is.

  Schema:
  ```sql
  CREATE TABLE IF NOT EXISTS weather_gismeteo_cities (
      location_id  TEXT NOT NULL PRIMARY KEY,
      slug         TEXT NOT NULL,
      gismeteo_id  INTEGER NOT NULL,
      label        TEXT NOT NULL DEFAULT ''
  );
  ```
  Seed (slugs and ids verbatim from `gismeteoCities`):
  ```sql
  INSERT OR IGNORE INTO weather_gismeteo_cities (location_id, slug, gismeteo_id, label) VALUES
    ('1526384','almaty',5205,'Almaty, Kazakhstan'),
    ('1526273','astana',5164,'Astana, Kazakhstan'),
    ('1518980','shymkent',5324,'Shymkent, Kazakhstan'),
    ('524901','moscow',4368,'Moscow, Russia');
  ```

- **Acceptance Criteria:**
   - [ ] File named exactly `202607.017.weather_gismeteo_cities.table_initiate.sql`.
   - [ ] Fresh-DB migrate produces exactly 4 rows with the `location_id → (slug,
         gismeteo_id)` pairs identical to the current `gismeteoCities` map.
   - [ ] `location_id` is the PRIMARY KEY (one coverage row per city, not per user).
   - [ ] Idempotent re-run.
- **Pitfalls & edge cases:** The four `location_id`/`slug`/`gismeteo_id` values are data
  tokens — a single typo silently breaks the URL and the city goes dark. Cross-check each
  triple against the `gismeteoCities` literal AND against the `cases` slice in
  `TestGismeteo_curatedMap` before committing. `label` is prose (English) and is
  operator-facing only — it must never be used to build the URL.
- **Complexity:** Easy

### Task 3: Domain — add `WeatherSource` and `WeatherGismeteoCity` value objects

- **Description:** In `internal/domain/`, add two struct value objects (no logic, per the
  domain-layer contract). New file `internal/domain/weathersource.go`:
  ```go
  // WeatherSource is the declarative per-provider configuration row from the
  // weather_sources table. Provider is one of ProviderOpenMeteo | ProviderGismeteo
  // (literal data tokens, never translated). ThrottleInterval is a Go duration
  // string (e.g. "3h"); empty means "use the provider's compiled-in default".
  type WeatherSource struct {
      Provider         string
      Title            string
      Active           bool
      BaseURL          string
      ThrottleInterval string
      Options          WeatherSourceOptions
  }

  // WeatherSourceOptions is the parsed options JSON column. UserAgent overrides the
  // request User-Agent header when non-empty.
  type WeatherSourceOptions struct {
      UserAgent string            `json:"user_agent,omitempty"`
      Headers   map[string]string `json:"headers,omitempty"`
  }

  // WeatherGismeteoCity is one gismeteo coverage row from weather_gismeteo_cities.
  // LocationID is the Open-Meteo geocoding id (== the weather_user_cities.location_id).
  type WeatherGismeteoCity struct {
      LocationID string
      Slug       string
      GismeteoID int
      Label      string
  }
  ```
  Keep the existing `ProviderOpenMeteo` / `ProviderGismeteo` constants in
  `weatherobservation.go` untouched.

- **Acceptance Criteria:**
   - [ ] Both structs compile; every exported identifier has a godoc comment starting
         with its name and ending with a period.
   - [ ] `Options` round-trips through `encoding/json` (a `WeatherSourceOptions` test or
         coverage via the repo test in Task 4).
   - [ ] No behavioural logic in the domain package (pure value objects).
- **Pitfalls & edge cases:** Do not add methods that reach into infrastructure. Keep
  `Active` as Go `bool`; the INTEGER→bool conversion happens in the repository scan.
- **Complexity:** Easy

### Task 7: Collector wiring — load config at startup, honour the `active` flag

- **Description:** In `cmd/collector/main.go` `wireWeather` (currently builds
  `NewOpenMeteo` + `NewGismeteo` unconditionally and always calls `WithGismeteo`):
  - Construct a `WeatherSourceRepository` and `WeatherGismeteoCityRepository`.
  - Load `weather_sources` and the gismeteo coverage map once at startup.
  - Open-Meteo: only wire the agent's primary provider when the `open-meteo` row is
    `active` (or absent → default active, to stay backward-safe). If open-meteo is
    inactive there is no weather collection at all — log it clearly.
  - Gismeteo: only call `collection.WithGismeteo(...)` when the `gismeteo` row is
    `active` **and** the coverage map is non-empty; build `NewGismeteo` with the injected
    coverage map, base URL, and UA from the row. Parse the row's `throttle_interval`
    (via `time.ParseDuration`) and pass it to `WithGismeteo`; on empty/parse-error fall
    back to `collection.DefaultGismeteoThrottleInterval` with a log line.
  - Keep `DefaultGismeteoThrottleInterval` / `DefaultWeatherThrottleInterval` as the
    fallback constants (do not delete them).
  - **Decision (D6):** the two new repositories are constructed in `main()` and passed into
    the weather wiring by interface, exactly like the existing repositories — do **not**
    construct them inside `buildRunners`/`wireWeather`; extend the wiring signature to take
    them. A repository *construction* failure is fatal (`log.Fatalf`, matching the existing
    repo wiring in `main()`). A *data* condition — `weather_sources` row absent, `active=0`,
    or an empty coverage map — is never fatal; it degrades per the rules above with a log
    line.

- **Acceptance Criteria:**
   - [ ] With the seed intact, collector behaviour is **identical to today**: open-meteo
         for all cities, gismeteo for the same 4 cities, 3h gismeteo throttle.
   - [ ] Setting `UPDATE weather_sources SET active=0 WHERE provider='gismeteo'` and
         restarting the collector disables the gismeteo phase with no code change and no
         crash (the `WithGismeteo` option is simply not applied).
   - [ ] An empty `weather_gismeteo_cities` table disables gismeteo gracefully (no panic,
         a single informative log line).
   - [ ] A malformed `throttle_interval` logs a warning and falls back to the default;
         it never aborts collector startup.
   - [ ] Coverage/config is loaded once per invocation, not once per location.
- **Pitfalls & edge cases:** The collector calls `RequireMigratedSchema` after opening the
  DB — the new tables exist by the time `wireWeather` runs because the deploy runs the
  migrator first (see Task 9 rollout note). Still, treat "row absent" as "default active"
  so a half-migrated or hand-edited DB degrades to current behaviour rather than going
  dark silently. Do not read config on every `Run` tick; the process is short-lived, one
  load at wiring time is correct.
- **Complexity:** Medium

- **Description:** `cmd/web/main.go` registers `inspector.NewGismeteoInspector()` as an
  advisory inspector with a hardcoded `gismeteoProbeURL = "https://www.gismeteo.kz/"`.
  Align it with the configurable base URL and active flag:
  - Load the `gismeteo` `weather_sources` row at boot (the `WeatherSourceRepository` is
    already constructed for the collector; `cmd/web` will construct its own).
  - **Decision (D4):** `cmd/web/main.go` currently builds `healthAgent` (~line 141) before
    the repositories (~line 154). Move the `WeatherSourceRepository` construction *above* the
    `healthAgent` block so the `gismeteo.active` flag and `base_url` are known when deciding
    whether to register the inspector. This is a local reordering — the repo only needs the
    already-open `db`; the rest of the "dependencies → repositories" order is unchanged.
  - When the gismeteo row is **inactive**, do not register the `GismeteoInspector` at all
    (a disabled provider should not appear as a health component).
  - When active, pass the row's `base_url` to the inspector via a new
    `NewGismeteoInspectorWithURL(baseURL string)` constructor (or reuse the existing
    `newGismeteoInspectorForTest` shape, promoted to exported). Empty `base_url` → keep

  - The inspector stays **advisory** — a gismeteo outage must never flip `/health/check`

- **Acceptance Criteria:**
   - [ ] `/health/check` still returns the `gismeteo` component and it never contributes
         to the aggregate 503 (existing advisory behaviour preserved — regression-checked
         by the existing inspector-agent tests).
   - [ ] With `gismeteo.active=0`, the `gismeteo` key is **absent** from the `services` map
         (**Decision D3**: an operator-disabled provider is not registered as a health
         component; the boot log records which weather providers are active so an absent key

   - [ ] The probe URL used matches the configured `base_url` when set.
   - [ ] `open-meteo` inspector behaviour is unchanged (Open-Meteo has no meaningful

- **Pitfalls & edge cases:** `cmd/web` does **not** parse `BEACON_PROXY_URL` (documented
  divergence in the `GismeteoInspector` godoc) — this task does not change that; the probe
  stays direct. Do not let a DB read failure at boot crash `cmd/web` — if the weather
  source row cannot be loaded, fall back to registering the inspector with the default URL
  (advisory, so worst case is a false "down" that cannot fail the gate). Keep the 3s
  bounded-timeout aggregation intact.
- **Complexity:** Medium

## Risks

- **Silent data-token typo in a seed** (Task 1/2): a wrong slug/id/UA compiles and
  migrates cleanly but sends a city dark or trips gismeteo's bot-fence. Mitigation: the
  Task 5 repository test asserts the four triples equal the (about-to-be-deleted)
  `gismeteoCities` literal; run it *before* deleting the Go map in Task 6 so the old and
  new sources are cross-checked in one CI run.
- **Half-migrated / hand-edited DB going dark** (Task 7): if config loads but a row is
  missing, weather could silently stop. Mitigation: "row absent → default active"
  fallback, and gismeteo empty-map → no-op with a log line, never a crash.
- **Migration immutability**: 016/017 filenames are frozen the instant they hit any
  environment. A mistake requires a *new* corrective migration, never a rename. Review the
  seed values against the current Go literals before merge.
- **Inspector false-negative masking a real outage** (Task 8): making the inspector
  respect `active` means a mistakenly-disabled provider hides its health line. Acceptable —
  the inspector is advisory and cannot fail the gate; the operator who set `active=0` owns
  that decision.
- **Scope temptation**: the `weather_sources` table invites "let's make open-meteo
  endpoints / a generic provider engine configurable too". Resist — see Trade-offs.

## Trade-offs

- **Two-table hybrid over a single generic `weather_sources` engine.** Chose (a)
  `weather_sources` (deploy knobs + active) **plus** (b) `weather_gismeteo_cities`
  (coverage) over a single `rate_sources`-style table that pretends any provider is
  addable via SQL. Reason: weather providers have bespoke Go parsing (gismeteo's
  Russian-tooltip→WMO regex); there is no generic extraction ruleset to store. A single
  table would imply a capability that does not exist and mislead operators.
- **Coverage table is the source of truth; `weather_user_cities.gismeteo_city_id` is left
  dead, not populated, not dropped.** Chose a dedicated one-row-per-city coverage table
  over populating the per-user column because (1) the column holds only the numeric id but
  URL building needs slug *and* id, (2) a per-user column duplicates coverage across every
  subscriber of the same city and conflates config with subscription data, (3) dropping
  the column now is a risky SQLite table rebuild that does not belong in a feature change.
  Cost: one documented-dead column remains until a future cleanup migration.
- **`gismeteoConditionRules` stays in Go.** Chose code over config because these are DOM-
  coupled parsing rules that change only when gismeteo's markup/vocabulary changes — the
  same event that forces a regex edit and a rebuild anyway. Externalising them would add a
  table and JSON-encoding surface for zero operator benefit (no operator hand-edits
  Russian-keyword→WMO mappings).
- **Load config once at startup, not per collection tick.** Chose startup load because the
  collector is short-lived per cron tick, making "startup" and "per invocation" identical
  and keeping the hot path query-free. Cost: a coverage change is picked up on the next
  tick, not mid-tick — an irrelevant delay for a cron-driven scraper.
- **Open-Meteo gets a config row it barely uses.** Chose to include it for a uniform
  `active` toggle and table symmetry, accepting that `base_url`/`options` are inert for it,
  over special-casing gismeteo as the only configurable provider. The inert columns are
  documented so no one wires open-meteo endpoints through them expecting an effect.

## Implementation Notes (post-review)

All nine tasks shipped as planned; the five-lens review plus a targeted re-review
added the following hardening on top of the original tasks:

- **Collector testability (Task 7).** `wireWeather`/`gismeteoOption` take narrow
  `weatherSourceLoader`/`gismeteoCoverageLoader` interfaces (realising decision D6's
  "by interface" intent) so the degradation matrix is unit-tested with fakes, no live
  DB. `flag.Parse()` moved from `init()` to `main()` so a package-main test binary works.
- **URL hardening (Task 6).** `NewGismeteo` trims a trailing slash from `base_url`
  before the empty-fallback (avoids a `//` 404); `buildGismeteoURL` path-escapes the slug.
- **`WeatherSourceOptions.Headers`** is documented as reserved/not-yet-consumed rather
  than wired — applying arbitrary request headers is out of scope for this plan.
- **Ops (Tasks 7/8).** The `cmd/web` weather-source repo fatal carries the
  `dependencies:` boot marker; a boot line records the registered gismeteo `base_url`;
  the inspector's boot-time-only evaluation is documented. CLAUDE.md notes the
  `/health/check` `gismeteo` key is absent only on an explicit `active=0`.
- **Consciously rejected.** A suggestion to redact verbatim advisory-inspector errors on
  the unauthenticated `/health/check` (blind-SSRF-oracle via `base_url`) was declined: it
  would break the documented `ok|<error>` contract and its tests, is a cross-cutting
  inspector-layer change out of scope here, requires root-equivalent DB write access to
  exploit, and matches the pre-existing `rate_sources.url` operator-trust model.
