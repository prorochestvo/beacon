# Task Breakdown

## Overview

Convert the four weather **threshold** alert kinds (`alert_frost`, `alert_heat`,
`alert_thaw`, `rain_alert`) plus `alert_thunderstorm` from **level-triggered + 20h/6h
cooldown** to **edge-triggered with a per-row latch**. An alert now fires **once** on the
transition into its condition, then stays silent until the condition **clears** (the day is
clearly on the other side) and re-arms; only after re-arming can it fire again. This is the
stateful "Design 2" that plan 261 (`plans/completed/260715.0001.weather-alert-ui-and-thaw-trigger.md`)
evaluated and deferred. The user has overridden that trade-off; this is the follow-up to
`r_0.0.3-alpha`.

`morning_summary` is **not** a threshold alert and is completely untouched — it keeps
`IsMorningDue` / `last_notified_at`.

### The model (finalised)

Per-row, per-kind **boolean latch**. Each subscription row carries a single `conditionMet`
predicate. Re-arm is the exact complement of fire, so the latch is a clean edge detector:

```
nextLatched = conditionMet(obs)                 (unless the obs is unevaluable → unchanged)
fire        = conditionMet(obs) && !prevLatched
```

The "Schmitt / full day clearly on the other side" property is **emergent, not a second
threshold**: cold-side kinds key on **TempMin** (coldest point of the day) and warm-side
kinds key on **TempMax** (warmest point). So re-arming frost needs `TempMin > T` — the
*coldest* moment already above threshold, i.e. the whole day warm; re-arming thaw needs
`TempMax ≤ 0` — the *warmest* moment at/below freezing, i.e. the whole day frozen. The
min/max split **is** the dead-band for the frost↔thaw pair; no per-kind hysteresis band is
stored.

### Finalised fire / re-arm table

| kind | `conditionMet` (fire predicate) | metric | re-arm (`!conditionMet`) | latched? |
|---|---|---|---|---|
| `alert_frost(T)` | `TempMin ≤ T` | daily min | `TempMin > T` | yes |
| `alert_heat(T)` | `TempMax ≥ T` | daily max | `TempMax < T` | yes |
| `alert_thaw` | `TempMax > 0` | daily max | `TempMax ≤ 0` | yes |
| `rain_alert(P)` | max hourly precip prob in fixed 6h window `≥ P` | hourly | max prob `< P` | yes |
| `alert_thunderstorm` | daily-dominant WMO `≥ 95` (band {95,96,99}) | daily code | WMO `< 95` | yes |

**`alert_thunderstorm` gets the latch** (fire-on-enter-storm, re-arm on first non-storm day).
Justification: it fits the identical edge model (`conditionMet = WMO≥95`); excluding it would
force the old cooldown machinery to survive *solely* for thunderstorm, which is strictly
worse than one uniform mechanism. Including it lets the cooldown code be deleted outright.

**thaw vs heat overlap:** thaw's fire (`TempMax > 0`) is nearly a heat alert at `T ≈ 0`. Kept
distinct on purpose — baked-in `0`, no `condition_value`, distinct "Thaw" wording. Called out,
not merged.

### Latch replaces the timer cooldown; a forecast_date gate caps refires

The latch **replaces** the timer cooldown for all five alert kinds. Once fired, a row cannot
fire again until `conditionMet` goes false (re-arm) — strictly stronger than a 20h timer. The
`weatherAlertCooldown*` constants and the `weatherAlertCooldown()` function are **deleted**.

On top of the latch sits a **per-`forecast_date` fire cap** (the anti-jitter backstop). The
collector rewrites the observation every ~6h and this feature targets near-zero crossings —
exactly where the daily `TempMin`/`TempMax` forecast can jitter across the fire/re-arm
boundary **within one day**. Without a cap, a latch that re-arms (forecast momentarily clears)
and re-meets (forecast crosses back) the *same* day would fire twice — "three frost alerts in
one cold evening," precisely the spam this feature exists to kill. The cap fires a given
subscription **at most once per `forecast_date`**:

```
latch edge says fire   AND   the latest obs forecast_date ≠ the forecast_date of the last fire
```

The latch still governs the enter/re-arm edge; the forecast_date gate is a second filter that
only limits **firing**. Re-arm transitions (no notification) still update latch state freely
within the day — only firing is capped.

**`last_notified_at` is repurposed for alert kinds**, *not* freed: a row is exactly one kind,
so morning_summary rows use `last_notified_at` as the wall-clock morning cursor (via
`IsMorningDue`) and alert rows use the *same column* as the **forecast_date of the last fire**
(stored as that local date's UTC-midnight instant). The two meanings never collide because no
single row is both kinds; `IsMorningDue` is only invoked on morning rows and the gate only on
alert rows. No separate column is needed.

### Storage & signature decisions (summary)

- **Latch storage:** new column `alert_latched INTEGER NOT NULL DEFAULT 0` on
  `weather_user_cities` (migration `202607.020`). A **boolean**, not a phase enum: each row is
  a single kind, so it only needs "fired-and-not-yet-re-armed"; there is no richer phase to
  represent, and the frost↔thaw dead-band lives in the min/max metric choice, not in state.
- **Fire-cap storage:** the existing `last_notified_at` column, reinterpreted for alert kinds
  as the last-fired `forecast_date` (see above). No new column.
- **Evaluator signature:** keep `EvaluateAlert(obs, now) (bool, string, error)` **unchanged**
  (backward compatible, still the uniform predicate). Add a sibling
  `EvaluateLatched(obs, now, prevLatched) (fire, nextLatched bool, reason string, err error)`.
  Both delegate to a new **tri-state** internal `evaluateAlertCondition` that distinguishes
  `met` / `notMet` / `unevaluable` — needed so a data gap (nil TempMin, empty hourly) does
  **not** get mistaken for "condition cleared" and wrongly re-arm the latch.

## Assumptions

1. **Plan number 262** is correct and free: highest existing `NNN-` prefix is 260; 261 shipped
   as the completed thaw plan (`260715.0001...`). No collision.
2. **The 7-day trace enters mid-story in the FROZEN phase**, meaning the frost row is already
   **latched** and the thaw row is **armed** at day 0. That is why the trace shows no frost
   event on day 1 despite `min ≤ 0`. The acceptance test seeds these initial latch values
   explicitly (see Task 2). A *fresh deploy* seeds every row `alert_latched=0` (armed), so an
   already-cold city fires **one catch-up alert** on the first tick — see Assumption 4.
3. **`notify_kind` has no CHECK constraint** (migration 011 line 13), so adding latch state
   needs no change to how kinds are validated. Confirmed against the schema.
4. **One-time catch-up fire on first tick after deploy is ACCEPTED (intended one-shot).**
   Decision confirmed by the coordinator: seed every row armed (`alert_latched=0`), no
   observation back-fill. The migration cannot seed the latch from live conditions (pure SQL
   can't read observations; wrong layer). Each existing alert row currently in-condition fires
   exactly once, then latches. Bounded (≤ one notification per in-condition subscription),
   one-time, deliberate. **Self-healing of the repurposed column:** existing alert rows may
   carry a stale wall-clock `last_notified_at` from the old cooldown code; on the catch-up fire
   it is overwritten with the forecast_date key, so a legacy value never spuriously trips the
   new fire-cap (a stale timestamp will not equal any forecast_date-midnight key).
5. **Re-subscribe (ON CONFLICT update) preserves the existing latch.** `RetainWeatherUserCity`
   does not list `alert_latched`, so a fresh insert defaults it to 0 (armed) and a conflict
   update leaves it untouched. Re-adding an already-latched subscription stays latched. Minor;
   noted, not "fixed" (see Trade-offs).
6. **`last_notified_at` is not surfaced for alert rows in any wire contract.** Verified:
   `internal/dto/weather.go` (`WeatherCityRow`) exposes neither `last_notified_at` nor any
   latch field, so no DTO change is required (Task 5 confirms). This is what lets the column be
   repurposed for alert kinds without a wire-contract change.
7. **Intra-day forecast jitter is capped NOW by a per-`forecast_date` fire gate (not
   deferred).** Decision confirmed by the coordinator. The collector rewrites the observation
   ~every 6h and the feature targets near-zero crossings, so the daily min/max can cross the
   fire/re-arm boundary several times within one day; the latch alone would permit each
   re-cross to fire. The gate reuses the `last_notified_at` column (freed from the deleted
   cooldown; Assumption 6) to store the last-fired forecast_date and suppresses any further fire
   while the latest observation's
   forecast_date matches it. The latch governs the edge; the gate caps firing to once per
   forecast_date. **The column carries both meanings cleanly** because a row is exactly one
   kind: morning rows → wall-clock morning cursor; alert rows → last-fired forecast_date. No
   separate field is needed.

     ```go
     // EvaluateLatched applies edge-trigger semantics on top of the alert condition.
     // fire is true only on the transition into the condition (met && !prevLatched).
     // nextLatched tracks the condition: latched while met, re-armed when it clears.
     // An unevaluable observation (data gap) leaves the latch unchanged and never fires.
     // On error the previous latch is preserved. reason is non-empty only when fire is true.
     func (c *WeatherUserCity) EvaluateLatched(obs WeatherObservation, now time.Time, prevLatched bool) (fire, nextLatched bool, reason string, err error) {
         cond, reason, err := c.evaluateAlertCondition(obs, now)
         if err != nil {
             return false, prevLatched, "", err
         }
         switch cond {
         case alertConditionUnevaluable:
             return false, prevLatched, "", nil // data gap: no state change, no fire
         case alertConditionNotMet:
             return false, false, "", nil       // condition cleared: re-arm
         default: // alertConditionMet
             return !prevLatched, true, reason, nil
         }
     }
     ```

- **Description:**

  3. Replace the per-city body of the alert loop with the latch + forecast_date-gate flow:
     ```go
     prev := city.AlertLatched
     fire, next, reason, evalErr := city.EvaluateLatched(*obs, now, prev)
     if evalErr != nil {
         errs = append(errs, fmt.Errorf("weather alert city=%s: evaluate: %w", city.ID, evalErr))
         continue
     }

     if !fire {
         // Re-arm (or any latch change without a fire) must still be persisted: this is the
         // day-5 / day-7 transition that produces NO notification. Only write on a real change
         // so steady state costs zero writes per tick.
         if next != prev {
             if setErr := a.cityRepo.SetWeatherAlertLatched(ctx, city.ID, next); setErr != nil {
                 errs = append(errs, fmt.Errorf("weather alert city=%s: persist re-arm: %w", city.ID, setErr))
             }
         }
         continue
     }

     // fire == true  ⇒  next == true and prev == false (guaranteed by EvaluateLatched).
     // Second gate: cap to one fire per forecast_date (anti-jitter backstop). For alert kinds
     // LastNotifiedAt holds the forecast_date of the last fire.
     fdKey, keyErr := domain.ForecastDateKey(obs.ForecastDate)
     if keyErr == nil && !city.LastNotifiedAt.IsZero() && city.LastNotifiedAt.Equal(fdKey) {
         // Same forecast_date already fired — this is a within-day jitter re-cross. Record the
         // latch edge (next == true) but do NOT notify again.
         if next != prev {
             if setErr := a.cityRepo.SetWeatherAlertLatched(ctx, city.ID, next); setErr != nil {
                 errs = append(errs, fmt.Errorf("weather alert city=%s: persist latch (gated): %w", city.ID, setErr))
             }
         }
         alertSuppressed++
         continue
     }
     if keyErr != nil {
         // Malformed/empty forecast_date is an anomaly: log, allow the fire (never drop an
         // alert), but the fire cap cannot be recorded — fall back to a latch-only write below.
         fmt.Fprintf(a.logger, "weather alert: city %s: unparseable forecast_date %q: %v\n", city.ID, obs.ForecastDate, keyErr)
     }

     msg, renderErr := RenderWeatherAlert(city, reason, *obs)
     if renderErr != nil {
         errs = append(errs, fmt.Errorf("weather alert city=%s: render: %w", city.ID, renderErr))
         continue
     }
     ev := &domain.RateUserEvent{UserType: domain.UserTypeTelegram, UserID: city.UserID, Message: msg}
     alertAttempted++
     if retainErr := a.eventRepo.RetainRateUserEvent(ctx, ev); retainErr != nil {
         errs = append(errs, fmt.Errorf("weather alert city=%s: queue event: %w", city.ID, retainErr))
         continue // do NOT latch or record the fire date; next tick retries
     }
     alertQueued++

     // Persist ONLY after a successful enqueue, so a queue failure re-fires next tick.
     var persistErr error
     if keyErr == nil {
         persistErr = a.cityRepo.MarkWeatherAlertFired(ctx, city.ID, fdKey) // latch=1 + record forecast_date
     } else {
         persistErr = a.cityRepo.SetWeatherAlertLatched(ctx, city.ID, true) // anomaly: latch only
     }
     if persistErr != nil {
         errs = append(errs, fmt.Errorf("weather alert city=%s: persist fire: %w", city.ID, persistErr))
     }
     ```

     interface; the morning phase still uses it.
  5. Keep the proof-of-execution log line's `suppressed` field (now = forecast_date-gated
     count): `alerts: q/a suppressed: N`.

          3. `min=-1` (met again, same `D`) → latch edge would fire, but `LastNotifiedAt==key(D)`

             true)` records the edge, `MarkWeatherAlertFired` **not** called.

          because the latch is still true from scrape 3, `fire=false` (no edge) → no event. Then
          a run where the row first re-armed on `D+1` (`min=+2`) and then `min=-1` on `D+1`
          fires once and calls `MarkWeatherAlertFired(id, key(D+1))` — proving the cap is

- **Pitfalls:**
  - **The re-arm write is the crux of the feature.** It is easy to keep the old "mutate only on
    successful queue" shape and silently drop the day-5/day-7 transitions — then a re-armed
    condition never persists and the alert never fires again. The `if next != prev` write on the
    `!fire` path is mandatory.
  - Only skip the latch write when `next == prev` (steady state) — this keeps writes bounded to
    real transitions, not one per tick. Do not write unconditionally.
  - Latch-after-enqueue ordering must be preserved exactly (mirror the old advance ordering):
    render → enqueue → **then** persist (`MarkWeatherAlertFired` / latch). A crash between
    enqueue and persist re-fires once next tick (at-least-once, same as today's advance
    semantics) — acceptable.
  - **The forecast_date gate must sit AFTER the latch edge, not replace it.** The latch decides
    *whether an edge occurred*; the gate only decides *whether that edge may notify*. On a gated
    (suppressed) fire you must still persist `next` (=true) via `SetWeatherAlertLatched`, or the
    row stays armed and every subsequent same-day tick re-hits the gate — harmless but noisy in
    logs and wasteful. Persisting the latch drives it to steady state (latched+met → no edge).
  - **Do not gate on wall-clock `now` or a real timestamp** — gate on `forecast_date` equality
    via `domain.ForecastDateKey`. Comparing `city.LastNotifiedAt.Equal(fdKey)` only works
    because `MarkWeatherAlertFired` stored exactly `fdKey`; never write `now` into
    `last_notified_at` for an alert row.
  - **Re-arm does not update the fire cursor.** The `!fire` re-arm path calls only
    `SetWeatherAlertLatched` (latch column), leaving `last_notified_at` = the last fire's
    forecast_date. That is deliberate: the cap must survive an intra-day re-arm so scrape 3 is
    still suppressed.
  - Unparseable `forecast_date` must never drop the alert: log, fire, and fall back to
    `SetWeatherAlertLatched(true)`. It cannot record the cursor, so that anomalous row is not
    jitter-protected until a well-formed forecast_date arrives — acceptable for an anomaly.
  - The `alertKinds` slice already lists all five kinds — no change there.
  - Do not add an `IsMorningDue`/time gate to the alert loop; alerts must be evaluated **every
    tick** to observe re-arm transitions. The existing loop already does this.

## Risks

- **Intra-day forecast flapping (Assumption 7) — handled by the fire cap, one residual edge.**
  The per-forecast_date gate (Task 4) suppresses same-day re-fires from near-threshold jitter.
  **Residual:** a fire late on day `D` that re-arms just before midnight and re-meets just after
  midnight fires again on `D+1` (different forecast_date → gate passes). That is a genuine
  re-entry on a new day after the condition cleared, so a second alert is defensible; the window
  is narrow. Not mitigated in v1 — flagged so it is a known, accepted behaviour, not a surprise.
- **Fire cap depends on the collector writing a correct `forecast_date`.** The gate keys on
  `obs.ForecastDate`. If the collector ever writes a wrong or blank forecast_date, the cap
  degrades to latch-only for that row (jitter unprotected) — it never over-suppresses a
  legitimate new-day alert. Failure mode is fail-open (may notify), never fail-closed (drop).
- **Re-arm write silently dropped (Task 4).** The single most likely implementation bug: keeping
  the old "persist only on successful queue" shape drops the no-notification latch transitions,
  and the alert never re-fires. Guarded by the explicit re-arm subtest.
- **Data-gap re-arm (Task 2).** Collapsing `Unevaluable`→`NotMet` would re-arm on nil-field
  observations and spuriously re-fire when data returns. Guarded by the data-gap subtest.
- **Catch-up storm on deploy (Assumption 4).** All rows start armed; every currently-in-condition
  subscription fires once on the first tick. Bounded and one-time, but if many users are
  in-condition at deploy (e.g. a cold snap) they each get one alert. Acceptable; deploy during a
  quiet-weather window if avoidable.
- **Filename immutability (Task 1).** Once `202607.020...sql` is applied to any prod DB it must
  never be renamed; a rename = duplicate apply. Standard project constraint.

## Trade-offs

- **Boolean latch over a phase enum.** Each subscription row is a single kind with a single
  predicate, so `armed/latched` is the complete state; the frost↔thaw "mutual" behaviour is two
  independent rows whose dead-band comes from min-vs-max, not from a shared tri-state. A phase
  enum would add unreachable states and invite the question "what sets the phase across rows?"
  — chosen the boolean. If a future kind ever needs true multi-phase hysteresis, migrate the one
  column to TEXT then.
- **New `EvaluateLatched` method over changing `EvaluateAlert`'s return.** Plan 261 valued the
  uniform `EvaluateAlert(obs, now)` signature shared by all kinds; the latch genuinely needs
  extra input (`prevLatched`) and extra output (`nextLatched`) that only the stateful caller
  wants. A decorator method keeps `EvaluateAlert` as the pure predicate (still used by the
  backward-compat path and tests) and confines statefulness to one clearly-named method. Rejected:
  widening `EvaluateAlert`'s tuple, which would churn every existing caller/test for no benefit.
- **Tri-state internal predicate over a boolean.** Adds one small unexported enum, but it is the
  only clean way to stop a data gap from masquerading as "condition cleared." The alternative —
  re-checking each kind's required fields inside `EvaluateLatched` — duplicates exactly the logic
  that makes an observation unevaluable. Chosen the enum; both public methods delegate to it, so
  there is one source of truth.
- **Latch replaces the timer cooldown; a forecast_date fire cap replaces its one useful side
  effect.** Deleting `weatherAlertCooldown*` removes two overlapping anti-spam mechanisms; the
  latch is strictly stronger for the enter/re-arm edge. The one behaviour the timer had that the
  bare latch lacks — suppressing an intra-day clear→re-meet refire — is preserved by the
  per-forecast_date cap, which is condition-aware (fires again only on a genuinely new forecast
  day) rather than a blunt N-hour timer. Net: one mechanism deleted, one sharper mechanism added.
- **`last_notified_at` repurposed for alert kinds over adding a new column or freeing it.** The
  coordinator's decision. A row is exactly one kind, so the column carries the morning cursor on
  morning rows and the last-fired forecast_date on alert rows with no collision, and no wire
  contract exposes it (Assumption 6). Rejected: (a) a dedicated `alert_last_fired_date` column —
  a second SQLite table change and more schema for a value the freed column already fits;
  (b) leaving `last_notified_at` unused on alert rows and losing the cheap anti-jitter cap. The
  cost is a mildly overloaded field name, contained by godoc documenting both meanings.
- **Accept one catch-up fire on deploy over a seed back-fill.** Seeding the latch from live
  conditions needs an observation read the migration layer cannot do; a startup back-fill job is
  disproportionate for a single, bounded, one-time ping. Accepted the catch-up.
- **Re-subscribe preserves the latch over resetting to armed.** Not listing `alert_latched` in
  the upsert is the minimal repo change and matches how `RetainWeatherUserCity` already avoids
  clobbering system columns. A re-subscribe keeping a latched row latched is a benign edge case;
  resetting would be a deliberate extra SET clause — not worth it now, noted for revisit.
