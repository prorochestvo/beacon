# Task Breakdown

## Overview

Plan 013 wired `chromedp` behind the `rulegen.Fetcher` interface, and live
smoke-tested it against the only `fetcher_kind="chromedp"` seed,
`KZ_BCC_BID_USD_KZT` (https://www.bcc.kz/kz/). Headless Chrome rendered an
additional ~44 KB of post-hydration DOM, the body passed `Sanitize`, the
co-location guard fired inside the 80 KB locate window, but both fallback LLM
attempts failed at the rule-validation / plausibility gates â€” the LLM could not
locate rate data in the rendered DOM.

Reviewer diagnosis (which we adopt): the wiring is correct; the LLM input is
poor. Two complementary fixes:

1. The current post-`WaitVisible("body")` wait is **1.5 s** (constant
   `defaultChromedpNetworkIdleMs` in
   `internal/application/rulegen/chromedpfetcher.go:56`). Modern SPAs typically
   take 3â€“5 s after `body` is visible to fetch rate JSON and render their
   table; 1.5 s often captures a skeleton state.
2. Even with a longer global wait, the proper signal is page-specific:
   "wait until selector X exists". This is the standard chromedp escape
   hatch.

This plan does **both**:

- Bump the default `NetworkIdleMillis` to **5000 ms**.
- Add a typed `WaitSelector` field to `domain.RateSourceOptions` (today a
  typed struct with a single `Reserve` field â€” confirmed at
  `internal/domain/ratesource.go:36-38`) and let `ChromedpFetcher` consume it
  via a per-call selector. The Generator constructs a fresh
  `ChromedpFetcher` for the active source's selector when needed; the public
  `Fetcher` interface (`Fetch(ctx, url) ([]byte, error)`) is unchanged.
- Update the BCC homepage seed row to set `wait_selector` to the most likely
  marker the rendered rate table uses (`div.text-lg`, matching the structural
  anchor used by every seeded BCC rule).
- Document the operator-side levers (`wait_selector`,
  `--max-fallback-attempts=4` for stubborn chromedp pages).

Scope is the BCC homepage. Adding new chromedp sources and refactoring to a
chromedp pool are out â€” see plan 015+.

### Task 2: Add `WaitSelector` to domain options and propagate through chromedp fetcher

- **Description:**

    follow the existing terse style.
  - File: `internal/application/rulegen/chromedpfetcher.go`. Add a typed
    field and option:
    ```go
    type ChromedpFetcher struct {
        // ... existing fields ...
        waitSelector string // empty means "wait for body only"
    }

    type ChromedpFetcherOptions struct {
        // ... existing fields ...
        // WaitSelector, if non-empty, replaces the default body+sleep wait
        // strategy with WaitVisible(selector). Use this for SPAs where the
        // rate table appears only after JS hydration. Empty falls back to
        // WaitVisible("body") + NetworkIdleMillis sleep.
        WaitSelector string
    }
    ```
    Plumb through `NewChromedpFetcher`. In `Fetch`, build the chromedp
    action slice conditionally:
    ```go
    actions := []chromedp.Action{
        chromedp.Navigate(url),
        chromedp.WaitVisible("body", chromedp.ByQuery),
    }
    if f.waitSelector != "" {
        actions = append(actions, chromedp.WaitVisible(f.waitSelector, chromedp.ByQuery))
    } else {
        actions = append(actions, chromedp.Sleep(time.Duration(f.networkIdleMs)*time.Millisecond))
    }
    actions = append(actions, chromedp.OuterHTML("html", &rendered, chromedp.ByQuery))
    if err := chromedp.Run(browserCtx, actions...); err != nil { ... }
    ```

    selector is strictly stronger than a fixed sleep. Stacking both wastes
    wall-clock budget on every fetch and obscures the failure mode (did
    the selector miss, or did the sleep expire first?). When the operator
    sets a selector, they have decided what "loaded" means; trust them.
  - Update the fetch log line to include the selector for diagnosability:
    `chromedp: url=%s wait_selector=%q rendered_bytes=%d` (empty string
    when none).

    pick the right chromedp fetcher per source. Cheapest approach
    (option (c) in the briefing): make the chromedp fetcher slot a
    **constructor function** instead of a single `Fetcher`:

        chromedpFetcherFor func(waitSelector string) Fetcher
        // ... existing fields ...
    }
    ```

    instead of a single `Fetcher`. In `Generate`, after selecting the

  ```go
  // chromedpfetcher.go (new field + conditional action list)
  type ChromedpFetcher struct {
      timeout       time.Duration
      chromiumPath  string
      networkIdleMs int
      waitSelector  string
      logger        io.Writer
  }

  func (f *ChromedpFetcher) Fetch(ctx context.Context, url string) ([]byte, error) {
      ctx, cancel := context.WithTimeout(ctx, f.timeout)
      defer cancel()

      // ... allocator + browser context setup unchanged ...

      actions := []chromedp.Action{
          chromedp.Navigate(url),
          chromedp.WaitVisible("body", chromedp.ByQuery),
      }
      if f.waitSelector != "" {
          actions = append(actions, chromedp.WaitVisible(f.waitSelector, chromedp.ByQuery))
      } else {
          actions = append(actions, chromedp.Sleep(time.Duration(f.networkIdleMs)*time.Millisecond))
      }
      var rendered string
      actions = append(actions, chromedp.OuterHTML("html", &rendered, chromedp.ByQuery))

      if err := chromedp.Run(browserCtx, actions...); err != nil {
          return nil, fmt.Errorf("chromedp fetch %s: %w", url, err)
      }
      fmt.Fprintf(f.logger, "chromedp: url=%s wait_selector=%q rendered_bytes=%d\n",
          url, f.waitSelector, len(rendered))
      return []byte(rendered), nil
  }
  ```

### Task 4: Documentation-only â€” `--max-fallback-attempts` guidance for chromedp pages

  - Content (English, no emojis):
    ```markdown
    ## Tuning for chromedp sources

    Sources with `fetcher_kind = "chromedp"` render through a headless Chrome
    instance, so the LLM receives the post-hydration DOM rather than the raw
    HTML. The rendered DOM is typically 30â€“50 % larger than the raw response
    and contains hydrated framework state, which makes pattern discovery
    harder than for plain HTTP sources. Two operator-side levers:

    1. **`options.wait_selector`** (per-source). Set the `wait_selector` key
       on the source's `options` JSON column to a CSS selector that appears
       only after the rate table has loaded. The fetcher will block on
       `WaitVisible(selector)` instead of falling back to a fixed
       post-`body` sleep. Example seed snippet:
       ```sql
       UPDATE rate_sources
       SET    options = json_set(COALESCE(options, '{}'), '$.wait_selector', 'div.text-lg')
       WHERE  name = 'KZ_BCC_BID_USD_KZT';
       ```
       Use the same structural marker the extraction rule will key off
       (e.g. `div.text-lg` for BCC rows). The 30 s wall-clock timeout still
       caps the wait; a missing selector surfaces as
       `chromedp fetch <url>: context deadline exceeded`.

    2. **`--max-fallback-attempts=4`** (per-invocation). Chromedp-rendered
       DOMs are noisier than plain HTML; if the first two fallback
       attempts both fail with rule-validation or plausibility errors,
       a larger budget often converges. Example:
       ```bash
       ./build/rulegen --force-fallback --max-fallback-attempts=4 KZ_BCC_BID_USD_KZT
       ```

    The default post-body sleep is 5 s (raised from 1.5 s in plan 014)
    and applies only when `wait_selector` is empty.
    ```

## Risks

- **Selector hypothesis is wrong.** `div.text-lg` is an educated guess
  based on every other seeded BCC source. If the homepage rate widget
  uses a different DOM, Task 5 fails and a follow-up migration is
  needed. Mitigated by the explicit fallback playbook in Task 5.
- **Longer waits compound on multi-source future use.** When
  `cmd/collector` integrates chromedp (plan 015+), every chromedp
  source will pay the new 5 s minimum (or its selector wait, capped at
  30 s). Acceptable today (one chromedp source, 6 h interval); revisit
  if the chromedp source count grows past ~5.
- **`WaitVisible` blocks forever on a missing selector until the wall-
  clock timeout.** A typo in `wait_selector` costs 30 s per attempt.
  Mitigated by surfacing the selector in the fetcher log line so the
  operator can spot the issue immediately.
- **Map-vs-struct decision is sticky.** Once we add typed fields to
  `RateSourceOptions`, switching to a map later means a migration on
  every existing row's JSON shape. The briefing recommended a map;
  we are diverging because the code is already a typed struct, and
  fighting that for one new field is worse than embracing it. If the
  options surface grows past ~5 keys, revisit (likely with a wrapped
  type that is half-struct, half-map for forward-compat keys).
- **Backward compatibility of `RateSourceOptions` round-trip.** The
  repository unmarshals `options` JSON into the struct; existing rows
  with `'{}'` continue to work (zero values). The risk is the *forward*
  direction â€” a future row with an unknown key (operator-edited DB)
  would silently drop the key on the next `RetainRateSource` call.
  Mitigated by funneling all schema changes through migrations.
- **The 5 s constant is a guess.** It might be 4 s, it might be 7 s.
  5 s is well-justified for a default but is not load-tested. Mitigated
  by `wait_selector`: the deterministic path doesn't depend on the
  constant at all.

## Trade-offs

- **Typed field vs. `map[string]interface{}` for options.** Chose a
  typed `WaitSelector string` field because `RateSourceOptions` is
  already a typed struct (`Reserve string`). Map-keyed access would
  require either changing the on-disk JSON shape (no â€” existing rows
  would need migration) or wrapping the struct (yes â€” but the wrapping
  is overhead for one new key). Static typing wins for now;
  re-evaluate when the option surface exceeds ~5 fields.

- **Replace-default-sleep semantics for `WaitSelector` vs. add-after.**
  Chose replace because (a) a deterministic selector subsumes a fixed
  sleep, (b) stacking both wastes wall-clock budget on every fetch,
  (c) the failure mode is clearer when there is exactly one "what does
  loaded mean" gate.

- **Per-source factory closure vs. mutating the chromedp fetcher.**
  Chose the factory closure (`func(string) Fetcher`) because the
  `Fetcher` interface is shared with plain HTTP and (eventually) the
  collector. Mutating the chromedp fetcher's `waitSelector` field in
  Generator code would create a hidden coupling and break the
  immutability of the fetcher object. The factory keeps per-call
  configuration encapsulated inside the chromedp implementation.

- **5 s default sleep vs. removing the sleep entirely.** Could have
  removed the sleep and required `wait_selector` on every chromedp
  source. Rejected: forces operators to chase a selector for every
  new chromedp page before they know whether the page even needs it.
  The bumped default is a reasonable fallback; the selector is the
  preferred path for known SPAs.

- **CLI flag for `--max-fallback-attempts` vs. per-fetcher-kind
  default.** Chose the existing CLI flag (kept at default 2) plus
  README guidance. A per-kind default would fragment the audit-loop
  semantics ("why does this source run more attempts?"); a CLI flag is
  loud and operator-controlled.

- **Editing the BCC homepage seed via `json_set` vs. a full INSERT OR
  REPLACE.** Chose `json_set` because the seed row already exists
  (from `202605.007`) and we only want to set one key. `INSERT OR
  REPLACE` would force restating every column, which (a) duplicates
  data, (b) creates merge conflicts with any future plan that also
  edits this row, and (c) drops a row's PK identity if the schema
  ever gains a generated column.
