# Task Breakdown

## Overview

`cmd/wasm/apiclient` integration and fetcher tests flake intermittently under the
full parallel `go test -race ./...` run, surfacing as a data race whose stack names
`http.Transport.CloseIdleConnections`. The most frequently reported victim is
`TestClient_Integration_SetSourceActive`, but the fault is class-wide, not specific
to that test.

Root cause (confirmed from the Go 1.26 stdlib source, `net/http/httptest/server.go`):

- Every test builds its fetcher via `apiclient.NewHTTPFetcher(srv.URL, nil)`. With a
  `nil` client, `NewHTTPFetcher` constructs `&http.Client{Timeout: 5s}` whose
  `Transport` is `nil`, so the request goes through the process-global
  `http.DefaultTransport`.
- Every test is `t.Parallel()`, so all tests in `package apiclient_test` (both
  `client_integration_test.go` and `fetcher_http_test.go`) run concurrently and share
  that one global transport.
- `httptest.Server.Close()` **unconditionally** calls
  `http.DefaultTransport.(closeIdleTransport).CloseIdleConnections()` (server.go
  lines 268-270). So when test A closes its server, it mutates the idle-connection
  pool of the same `DefaultTransport` that test B is concurrently issuing a request
  against â€” the race the detector reports.

The fix is to stop sharing the global transport: give each test an HTTP client whose
`Transport` is private to its own `httptest.Server`. `httptest.Server.Client()` returns
exactly that â€” for a plain (non-TLS) server, `s.client` is `&http.Client{Transport: tr}`
with a dedicated `tr := &http.Transport{}` created in `Start()`. As a bonus, `Close()`
(server.go lines 273-277) then deterministically closes the idle connections of *that*
transport, so no client-side connections linger.

This is a test-only change; production code (`NewHTTPFetcher`, the WASM `domFetcher`)
is untouched. Test-parallelism global-state coupling belongs fixed in the tests, not
worked around in a production default.

## Assumptions

- The intermittent `-race` failure the handoff calls a "`CloseIdleConnections`
  transport race" is the `DefaultTransport` sharing described above. It reproduces only
  under cross-package scheduling pressure (`./...`); 300x single-package `-count` runs
  do not trip it, consistent with a rare timing window.
- No production caller invokes `NewHTTPFetcher(url, nil)` concurrently in a way that
  calls `CloseIdleConnections`; the fault is purely a test artifact of
  `httptest.Server.Close()`.

## Tasks

### Task 1: Isolate every apiclient test's HTTP transport from the global default

- Description:
  - `client_integration_test.go`: change the shared `newIntegrationClient` helper to
    pass `srv.Client()` instead of `nil` to `NewHTTPFetcher`. This one edit covers all
    eight `TestClient_Integration_*` functions, since they all build through the helper.
  - `fetcher_http_test.go`: change all `apiclient.NewHTTPFetcher(srv.URL, nil)` (and the
    one `srv.URL+"/"` variant) call sites to pass `srv.Client()`. These tests do not
    share a helper, so each site is converted individually.
  - Add a short comment at the helper (and where non-obvious) explaining that
    `srv.Client()` is used deliberately to avoid `http.DefaultTransport` sharing across
    parallel tests.
- Acceptance Criteria:
  - No test in `package apiclient_test` constructs a fetcher that falls back to
    `http.DefaultTransport` (`grep` for `NewHTTPFetcher(.*nil)` in the test files is
    empty).
  - `CGO_ENABLED=1 go test -race -count=200 ./cmd/wasm/apiclient/` is green.
  - Several full `CGO_ENABLED=1 go test -race ./...` runs are green.
  - `make test` and `make build` are green.
- Pitfalls & edge cases:
  - The two "network error" subtests close the server *before* building the fetcher;
    `srv.Client()` is still valid there (the client is created in `Start()`, which
    `NewServer` already ran) and the dial still fails as intended.
  - `srv.Client()` carries no `Timeout` (unlike the previous `&http.Client{Timeout: 5s}`).
    Acceptable: the handlers never hang, and `t.Context()` / `go test -timeout` bound the
    tests. Not worth re-adding per-site.
- Complexity: Easy

## Execution Order

1. Task 1 (single, atomic change across the two test files).

## Risks

- Low. Test-only, mechanical. The behaviour under test (request/response round-trips,
  error propagation) is unchanged; only the transport backing each request is now
  per-server instead of process-global.

## Trade-offs

- Chose `srv.Client()` over changing `NewHTTPFetcher`'s nil-default to a fresh
  `&http.Transport{}`: the latter is a one-line production change with zero test edits,
  but (a) it fixes a test-parallelism problem in production code â€” wrong layer â€” and
  (b) the fetcher's private transport would not be torn down by `srv.Close()`, leaving
  idle connections to linger until GC. `srv.Client()` keeps the fix in the tests and
  lets `httptest.Server.Close()` clean up the exact transport it used.
