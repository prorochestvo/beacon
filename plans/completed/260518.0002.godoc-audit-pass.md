# Task Breakdown

## Overview

Documentation-only audit pass across the entire `fx_rate_monitor` Go codebase to
make sure every exported identifier carries a correct, idiomatic godoc comment.
No behavior changes. No refactors. No tests added or removed (existing tests must
keep passing, `go vet ./...` must stay clean).

The work is broken into per-package waves so the engineer can land each batch as
a small, easy-to-review commit. Each wave is independently shippable; the only
hard ordering is the final wave, which runs the verification commands across the
whole repo and adds any missing package-level (`// Package foo ...`) comments
discovered along the way.

## Assumptions

### What "correct godoc" means in this repo

1. **Every exported identifier** (Type, Func, Method, Var, Const, package-level
   declaration) gets a doc comment. "Exported" = starts with an uppercase letter.
2. **Comment starts with the identifier's name** and is a complete sentence ending
   with a period.
   - Good: `// Encode returns the base64-encoded form of v.`
   - Bad: `// encode v` / `// Encodes the value` / `// Encode` (no period).
3. **Package comments** live on one file per package, formatted
   `// Package foo <what foo does>.`. Add only if missing — do not duplicate.
   Prefer the existing primary file (e.g. `ratesource.go` for `package domain`)
   over creating a `doc.go` unless the package has no obvious anchor file.
4. **Methods** document behavior, returned errors, side effects, concurrency
   guarantees, and any non-obvious invariants. They do **not** restate the
   signature. If the only thing a comment would say is `// Foo is a Foo.`, omit
   the comment and leave the existing state unchanged. Better to skip than to
   add noise.
5. **Unexported symbols** only get comments when the intent is non-obvious. Do
   **not** bulk-add comments to unexported helpers — that violates the
   "no fluff" rule in `CLAUDE.md`.
6. **Constructors** (`New<T>`, `Open<T>`, `Must<T>`, etc.) state what they
   return and any setup/teardown contract (e.g. "caller must Close the returned
   client when done").
7. **Interfaces** get a doc on the interface itself **and** on every method
   declared inside.
8. **Error sentinels** (`var ErrX = errors.New(...)`) get a one-liner describing
   the condition under which the error is returned.
9. **Existing WHY-comments** that explain non-obvious constraints, bug history,
   or trade-offs are preserved verbatim. If the existing comment is correct,
   leave it alone. Never overwrite a substantive comment with a generic
   restatement.
10. **No section-divider comments** (`// --- foo ---`) are introduced. The
    repo's `CLAUDE.md` forbids them.

### Scope boundaries

- **Test files (`*_test.go`)** are out of scope for the audit. Test helpers stay
  as-is unless an exported test helper is intended for reuse across packages
  (none currently exist).
- **Generated files** (`migrations/embed.go` is the only embed-style file; it
  declares one exported `var MigrationsFS embed.FS`) get a one-line comment but
  no further edits.
- **Build tags / GOOS-specific files** (`cmd/wasm`, `internal/wasm/*`) are in
  scope; the auditor should run `GOOS=js GOARCH=wasm go vet ./internal/wasm/...`
  in addition to the default `go vet` to confirm WASM-tagged exports still
  build clean.
- **`pkg/api/`** is the only `pkg/` subtree — these are wire DTOs shared by
  service and WASM client; they need godoc because they are the public contract.

## Tasks

### Task 1: Audit `internal/domain` and `pkg/api`
- **Description:** Smallest surface, zero internal dependencies. Add or correct
  godoc on every exported type, field, function, and constant in
  `internal/domain/{executionhistory.go, ratesource.go, rateuserevent.go,
  rateusersubscription.go, ratevalue.go, user.go}` and in `pkg/api/{events.go,
  history.go, me.go, notifications.go, rates.go, sources.go, stats.go,
  subscriptions.go}`. Add a package comment to `internal/domain` (`// Package
  domain defines value types shared across the application.`) and to `pkg/api`
  (`// Package api defines the JSON DTOs exchanged between the HTTP server and
  the WASM client.`) if either is missing.
- **Acceptance Criteria:**
  - [ ] Every exported identifier in `internal/domain` has a godoc starting with
    the identifier name and ending with a period.
  - [ ] Every exported identifier in `pkg/api` has a godoc starting with the
    identifier name and ending with a period.
  - [ ] Each of the two packages has exactly one `// Package <name>` comment.
  - [ ] `go vet ./internal/domain/... ./pkg/api/...` is clean.
  - [ ] `make test` still passes.
- **Pitfalls & edge cases:** DTO struct fields in `pkg/api` are exported. Godoc
  convention does **not** require a comment per field, but a JSON DTO is part
  of the public wire contract — add field comments only where the meaning is
  not obvious from the field name + json tag (e.g. `Period string` deserves
  `// Period is one of "week", "month", "year".`). Do not bloat trivially-named
  fields like `ID`, `Name`, `CreatedAt`.
- **Complexity:** Easy

### Task 2: Audit `internal/repository`
- **Description:** All persistence code. Files: `executionhistory.go`,
  `ratesource.go`, `rateuserevent.go`, `rateusersubscription.go`, `ratevalue.go`,
  `stats.go`, `mian.go` (note: typo in filename; do **not** rename — this is a
  doc pass). Each repository struct, its `New<Repo>` constructor, every CRUD
  method, every exported helper function, and every exported `const` for table
  / column names gets a godoc.
- **Acceptance Criteria:**
  - [ ] Every exported identifier in `internal/repository` has a godoc starting
    with the identifier name and ending with a period.
  - [ ] Each repository constructor's godoc names the table the repo operates
    on (e.g. `// NewRateSourceRepository returns a repository for the
    rate_sources table.`).
  - [ ] Each query method's godoc documents its filter semantics and what it
    returns when no row matches (sentinel error vs zero value vs nil slice).
  - [ ] `go vet ./internal/repository/...` is clean.
  - [ ] `make test` still passes.
- **Pitfalls & edge cases:** The `const` blocks for table and column names
  (per CLAUDE.md these gate schema renames at compile time) are unexported in
  most files — skip them. Only add comments to **exported** consts. Do not edit
  SQL string literals or method bodies.
- **Complexity:** Medium
- **Code Example:**
  ```go
  // NewRateSourceRepository returns a repository for the rate_sources table.
  // The returned value is safe for concurrent use; each call serialises
  // through the underlying *sqlitedb.DB.
  func NewRateSourceRepository(db *sqlitedb.DB) *RateSourceRepository { ... }

  // FindByName returns the rate source with the given name. It returns
  // sql.ErrNoRows if no row matches.
  func (r *RateSourceRepository) FindByName(ctx context.Context, name string) (domain.RateSource, error) { ... }
  ```

### Task 3: Audit `internal/application/service`
- **Description:** The two service files exposed via HTTP / Telegram:
  `raterestapi.go`, `telegramapi.go`, `telegramapi_regen.go`. Document every
  exported service struct, its constructor, every public method, and every
  exported error sentinel. Service methods are the public contract used by the
  gateway layer — godoc must spell out which errors are `internal.PublicError`
  (safe to show to user) versus plain errors (mapped to the fallback message
  per the project's error-handling contract in CLAUDE.md).
- **Acceptance Criteria:**
  - [ ] Every exported identifier in `internal/application/service` has a godoc
    starting with the identifier name and ending with a period.
  - [ ] Each method that can return a `PublicError` says so in its godoc
    (e.g. "Returns a PublicError when the source name is unknown.").
  - [ ] `go vet ./internal/application/service/...` is clean.
  - [ ] `make test` still passes.
- **Pitfalls & edge cases:** Several service methods accept large config
  structs as parameters — document the meaningful fields in the struct's own
  godoc, not in every method that consumes it. Do not duplicate.
- **Complexity:** Medium

### Task 4: Audit `internal/application/{collection, notification, rulegen, labelfmt, sourceaudit}`
- **Description:** The remaining application-layer packages. `rulegen` is the
  largest (`chromedpfetcher.go`, `executor.go`, `generator.go`, `locks.go`,
  `plausibility.go`, `prompt.go`, `sanitizer.go`). Document every exported
  type, constructor, method, and function. Each of these five packages needs
  a `// Package <name>` comment if missing.
- **Acceptance Criteria:**
  - [ ] Every exported identifier across the five packages has a godoc starting
    with the identifier name and ending with a period.
  - [ ] Each of `collection`, `notification`, `rulegen`, `labelfmt`,
    `sourceaudit` has exactly one `// Package <name>` comment.
  - [ ] `go vet ./internal/application/...` is clean.
  - [ ] `make test` still passes.
- **Pitfalls & edge cases:** `rulegen` has chromedp interactions and locks —
  concurrency / cancellation behavior must be documented on exported types
  (e.g. "Generator is safe for concurrent use; only one generation per source
  name runs at a time, gated by the locks in this package."). Do not rewrite
  any existing WHY-comments that document chromedp timing decisions — they
  came from prior incidents.
- **Complexity:** Medium

### Task 5: Audit `internal/gateway` and `internal/gateway/httpV1{, /handlers, /routes, /swagger}`
- **Description:** Document the gateway composition entry point (`gateway.go`),
  the v1 router (`httpV1/router.go` or equivalent), and every exported handler
  in `httpV1/handlers/{handlers.go, rulegen_handler.go}`. Each handler's godoc
  names the route it serves and any auth requirement (e.g. "Serves GET
  /api/me/subscriptions; requires Telegram WebApp initData auth.").
- **Acceptance Criteria:**
  - [ ] Every exported identifier in `internal/gateway/...` has a godoc starting
    with the identifier name and ending with a period.
  - [ ] Every handler godoc names the HTTP method + route it serves.
  - [ ] Every handler godoc that requires auth (initData HMAC, admin chat-id
    check) says so explicitly.
  - [ ] `go vet ./internal/gateway/...` is clean.
  - [ ] `make test` still passes.
- **Pitfalls & edge cases:** Swagger generation tags may live in struct
  comments — leave any `// @...` swagger directives intact. The error-handling
  contract from CLAUDE.md applies here: handler godocs should mention that
  unexpected errors map to the fallback message; PublicError instances are
  surfaced verbatim.
- **Complexity:** Medium

### Task 6: Audit `internal/infrastructure/{sqlitedb, sqlitedb/sqlitedbtest, telegrambot, artificialintelligence, scheduler}`
- **Description:** External-client code. Each package likely exposes a
  constructor (`Open<X>`, `New<X>`) that owns a resource lifecycle — document
  the Close / cancellation contract explicitly. For `sqlitedbtest` (test
  helpers), document each exported helper since these are designed to be
  imported by other packages' tests.
- **Acceptance Criteria:**
  - [ ] Every exported identifier across the five infrastructure packages has a
    godoc starting with the identifier name and ending with a period.
  - [ ] Each constructor names the resource it owns and who is responsible for
    closing it.
  - [ ] `go vet ./internal/infrastructure/...` is clean.
  - [ ] `make test` still passes.
- **Pitfalls & edge cases:** `sqlitedb` exposes `Migrator`, `DB`,
  `RequireMigratedSchema`, and the transaction helper — these are referenced by
  name in CLAUDE.md so the godoc must match the documented behavior (e.g.
  `RequireMigratedSchema` "returns an error if the __schema_migrations table is
  missing or empty"). The `stubSQLiteDB` mutex in `sqlitedbtest` is mentioned in
  agent memory (`feedback_keep_sqlite_test_mutex.md`) as load-bearing; if a
  comment touches it, it must explain *why* the mutex is required, not strip it.
- **Complexity:** Medium

### Task 7: Audit `internal/tools/{tgwebapp, rateextractor, rateforecaster, threadsafe}`
- **Description:** Cross-cutting utilities. `tgwebapp` houses the initData HMAC
  validator referenced by CLAUDE.md — its godoc must document the
  `secret_key = HMAC_SHA256("WebAppData", botToken)` derivation so future
  readers do not reinvent it. `threadsafe` is concurrency primitives —
  document the synchronisation guarantee on every exported type.
- **Acceptance Criteria:**
  - [ ] Every exported identifier across the four tool packages has a godoc
    starting with the identifier name and ending with a period.
  - [ ] The initData validator godoc names the signing algorithm and the
    header / query-string fallback the gateway uses.
  - [ ] Every `threadsafe` exported type documents its concurrency guarantee
    (e.g. "Safe for concurrent use by multiple goroutines.").
  - [ ] `go vet ./internal/tools/...` is clean.
  - [ ] `make test` still passes.
- **Pitfalls & edge cases:** Do not move or rename the initData validator —
  CLAUDE.md and several handlers reference it by path. Do not change the
  algorithm description in the comment unless the implementation actually
  differs from CLAUDE.md (in which case escalate to the user; do not
  silently "fix" either side in a doc pass).
- **Complexity:** Easy

### Task 8: Audit `internal/wasm/{apiclient, application, dom, ui}` and `cmd/wasm`
- **Description:** WASM-tagged client code. Same rules apply; verify with
  `GOOS=js GOARCH=wasm go vet ./internal/wasm/... ./cmd/wasm/...` since the
  default `go vet` will skip wasm-tagged files.
- **Acceptance Criteria:**
  - [ ] Every exported identifier in `internal/wasm/...` and `cmd/wasm/` has
    a godoc starting with the identifier name and ending with a period.
  - [ ] Each of the four `internal/wasm/*` packages has a `// Package <name>`
    comment.
  - [ ] `GOOS=js GOARCH=wasm go vet ./internal/wasm/... ./cmd/wasm/...` is
    clean.
  - [ ] `make test` still passes (host tests do not exercise WASM packages
    directly; the wasm vet is the verification).
- **Pitfalls & edge cases:** `syscall/js` types do not lend themselves to
  short comments — keep godocs focused on intent (what the wrapper does), not
  on the JS interop mechanics. `dom` likely has small wrappers — do not bulk
  add `// X wraps js.Value` comments; document only the non-obvious ones.
- **Complexity:** Easy

### Task 9: Audit `cmd/{web, collector, notifier, migrator, rulegen, sourceaudit}`
- **Description:** Entry points. These are `main` packages so almost no
  exported identifiers exist by definition. The work is: (a) ensure each `main`
  package has a one-line `// Command <name> ...` comment above the `package
  main` declaration in the file that holds `func main()`; (b) any exported
  helper that has leaked into a cmd package gets a godoc. Convention for
  command docs is `// Command web serves the HTTP API and the embedded Mini
  App static files.` etc.
- **Acceptance Criteria:**
  - [ ] Every `cmd/*/main.go` (or equivalent entry file) starts with a
    `// Command <name> ...` doc comment.
  - [ ] Every exported identifier in any `cmd/` subpackage has a godoc.
  - [ ] `go vet ./cmd/...` is clean (skip `cmd/wasm` which Task 8 covers under
    WASM tags).
  - [ ] `make test` still passes.
- **Pitfalls & edge cases:** Do not relocate `func main()` to a `doc.go`; the
  command doc lives on whichever file already holds `package main`. If a cmd
  binary has multiple `.go` files only one of them needs the command-level
  doc.
- **Complexity:** Easy

### Task 10: Audit top-level `internal` (`errors.go`, `logger.go`) and `migrations`
- **Description:** Document the `internal` package itself (the top-level files
  `internal/errors.go` and `internal/logger.go`). Specifically: `PublicError`
  and `NewPublicError` are referenced in the project's error-handling contract
  in CLAUDE.md — their godoc must spell out the public-vs-internal split.
  Document the exported logger interface / constructor. In `migrations/`,
  document the package and the single exported `MigrationsFS`.
- **Acceptance Criteria:**
  - [ ] `internal` package has a `// Package internal ...` comment on whichever
    of `errors.go` / `logger.go` is the natural anchor.
  - [ ] `PublicError`, `NewPublicError`, and any `Details()` / `Error()` methods
    have godoc that aligns with the error-handling table in CLAUDE.md.
  - [ ] The logger's exported type and constructor have godoc.
  - [ ] `migrations` package has a `// Package migrations embeds the SQL files
    applied by cmd/migrator.` comment, and `MigrationsFS` has a one-line
    godoc.
  - [ ] `go vet ./internal ./migrations` is clean.
  - [ ] `make test` still passes.
- **Pitfalls & edge cases:** The `PublicError` docstring is the contract — keep
  it terse and aligned with the CLAUDE.md error-handling section. Do not
  describe behavior the type does not actually have.
- **Complexity:** Easy

### Task 11: Full-repo verification and gap sweep
- **Description:** Run a final pass across the whole repository to catch
  anything Tasks 1–10 missed. Use `go vet` and a grep-based check
  (`golang.org/x/lint/golint` is deprecated, so a manual sweep over
  `go doc -all <pkg>` output is the verification of choice). Add any missing
  `// Package <name>` comments. Re-run `make test`. If new packages have been
  added during the audit window, fold them in here.
- **Acceptance Criteria:**
  - [ ] `go vet ./...` is clean across the whole module.
  - [ ] `GOOS=js GOARCH=wasm go vet ./internal/wasm/... ./cmd/wasm/...` is
    clean.
  - [ ] `make test` passes.
  - [ ] `make lint` passes.
  - [ ] Every package has exactly one package-level doc comment.
  - [ ] Spot-check: `go doc <pkg>` output for `internal/domain`,
    `internal/repository`, `internal/application/service`,
    `internal/gateway/httpV1/handlers`, `pkg/api`, `internal/tools/tgwebapp`
    shows a non-empty doc for every exported identifier.
- **Pitfalls & edge cases:** Resist the urge to refactor anything noticed
  during the sweep. File any non-doc issues in a follow-up plan, do not fix
  them here. Mark the plan complete by moving it to `plans/completed/` per
  the lifecycle in CLAUDE.md.
- **Complexity:** Easy

## Execution Order

Strictly sequential, smallest-surface and zero-dependency packages first so
later waves can rely on a known-clean foundation. Waves 2–10 are independent of
each other in terms of source dependencies (they all build on Wave 1's domain
types), but landing them in order keeps each PR small and reviewable.

1. Task 1 — `internal/domain`, `pkg/api` (foundation; everything imports
   `domain`).
2. Task 2 — `internal/repository` (depends on `domain`).
3. Task 3 — `internal/application/service` (depends on `repository`, `domain`).
4. Task 4 — `internal/application/{collection, notification, rulegen,
   labelfmt, sourceaudit}` (orchestration packages).
5. Task 5 — `internal/gateway/...` (HTTP surface; depends on service).
6. Task 6 — `internal/infrastructure/...` (external clients).
7. Task 7 — `internal/tools/...` (utilities).
8. Task 8 — `internal/wasm/...` and `cmd/wasm` (WASM-tagged; requires the
   wasm-specific vet step).
9. Task 9 — `cmd/{web, collector, notifier, migrator, rulegen, sourceaudit}`
   (entry points; trivial surface but needs `// Command` doc lines).
10. Task 10 — top-level `internal` and `migrations`.
11. Task 11 — full-repo verification + cleanup.

Tasks 2 through 10 could in principle be parallelised (no source-level
dependency between them once Wave 1 is in), but the user gets a cleaner review
trail by landing them in order. If two engineers are available, the safe
parallel split is: one engineer on Tasks 2–5 (server side), the other on
Tasks 6–10 (infrastructure / tools / wasm / cmds). Task 11 must run last in
either case.

## Risks

- **Scope creep.** Some packages (notably `internal/application/rulegen` and
  `internal/application/service`) have many exported methods. A literal
  one-comment-per-method pass can balloon into hundreds of lines of diff per
  PR. Mitigation: bias toward terse, single-line godocs. Reject any comment
  that is just a restatement of the signature — leave the symbol uncommented
  rather than adding noise.
- **Overwriting load-bearing existing comments.** Some files already carry
  WHY-comments that document non-obvious decisions (e.g. the chromedp wait
  strategy, the `stubSQLiteDB` mutex per agent memory
  `feedback_keep_sqlite_test_mutex.md`, the `dsninjector` parse contract).
  Mitigation: read each comment before editing; if it is substantive, keep it
  verbatim and only adjust the leading sentence if the godoc convention
  requires it.
- **Verbose-doc bounce-back from reviewer.** The `gocode-reviewer` will flag
  any comment that violates `CLAUDE.md`'s "no fluff" rule. Mitigation: every
  task explicitly lists "no fluff comments on unexported helpers" as a
  pitfall.
- **Behavioral drift via comment edits.** Touching a comment can lead the
  engineer to "fix" the surrounding code. Mitigation: this plan is
  documentation-only. Any code change discovered as necessary during the
  audit goes into a separate follow-up plan; the engineer flags it and
  proceeds without applying it.
- **WASM tags hide stale code.** Default `go vet ./...` skips `//go:build js`
  files, so undocumented exports there will not surface until Task 8 runs the
  WASM-tagged vet. Mitigation: Task 8 makes the wasm vet step explicit; Task
  11 reruns it as a final guard.
- **Schema-rename-by-rename.** The repository constants are intentionally
  exported in some places to gate schema renames at compile time. A doc edit
  that accidentally renames a constant breaks the compile-time guard. Mitigation:
  edits in Task 2 are restricted to comment text only — no identifier renames,
  no SQL changes.

## Trade-offs

- **Full repo audit vs spot-fix-on-touch.** The user explicitly asked for the
  full-audit option, so we chose full audit. Spot-fix-on-touch would have been
  cheaper but leaves an unknown amount of undocumented public surface in
  packages that are rarely modified (e.g. `pkg/api`, `internal/tools/tgwebapp`).
- **One PR per wave vs one giant PR.** Per-wave PRs picked because (a) the
  reviewer can verify each package's doc style against CLAUDE.md without
  scrolling through hundreds of files, (b) any regression can be reverted at
  the wave granularity. The trade-off is more PR overhead.
- **Per-field godocs on `pkg/api` DTOs.** Chose "comment only non-obvious
  fields" rather than "comment every field" to honor the no-fluff rule. The
  cost is some loss of self-documentation in the public wire DTO; mitigated by
  documenting the parent struct thoroughly.
- **`// Package` anchor on existing primary file vs new `doc.go`.** Chose the
  existing primary file to avoid churn. The trade-off is that the package doc
  is co-located with implementation rather than separately discoverable.
  Project precedent: no existing `doc.go` files in the repo.
- **Skip unexported symbols by default.** Idiomatic Go does not require doc on
  unexported identifiers; adding them universally would have produced a
  significantly larger diff for very little reader benefit. Engineers may
  still add comments on unexported helpers where the intent is genuinely
  non-obvious — judgement call, documented per-comment.
