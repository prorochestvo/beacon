# Task Breakdown

## Overview

Add a Telegram bot command `/regen <source> [--force-fallback] [--max-fallback=N]` that lets the admin trigger the rule generator from a Telegram DM and receive the outcome as a chat reply. The HTTP endpoint `POST /api/sources/{name}/rules/generate` (plan 015) and the BID/ASK prompt semantics (plan 016) are already in place; this plan wires a third invocation surface — the bot itself — into the same `rulegen.Generator` that the HTTP handler uses.

The bot's update dispatcher already exists in `internal/application/service/telegramapi.go`. `TelegramApi.Run` (line 43) starts a long-poll goroutine and routes incoming messages through `handleMessage` (line 65). That function already handles `/start` and `/subscriptions`; `/regen` becomes a third command, but is gated on admin chat id so non-admins fall through to the existing "Please use /subscriptions to start." fallback.

Two implementation paths were considered:

- **(a) Direct dispatch** — `TelegramApi` holds a `rulegen.Generator` and calls `Generate(...)` directly when it sees `/regen`.
- **(b) HTTP loopback** — the bot synthesises a Telegram `initData` payload, signs it with the bot token, and POSTs to `http://localhost:<port>/api/sources/{name}/rules/generate`.

This plan locks in **(a) direct dispatch**. Reasons in the Trade-offs section.

Path (a) requires the per-source lock to be shared between the HTTP handler and the Telegram command — otherwise a bot-initiated regen could race an HTTP-initiated regen for the same source. The `rulegenLockManager` therefore moves from `internal/gateway/httpV1/handlers/rulegen_locks.go` to `internal/application/rulegen/locks.go` and is constructed once in `cmd/web/main.go` and injected into both the HTTP handler and `TelegramApi`.

## Assumptions

- The bot client `*integration.TelegramBotClient` runs inside `cmd/web` only. `cmd/notifier` constructs the same client for outbound notifications but never calls `Listen` and does not need `/regen` support.
- The update receive-loop is `(*TelegramBotClient).Listen(ctx, handler)` in `internal/infrastructure/telegrambot/tbotclient.go:197`. The dispatcher that wraps it is `(*TelegramApi).Run(ctx)` in `internal/application/service/telegramapi.go:43`, which kicks off `go h.telegramClient.Listen(ctx, handle)` and routes messages through `handleMessage`.
- Existing message replies use HTML (`SendHTMLMessage`, `EditHTMLMessageWithKeyboard`) — `/regen` will follow the same convention.
- `(*TelegramBotClient).EditMessageText(ctx, chatID, messageID, text)` already exists (line 159) and uses `ParseMode = ModeHTML`. "Working on it…" can be sent first, then edited in place with the final result.
- `TelegramBotClient.AdminChatID() int64` is the source of truth for the admin gate, matching how the HTTP handler authorises in `rulegen_handler.go:87`.
- The `rulegen.Generator` and the factory closure `buildGenerator(maxPrimary, maxFallback int) (v1handlers.RulegenGenerator, error)` already exist in `cmd/web/main.go:181-190` and are passed into the handlers there. They can be passed into the new `TelegramApi` constructor the same way.
- No new third-party dependency is required. `github.com/OvyFlash/telegram-bot-api` and `github.com/seilbekskindirov/monitor/internal/application/rulegen` are already in `go.mod`.
- Admin-mismatch behaviour: silently fall through to the existing `handleMessage` default branch (which prints "Please use /subscriptions to start."). This is the least-information-leak option and matches how `/start` typos are handled today.
- The bot replies in HTML. Source names contain `_` characters (e.g. `KZ_QAZPOST_BID_USD_KZT`), which are safe in HTML but require escaping in MarkdownV2 — HTML wins for fewer foot-guns.

### Task 3: Implement `/regen` command parser

- **Description:** Add a pure function `parseRegenCommand(text string) (sourceName string, forceFallback bool, maxFallback int, err error)` in a new file `internal/application/service/telegramapi_regen.go`. Inputs and expected outputs:

  | Input | sourceName | forceFallback | maxFallback | err |
  |-------|------------|---------------|-------------|-----|
  | `/regen KZ_QAZPOST_BID_USD_KZT` | `KZ_QAZPOST_BID_USD_KZT` | `false` | `0` | nil |
  | `/regen KZ_QAZPOST_BID_USD_KZT --force-fallback` | same | `true` | `0` | nil |
  | `/regen KZ_QAZPOST_BID_USD_KZT --max-fallback=4` | same | `false` | `4` | nil |
  | `/regen KZ_QAZPOST_BID_USD_KZT --force-fallback --max-fallback=2` | same | `true` | `2` | nil |
  | `/regen` | "" | false | 0 | `errMissingSourceName` |
  | `/regen X Y` | "" | false | 0 | `errTooManyPositionalArgs` |
  | `/regen X --unknown` | "" | false | 0 | `errUnknownFlag` |
  | `/regen X --max-fallback=abc` | "" | false | 0 | `errInvalidMaxFallback` |
  | `/regen X --max-fallback=0` | "" | false | 0 | `errInvalidMaxFallback` (must be 1..10) |
  | `/regen X --max-fallback=11` | "" | false | 0 | `errInvalidMaxFallback` |

  Whitespace-split with `strings.Fields(text)`. Reject any token after the source name that does not start with `--`. Errors are sentinel `var err... = errors.New(...)` values so the dispatch handler can produce a tailored reply per error kind without `strings.Contains`-style sniffing.
- **Acceptance Criteria:**
  - [ ] `parseRegenCommand` lives in `internal/application/service/telegramapi_regen.go`, is package-private, and has zero dependencies on `tgbotapi` or any I/O.
  - [ ] Sentinel errors are defined as package-private vars: `errMissingSourceName`, `errTooManyPositionalArgs`, `errUnknownFlag`, `errInvalidMaxFallback`.
  - [ ] Unit tests in `internal/application/service/telegramapi_regen_test.go` cover all rows from the table above using one `TestParseRegenCommand` function with `t.Run` subtests (per CLAUDE.md "one Test* per method").
  - [ ] Bound check on `maxFallback`: only `1..10` is valid (mirrors the HTTP handler's bound in `rulegen_handler.go:103`).
  - [ ] All tests pass with `go test -race`.
- **Pitfalls & edge cases:**
  - `/REGEN X` and `/Regen X` — accept any casing of the command word, but treat source names case-sensitively (seed catalog uses upper-case names like `KZ_QAZPOST_BID_USD_KZT`; normalising would mask real typos).
  - Telegram may append the bot's @username to the command in groups (`/regen@my_bot KZ_QAZPOST_...`). Strip `@<anything>` from the first token before comparing.
  - `--max-fallback=` with no value (empty after `=`) should match `errInvalidMaxFallback`, not `errUnknownFlag`. Test it.
  - The parser must NOT call `log.Printf` — errors are sentinel values, the caller decides what to log.
- **Complexity:** Easy
- **Code Example:**
  ```go
  // parseRegenCommand splits the command text and extracts the source name + flags.
  // Returns one of the package-private sentinel errors on malformed input; the
  // caller maps each sentinel to a user-facing reply.
  func parseRegenCommand(text string) (string, bool, int, error) {
      fields := strings.Fields(text)
      if len(fields) == 0 {
          return "", false, 0, errMissingSourceName
      }
      // strip @botname suffix from first token
      head := fields[0]
      if i := strings.IndexByte(head, '@'); i >= 0 {
          head = head[:i]
      }
      if !strings.EqualFold(head, "/regen") {
          return "", false, 0, errMissingSourceName
      }
      if len(fields) < 2 {
          return "", false, 0, errMissingSourceName
      }
      sourceName := fields[1]

      var (
          forceFallback bool
          maxFallback   int
      )
      for _, tok := range fields[2:] {
          switch {
          case tok == "--force-fallback":
              forceFallback = true
          case strings.HasPrefix(tok, "--max-fallback="):
              raw := strings.TrimPrefix(tok, "--max-fallback=")
              n, err := strconv.Atoi(raw)
              if err != nil || n < 1 || n > 10 {
                  return "", false, 0, errInvalidMaxFallback
              }
              maxFallback = n
          case strings.HasPrefix(tok, "--"):
              return "", false, 0, errUnknownFlag
          default:
              return "", false, 0, errTooManyPositionalArgs
          }
      }
      return sourceName, forceFallback, maxFallback, nil
  }
  ```

### Task 4: Wire `/regen` into the message dispatcher and implement the handler

- **Description:** Extend `handleMessage` in `internal/application/service/telegramapi.go:65` to detect `/regen` (case-insensitive on the first token, allow `@botname` suffix) and route to a new method `handleRegen(ctx, msg)`. The new method lives in `internal/application/service/telegramapi_regen.go`.

  Behaviour:
  1. Guard: if `msg.Chat.ID != h.adminChatID` → fall through silently (no reply). Non-admins get nothing back, matching the principle of minimum information leak.
  2. Guard: if `h.rulegenGenerator == nil` (regen disabled in this build) → reply "Rule regeneration is not enabled." and return.
  3. Parse command via `parseRegenCommand(msg.Text)`. On error, reply with a one-liner mapped per sentinel:
     - `errMissingSourceName` → `Usage: /regen <source> [--force-fallback] [--max-fallback=N]`
     - `errTooManyPositionalArgs` → `Too many arguments. Expected exactly one source name.`
     - `errUnknownFlag` → `Unknown flag. Supported: --force-fallback, --max-fallback=N (1..10).`
     - `errInvalidMaxFallback` → `--max-fallback must be an integer between 1 and 10.`
  4. Try to acquire the lock: `release, ok := h.rulegenLocks.TryAcquire(sourceName)`. On `!ok` reply `Rule generation already in progress for <code>{source}</code>.` and return.
  5. Send "Working on rules for `<source>`…" via `SendHTMLMessage`. Capture the returned `messageID` so the handler can `EditMessageText` later. NOTE: the existing `SendHTMLMessage` does not return the message id — see the implementation pitfall below.
  6. Spawn a goroutine that:
     a. `defer release()`.
     b. Picks the generator: default if `maxFallback == 0`, otherwise calls `h.generatorFactory(3, maxFallback)`. If factory errors, edit the "Working on…" message with `Failed to build generator: {err}` and return.
     c. Creates a derived context with the same 120-second timeout the HTTP handler uses (`rulegen_handler.go:41`). Use `context.WithTimeout(context.Background(), 120*time.Second)` — NOT the bot's `ctx`, because the bot's ctx is cancelled when long-poll stops on shutdown, which would orphan an in-flight regen. Background context with explicit timeout is safe because (i) the underlying LLM client and fetchers respect their own per-request timeouts, and (ii) the persistence step is idempotent — even an orphaned-but-completed call leaves the DB consistent.
     d. Calls `gen.Generate(timeoutCtx, sourceName, forceFallback)`.
     e. Builds the final reply (success or failure HTML body, see formats below) and calls `EditMessageText(ctx, chatID, workingMsgID, finalText)`.
  7. Reply formats:
     - Success:
       ```
       ✅ <code>{source}</code> → <b>{value}</b>
       provider: {provider} [{model}]
       attempts: {n} ({"fallback" or "no fallback"})
       ```
     - Failure (mapped from `errors.Is`):
       - `rulegen.ErrSourceNotFound` → `❌ <code>{source}</code>: source not found`
       - `rulegen.ErrAttemptsExhausted` → `❌ <code>{source}</code>: all attempts exhausted (primary=3, fallback={configured})`
       - `rulegen.ErrUnsupportedFetcherKind` → `❌ <code>{source}</code>: required fetcher not available in this build`
       - context deadline → `❌ <code>{source}</code>: timed out after 120s`
       - any other error → `❌ <code>{source}</code>: internal error (see logs)` AND `log.Print(...)` the full error
  8. Source names should be escaped before placing inside `<code>...</code>` — Telegram HTML reserves `<`, `>`, `&`. Use a small inline escape helper or `html.EscapeString`. The seeded source names contain only `A-Z`, `_`, `0-9` so escaping is defensive, but do it anyway.

- **Acceptance Criteria:**
  - [ ] `handleMessage` routes `/regen`, `/Regen`, `/regen@botname`, etc. to `handleRegen`.
  - [ ] Non-admin senders get no reply for `/regen` (silent ignore at the first guard).
  - [ ] When regen is disabled (generator is nil), admin gets "Rule regeneration is not enabled." reply.
  - [ ] On parse error, the correct per-sentinel message is sent.
  - [ ] On lock contention, the "already in progress" message is sent.
  - [ ] On success, the "Working on…" message is **edited in place** with the success format above — no second message is sent.
  - [ ] On failure, the "Working on…" message is edited with the failure format.
  - [ ] The long-running call runs in a goroutine; `handleMessage` returns immediately after spawning it.
  - [ ] The goroutine uses `context.Background()` + a 120 s timeout (NOT the bot's `ctx`).
  - [ ] The lock is released via `defer release()` inside the goroutine — verified by a test that calls `handleRegen` twice in sequence for the same source and observes the second call succeeds (not lock-contended) once the first finishes.
  - [ ] `make test` and `make lint` pass.
- **Pitfalls & edge cases:**
  - **`SendHTMLMessage` does not return the message id.** The current method signature is `SendHTMLMessage(ctx, chatID, text) error` — the message id is discarded inside `emit`. To support send-then-edit we need a new method on `TelegramBotClient`: `SendHTMLMessageReturning(ctx, chatID, text) (messageID int, err error)`. Add it alongside the existing method (do NOT modify the existing one — `notification.RateDispatchAgent` and the existing `notifyText` would break). Also extend the `telegramClient` interface in `telegramapi.go:591` to add the new method.
  - **`context.Background()` for the goroutine.** Plan-015's HTTP handler uses `r.Context()` because the HTTP request must die when the client disconnects. The Telegram command has no client to disconnect — the admin sent a message and walked away. Using the bot's `ctx` would cancel the regen on graceful shutdown mid-call, leaving the user with neither a success nor a failure reply. Background + timeout is correct.
  - **`log.Printf` inside the goroutine** is the only logging surface — there is no request-scoped log context. Match the existing `log.Printf("telegram: …")` prefix used throughout `telegramapi.go`.
  - **No panic recovery in the goroutine.** Per CLAUDE.md ("No skipped errors"), do not silently `recover()` panics. The bot process should fail fast if rulegen panics — this is an admin-only command, not a hot path.
  - **Telegram has a 4096-char message limit.** A long error message inside `<code>` could exceed it. Source names are short and error strings are short — but truncate the final reply to 4000 chars defensively.
  - **The "Working on…" send can itself fail** (network blip). If `SendHTMLMessageReturning` returns an error, log it and DO NOT proceed to `Generate` — the user got no feedback and the lock is held; release and abort. Test this path.
- **Complexity:** Hard

## Risks

1. **Send-then-edit requires a new bot client method.** Adding `SendHTMLMessageReturning` to `*TelegramBotClient` is a non-trivial signature change. Risk: other callers grow to depend on the new method and the older `SendHTMLMessage` becomes ambiguous. **Mitigation:** keep both methods; do NOT delete the existing one. The new method is opt-in for callers that need the message id.
2. **`telegramClient` interface in `telegramapi.go:591` becomes one method larger.** Every test mock implementing it gains a new method. **Mitigation:** Task 5's compile-time interface checks (`var _ telegramClient = ...`) catch this at build time — failures are not silent.
3. **Background context decision could orphan the rulegen call on a panicked bot process.** If the bot crashes mid-call, the rulegen goroutine continues until the 120 s timeout and the DB write either lands or doesn't. **Mitigation:** `RetainRateSource` is idempotent (it's an upsert); a partial completion is consistent — the audit log still records the attempt.
4. **Telegram's flood-control may rate-limit edits if the admin spams `/regen`.** The bot will receive "Too Many Requests" from Telegram. **Mitigation:** the per-source lock prevents same-source spam; cross-source spam is bounded by the admin's typing speed. Out of scope to add explicit rate-limiting per CLAUDE.md note.
5. **Group-chat usage.** If the admin invites the bot to a group and types `/regen`, the bot sees `update.Message.Chat.ID == <group chat id>`, NOT the admin's personal chat id. The admin gate fails, command is silently ignored. **Mitigation:** explicit decision — `/regen` is DM-only. Document in `cmd/rulegen/README.md`.

## Trade-offs

- **Direct dispatch (path a) over HTTP loopback (path b).** Chose direct dispatch because:
  - The Generator is already constructed in `cmd/web/main.go:190` and threaded into the HTTP handler. Threading it into `TelegramApi` is a one-field extension of the existing constructor.
  - HTTP loopback would require synthesising a Telegram `initData` payload inside the bot — duplicating the validator logic in reverse and burning a network roundtrip on every call.
  - The auth check is a two-line comparison (`msg.Chat.ID == adminChatID`), not a middleware chain.
  - **What we give up:** every guard added to the HTTP handler in the future (rate-limiting, audit logging, idempotency tokens) has to be added in two places — once on the HTTP path, once on the bot path. The shared `LockManager` (Task 1) is the largest such guard today, which is why we extract it. Future shared guards should go alongside it in `internal/application/rulegen`.

- **`LockManager` in `internal/application/rulegen` over `internal/infrastructure` or a dedicated `lockmgr` package.** Chose `rulegen` because the lock is semantic to rule generation — it serialises a specific application-layer operation. Putting it in a generic `lockmgr` package would be premature; we'd need a second use case to justify the abstraction.

- **HTML reply format over MarkdownV2.** Chose HTML because (i) source names contain `_` which is a MarkdownV2 metacharacter requiring escape, (ii) the existing bot code uses HTML throughout (`SendHTMLMessage*`, `EditHTMLMessageWithKeyboard`), (iii) HTML's `<code>` is a clean way to render the source name without inviting copy-paste issues.

- **Send-then-edit over send-then-reply.** Chose send-then-edit because the chat ends up with one message per `/regen` invocation, not two. The cost is a new bot-client method (`SendHTMLMessageReturning`). The benefit is a cleaner chat history for the admin, who may run `/regen` dozens of times during a regeneration sweep.

- **Silent ignore for non-admins over polite "command not available" reply.** Chose silent ignore to match Telegram's principle of minimum information leak — non-admins should not learn that `/regen` exists. The existing `handleMessage` fallback ("Please use /subscriptions to start.") catches them anyway and steers them to the supported flow.

- **`context.Background()` + explicit 120 s timeout for the goroutine, not the bot's ctx.** Chose Background because the long-poll's `ctx` cancels on bot shutdown, which would orphan in-flight regen calls without writing a reply to the admin. The 120 s ceiling matches the HTTP handler's ceiling (`rulegen_handler.go:41`) so the two surfaces have identical generation budgets.

- **No async job queue.** A `/regen` command takes up to 120 s. Keeping it synchronous (admin waits for the edit) is simpler than building a job table + a `/regen-status` command. If the volume grows, revisit — but for an admin-triggered command with one authorised user, synchronous is right.
