# Task Breakdown

## Overview

Second increment of weather monitoring: add **gismeteo** as a comparison
provider alongside the already-shipped Open-Meteo MVP. The user keeps gismeteo
because he trusts its 3–4 day forecast; gismeteo data appears next to Open-Meteo
in the daily morning summary, never replacing it.

The MVP (plan `260630.0001.weather-monitoring.md`, Tasks 1–10 + the Open-Meteo
half of Task 14) is shipped and committed on this branch. This plan delivers the
deferred Tasks 11–14, **regrounded against the code that actually shipped**. The
regrounding changed several details from the original spec — those changes are
called out per task and in Trade-offs.

What the MVP already provides that this increment builds on:

- `domain.ProviderGismeteo` / `domain.ProviderOpenMeteo` constants
  (`internal/domain/weatherobservation.go`).
- `WeatherUserCity.GismeteoCityID *int` field and the `gismeteo_city_id INTEGER`
  column (migration `202606.011`) — already selected and scanned by
  `ObtainDistinctWeatherLocations` and persisted by `RetainWeatherUserCity`.
- `weather_observations` table is provider-agnostic (a `provider TEXT` column,
  all forecast fields nullable) — gismeteo rows need **no schema change**.
- `WeatherObservationRepository.ObtainLatestObservation(ctx, locationID, provider)`
  / `RetainWeatherObservation` already work for any provider.
- `RenderMorningSummary(city, obs ...WeatherObservation)` is **already variadic
  and already renders the two-provider layout**, with passing tests
  (`weatherrender_test.go`: "two providers renders both with labels and content",
  `TestWeatherProviderLabel`). Task 13 is therefore a pure data-path change in the
  check agent — the renderer needs no edit.
- The advisory health-inspector tier shipped:
  `inspector.NewAgentWithAdvisory(timeout, critical, advisory)` and
  `OpenMeteoInspector` (with a `newOpenMeteoInspectorForTest` seam). The gismeteo
  inspector mirrors it exactly.
- `cmd/collector` already reads `BEACON_CHROMIUM_PATH` into `ChromiumPath`,
  resolves `proxyURL`, builds `weatherCityRepo` / `weatherObsRepo` / the
  Open-Meteo provider, and constructs the `WeatherAgent` in `buildRunners`.

Increment shape (five tasks + one doc task; **no migration**):

1. Extract a shared chromedp fetch primitive (`internal/tools/chromedpfetch`).
2. gismeteo provider + curated `location_id → {slug, id}` map
   (`internal/infrastructure/weather/gismeteo.go`).
3. Wire gismeteo into `WeatherAgent` and `cmd/collector` (batched, throttled).
4. Cross-provider comparison: `WeatherCheckAgent` loads both providers and passes
   them to the already-variadic renderer.
5. `GismeteoInspector` (advisory HTTP reachability), registered in `cmd/web`.
6. Documentation updates (CLAUDE.md health-check components + coverage
   limitation).

## Assumptions

- **No new migration.** `gismeteo_city_id` exists (011); `weather_observations`
  already stores any provider with all-nullable forecast fields (012). Migrations
  013–016 remain reserved/free for weather but are **not** used here. Confirmed by
  reading the two migration files and the two weather repositories.
- gismeteo has no public API and no public "city name → numeric id" resolver, so
  coverage is limited to a hand-curated map. This is the documented limitation:
  gismeteo ⊊ Open-Meteo.
- The gismeteo URL shape is `https://www.gismeteo.kz/weather-<slug>-<id>/`. The
  **slug is required to build the URL and is not stored in the database** — only
  the numeric id column exists. Consequence: the curated map (which holds the
  slug) is the source of truth at fetch time, and the map is keyed by
  `location_id` (the same `LocationKey(geo)` value used everywhere else — the
  Open-Meteo geocoding id as a decimal string). See Trade-offs.
- Host = **`.kz`** (Almaty / KZT user base; Russian-language, metric units).
  `.ru` / `.com` differ in default language and unit conventions. Documented.
- gismeteo egress runs only from `cmd/collector` and honours `BEACON_PROXY_URL`
  via the primitive's `chromedp.ProxyServer` option (Chromium does not inherit Go
  proxy env). The `GismeteoInspector` runs from `cmd/web`, which does **not** parse
  `BEACON_PROXY_URL`, so the inspector probes the direct host→gismeteo path —
  same documented caveat as `OpenMeteoInspector`.
- The codebase has **no DOM/CSS parser dependency** (no goquery, no
  `golang.org/x/net/html`); the existing extraction toolkit is regex + JSONPath
  (`internal/tools/rateextractor/matchers.go`). gismeteo field scraping uses
  regex-per-field (the approach the brief compares to "the KASE regex"). See
  Trade-offs for the goquery alternative.
- Telegram is the only delivery channel; the FX `rate_user_events` queue +
  `RateDispatchAgent` carry weather summaries unchanged (decided and shipped in
  the MVP).

## Tasks

### Task 1: Shared chromedp fetch primitive (`internal/tools/chromedpfetch`)

- **Description:** gismeteo needs the headless-Chrome path, but the existing
  `ChromedpRateExtractor` (`internal/tools/rateextractor/chromedp.go`) couples the
  reusable fetch core to the FX rule pipeline (`applyRulesAndStore`) and to
  `*domain.RateSource`. Extract only the fetch core into a new, dependency-light
  package `internal/tools/chromedpfetch` that takes a URL + wait selector and
  returns rendered bytes — no `domain.RateSource`, no repository.

  Expose:
  - `type RenderSpec struct { Key, URL, WaitSelector string }` — `Key` is an
    opaque caller-chosen label (gismeteo passes the `location_id`) used to key the
    result map.
  - `type Result struct { HTML []byte; Err error }`.
  - `Render(ctx context.Context, url, waitSelector string) ([]byte, error)` —
    single URL, own one-shot allocator.
  - `RenderBatch(ctx context.Context, specs []RenderSpec) map[string]Result` —
    shares one `ExecAllocator` (one Chromium subprocess) across the batch, keyed
    by `RenderSpec.Key`.
  - `NewChromedpFetcher(chromiumPath, proxyURL string, logger io.Writer) *ChromedpFetcher`.

  Carry over verbatim from `rateextractor/chromedp.go` (these behaviours are
  load-bearing and already reviewed):
  - `buildExecAllocatorOptions`: `slices.Clone(chromedp.DefaultExecAllocatorOptions[:])`
    + `Headless` + `DisableGPU` + `NoSandbox` + `Flag("disable-blink-features",
    "AutomationControlled")`, then optional `ProxyServer(proxyURL)` and optional
    `ExecPath(chromiumPath)`. Keep `fixedExecAllocatorOptionCount = 4`.
  - `validateNavigableURL`: http/https only, non-empty host, reject userinfo.
  - Per-URL 30 s navigate timeout (`defaultChromedpTimeout`), scoped inside the
    per-URL fetch so one slow URL never starves the batch.
  - Wait strategy: `Navigate` → `WaitVisible("body")` → if `waitSelector != ""`
    then `WaitVisible(waitSelector)` else `Sleep(defaultChromedpNetworkIdleMs=5000)`
    → `OuterHTML("html", ...)`.
  - Per-process failed-URL tombstone (`failedURLs map[string]error` + mutex +
    `loadFailedURL` / `recordFailedURL`); record on both validation failure and
    fetch failure; short-circuit subsequent fetches of the same URL.
  - Empty batch → fast no-op (Chromium never launched); cancelled parent ctx →
    every remaining spec tagged with `ctx.Err()` rather than silently dropped.
  - Cancel the allocator (and its subprocess) via `defer` before return on all
    paths including panic.

  **Rate-extractor delegation is an OPTIONAL, separately-bounded follow-up and is
  NOT recommended for this increment** (see Trade-offs and the Risks section).
  Ship the primitive standalone. The duplication is pre-existing — there are
  already two near-identical copies (`rateextractor/chromedp.go` and
  `rulegen/chromedpfetcher.go`) — and consolidating all three is out of scope.

- **Acceptance Criteria:**
   - [ ] `internal/tools/chromedpfetch` compiles with `Render`, `RenderBatch`,
         `RenderSpec`, `Result`, `NewChromedpFetcher`, godoc on every exported
         identifier (starts with the name, ends with a period).
   - [ ] The package imports neither `internal/domain` nor any repository — it
         depends only on `chromedp`, stdlib, and `internal` (for trace errors).
   - [ ] Tests (mirroring `rateextractor/chromedp_test.go`, no real Chromium):
         empty batch → `nil`/empty map, Chromium not launched; cancelled parent
         ctx tags every spec with `context.Canceled` and includes the spec `Key`;
         `validateNavigableURL` rejects `""`, `file://...`, `javascript:...`, and
         a URL with userinfo; `buildExecAllocatorOptions` option counts for
         `{no-proxy, proxy, proxy+path, path-only}` against
         `len(DefaultExecAllocatorOptions)+fixedExecAllocatorOptionCount`;
         tombstone: a pre-recorded failed URL short-circuits in both `Render` and
         `RenderBatch` and surfaces the cached error (`errors.Is`).
   - [ ] `make test` stays green; `make lint` reports no forbidden imports.
   - [ ] `chromedp_test.go` in `rateextractor` is untouched and still passes
         (because the rate extractor is not modified).
- **Pitfalls & edge cases:** Do not alias `chromedp.DefaultExecAllocatorOptions` —
  use `slices.Clone`, or an upstream length change lets `append` stomp the global.
  The tombstone cache is per-process and built for one-shot binaries; document
  "do not reuse a `ChromedpFetcher` across cron ticks in a long-lived daemon" (the
  collector is one-shot, so this holds). The `--no-sandbox` + shared
  `--user-data-dir` trust boundary must be preserved in the godoc: only batch
  operator-vetted, credential-free https URLs together — gismeteo URLs come from
  the curated map, satisfying this. `RenderBatch` must process specs sequentially
  (browser contexts from one allocator share one CDP websocket and are not
  concurrency-safe).
- **Complexity:** Hard
- **Code Example:**
  ```go
  // RenderBatch fetches every spec under one shared Chromium subprocess and
  // returns a result map keyed by RenderSpec.Key. An empty batch is a fast no-op:
  // Chromium is never launched. A cancelled ctx tags the remaining specs with
  // ctx.Err() so the caller can still record per-key outcomes.
  func (f *ChromedpFetcher) RenderBatch(ctx context.Context, specs []RenderSpec) map[string]Result {
      if len(specs) == 0 {
          return nil
      }
      allocCtx, cancelAlloc := chromedp.NewExecAllocator(ctx, f.buildExecAllocatorOptions(f.proxyURL)...)
      defer cancelAlloc()

      out := make(map[string]Result, len(specs))
      for _, spec := range specs {
          if err := ctx.Err(); err != nil {
              out[spec.Key] = Result{Err: fmt.Errorf("chromedpfetch: key %s: %w", spec.Key, err)}
              continue
          }
          html, err := f.renderInAllocator(allocCtx, spec.URL, spec.WaitSelector)
          out[spec.Key] = Result{HTML: html, Err: err}
      }
      return out
  }
  ```

### Task 2: gismeteo provider + curated city map (`internal/infrastructure/weather/gismeteo.go`)

- **Description:** Add a gismeteo provider in the existing `weather`
  infrastructure package, mirroring the structure of `openmeteo.go`.

  - Curated map, compiled into the binary:
    ```go
    // gismeteoCity holds the gismeteo URL components for one supported city.
    type gismeteoCity struct {
        slug string // gismeteo URL slug, e.g. "almaty-5205" path segment minus the id
        id   int    // gismeteo numeric city id
    }
    // gismeteoCities maps a canonical location_id (the Open-Meteo geocoding id as
    // a decimal string, == LocationKey(geo)) to its gismeteo URL components.
    var gismeteoCities = map[string]gismeteoCity{ /* curated */ }
    ```
    The keys are `LocationKey(geo)` values (Open-Meteo ids) so the map lines up
    with the `location_id` already stored on every `weather_user_cities` row.
  - `Gismeteo` struct + `NewGismeteo(fetcher gismeteoFetcher) *Gismeteo`, where
    `gismeteoFetcher` is a one-method interface (`RenderBatch` / `Render`)
    satisfied by `chromedpfetch.ChromedpFetcher` and stubbable in tests with a
    fixture-returning fake.
  - `Supports(locationID string) bool` — reports map membership.
  - `Forecast(ctx context.Context, locationID string) (*domain.WeatherObservation, error)`
    — resolve the map, build the URL, render via the fetcher, scrape. Returns
    `internal.ErrNotFound` (or a typed "unsupported") when `locationID` is not in
    the map.
  - `ForecastBatch(ctx context.Context, locationIDs []string) (map[string]*domain.WeatherObservation, map[string]error)`
    — filter to supported ids, build one `[]chromedpfetch.RenderSpec` (Key =
    location_id, WaitSelector = the temperature container selector), call the
    fetcher's `RenderBatch` once (one Chromium subprocess), scrape each result.
  - `buildGismeteoURL(c gismeteoCity) string` → `https://www.gismeteo.kz/weather-<slug>-<id>/`.
  - Scrape today's **high / low / precip / condition** into
    `WeatherObservation{Provider: domain.ProviderGismeteo, LocationID: ..., CapturedAt: time.Now().UTC(), ForecastDate: <today in city tz>}`,
    leaving every field gismeteo does not expose as `nil` (NULL). Use
    `decodeGismeteoForecast(html []byte, locationID string)` as a pure,
    fixture-testable decode step (mirrors `decodeOpenMeteoForecast`).
  - **Selectors / regex isolated in named constants** at the top of the file
    (e.g. `selGismeteoTemp`, `reGismeteoTempMax`, `reGismeteoPrecip`,
    `reGismeteoCondition`). Authoring the live patterns against the current
    gismeteo DOM is an **engineer step** (exactly like the KASE regex was) and is
    expected to be iterated with a captured fixture; the plan fixes the structure,
    not the literal patterns.
  - **`ForecastDate`:** gismeteo's page is implicitly "today in the city's local
    day." Derive `ForecastDate` as `time.Now().In(cityLoc).Format("2006-01-02")`.
    The provider does not know the city timezone from a numeric id, so either
    (a) the agent passes the city's `Timezone` into `ForecastBatch`, or (b) the
    provider stores `ForecastDate` as `time.Now().UTC()` date and the check agent
    compares on UTC date. **Recommend (a):** thread the city `*time.Location`
    through so the gismeteo `ForecastDate` matches Open-Meteo's city-local
    `ForecastDate` and the freshness comparison in Task 4 is apples-to-apples.

- **Acceptance Criteria:**
   - [ ] `TestGismeteo_Forecast` / `decodeGismeteoForecast` parse a committed
         rendered-HTML fixture under `internal/infrastructure/weather/testdata/`
         into the expected fields; fields gismeteo omits are `nil` (NULL), not 0;
         a parse failure returns a wrapped error carrying
         `internal.NewTraceError()`, never a panic.
   - [ ] `TestGismeteo_curatedMap` asserts every entry has a non-empty `slug` and
         positive `id`, and that map keys are non-empty.
   - [ ] `Supports` returns false for an unknown `location_id` and the caller can
         still serve the city via Open-Meteo only.
   - [ ] `ForecastBatch` issues exactly one `RenderBatch` call regardless of how
         many supported ids it receives; one failing location appears in the
         returned error map without aborting the others (assert with a fake
         fetcher returning a per-key error).
   - [ ] `Provider == domain.ProviderGismeteo` on every produced observation; the
         literal token is never translated.
   - [ ] gismeteo egress honours the proxy: constructed from the collector's
         `proxyURL`, the fetcher appends `ProxyServer` (covered by the Task 1
         option-count test plus a wiring assertion in Task 3).
- **Pitfalls & edge cases:** **gismeteo DOM is fragile** — a redesign breaks the
  regex; keep every pattern in a named constant and fail loudly (the collector
  logs per-location errors — see Task 3). The numeric `gismeteo_city_id` column
  alone cannot build the URL (no slug), so do **not** drive fetching from the DB
  column — drive it from the map keyed by `location_id`. `.kz` returns
  Russian-language, metric values; parse the numeric value, not the localized
  condition word, for `WeatherCode` (or map a small set of gismeteo condition
  classes to the nearest WMO code — keep that mapping table beside the selectors).
  Chromium cold-start dominates runtime — never call `Forecast` per location in a
  loop from the collector; always go through `ForecastBatch`.
- **Complexity:** Hard

### Task 3: Wire gismeteo into `WeatherAgent` and `cmd/collector`

- **Description:** Extend the collection agent
  (`internal/application/collection/weatheragent.go`) to fetch gismeteo for
  supported, due locations, batched into one Chromium subprocess, then wire it in
  `cmd/collector/main.go`.

  - Add an **optional** gismeteo provider to `WeatherAgent` (nil = gismeteo
    disabled, preserving exact MVP behaviour and keeping existing tests valid):
    ```go
    type weatherGismeteoProvider interface {
        Supports(locationID string) bool
        ForecastBatch(ctx context.Context, locationIDs []string, tz string) (map[string]*domain.WeatherObservation, map[string]error)
    }
    ```
    Add a `gismeteoThrottleInterval time.Duration` and `gismeteo weatherGismeteoProvider`
    field. Update `NewWeatherAgent` to accept both (a nil provider skips the whole
    gismeteo phase).
  - In `Run`, after the existing Open-Meteo loop, add a gismeteo phase:
    1. From the distinct locations, keep those where `gismeteo.Supports(loc.LocationID)`.
    2. Throttle each on its latest **gismeteo** observation
       (`ObtainLatestObservation(locationID, domain.ProviderGismeteo)`), reusing
       the existing `isDue`-style gate but with `gismeteoThrottleInterval`.
    3. If any remain due, call `ForecastBatch` **once** for all of them.
    4. `RetainWeatherObservation` each result under `context.Background()` (same
       SIGTERM-survival rationale as the Open-Meteo path); join per-location
       errors; one failure never aborts the rest.
    5. Extend the per-tick proof-of-execution marker to include gismeteo counts
       (e.g. `weather gismeteo: fetched=.. skipped=.. failed=.. supported=..`).
  - Add `DefaultGismeteoThrottleInterval` (recommend a few hours — Chromium
    cold-start is expensive and gismeteo only feeds the daily comparison). The
    binding constraint: it must be short enough that a same-`forecast_date`
    gismeteo observation reliably exists **before the earliest user `NotifyHour`**
    (the default is 07:00). A 3 h interval with an hourly cron guarantees a
    today-dated gismeteo run by ~06:00; document this constant's reasoning.
  - `cmd/collector/main.go` `buildRunners`: construct
    `chromedpfetch.NewChromedpFetcher(ChromiumPath, proxyURL, logger)`, then
    `weatherinfra.NewGismeteo(fetcher)`, and pass it (plus
    `DefaultGismeteoThrottleInterval`) into `NewWeatherAgent`. `ChromiumPath` and
    `proxyURL` are already resolved in `main` and already passed to the FX
    `NewRateAgent` — reuse them.

- **Acceptance Criteria:**
   - [ ] `TestWeatherAgent_Run` gains gismeteo subtests: nil gismeteo provider →
         behaves exactly as the MVP (no gismeteo calls); one supported + due
         location → one gismeteo observation stored with
         `Provider == domain.ProviderGismeteo`; an unsupported location → no
         gismeteo fetch attempted; a supported location with a fresh gismeteo
         observation → throttled/skipped (no `RenderBatch` call); a per-location
         gismeteo error is joined into the result while other locations still
         persist.
   - [ ] The gismeteo phase calls `ForecastBatch` **at most once per `Run`** (one
         Chromium subprocess per tick) — assert via a counting fake.
   - [ ] `cmd/collector` builds; `buildRunners` wires the fetcher + gismeteo
         provider with `ChromiumPath` and `proxyURL`; zero weather subscriptions
         → no Chromium launch (the supported+due set is empty → `RenderBatch` is
         never called).
   - [ ] Existing Open-Meteo `WeatherAgent` subtests pass unchanged.
- **Pitfalls & edge cases:** Compute the supported+due set **before** calling
  `ForecastBatch` so Chromium launches only when there is real work — an empty
  supported set must short-circuit (the Task 1 empty-batch no-op is the backstop,
  but skip the call entirely to avoid even constructing the allocator). Do not
  throttle gismeteo on the Open-Meteo observation timestamp — use the gismeteo
  provider's own latest observation, or the two providers will desync. Log
  per-location gismeteo errors by `location_id` only (never coordinates), matching
  the MVP's privacy-scrubbed logging (divergence #4/#5 of the MVP). Weather has no
  `execution_history` table (MVP deferred it); the collector log is the only
  observability surface for gismeteo failures — make the marker explicit.
- **Complexity:** Medium
- **Code Example:**
  ```go
  // gismeteo phase, appended after the Open-Meteo loop in WeatherAgent.Run.
  if a.gismeteo != nil {
      var due []string
      tzByLoc := make(map[string]string)
      for _, loc := range locations {
          if !a.gismeteo.Supports(loc.LocationID) {
              continue
          }
          if a.gismeteoIsDue(ctx, loc.LocationID, now) {
              due = append(due, loc.LocationID)
              tzByLoc[loc.LocationID] = loc.Timezone // city tz for ForecastDate alignment
          }
      }
      if len(due) > 0 {
          obsByLoc, errByLoc := a.gismeteo.ForecastBatch(ctx, due, /* tz handling */)
          for locID, obs := range obsByLoc {
              obs.LocationID = locID
              if err := a.obsRepo.RetainWeatherObservation(context.Background(), obs); err != nil {
                  errs = append(errs, fmt.Errorf("gismeteo location %s: retain: %w", locID, err))
              }
          }
          for locID, err := range errByLoc {
              errs = append(errs, fmt.Errorf("gismeteo location %s: %w", locID, err))
          }
      }
  }
  ```

### Task 4: Cross-provider comparison in `WeatherCheckAgent`

- **Description:** With gismeteo observations now being collected, have
  `WeatherCheckAgent.Run` (`internal/application/notification/weathercheckagent.go`)
  load the latest gismeteo observation for a due city and, when it is **fresh
  enough**, pass both observations to `RenderMorningSummary`. The renderer is
  already variadic and already tested for the two-provider layout — **do not edit
  the renderer.**

  After the existing Open-Meteo observation load (the source of truth), add:
  1. Attempt `obsRepo.ObtainLatestObservation(city.LocationID, domain.ProviderGismeteo)`.
     The check agent does **not** need the curated map — it simply checks whether
     a gismeteo observation exists.
  2. Freshness gate: include gismeteo only when
     `gismeteoObs.ForecastDate == openMeteoObs.ForecastDate` **and**
     `now.Sub(gismeteoObs.CapturedAt) <= weatherGismeteoMaxAge` (define the const,
     e.g. 24 h). `forecast_date` equality is the primary guard so yesterday's
     gismeteo scrape never appears next to today's Open-Meteo; the `CapturedAt`
     bound is a belt-and-braces backstop.
  3. If fresh → `RenderMorningSummary(city, openMeteoObs, gismeteoObs)`; else →
     `RenderMorningSummary(city, openMeteoObs)` (unchanged single-provider path).
  4. A missing gismeteo observation (`internal.ErrNotFound`) or a stale one is a
     normal, non-error fallback — **never** block, skip, or fail the summary on
     gismeteo. Open-Meteo remains the gating provider for "do we send at all."

  This requires **no change** to `NewWeatherCheckAgent`'s signature, no change to
  `cmd/notifier`, and no change to the renderer.

- **Acceptance Criteria:**
   - [ ] `TestWeatherCheckAgent_Run` gains subtests: due city with fresh
         Open-Meteo + fresh gismeteo (same `forecast_date`) → **both** passed to
         the renderer (assert the queued message contains the gismeteo provider
         label "Gismeteo"); due city with only Open-Meteo → single-provider
         summary (no "Gismeteo" label); due city with a **stale** gismeteo
         observation (older `forecast_date` or `CapturedAt` beyond
         `weatherGismeteoMaxAge`) → treated as absent, single-provider summary;
         gismeteo lookup returning a non-`ErrNotFound` error → logged, still sends
         the Open-Meteo-only summary (gismeteo never blocks).
   - [ ] The existing single-provider subtests still pass unchanged.
   - [ ] `last_notified_at` advance and event-queue ordering are unchanged
         (advance only after a successful enqueue).
- **Pitfalls & edge cases:** The current test mock
  `mockWeatherCheckObsRepo.ObtainLatestObservation` ignores the `provider`
  argument (`_ string`) and returns one fixed observation — it **must** be made
  provider-aware (e.g. `obsByProvider map[string]*domain.WeatherObservation`) to
  test the dual-provider path; otherwise the gismeteo lookup returns the
  Open-Meteo fixture and the test is meaningless. Compare `forecast_date` as the
  stored string (both providers store the same `"2006-01-02"` city-local format —
  verified for Open-Meteo; Task 2 must produce the same format). Do not assume
  both observations carry every field; the renderer already renders `nil` fields
  as "—", so a sparse gismeteo block is fine. Keep gismeteo strictly the
  comparison: if Open-Meteo is absent, send nothing (do not promote gismeteo to
  source of truth).
- **Complexity:** Easy

### Task 5: `GismeteoInspector` (advisory) + register in `cmd/web`

- **Description:** Add `internal/application/inspector/gismeteo_inspector.go`,
  mirroring `openmeteo_inspector.go` exactly, and register it as **advisory** in
  `cmd/web/main.go`.

  - `GismeteoInspector` with a plain `*http.Client` (2 s timeout, shorter than the
    3 s agent sweep), `Name() string { return "gismeteo" }`, and
    `CheckUP(ctx) error` that issues a `GET https://www.gismeteo.kz/` with a
    browser-like `User-Agent` and asserts the response is **non-5xx** (reachable).
    A 200/301/302/403 all count as "reachable" — gismeteo may bounce non-browser
    clients with a redirect or 403, which is still "host is up." Only a 5xx or a
    transport error is a failure.
  - `NewGismeteoInspector()` + a `newGismeteoInspectorForTest(client, probeURL)`
    seam (mirror the Open-Meteo inspector's test constructor) so tests inject an
    `httptest.Server` with no live network.
  - It deliberately does **not** drive chromedp: a full render's cold-start can
    exceed the 3 s health budget. Document this in the godoc, as the Open-Meteo
    inspector documents its direct-egress caveat.
  - Register in `cmd/web/main.go` inside the existing

## Risks

- **gismeteo DOM fragility (high).** No API, JS-rendered, regex-scraped against a
  DOM that changes on redesign; coverage limited to a hand-curated map.
  Mitigation: isolate every selector/regex in a named constant; fail loudly into
  the collector log with explicit per-location markers (weather has no
  `execution_history`, so the log is the only signal); keep gismeteo strictly the
  *comparison* and Open-Meteo the *source of truth* so a total gismeteo breakage
  degrades to the shipped MVP behaviour, never a missed summary.
- **Chromium cold-start cost (medium).** Each gismeteo tick that does work spawns
  a Chromium subprocess on the ARM host. Mitigation: a dedicated, longer
  `DefaultGismeteoThrottleInterval`; batch all due gismeteo locations into one
  `RenderBatch` (one subprocess per tick); never launch when the supported+due set
  is empty.
- **chromedp-extractor delegation touching working FX code (medium → avoided).**
  Refactoring `ChromedpRateExtractor` to delegate to the new primitive risks the
  one piece of code that scrapes real bank rates in production, and its tests
  assert on private methods (`buildExecAllocatorOptions`, `loadFailedURL`,
  `fetchRenderedPageInAllocator`) that delegation would have to preserve as
  pass-throughs (keeping the duplication) or rewrite (violating "keep assertions
  green"). Mitigation: **ship the primitive standalone**; leave the rate extractor
  untouched; defer any consolidation to a dedicated plan.
- **Curated-map maintenance burden (medium).** Each supported city needs both its
  Open-Meteo geocoding id (the map key, == `location_id`) and its gismeteo slug +
  numeric id, hand-entered. Mitigation: the map is version-controlled and
  unit-tested for structure; document the limitation; map growth covers existing
  subscribers automatically on the next tick (because fetch resolves from the map,
  not a stored column).
- **`.kz` host language/units (low).** `.kz` returns Russian-language conditions
  and metric units; `.ru`/`.com` differ. Mitigation: pin `.kz`, document it, and
  parse numeric values + map gismeteo condition classes to WMO codes rather than
  matching localized words where possible.
- **`cmd/web` inspector egress is direct (low).** `GismeteoInspector` probes
  without the proxy (cmd/web does not parse `BEACON_PROXY_URL`), so it tests a
  different path than the proxied collector. Mitigation: advisory severity (a
  false "down" cannot fail the deploy gate); documented caveat; real gismeteo
  health is observable from collector logs.

## Trade-offs

- **Ship the chromedp primitive standalone; do not delegate the rate extractor.**
  Delegation buys near-zero deduplication (the rate-extractor tests pin private
  methods that would survive as wrappers anyway) while risking production FX
  scraping. Standalone is new code with zero FX blast radius. The pre-existing
  triple duplication (rateextractor / rulegen / chromedpfetch) is left for a
  dedicated consolidation plan.
- **The curated map (keyed by `location_id`) is the source of truth at fetch
  time; the `gismeteo_city_id` column is not.** The gismeteo URL needs the slug,
  which is map-only (never persisted), so the collector must consult the map
  regardless. Keying the map by `location_id` (== `LocationKey(geo)`, already on
  every city row) makes map growth cover existing subscribers with no backfill.
  Rejected: making the DB column authoritative (would require persisting the slug
  too and re-running geocoding/curation on subscribe, and still suffers the
  backfill trap).
- **Corrected provider signature: `Forecast(ctx, locationID string)`**, not the
  original spec's `Forecast(ctx, gismeteoCityID int)`. The numeric id cannot
  select a map entry (the map is keyed by `location_id`) nor build the URL (no
  slug). `location_id` is the only input that makes the map usable and aligns with
  the rest of the weather data path.
- **Regex-per-field scrape, not a DOM/CSS parser.** The repo has no goquery /
  `golang.org/x/net/html` dependency, and the established pattern for rendered
  pages is regex (`internal/tools/rateextractor/matchers.go`). Regex keeps the
  increment dependency-free and matches "the KASE regex" precedent. Alternative
  (add `golang.org/x/net/html` or goquery for robust traversal) is noted but
  rejected to avoid a new dependency for one fragile provider; revisit if the
  scrape proves too brittle.
- **Freshness = `forecast_date` equality + a `CapturedAt` age bound.** Comparing
  the stored `"2006-01-02"` city-local `forecast_date` is the robust guard against
  showing yesterday's gismeteo next to today's Open-Meteo; the age bound is a
  backstop. gismeteo never gates "do we send" — Open-Meteo does.
- **Dedicated, longer gismeteo throttle.** Reusing the 1 h Open-Meteo throttle
  would launch Chromium hourly. A few-hour gismeteo throttle cuts subprocess
  launches while still guaranteeing a same-day gismeteo observation before the
  earliest `NotifyHour` (default 07:00). The constant's value carries that
  constraint in its godoc.
- **gismeteo provider is optional (nil) in `WeatherAgent`.** Preserves exact MVP
  behaviour, keeps existing agent tests valid without a gismeteo fake, and lets
  the collector wire gismeteo independently.
- **Reachability inspector, not a render probe, and advisory.** A full chromedp
  render's cold-start can exceed the 3 s health budget, and a third-party outage
  must not fail the deploy gate; a non-5xx HTTP reachability check registered
  advisory satisfies both.
- **Populating `gismeteo_city_id` + the UI hint is deferred.** It forces the
  `WeatherGatewayDeps` constructor refactor and a resolver injection for a
  cosmetic hint, while the map already drives fetching. Documented as a known
  limitation instead (within the brief's "surface in UI OR document" latitude).

## Post-Review Divergences & Resolutions

Recorded per the "keep plans honest" rule. The increment shipped with a green
`make test` (-race + WASM). The biggest divergence from this plan is that **chromedp
was dropped entirely in favour of plain HTTP** — see below.

### Major divergence: plain HTTP, not chromedp

Empirical verification during implementation found that `gismeteo.kz` **server-renders
the daily forecast values into custom-element attributes in the raw HTTP response**
(`<temperature-value value="N">`, `<precipitation-value>`, the condition tab) — a plain
GET returns 200 (even on a non-browser UA) with the data present, no JS hydration
required. Consequently:

- **Task 1 (the shared `chromedpfetch` primitive) was built and then DELETED.** It has
  no consumer once gismeteo uses plain HTTP, and the project's existing
  `ChromedpRateExtractor` was deliberately left untouched. Net result: weather has NO
  Chromium dependency at all — simpler, more robust, no cold-start, no hydration
  mismatch.
- The gismeteo provider fetches over a proxy-aware `*http.Client` (mirroring the
  Open-Meteo provider), capped by `io.LimitReader`. One anchored regex per field
  (user's explicit preference), curated `location_id → {slug,id}` Go map, host
  `gismeteo.kz`. Eligibility comes from the map, not the unused `gismeteo_city_id`
  column.

### Review fixes folded in (five-lens + targeted re-review)

1. **`ForecastDate` city-local (correctness).** The decoder stamps the UTC date; the
   collector agent patches it to the city-local date (`now.In(cityLoc)`) using the
   city's stored timezone, before persisting — so it aligns with Open-Meteo's
   `timezone=auto` date and the `loadFreshGismeteo` equality guard does not silently
   reject gismeteo for UTC+ cities during the midnight-crossover window.
2. **DOM-drift fails loudly.** `decodeGismeteoForecast` returns a wrapped error when all
   four fields are nil (a likely site-redesign signal) instead of a misleading
   nil-field "success" that would surface as `fetched=1` and a "— / — / —" summary.
3. **Real multi-tab fixture + anchored temp regexes.** The hand-built stub was replaced
   with a representative slice of the real `gismeteo.kz` response (multiple day tabs);
   `reGismeteoTempMax`/`reGismeteoTempMin` now anchor on `class="weathertab is-active"`
   (`(?s)` + non-greedy) so today's values are selected regardless of day order.
   Precipitation is not anchored because the real page exposes only today's 3-hourly
   bars (verified). Parse-error paths are now covered by tests.
4. **Observability + consistency Minors/Nits.** Inspector body-drain; inline
   per-location gismeteo logging; `loadFreshGismeteo` logs each freshness rejection;
   the per-tick summary log is gated on `supported > 0`; single-pass `supported`
   counter; `newGismeteoWithClient` unexported; exact-UA test assertion; a
   transport-failure test; precip uses `FindAllSubmatch` (no second `Match`); the
   non-2xx error uses the post-redirect URL; CLAUDE.md notes the unused
   `gismeteo_city_id` column.

### Rejected review finding (false positive)

The targeted re-review flagged `runGismeteoPhase` using `context.Background()` for
`RetainWeatherObservation` as inconsistent with the Open-Meteo path. This is incorrect:
**both** retain paths intentionally use `context.Background()` so a SIGTERM does not
discard a successfully-fetched observation (the one-shot-collector pattern, documented
in a comment on the Open-Meteo path). Changing gismeteo to `ctx` would introduce the
inconsistency, not remove it. No change made.

`TestClient_Integration_SetSourceActive`, a `CloseIdleConnections` transport race under
the full parallel `-race` run) failed once during verification and passed on isolation
and on a clean re-run. It is not introduced by this increment.
