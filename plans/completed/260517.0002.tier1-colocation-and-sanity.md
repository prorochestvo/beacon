# Task Breakdown

## Overview

Plan 011 introduced tier-1 (structural) / tier-2 (currency) priority in
`internal/application/rulegen/sanitizer.go::Locate`. Live smoke-tests after
that landed exposed two real problems and a third that has been latent for a
while:

1. **Tier-1 over-broad anchors fire in the wrong place.** On `bcc.kz`, the
   tier-1 anchor ``<div class="text-lg`` matches a marketing heading near
   byte 40,000 AND the rate-row label near byte 333,000. Smallest-offset
   wins, the window centres on the marketing block, the actual rate table
   falls outside the 80 KB window, and the LLM cannot extract anything.
   Stopgap commit `9fb6a39` removed ``<div class="text-lg`` from the
   tier-1 list. This works for now (tier-2 wins on bcc.kz FX because the
   bare "USD" code at byte 324k is within 9 KB of the rate table) but it
   throws away a real signal ŌĆö that anchor IS the right one INSIDE the
   rate table; it's just also used elsewhere on the page. We need a
   stricter test that keeps the anchor and rejects the marketing hit.

2. **The post-extraction plausibility check is too loose.** Today
   `executor.Execute` and the other two extractor sites accept any finite
   positive float in `(0, math.MaxInt32]`. A real smoke-test run produced
   `value=19.1671` for USD/KZT (real ~469) and the value was happily
   persisted. We need a per-pair plausibility window so 19.1671 is
   rejected and re-prompted instead of stored.

3. **`KZ_BCC_BID_USD_KZT` (the bcc.kz homepage source) is structurally
   unfetchable via plain HTTP.** It lazy-loads rates via JS, so no amount
   of regex tuning will recover anything from the raw body. Running
   rulegen against it always burns LLM tokens and always fails. The
   source row needs an explicit "this needs a headless browser, do not
   try" marker, and `cmd/rulegen` needs to honour it.

This plan addresses all three with a single co-location guard, a hardcoded
per-pair plausibility table, and a `fetcher_kind` source column. No DOM
parsing, no chromedp, no relative-to-history validation. The audit-loop
mechanics and LLM client code from plans 010/011 are untouched.

## Assumptions

- **Co-location distance is 5 KB on each side of the anchor offset** (so a
  10 KB window in total). Justification: the existing seed regexes use
  `[\s\S]{0,400}?` between structural and value markers (see
  `migrations/202605.007.rate_sources.seed_initial.sql` lines 14ŌĆō25),
  meaning the currency-code ŌåÆ value distance in real templates is on the
  order of a few hundred bytes. 5 KB gives roughly 12├Ś headroom for cases
  where multiple unrelated currency codes sit between the structural
  anchor and the row of interest, while remaining tight enough that a
  marketing heading 280 KB away from the rate table never qualifies.
- **Co-location semantics: "any currency anchor within ┬▒N bytes of the
  tier-1 hit"** ŌĆö a single co-located currency code is enough to qualify a
  tier-1 candidate. "Every anchor must be co-located" is too strict
  (would defeat the tier-1 fallback when only one of base/quote appears
  near the anchor, e.g. a USD row inside a KZT-quoted table). "Majority
  of structural anchors" is over-engineered.
- **The plausibility table lives in a new file
  `internal/application/rulegen/plausibility.go`** next to `executor.go`,
  for the same reason `prompt.go` and `sanitizer.go` live there: they are
  rulegen-internal concerns. The other two extraction sites
  (`internal/tools/rateextractor` and `internal/application/sourceaudit`)
  are deliberately NOT changed in this plan ŌĆö their values are already
  cross-checked against history in production, and changing them
  triggers a larger refactor that is out of scope. Document this scope
  boundary in the plausibility file's package comment.
- **Map key shape: `struct CurrencyPair{ Base, Quote string }`** ŌĆö a
  struct key is `grep`-friendly (you can search for `Base: "USD"`),
  composite-literal-friendly, and avoids string-format mistakes
  (`"USD/KZT"` vs `"USD-KZT"` vs `"USDKZT"`). Stringly-typed keys are
  fragile and we have no compelling reason to use them.
- **Initial table covers every pair currently seeded** (see
  `migrations/202605.007.rate_sources.seed_initial.sql`):
  AED, CAD, CHF, EUR, GBP, GOLD, JPY, RUB, RUR, SILVER, TRY, USD, UZS ŌĆö
  all paired with KZT. Unknown pairs fall through to the universal
  `(0, MaxInt32]` check (current behaviour); this is documented in the
  package comment.
- **`fetcher_kind` allowed values: `"plain"`, `"headless"`.** "plain" is
  the existing HTTP fetcher; "headless" is a marker meaning "do not try
  with plain HTTP ŌĆö needs JS rendering we haven't built yet". RSS/JSON
  fetchers are not introduced here. The column is `NOT NULL DEFAULT
  'plain'` so all existing rows continue to work, and only
  `KZ_BCC_BID_USD_KZT` gets bumped to `"headless"` by this migration.
- **Migration NNN is `202605.010`** ŌĆö last applied is
  `202605.009.rate_sources.add_rule_metadata.sql` (confirmed via
  `ls migrations/`).
- **No new dependencies.** No DOM parser, no chromedp, no goquery, no
  `golang.org/x/net/html`. Byte-level only.

## Tasks

### Task 1: Add co-location guard to tier-1 anchor matching

- **Description:**
  Modify `Locate` in `internal/application/rulegen/sanitizer.go` so that
  a tier-1 anchor hit at byte offset `i` only counts when at least one
  non-empty currency anchor from the `currency []string` slice appears
  within `┬▒coLocationBytes` of `i` in the body. The smallest-offset rule
  still applies, but only among tier-1 hits that pass the co-location
  test. If no tier-1 hit qualifies, fall through to tier-2 (current
  currency-only behaviour). Introduce
  `const defaultCoLocationBytes = 5 * 1024` and pass it through `Sanitize`
  ŌåÆ `Locate` (do NOT add a new public parameter to `Sanitize`; thread the
  constant in via the existing call site so the public API of `Sanitize`
  stays a 3-argument function).

  The new `Locate` signature is:

  ```go
  func Locate(body []byte, structural, currency []string, windowBytes, coLocationBytes int) ([]byte, bool)
  ```

  Internally, replace the current "tier-1 smallest hit wins outright" loop
  with: iterate ALL distinct tier-1 anchors, record every hit offset,
  filter out hits that have no currency-anchor occurrence within
  `[max(0, i-coLocationBytes), min(len(body), i+coLocationBytes)]`, then
  pick the smallest surviving offset. If none survive, fall through to
  tier-2 exactly as today.

- **Acceptance Criteria:**
   - [ ] `defaultCoLocationBytes = 5 * 1024` constant defined in
     `sanitizer.go` with a comment explaining the rationale (cites the
     `[\s\S]{0,400}?` slack in production regexes).
   - [ ] `Locate` accepts a new `coLocationBytes int` parameter; passing
     `0` or a negative value MUST short-circuit the co-location check
     (i.e. behave like plan 011 / current behaviour). Document this in
     the godoc and add a subtest for it.
   - [ ] `Sanitize` continues to take `(body, structural, currency)` ŌĆö
     the public 3-arg signature is unchanged. Internally it calls
     `Locate(out, structural, currency, locateWindowBytes, defaultCoLocationBytes)`.
   - [ ] Existing `TestLocate` and `TestSanitize` subtests still pass
     with the added parameter wired through.
   - [ ] New `TestLocate` subtests:
     - "tier-1 hit with co-located currency wins": tier-1 anchor at
       offset 5,000 with "USD" at offset 5,300 ŌåÆ tier-1 wins.
     - "tier-1 hit without co-located currency is skipped": tier-1
       anchor at offset 1,000 with no currency code within ┬▒5 KB, plus
       a second tier-1 hit at offset 100,000 with "USD" at 100,400 ŌåÆ
       the far hit wins.
     - "tier-1 candidates: smallest offset among co-located wins":
       three tier-1 hits, two co-located, picks the smallest of the two.
     - "tier-1 falls through to tier-2 when zero qualify": every tier-1
       hit lacks a nearby currency anchor ŌåÆ tier-2 currency anchor wins.
     - "`coLocationBytes=0` disables the guard": same body as the
       skip-case above, but with `coLocationBytes=0` ŌåÆ the close-but-not-
       co-located tier-1 hit at offset 1,000 wins.
     - "duplicate tier-1 anchors are still deduplicated".
   - [ ] New `TestSanitize` subtest: feed a synthetic body where
     ``<div class="text-lg`` appears twice ŌĆö once with no nearby USD/KZT
     and once with USD inside it. The output window must contain the
     USD-adjacent occurrence, not the lonely one.
   - [ ] Restore ``<div class="text-lg`` to the structural anchor list
     in `internal/application/rulegen/generator.go` (revert the
     workaround introduced by commit `9fb6a39`). Update the comment
     block immediately above the `structural` slice to explain that
     co-location now guards it.
   - [ ] The fixture-based test
     `TestSanitize/bcc_fixture_structural_anchor_takes_priority_over_currency_tier`
     in `sanitizer_test.go` still passes against
     `tmp/testdata/sources/bcc.html` and continues to find the
     ``<div class="text-lg`` window. Confirm by reading the existing
     assertion at sanitizer_test.go:159.
   - [ ] `make test` and `make lint` both pass.

- **Pitfalls:**
   - **Don't add `coLocationBytes` as a public parameter on `Sanitize`.**
     Plan 011 already established `Sanitize` as a 3-argument function;
     every caller would need updating and it leaks an internal tuning
     knob to the API. Keep `Sanitize` at 3 args, thread the constant
     through internally.
   - **Window clamping for the co-location check is independent of the
     output window.** `coLocationBytes` clamps the SEARCH range around
     the anchor offset; `windowBytes` clamps the RETURNED slice. Don't
     conflate them.
   - **Iterate distinct anchors, but iterate ALL of their occurrences,
     not just the first.** Today `firstHit` returns the smallest offset
     across distinct anchors. For tier-1 co-location you need to look
     at every occurrence of every distinct anchor, because the
     marketing-vs-rate-table case has TWO occurrences of the SAME
     anchor string. Use `bytes.Index` in a loop with a moving cursor
     (`for cursor < len(body)`) so you visit every occurrence of each
     anchor. Cap the per-anchor scan with a sanity bound (e.g. 1,000
     occurrences) to avoid O(n┬▓) on adversarial bodies ŌĆö log and skip
     if exceeded.
   - **`firstHit` is still useful for the tier-2 fallback.** Don't
     delete it; the tier-2 logic stays exactly as it is today
     (no co-location requirement on tier-2 ŌĆö the whole point of tier-2
     is "you have no structural hit, take whatever currency match you
     can find").
   - **Empty `currency` slice with co-location enabled means NO tier-1
     hit can qualify** (there is no currency anchor to be co-located
     with). Make sure the code falls through to tier-2 cleanly in this
     case rather than returning false; add a subtest covering it.
   - **Don't change `Locate`'s behaviour when `structural` is empty**;
     the existing "skip tier-1 entirely if nothing to match" path is
     correct.

- **Complexity:** Medium

- **Code Example:**

  ```go
  // Locate returns the smallest window of body centred on the earliest
  // *qualifying* structural anchor match. A tier-1 hit at offset i
  // qualifies only when at least one non-empty currency anchor occurs
  // inside [i-coLocationBytes, i+coLocationBytes]. Passing
  // coLocationBytes <= 0 disables the co-location check (legacy behaviour).
  func Locate(body []byte, structural, currency []string, windowBytes, coLocationBytes int) ([]byte, bool) {
      if hit, ok := smallestQualifyingTier1Hit(body, structural, currency, coLocationBytes); ok {
          return windowAround(body, hit, windowBytes), true
      }
      if hit, ok := firstHit(body, currency); ok {
          return windowAround(body, hit, windowBytes), true
      }
      return body, false
  }

  func smallestQualifyingTier1Hit(body []byte, structural, currency []string, coLocationBytes int) (int, bool) {
      seen := make(map[string]bool, len(structural))
      best := -1
      for _, a := range structural {
          if a == "" || seen[a] {
              continue
          }
          seen[a] = true
          needle := []byte(a)
          cursor := 0
          for safety := 0; safety < 1000; safety++ {
              rel := bytes.Index(body[cursor:], needle)
              if rel < 0 {
                  break
              }
              i := cursor + rel
              if coLocationBytes <= 0 || hasCurrencyWithin(body, i, currency, coLocationBytes) {
                  if best < 0 || i < best {
                      best = i
                  }
              }
              cursor = i + 1
          }
      }
      if best < 0 {
          return 0, false
      }
      return best, true
  }

  func hasCurrencyWithin(body []byte, centre int, currency []string, radius int) bool {
      start := centre - radius
      if start < 0 {
          start = 0
      }
      end := centre + radius
      if end > len(body) {
          end = len(body)
      }
      window := body[start:end]
      for _, c := range currency {
          if c == "" {
              continue
          }
          if bytes.Contains(window, []byte(c)) {
              return true
          }
      }
      return false
  }
  ```

     and we don't want to false-reject. The narrower the better as
     long as we don't trip on real values.

- **Complexity:** Easy

  type currencyPair struct {
      Base  string
      Quote string
  }

  type valueRange struct {
      Lo float64
      Hi float64
  }

  var plausibleRanges = map[currencyPair]valueRange{
      // KZT-quoted majors
      {Base: "USD", Quote: "KZT"}: {Lo: 100, Hi: 1000},
      {Base: "EUR", Quote: "KZT"}: {Lo: 100, Hi: 1000},
      {Base: "GBP", Quote: "KZT"}: {Lo: 100, Hi: 1000},
      {Base: "CHF", Quote: "KZT"}: {Lo: 100, Hi: 1000},
      {Base: "CAD", Quote: "KZT"}: {Lo: 100, Hi: 1000},
      {Base: "AED", Quote: "KZT"}: {Lo: 50,  Hi: 500},
      // CIS / emerging
      {Base: "RUB", Quote: "KZT"}: {Lo: 1,    Hi: 30},
      {Base: "RUR", Quote: "KZT"}: {Lo: 1,    Hi: 30}, // legacy alias used by qazpost
      {Base: "TRY", Quote: "KZT"}: {Lo: 1,    Hi: 50},
      // Small per-unit quotes
      {Base: "JPY", Quote: "KZT"}: {Lo: 0.5,  Hi: 10},
      {Base: "UZS", Quote: "KZT"}: {Lo: 0.01, Hi: 1},
      // Metals (per troy ounce in KZT)

      {Base: "SILVER", Quote: "KZT"}: {Lo: 1_000,   Hi: 100_000},
  }

  func plausibleRangeFor(base, quote string) (float64, float64, bool) {
      r, ok := plausibleRanges[currencyPair{Base: base, Quote: quote}]
      if !ok {
          return 0, 0, false
      }
      return r.Lo, r.Hi, true
  }

## Risks

- **Co-location distance too tight (under-matches).** If a real bank
  page puts the structural anchor and the currency code more than 5 KB
  apart (e.g. a sidebar layout with the table heading at the top and
  the rows below a large header image's `alt` text), tier-1 fails to
  qualify and we fall through to tier-2 ŌĆö degraded but not broken.
  Mitigation: log every tier-1 candidate's offset + co-location
  result at `info` level so we can see in production whether real
  sources are being skipped. Acceptable next step is to widen
  `defaultCoLocationBytes` to 10 KB.
- **Co-location distance too loose (false positives).** If we set it
  to 50 KB, the marketing-block hit on bcc.kz at byte 40k would still
  qualify (USD code somewhere in the 90 KB window). 5 KB is the
  middle ground; we deliberately tune toward false negatives because
  the tier-2 fallback exists.
- **Per-pair table drift.** If a currency pair's real-world rate
  legitimately moves outside our hardcoded range (hyperinflation,
  redenomination), every rulegen run for that pair will fail. The
  table is a config-as-code surface; updating it is a one-line PR
  and we document this. Mitigation: in `make lint` or in CI, fail
  the build if `migrations/202605.007.rate_sources.seed_initial.sql`
  introduces a base_currency that has no entry in `plausibleRanges`
  (this would be a future plan, not in scope here, but flag it).
- **`fetcher_kind` is a single-row UPDATE in the migration.** If a
  future seed migration adds another SPA source without bumping
  `fetcher_kind`, it inherits the `'plain'` default and rulegen will
  burn tokens trying to extract from a hydrated shell. Mitigation:
  document in `cmd/rulegen/README.md` and in the seed-migration
  template (if one exists) that new SPA sources must set the column
  explicitly.
- **Restoring ``<div class="text-lg`` to the anchor list re-introduces
  the same anchor that the stopgap commit removed.** The co-location
  guard is what makes this safe; if Task 1's guard logic has a bug,
  bcc.kz FX regresses. Mitigation: keep the
  `bcc_fixture_structural_anchor_takes_priority_over_currency_tier`
  test passing AND add a new fixture-based subtest that exercises
  the specific marketing-vs-rate-table disambiguation.
- **Executor signature change is wire-breaking inside the package.**
  Anyone implementing the `RuleExecutor` interface (today: only the
  package's `defaultRuleExecutor`, but tests may stub it) needs
  updating. A simple grep confirms today's blast radius is tiny;
  document this in the PR description.

## Trade-offs

- **Co-location distance: 5 KB chosen over 10 KB.** 5 KB is 12├Ś the
  longest seed-regex slack (`[\s\S]{0,400}?`), which is conservative.
  10 KB would absorb more layout variation but would have re-admitted
  the bcc.kz marketing hit (40k ŌåÆ USD at 324k is 284k away ŌĆö both
  are out of range either way, BUT a page where marketing sits at
  byte 320k with a USD price tag inside it would slip through 10 KB
  but not 5 KB). Picked 5 KB. Easy to tune later via the constant.
- **Co-location: "any" rather than "all" or "majority".** "All"
  collapses on legitimately one-sided sources (a USD/KZT page where
  the structural anchor only sits near USD because KZT is the
  implicit quote). "Majority" is over-engineered for a 2-element
  currency slice. "Any" is the right semantic for tier-1's purpose
  (rule out unrelated structural anchors) without over-constraining.
- **Plausibility table location: `internal/application/rulegen/`
  rather than `internal/domain/`.** Domain types should describe
  invariants of the business, not policy of one extraction site.
  Three call sites extract rates; only rulegen needs the per-pair
  guard today (sourceaudit and rateextractor have their own
  history-based checks elsewhere). Putting the table in domain
  would force them to either adopt it or import it without using
  it, both worse than keeping it scoped.
- **Plausibility key shape: struct instead of string `"USD/KZT"`.**
  Struct keys are compile-checked, allocation-free at the call site
  (the struct literal is constant-folded by the compiler), and they
  pattern-match cleanly with the `(BaseCurrency, QuoteCurrency)`
  pairs already in `domain.RateSource`. Stringly-typed keys would
  require `fmt.Sprintf` or careful concatenation at every call site.
- **`fetcher_kind` as a SQLite TEXT column rather than INTEGER enum.**
  TEXT is grep-friendly, dumpable, and migration-friendly. The
  enforcement happens in the repository write path with a switch
  statement, not at the DB layer (SQLite has no CHECK enforcement
  by default in older driver versions). A TEXT column with a
  documented allowed set is the project's standing pattern (see
  `kind` column, lines 8/15 of
  `202605.001.rate_sources.table_initiate.sql`).
- **`ErrUnsupportedFetcherKind` as an exported sentinel rather than a
  typed struct.** A sentinel keeps `errors.Is` checks trivial and
  doesn't need extra fields (the wrapped error message already
  carries the source name and the offending kind). A typed error
  would be over-engineered for a single boolean condition.
- **Guard lives inside `Generator.Generate`, not in `cmd/rulegen/main.go`.**
  Future callers (a hypothetical `make rulegen-all` or a controller
  endpoint) get the guard for free. Single source of truth.
- **No changes to `internal/tools/rateextractor` and
  `internal/application/sourceaudit`.** Both sites have their own
  history-based plausibility checks (relative-value-of-yesterday in
  the collector path; per-source historical ranges in audit). The
  per-pair static table is a rulegen-time check; bringing it
  everywhere would be a refactor of three loosely-coupled call
  sites and is out of scope per plan 009's earlier conclusion that
  history-based validation is brittle. Documented explicitly in the
  plausibility file's package comment.
