# Task Breakdown

## Overview

Two related Mini App features delivered in one plan because both touch the same
chart pipeline and the same `/api/me/...` HMAC-auth surface.

**Feature A — Period selector.** Replace the hardcoded 7-day window on the
sparkline-list chart with a user-selectable period chosen from a whitelist
`{7, 30, 90, 180, 360}` days. Wire the selection end-to-end: integer query
parameter on the two chart endpoints, parameter on the service methods,
parameter on the WASM apiclient calls, and a row of five chips/buttons on both
the authenticated screen (`renderMeSubscriptions`) and the guest screen
(`renderPublicSubscriptions`). Backwards compatible: omitting `period` yields
the existing 7-day behaviour. The response `Window` string reflects the
selected period (e.g. `"30 days"`).

**Feature B — Subscriptions editor (separate SPA screen).** Add full CRUD over
the caller's own subscriptions on a new dedicated screen, mounted via the
existing `mountScreen()` lifecycle. The editor needs per-condition rows with
stable IDs (the existing `GET /api/me/subscriptions` joins all conditions for a
source into one row), so we expose them through a new query mode on the
existing endpoint and three write endpoints (`POST`, `PATCH`, `DELETE`).
Authentication uses the same `X-Telegram-Init-Data` header HMAC as every other
`/api/me/...` route. Ownership is enforced by comparing the initData chat_id to
the persisted `user_id`; a mismatch returns 404 so the endpoint does not
disclose existence of subscriptions belonging to other users.

Backend lands before WASM consumes it. Each task leaves `make test` green.

## Assumptions

1. **No schema changes are required.** `rate_user_subscriptions` already stores
   `id`, `user_type`, `user_id`, `source_name`, `condition_type`,
   `condition_value`, `latest_notified_rate`, `updated_at`, `created_at`.
   `RetainRateUserSubscription` already inserts when `record.ID == ""` and
   updates otherwise, generating an `RUS...` ID via `generateRateUserSubscriptionID`.
   `RemoveRateUserSubscription(ctx, *domain.RateUserSubscription)` already
   exists and deletes by ID. The plan calls out a small repo addition
   (`ObtainRateUserSubscriptionByID`) needed by the PATCH/DELETE ownership
   check; no migrations.
2. **`rate_user_events` is NOT scoped by `subscription_id`.** It has FK
   `source_name → rate_sources(name) ON DELETE CASCADE` but no link to the
   subscription row. Deleting a subscription therefore leaves any historical
   `rate_user_events` for that user/source intact. This is acceptable (events
   are historical truth, not subscription state), but the plan calls it out so
   no engineer assumes the cascade does cleanup it does not.
3. **DM-only.** As noted by the existing TODO at handlers.go:564, subscriptions
   are keyed by Telegram chat_id which equals user_id only for direct chats.
   The editor inherits that limitation unchanged; do not introduce a "select
   chat" UI.
4. **`/api/sources` is safe to surface to Mini App users for the source-picker
   dropdown.** The response already excludes user-keyed data; it exposes
   `name`, `title`, `base_currency`, `quote_currency`, `interval`, `active`,
   `last_success`, `last_error`, `last_run_at`. Of those, `last_error` is the
   only borderline field (operator-facing). See Open Decisions for the chosen
   resolution.
5. **Period parameter is an integer whitelist, not a free integer.** Exactly
   `{7, 30, 90, 180, 360}`; anything else (including `"7d"`, `"30 "`, `"45"`)
   is 400 with a `PublicError`. The whitelist is the contract; the renderer
   uses the same constant set.
6. **Telegram WebApp `showConfirm`** is the preferred delete-confirmation dialog
   when `window.Telegram.WebApp.showConfirm` is available; fallback to
   `window.confirm` keeps it working in a local-dev browser tab.
7. The user has accepted that opening the editor re-fetches the chart on
   return, so we do NOT need to mutate the chart in place after a CRUD action.

## Open Decisions (resolved)

**D1 — Per-condition list endpoint shape.**
*Options:*
- (A) Extend `GET /api/me/subscriptions` with `?expanded=true` toggling
  grouped/per-condition response.
- (B) Add a separate endpoint, e.g. `GET /api/me/subscriptions/raw`.

*Choice: B (new endpoint at `GET /api/me/subscriptions/raw`).*

*Rationale:*
- Different consumers, different shapes. The current `MeSubscriptionsResponse`
  is grouped per source, paginated, enriched with `LatestPrice`/`LatestAt`, and
  carries a `q` search filter. The editor list is per-condition, needs no
  enrichment, and (for now) needs no search. Conflating both shapes behind a
  query flag forces a sum type in the response that the strongly-typed Go DTO
  cannot express cleanly without an `omitempty` minefield or two separate
  decoders client-side.
- Cache and observability: two endpoints get their own access-log lines,
  rate-limit budgets, and future cache TTLs without an `?expanded=` discriminator
  that operators have to remember.
- Cost is trivial: one new handler, one new DTO. Existing endpoint stays
  byte-identical to its current contract — zero backward-compatibility risk.

**D2 — Available-sources endpoint for the picker.**
*Options:*
- (A) Reuse `GET /api/sources` (no auth).
- (B) Add a slimmer `GET /api/me/sources/available` (auth via initData) that
  strips operator-only fields (`last_error`, `last_run_at`).

*Choice: A (reuse `GET /api/sources`).*

*Rationale:*
- `/api/sources` already exposes the fields the picker needs (`name`, `title`,
  `base_currency`, `quote_currency`, `active`). The "operator-only"
  `last_error`/`last_run_at` fields contain no PII; they're publicly inferable
  from `/api/public/rates/chart` activity anyway.
- Adds zero new surface area. Bug surface stays smaller.
- The editor filters out `active=false` rows client-side so the user can't
  create a subscription against a dead source.
- If we later decide to hide operator fields, we can introduce a slim endpoint
  without breaking the editor — the WASM apiclient already isolates the URL.

## Tasks

### Task 1: Generalise ChartWindow into a runtime parameter on the chart service

- **Description:** Replace the hardcoded `ratepair.ChartWindow` constant with
  a `periodDays int64` parameter that flows from handler → service. Specifically:
  - Add `ObtainMeChartForPeriod(ctx, userID, periodDays)` and
    `ObtainPublicChartForPeriod(ctx, page, limit, periodDays)` to the chart
    service in `internal/application/chart/service.go`. The new methods take
    `periodDays` (one of 7/30/90/180/360) and compute
    `since = now.Add(-time.Duration(periodDays) * 24 * time.Hour)`.
  - Keep the existing `ObtainMeChart`/`ObtainPublicChart` as one-line wrappers
    that delegate to the `ForPeriod` variants with `periodDays=7`. This keeps
    every existing caller (and every existing test) compiling unchanged.
  - Leave `ratepair.ChartWindow = 7 * 24 * time.Hour` in place for now; it's a
    semantic default referenced in godoc only. Mark it `// Deprecated:` so
    future grepping finds the migration trail. (Do NOT remove yet — Task 5
    decides whether anything still needs it.)
  - `bucketCount = 12` stays — the visual density of the sparkline does not
    change with window length. A 30-day chart still has 12 buckets, each
    representing ~2.5 days. This is intentional; period selection is about
    horizon, not resolution.
- **Acceptance Criteria:**
  - [ ] `ObtainMeChartForPeriod` and `ObtainPublicChartForPeriod` exist on
        `*Service` with the signatures above and behave identically to the
        existing methods when `periodDays == 7`.
  - [ ] `ObtainMeChart` and `ObtainPublicChart` still exist with their original
        signatures; they delegate to the `ForPeriod` variants and pass `7`.
  - [ ] Existing tests in `internal/application/chart/service_test.go` (and any
        sibling files) pass unchanged.
  - [ ] New unit tests cover `ObtainMeChartForPeriod`/`ObtainPublicChartForPeriod`
        with at least two distinct period values (e.g. 7 and 90) and verify
        `since = now - period*24h` is the value passed to `ValuesLoader`.
  - [ ] `make test` passes.
- **Pitfalls & edge cases:**
  - Do NOT change the bucket count based on period — every reviewer-A finding
    on the "1 sample per day vs 12 buckets total" question will rediscover that
    `bucketCount` is intentionally fixed; the godoc comment on `bucketCount`
    must be updated so the next engineer doesn't re-litigate it.
  - The "sparse" determination still applies (< 2 deduped raw samples within
    the window). A 360-day window with two stray samples is still sparse — the
    behaviour is intentional, since extending the window doesn't manufacture
    data.
  - `time.Duration(periodDays) * 24 * time.Hour` overflows `time.Duration`
    only beyond ~106 years; 360 days is fine. Keep the integer multiplication
    explicit so reviewer-E cannot mistake it for a math.MaxInt64 hazard.
- **Complexity:** Easy

      `internal.NewPublicError("invalid condition: " + err.Error())`.
    - Optional: verify the source exists via
      `meSourceRepo.ObtainRateSourceByName(ctx, body.SourceName)`. If
      `(nil, nil)` → 400
      `internal.NewPublicError("unknown source")`. Cheap guard; without it
      the FK fails opaquely.
    - Call `meSubRepo.RetainRateUserSubscription(ctx, &record)`.
    - 201 Created with `{"id": "<generated>"}`.
  - Tests in one `TestCreateMeSubscription` with subtests: happy path (201),
    401, 400 body decode, 400 validate (each condition type — delta, daily,
    interval, cron), 400 unknown source, 500 repo failure.
- **Acceptance Criteria:**
  - [ ] `POST /api/me/subscriptions` returns 201 with the generated ID.
  - [ ] Server-side validation runs `domain.RateUserSubscription.Validate()`
        and surfaces the underlying error as a `PublicError` body.
  - [ ] `UserID` is forced from the verified initData chat_id, never from the
        request body.
  - [ ] CLAUDE.md Routes section updated with the new endpoint and the body
        shape.
  - [ ] 401 path matches existing handlers byte-for-byte (no rewording).
- **Pitfalls & edge cases:**
  - `condition_type` from the wire is a string; cast to
    `domain.SubscriptionConditionType` then run `Validate()`. The `default`
    branch in `Validate()` handles unknown types cleanly — surface that
    error verbatim in the PublicError.
  - For `delta` condition, the `Validate()` path rejects negative thresholds
    but accepts `"0"`. That's existing behaviour; do NOT add a stricter check
    here. Consistency with the bot is more important than aesthetics.
  - Per the no-PII memory: the request DTO must NOT have `username`,
    `display_name`, etc. Reject any future schema growth at code review.
  - There's no idempotency key. A double-POST creates two subscription rows
    with the same `(user_id, source_name, condition_type, condition_value)`
    tuple — the schema does not enforce uniqueness. This is consistent with
    the existing bot UX and out of scope for this plan.
- **Complexity:** Medium

### Task 9: REST — `PATCH /api/me/subscriptions/{id}` and `DELETE /api/me/subscriptions/{id}`

- **Description:** Edit and delete with ownership enforcement.
  - Route constant
    `MeSubscriptionByID = "/api/me/subscriptions/{id}"` in routes.go.
  - Two handlers:
    - `UpdateMeSubscription` (PATCH). Body shape:
      ```go
      type MeSubscriptionUpdateRequest struct {
          ConditionType  string `json:"condition_type"`
          ConditionValue string `json:"condition_value"`
      }
      ```
      Editing `SourceName` is deliberately out of scope — changing a source
      is conceptually a delete+create and easier to reason about as such.
    - `DeleteMeSubscription` (DELETE). No body.
  - Both flows:
    1. Validate HMAC; 401 on failure.
    2. `id := r.PathValue("id")`. Empty → 400.
    3. `sub, err := meSubRepo.ObtainRateUserSubscriptionByID(ctx, id)`. On
       err → 500. On `(nil, nil)` → 404
       `internal.NewPublicError("subscription not found")`.
    4. Ownership check: `if sub.UserID != chatIDString` → 404 with the SAME
       "subscription not found" PublicError. Reviewer-D will check this is
       not a 403 — leaking existence is the regression we're avoiding.
    5. PATCH: decode body, set `sub.ConditionType = ...`,
       `sub.ConditionValue = ...`, run `sub.Validate()`. 400 PublicError on
       fail. Call `RetainRateUserSubscription(ctx, sub)`. Return 204.
    6. DELETE: call `RemoveRateUserSubscription(ctx, sub)`. Return 204.
  - Tests: one `TestUpdateMeSubscription` and one `TestDeleteMeSubscription`,
    subtests covering: 204 success, 401, 404 missing, 404 cross-user (the
    ownership test fixture creates a sub for user A and tries to PATCH it as
    user B), 400 validate, 500 repo failures.
- **Acceptance Criteria:**
  - [ ] `PATCH /api/me/subscriptions/{id}` and
        `DELETE /api/me/subscriptions/{id}` exist.
  - [ ] Cross-user access returns 404 with the same message as a genuine miss.
  - [ ] PATCH runs `Validate()` and surfaces failures as PublicError 400.
  - [ ] Both endpoints return 204 No Content on success.
  - [ ] CLAUDE.md Routes section updated; ownership-via-404 noted.
  - [ ] Tests cover the ownership 404 path explicitly.
- **Pitfalls & edge cases:**
  - SQLite does not have native bool-test paths for "row exists" vs
    "ownership"; we accept two reads on the same row (one in the by-ID
    lookup, one inside `RetainRateUserSubscription`'s count query). That's
    a minor inefficiency, not a correctness issue. Reviewer-E may flag it;
    defer fixing as a future plan unless real latency complaints arrive.
  - `RetainRateUserSubscription` overwrites `UpdatedAt` and ID-generates
    when ID is empty. Since we pass the loaded record with its ID set, it
    correctly performs an UPDATE, not an INSERT.
  - **Orphan events:** Deleting a subscription does NOT delete the user's
    `rate_user_events` rows for that source — events are FK'd to the source,
    not the subscription. Note this in a code comment so the next engineer
    doesn't assume a cascade fixes it. Cleaning up orphan events is out of
    scope and not currently desired (events are historical truth).
  - 4 KiB MaxBytesReader on PATCH body, consistent with Task 8.
- **Complexity:** Medium

## Risks

1. **WASM bundle size growth.** The editor adds new application + ui
   packages and pulls `robfig/cron/v3` into the WASM artifact for
   client-side cron validation. Mitigation: build the WASM target and check
   `ls -la cmd/web/static/app.wasm` after Task 11 lands. If the growth is
   > 100 KiB compressed, drop client-side cron validation and rely on
   server PublicError instead. Trade-off documented in the controller's
   godoc.
2. **Ownership 404 is the right answer, but easy to get wrong.** Reviewer-D
   should explicitly verify that a cross-user PATCH/DELETE response is
   indistinguishable from a genuine not-found at the HTTP-status level AND
   the body level. A "subscription not found" message on a cross-user 404
   and a "subscription does not belong to you" message on a genuine 404
   would be a covert disclosure regression.
3. **Inflight chart fetch racing a period change.** Single-threaded WASM
   makes literal data-race impossible, but a slow fetch's result writing
   to state AFTER a newer fetch has already completed is a real risk. The
   existing `MeSubscriptionsPage.LoadHistory` solves it via target
   snapshotting; mirror the same pattern in `LoadSparklineChart` if it
   doesn't already (verify during Task 4).
4. **Migrations.** No schema change. If a reviewer believes one is needed
   for, e.g., a unique constraint on `(user_id, source_name, condition_type,
   condition_value)`, that's a separate plan — out of scope here. The
   current insert-on-empty-ID flow is preserved.
5. **`make test` regression on chart goldens.** Several `cmd/wasm/ui` tests
   compare rendered HTML against fixed strings. The chip row and Window
   text changes touch those strings; update goldens explicitly in the same
   task that introduces the rendering change (Task 5). Bulk-replace via
   `replace_all` will silently swallow drift; reviewer-B catches it.
6. **Telegram `showConfirm` callback lifecycle.** Forgetting to
   `Release` the `js.FuncOf` callback after `showConfirm` fires leaks one
   entry per delete confirmation. Reviewer-A specifically checks Task 12
   for this.

## Trade-offs

1. **Separate `/api/me/subscriptions/raw` endpoint vs. `?expanded=` flag.**
   Chose the new endpoint over a flag. Reasons in Open Decision D1.
   Implication: one extra route to maintain forever, but in exchange the
   existing `MeSubscriptionsResponse` contract stays untouched and the
   editor's wire shape is decoupled from the read screen's.
2. **Reuse `/api/sources` for the picker vs. slim `/api/me/sources/available`.**
   Chose reuse. Reasons in Open Decision D2. Implication: if we later want
   to hide operator-only fields from end users, we'll add the slim endpoint
   then. Today, code we don't write can't break.
3. **Bucket count stays 12 regardless of period.** A 360-day chart at 12
   buckets is 1 point per 30 days. Sparser-looking sparklines on long
   horizons are a deliberate UX choice — fewer points reduce visual
   noise. If real users complain, revisit by making `bucketCount` scale
   with period in a follow-up plan.
4. **Editor is a separate screen, not a modal.** Modal mode would re-use
   the existing modal infrastructure but constrain form layout on small
   viewports. A full screen reads natively on mobile, supports a wide
   source dropdown, and matches user expectation. Cost: more navigation
   plumbing (`mountScreen`/`unmount`/back button), already paid by the
   existing `me_pair_history` flow.
5. **Client-side validation duplicates `domain.Validate()`.** Required to
   give the user inline feedback before the network round-trip. The
   duplication is a known maintenance cost: any future change to a
   condition-type semantic must be reflected in both. Lower-stakes than it
   sounds since the server's validation is the source of truth — the
   client-side version's worst failure mode is letting through a request
   that the server then rejects with a PublicError.
6. **No idempotency on `POST /api/me/subscriptions`.** A double-tap from a
   spotty mobile connection creates two identical subscriptions. Matches
   existing bot behaviour. Adding an idempotency key (or a unique
   constraint at the schema level) is a separate cleanup not included
   here.
7. **Deleting a subscription leaves historical events orphaned.** Events
   are FK'd to `rate_sources`, not to `rate_user_subscriptions`. Cleaning
   them up would require either schema change (add `subscription_id` and
   cascade) or a one-off DELETE alongside the subscription remove. Out of
   scope: events are historical truth, and the cost of querying for them
   later (a user opening their old events from a since-deleted
   subscription) is low.
