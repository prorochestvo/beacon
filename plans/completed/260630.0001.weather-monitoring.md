# Task Breakdown

## Overview

Add per-user weather monitoring to beacon. Each user picks their own city; the
system collects forecasts from multiple providers and delivers a daily morning
summary over Telegram. Conceptually this mirrors the FX side (multi-provider,
user subscriptions, an extensible set of notification kinds), but weather is
multi-dimensional (temperature, feels-like, humidity, wind, precipitation,
condition code, hourly + daily forecast) and per-user-city, so it does **not**
fit the `rate_sources` / `rate_values` "one source = one scalar" model.

**Architecture decision (detailed in Trade-offs): build a parallel weather
domain (Option B), not extend the FX model (Option A).** Storage, addressing,
and collection are weather-specific; delivery is generic. The plan therefore
adds new weather tables, repositories, providers, and collection/check agents,
while **reusing** the existing generic infrastructure: the notification pool
(`rate_user_events`), the `RateDispatchAgent` Telegram transport, the one-shot
collector/notifier cron model, the `BEACON_PROXY_URL` egress, the
Telegram-WebApp-initData auth, and the health-check inspector pattern.

**Providers:**
1. **Open-Meteo** — free, keyless, global. One JSON GET returns current +
   hourly + daily as parallel arrays indexed by `time[]`. Fits the app's
   "call a JSON API" fetch mode; no headless browser. Universal because any
   city resolves to coordinates via Open-Meteo geocoding (also keyless).
2. **gismeteo** — no public API; values are JS-rendered, so it requires the
   headless-Chrome (chromedp) path. There is no public "city name → gismeteo
   numeric ID" resolver, so gismeteo support is limited to a **curated
   city→ID map**: gismeteo works only for cities in that map, whereas
   Open-Meteo is fully global.

**Increment boundary:**
- **MVP (first shippable increment):** domain model + Open-Meteo provider +
  geocoding city picker + city subscription + **daily morning-summary cron**.
  This is the clean, low-risk path (keyless JSON, no chromedp).
- **Second increment (same plan):** gismeteo provider + cross-provider
  comparison in the summary (the user's stated reason for keeping gismeteo:
  he trusts its 3–4 day forecast).
- **Deferred beyond this plan (domain must not preclude them):** rain-incoming
  alert, threshold alerts (heat / frost / high-wind / thunderstorm), on-demand
  current-weather screen in the Mini App. The user listed all of these as
  desired "later, by analogy with rate subscriptions."

## Assumptions

- The two providers stay keyless. No new `dsninjector` DSN / env var is
  introduced. (Verified: Open-Meteo is keyless; gismeteo is scraped.)
- Weather collection runs inside `cmd/collector` (already wired for
  `BEACON_PROXY_URL` and `BEACON_CHROMIUM_PATH`, one-shot, cron-wrapper-driven).
  Weather notification (check + queue) runs inside `cmd/notifier` (already has
  the dispatch agent, the profile repo, and embedded `time/tzdata`).
- The notification queue/transport is generic enough to carry a free-form
  weather summary. (Verified: `domain.RateUserEvent.Message` is a free-form
  string; `RateDispatchAgent.Run` drains every pending event and calls
  `SendHTMLMessage`; `rate_user_events.source_name` is nullable.)
- "Local morning" for the summary uses the **subscribed city's** IANA timezone
  (returned by Open-Meteo geocoding and stored on the city row). For a user
  tracking their own city this equals their local time. See Trade-offs for the
  profile-timezone alternative.
- Storing a chosen city's **public coordinates** is acceptable under the
  project's data-privacy policy (it is a user-volunteered weather *preference*,
  not device location). See Risks/Trade-offs for the full classification.
- `cmd/web` continues not to parse `BEACON_PROXY_URL`; the geocoding lookup and
  the Open-Meteo health probe issued from `cmd/web` go direct. (Documented
  caveat in Risks.)
- Migration counter: this plan owns **011–016** (`202606.011.*` … `202606.016.*`).
  009–010 belong to a parallel stocks plan and are not used here. Only 011 and
  012 are actually needed (see Task 2); 013–016 are left free.
- Telegram is the only delivery channel (`domain.UserTypeTelegram`), matching
  the FX side.

## Tasks

### Task 1: Weather domain types, condition map, and timezone-aware due-check

- **Description:** Add the weather domain in `internal/domain/`. New value types
  (no business I/O, mirroring the existing domain style):
  - `weatheruser.go`: `WeatherUserCity` — `ID`, `UserType`, `UserID`,
    `LocationID`, `DisplayName`, `Latitude`, `Longitude`, `Timezone` (IANA),
    `Country`, `Admin1`, `GismeteoCityID *int` (nil until the curated map is
    consulted in the gismeteo increment), `NotifyKind WeatherNotifyKind`,
    `NotifyHour int` (local 0–23), `LastNotifiedAt time.Time` (zero = never),
    `UpdatedAt`, `CreatedAt`.
  - `WeatherNotifyKind` string enum with one MVP value
    `WeatherNotifyMorningSummary = "morning_summary"`. This is the extension
    point that future kinds (`rain_alert`, `threshold_heat`, …) plug into,
    analogous to `SubscriptionConditionType`.
  - `weatherobservation.go`: `WeatherObservation` — `ID`, `LocationID`,
    `Provider` (`"open-meteo"` / `"gismeteo"` — literal data tokens, matched in
    code, never translated), `Latitude`, `Longitude`, `CapturedAt`,
    `ForecastDate` (YYYY-MM-DD, city-local), plus nullable forecast fields
    (`TempMax`, `TempMin`, `PrecipSum`, `PrecipProbMax`, `WeatherCode`,
    `Sunrise`, `Sunset`, and the current-snapshot fields `TempCurrent`,
    `TempFeels`, `Humidity`, `WindSpeed`, `WindDir`, `Precip`, `CloudCover`).
    Use pointer or `sql.Null*`-friendly types for the nullable columns so a
    provider that omits a field stores NULL rather than a misleading zero.
  - `WMOWeatherCode(code int) (text string, emoji string)`: the WMO code →
    human description + emoji map (0 clear; 1–3 mainly clear → overcast;
    45/48 fog; 51–67 drizzle/rain; 71–86 snow; 95–99 thunderstorm). Descriptions
    are English prose; emojis are display tokens.
  - `IsMorningDue(now time.Time) (bool, error)` on `WeatherUserCity`:
    timezone-aware analog of `RateUserSubscription.IsDailyDue`. Loads the city's
    own `Timezone`, computes today's local fire time at `NotifyHour:00`, and
    fires once per local day. **Does not reuse `IsDailyDue`**, which is
    hard-coded to UTC.
  - Add `identity.Kind` constants in `internal/domain/identity/identity.go`:
    `KindWeatherUserCity = "WUC"`, `KindWeatherObservation = "WOB"`. Prefixes
    are frozen-on-disk going forward, like the existing ones.
- **Acceptance Criteria:**
   - [ ] `WeatherUserCity`, `WeatherObservation`, `WeatherNotifyKind`, and the
         new `identity.Kind` constants compile and carry godoc on every exported
         identifier (starts with the name, ends with a period).
   - [ ] `TestWeatherUserCity_IsMorningDue` (one top-level test, scenarios as
         subtests) covers: before local fire time → not due; after fire time,
         never notified → due; after fire time, already notified today (local)
         → not due; notified yesterday → due; unknown/empty timezone → returns
         a non-nil error (caller decides fallback). Uses a fixed `now` and a
         non-UTC zone (e.g. `Asia/Almaty`) to prove local-day semantics.
   - [ ] `TestWMOWeatherCode` asserts representative codes map to the expected
         (text, emoji) pairs and that an unknown code returns a safe default
         (e.g. "Unknown", a neutral emoji) rather than an empty string.
   - [ ] `TestNew` in the identity package is extended (or its kind list) to
         include the two new kinds; prefixes are asserted.
- **Pitfalls & edge cases:** Local-day boundaries vs UTC (a 07:00 Almaty fire is
  01:00 UTC the same calendar day, or the previous UTC day for western zones) —
  compute everything in `loc`, never mix. DST transitions: rely on
  `time.LoadLocation` + `time.Date(..., loc)` which handle the offset; do not
  hand-roll offsets. `time.LoadLocation` needs tzdata — `cmd/notifier` already
  imports `_ "time/tzdata"`, so the agent has it, but the domain test must too
  (add the blank import in the test or rely on the host; prefer the embedded
  import to stay host-independent). Nullable observation fields: a zero
  temperature is a real value, so distinguish NULL from 0 with pointers.
- **Complexity:** Medium
- **Code Example:**
  ```go
  // IsMorningDue reports whether the daily morning summary should fire now,
  // evaluated in the city's local timezone. now must be UTC. It fires once per
  // local calendar day at NotifyHour. Returns an error if the stored timezone
  // is not loadable.
  func (c *WeatherUserCity) IsMorningDue(now time.Time) (bool, error) {
      loc, err := time.LoadLocation(c.Timezone)
      if err != nil {
          return false, fmt.Errorf("weather city %s: load timezone %q: %w", c.ID, c.Timezone, err)
      }
      local := now.In(loc)
      fire := time.Date(local.Year(), local.Month(), local.Day(), c.NotifyHour, 0, 0, 0, loc)
      if local.Before(fire) {
          return false, nil
      }
      if c.LastNotifiedAt.IsZero() {
          return true, nil
      }
      return c.LastNotifiedAt.In(loc).Before(fire), nil
  }
  ```

### Task 2: Migrations 011 (weather_user_cities) and 012 (weather_observations)

- **Description:** Add two migration files under `migrations/`, following the
  frozen naming convention `<YYYYMM>.<NNN>.<table>.<description>.sql`:
  - `202606.011.weather_user_cities.table_initiate.sql` — per-user city
    subscription. Columns mirror `WeatherUserCity`. **No foreign key** (weather
    has no global source catalog; `location_id` is a plain key — this is exactly
    what keeps weather out of the FX `rate_sources` cascade web). Indexes on
    `(user_type, user_id)` and `(location_id)`. A `UNIQUE (user_type, user_id,
    location_id, notify_kind)` constraint prevents duplicate subscriptions.
  - `202606.012.weather_observations.table_initiate.sql` — provider forecast
    snapshots, shared across all users tracking a `location_id`. Columns mirror
    `WeatherObservation`; forecast fields are nullable. Indexes on
    `(location_id, provider, captured_at DESC)` (latest snapshot lookup) and
    `(location_id, provider, forecast_date)`.
  - No new event table: weather reuses `rate_user_events` (Task 6).
- **Acceptance Criteria:**
   - [ ] Both files apply cleanly via `make migrate` on a fresh DB and are
         idempotent (`IF NOT EXISTS` on tables and indexes).
   - [ ] `migrator_test.go` (or the existing migration test harness) sees the
         two new files and the applied-count assertion is updated.
   - [ ] Column types are SQLite-appropriate: `TEXT` for ids/strings/timestamps,
         `REAL` for floats, `INTEGER` for `notify_hour`, `weather_code`,
         `gismeteo_city_id`. Timestamps stored as RFC3339 `TEXT` (matches every
         existing table).
   - [ ] No FK references to `rate_sources` from either table.
- **Pitfalls & edge cases:** Filenames are **immutable once applied** — get the
  number and table name right the first time. The global `<NNN>` counter is at
  008; 009–010 are reserved for the stocks plan, so weather starts at 011 — do
  not reuse 009/010. Nullable forecast columns must omit `NOT NULL`. Keep the
  `UNIQUE` constraint aligned with the repository's upsert/ownership logic
  (Task 3) so a re-subscribe updates rather than errors.
- **Complexity:** Easy

### Task 4: Open-Meteo provider (geocoding + forecast) infrastructure

- **Description:** Add `internal/infrastructure/weather/openmeteo.go` (new
  `weather` infrastructure package). Two responsibilities, one proxy-aware HTTP
  client:
  - `Geocode(ctx, name string, count int) ([]GeoResult, error)` — GET
    `https://geocoding-api.open-meteo.com/v1/search?name=..&count=..&language=ru`,
    decode `results[]` into `GeoResult{ID, Name, Latitude, Longitude, Country,
    CountryCode, Admin1, Timezone, Population}`. Used at subscribe time to turn a
    typed city into coordinates + IANA timezone.
  - `Forecast(ctx, lat, lng float64) (*domain.WeatherObservation, error)` — GET
    the forecast URL with the exact query string the feature needs:
    `current=temperature_2m,apparent_temperature,relative_humidity_2m,wind_speed_10m,wind_direction_10m,precipitation,weather_code,cloud_cover`
    `&daily=temperature_2m_max,temperature_2m_min,precipitation_sum,precipitation_probability_max,weather_code,sunrise,sunset`
    `&hourly=precipitation_probability,temperature_2m&timezone=auto`. Decode the
    parallel arrays, take index 0 of `daily[]` for "today", and map into a
    `WeatherObservation{Provider: "open-meteo", ...}`. `timezone=auto` makes
    sunrise/sunset/daily local; capture `utc_offset_seconds` if needed for
    display.
  - HTTP client construction mirrors `rateextractor.NewRateExtractor`: a
    `*http.Transport` whose `Proxy` is set from a passed `proxyURL` (empty =
    direct), a bounded `Timeout`, and a `User-Agent` header. Do **not** consult
    `HTTPS_PROXY`/`HTTP_PROXY` env — proxy is injected explicitly, same policy
    as the rest of the app.
  - The location key: provide `LocationKey(geo GeoResult) string` returning the
    canonical `location_id` used across cities and observations — prefer the
    Open-Meteo geocoding `id` as a string; fall back to lat/lng rounded to 4
    decimals (`"43.2389,76.8897"`) when no id is present. Document this so the
    collector and the city subscription agree on the key.
- **Acceptance Criteria:**
   - [ ] `TestOpenMeteo_Geocode` and `TestOpenMeteo_Forecast` run against an
         `httptest.Server` serving captured fixture JSON (committed under
         `internal/infrastructure/weather/testdata/`), asserting the decode and
         the field mapping. No live network in tests.
   - [ ] Forecast decode is robust to short/empty `daily[]` arrays (returns a
         clear error, never panics on `daily.Temperature[0]`).
   - [ ] Non-2xx responses and malformed JSON return wrapped errors carrying
         `internal.NewTraceError()`, consistent with the codebase.
   - [ ] A non-empty `proxyURL` routes requests through the configured proxy
         (assert via a stub proxy/transport in the test).
   - [ ] `WeatherObservation.Provider == "open-meteo"` (literal) and
         `WeatherCode` carries the raw WMO integer; the description is NOT
         resolved here (the WMO map is applied at render time, Task 7).
- **Pitfalls & edge cases:** Open-Meteo returns metrics as parallel arrays
  indexed by `time[]`; never assume alignment without checking lengths. `daily`
  arrays are one-element-per-day — index 0 is today only under `timezone=auto`;
  document the assumption. `weather_code` is WMO, not a provider-specific code.
  `language=ru` affects geocoding *display* names only (prose returned to the
  user for the picker); it does not change ids/coords. Respect `ctx`
  cancellation/timeout on every request. Keep the fixture JSON small but
  structurally faithful (current + daily + hourly blocks).
- **Complexity:** Medium

### Task 5: Weather collection agent + wire into cmd/collector

- **Description:** Add `internal/application/collection/weatheragent.go`:
  `WeatherAgent` with `Run(ctx) error`, constructed with the Open-Meteo provider,
  the city repo (for `ObtainDistinctWeatherLocations`), the observation repo, and
  a logger. On each invocation it:
  1. Loads the distinct locations from `weather_user_cities`.
  2. Self-throttles per location (skip if the latest observation for
     `(location_id, "open-meteo")` is newer than a configured interval, e.g. 1h —
     analogous to the FX `isDue` execution-history gate, so frequent cron
     invocations don't hammer the API).
  3. Fetches Open-Meteo for each due location and `RetainWeatherObservation`.
  4. Returns a joined per-location error (one bad location never aborts the rest),
     mirroring `RateAgent`.
  Then wire it into `cmd/collector/main.go` `buildRunners`: construct the
  Open-Meteo provider with the already-resolved `proxyURL` and add the
  `WeatherAgent` to the returned `[]runner`. The existing per-runner
  panic-recover shim in `main` already isolates it. gismeteo is **not** collected
  here yet (Task 12).
- **Acceptance Criteria:**
   - [ ] `TestWeatherAgent_Run` (subtests): no locations → no-op, no error;
         one location → one observation stored; provider error on one location →
         that location's error is in the joined result, others still stored;
         throttle → a location with a fresh observation is skipped.
   - [ ] The collector binary builds and `buildRunners` returns both the FX
         `RateAgent` and the `WeatherAgent`.
   - [ ] The agent uses `proxyURL` from `cmd/collector` (so Open-Meteo egress
         honours `BEACON_PROXY_URL`).
   - [ ] No outbound call when there are zero weather subscriptions (cheap
         no-op, like the chromedp empty-batch path).
- **Pitfalls & edge cases:** Do not let one slow/failing location stall the
  batch — bound each fetch by the provider client timeout and join errors.
  Persistence of observations should survive a mid-run SIGTERM the same way
  `RateAgent.execution` uses `context.Background()` for the write that follows a
  completed fetch (consider the same pattern so a fetched-but-unstored
  observation isn't lost). Throttle interval must be < the morning-summary
  window so a fresh-enough observation always exists by fire time.
- **Complexity:** Medium

### Task 6: Weather morning-summary check agent + wire into cmd/notifier

- **Description:** Add `internal/application/notification/weathercheckagent.go`:
  `WeatherCheckAgent` with `Run(ctx) error`, constructed with the city repo, the
  observation repo, the event repo (`rateCheckEventRepository`, the same narrow
  `RetainRateUserEvent` interface `RateCheckAgent` uses), and a logger. On each
  invocation:
  1. Load candidate cities (`ObtainDueWeatherUserCities(ctx,
     WeatherNotifyMorningSummary)`).
  2. For each, evaluate `city.IsMorningDue(now)` with `now := time.Now().UTC()`.
     On a timezone-load error, log once and skip (never fatal) — same
     "wrong-tz-beats-missed-notification" stance as `resolveUserTimezone`.
  3. For due cities, load the latest Open-Meteo observation for the location,
     render the summary (Task 7), and queue a `domain.RateUserEvent{UserType:
     Telegram, UserID: city.UserID, Message: <html>, SourceName: ""}` via
     `RetainRateUserEvent`. **SourceName is empty** → stored as NULL → no FK to
     `rate_sources`; the existing dispatcher delivers it unchanged.
  4. `AdvanceLastNotifiedAt(ctx, city.ID, now)` so it won't re-fire today.
  5. Emit a one-line proof-of-execution marker via the injected logger, like
     `RateCheckAgent`.
  Then wire it into `cmd/notifier/main.go`: build the two new repos
  (`WeatherUserCityRepository`, `WeatherObservationRepository`), construct
  `WeatherCheckAgent`, and run it **before** `dispatchAgent.Run(ctx)` so freshly
  queued weather events are delivered in the same invocation. No change to
  `RateDispatchAgent` — it already drains all pending events.
- **Acceptance Criteria:**
   - [ ] `TestWeatherCheckAgent_Run` (subtests): due city with an observation →
         one event queued with non-empty `Message` and empty `SourceName`, and
         `last_notified_at` advanced; not-due city → nothing queued; due city
         with **no** observation → skipped with a logged warning, no event, and
         `last_notified_at` NOT advanced (so it can fire once data arrives);
         timezone-load error → skipped, not fatal.
   - [ ] An integration-style test confirms a weather event with empty
         `SourceName` round-trips through `RateUserEventRepository.RetainRateUserEvent`
         and is returned by `ObtainUnprocessedRateUserEvents` (proves the
         reuse-the-FX-queue decision holds).
   - [ ] `cmd/notifier` builds and runs `WeatherCheckAgent` then the FX check
         then dispatch; a clean `context.Canceled` on shutdown is not
         double-logged (match the existing pattern).
- **Pitfalls & edge cases:** Ordering — advance `last_notified_at` only after the
  event is successfully queued; if `RetainRateUserEvent` fails, do not advance
  (so the next run retries), matching how the FX agent only relies on retained
  events. Idempotency within a single local day is enforced by `IsMorningDue` +
  `last_notified_at`; if the notifier runs every few minutes, only the first run
  after the fire time queues. A user with no observation yet (just subscribed,
  collector hasn't run) must not be marked notified. Empty `SourceName` must map
  to SQL NULL (verify the repo's existing `sourceNameForDB` behaviour covers
  this — the FX multi-source path already relies on it).
- **Complexity:** Medium
- **Code Example:**
  ```go
  // queue a weather summary as a generic outbound Telegram event; SourceName
  // is left empty so it stores as NULL (no rate_sources FK) and the existing
  // RateDispatchAgent delivers it without any weather-specific transport code.
  ev := &domain.RateUserEvent{
      UserType: domain.UserTypeTelegram,
      UserID:   city.UserID,
      Message:  htmlSummary, // rendered in Task 7
  }
  if err := a.eventRepo.RetainRateUserEvent(ctx, ev); err != nil {
      errs = append(errs, fmt.Errorf("queue weather summary city=%s: %w", city.ID, err))
      continue // do NOT advance last_notified_at on failure
  }
  if err := a.cityRepo.AdvanceLastNotifiedAt(ctx, city.ID, now); err != nil {
      errs = append(errs, err)
  }
  ```

### Task 10: Privacy review + minimal-storage enforcement (documentation task)

- **Description:** This is a deliberate, documented data-privacy decision, not
  code beyond what Tasks 1–3 already store. Capture it in the repo's data-privacy
  documentation (a short addition to `CLAUDE.md`'s "Data & Privacy" section,
  classifying the new field):

    coordinates of the chosen city**, IANA timezone, and (optionally) country /
    admin1 for display; plus the notify config and `last_notified_at`. Keyed by
    Telegram `chat_id` (`user_id`), exactly like every other user-scoped table.

    "precise location (lat/lng)". The stored value here is **not** device GPS or
    observed location — it is the public centroid of a city the user *typed and
    picked* as a weather preference. It is user-volunteered and functionally
    closer to the already-approved `timezone`/`locale` preferences than to
    location surveillance. The off-limits intent is "no tracking of where the
    user actually is"; a chosen-city preference does not reveal that.

    minimum required to call the forecast API, with explicit guardrails below.
  - **Guardrails / what is NOT stored:** no device location, no GPS, no IP
    address, no user-agent, no `@username` / display name / phone / email. The
    coordinates are city-granularity (geocoding centroid), never device

    geocoding proxy. `X-Telegram-Init-Data` stays out of access logs (header
    only).

- **Pitfalls & edge cases:** Easy to drift: a future "nearest city to my

## Risks

- **gismeteo fragility (high).** No API, JS-rendered, scraped via chromedp
  against selectors that break whenever the site changes; coverage limited to a
  hand-curated city→ID map. Mitigation: isolate selectors in constants, fail
  loudly into execution-history/failed-events, treat gismeteo as the *comparison*
  and Open-Meteo as the *source of truth*, and keep the whole gismeteo path in
  the second increment so the MVP never depends on it.
- **chromedp reuse refactor (medium).** Extracting the fetch primitive (Task 11)
  touches working FX code if the delegation step is taken. Mitigation: the
  primitive is new code (low risk); the rate-extractor delegation is an optional,
  separately-bounded follow-up that must keep every existing `chromedp_test.go`
  assertion green, and can be skipped if review deems it risky.
- **Privacy classification (medium).** Storing city coordinates touches the
  off-limits "precise location" line. Mitigation: Task 10 documents the
  classification (chosen-city *preference*, not device location), enforces
  minimal/city-granularity storage, and forbids device-location features without
  re-review.
- **Third-party outage failing the deploy gate (medium).** A live Open-Meteo
  probe in `/health/check` could 503 the readiness gate during an Open-Meteo
  outage and trigger a rollback for something the operator can't fix. Mitigation:
  weather inspectors are **advisory** (Task 14) — reported but non-gating.
- **`cmd/web` outbound egress without proxy (low/medium).** Geocoding (Task 8)
  and the health probe (Task 14) issue outbound calls from `cmd/web`, which does
  not parse `BEACON_PROXY_URL`. If the host mandates proxy egress for these
  providers, direct calls will fail. Mitigation: documented caveat; if needed,
  add proxy parsing to `cmd/web` (a deliberate deviation from the current "web
  doesn't parse proxy" rule) or derive weather health from collection success.
- **Local-time / DST correctness (low).** Morning-summary timing is timezone- and
  DST-sensitive. Mitigation: all date math in the city's `*time.Location` via
  `time.Date(...)`; embedded `time/tzdata`; explicit subtests for non-UTC zones
  and the never/already-notified-today cases.
- **Reusing `rate_user_events` for weather (low).** Weather rows carry NULL
  `source_name` and appear in `/api/notifications`. Mitigation: the column is
  already nullable and the dispatcher is transport-only; a future split is a
  clean migration if semantics diverge.

## Trade-offs

- **Option B (parallel weather domain) over Option A (extend rate_sources /
  rate_values) — chosen.** `rate_values` stores exactly one scalar
  (`price REAL`) per row, and `rate_sources` is a **global, operator-seeded**
  catalog whose rows carry regex/JSONPath extraction *rules* and are consumed by
  the rulegen/audit tooling. Weather is (1) **multi-dimensional** (temp,
  feels-like, humidity, wind, precip, condition code, daily + hourly) — it cannot
  be a single `price`, and the rule pipeline (`applyRulesAndStore`) extracts one
  float, so it cannot parse a structured weather JSON; (2) **per-user-city** —
  each user picks a city, so the addressing is user data, not a global catalog,
  and forcing it into `rate_sources` would pollute the operator catalog and
  entangle user cities with the `rate_sources` ON DELETE CASCADE shared by
  `rate_values` / `rate_user_subscriptions` / `rate_user_events`; (3)
  **cross-provider comparison** of multiple fields, not MAX/MIN of one number.
  Option A would require either a combinatorial explosion of pseudo-sources
  (one per city × provider × metric) or gutting the meaning of the FX tables.
  Option B isolates the genuine differences (storage shape, addressing,
  collection) while Option B-with-reuse keeps the genuinely-generic parts shared.
- **Reuse the FX notification queue + dispatcher, not a weather-specific one.**
  `domain.RateUserEvent.Message` is already free-form and `RateDispatchAgent` is
  transport-only (it drains all pending events and calls `SendHTMLMessage`), and
  `rate_user_events.source_name` is nullable. Weather queues events with empty
  `SourceName` → zero new dispatch code, one TTL/vacuum, one Telegram
  rate-limiter. Cost: weather rows carry NULL source attribution and show up in
  `/api/notifications`. Accepted; split later if needed.
- **Weather collection in `cmd/collector`, weather check in `cmd/notifier` — not
  a new binary.** The collector already resolves `BEACON_PROXY_URL` and
  `BEACON_CHROMIUM_PATH`, is one-shot, and is driven by a host cron wrapper; the
  notifier already has the dispatcher, the profile repo, and embedded tzdata.
  Adding a new binary would mean a new cron wrapper and deploy plumbing for no
  benefit, since the weather agent self-throttles per location. Rejected the
  separate-binary option on ops-surface grounds.
- **Timezone-aware due-check instead of reusing `IsCronDue`.** The FX cron logic
  (`RateUserSubscription.IsCronDue` / unexported `parseCronSchedule`) is hard-coded
  to UTC, but the morning summary must fire at the **city's local** morning. The
  MVP needs only "daily at local HH:MM", so Task 1 adds a small timezone-aware
  `IsMorningDue` rather than refactoring the FX cron path. If a future weather
  kind needs full cron, export `parseCronSchedule` into a shared helper and
  evaluate `schedule.Next(last.In(loc))` — deferred until actually needed, to
  avoid touching working FX code now.
- **"Local morning" = the subscribed city's timezone, not the user's profile
  timezone.** The city's IANA timezone comes back from geocoding and is stored on
  the city row, so no extra lookup is needed, and a weather summary "for the day
  in city X" naturally uses city X's morning. For a user tracking their own city
  these coincide. Alternative: use `rate_user_profiles.timezone` (the FX
  notifications already do, via `resolveUserTimezone`). **Decision point for the
  user:** which timezone defines "morning" when a user tracks a city in a
  different zone than they live in. Default chosen: city timezone.
- **Curated gismeteo city→ID map as a compiled Go map, not a DB seed table.**
  gismeteo support is inherently code-coupled (bespoke scraping), the operator is
  the sole maintainer, and a Go map is version-controlled and testable with no
  migration. Alternative (DB seed table, consistent with how `rate_sources` are
  seeded) is noted but rejected for the MVP-of-gismeteo to avoid an extra
  migration and operator DB editing. This is the documented limitation: gismeteo
  works only for curated cities; Open-Meteo is global.
- **Typed forecast columns for the MVP, hourly deferred.** `weather_observations`
  gets typed columns for the fields the morning summary uses (plus the current
  snapshot Open-Meteo returns in the same call). Hourly arrays (needed only for
  the deferred rain alert) are not stored yet — adding them later is an additive
  migration. Alternative (a `raw_json` blob for zero-migration forward-compat) is
  rejected to keep rows lean and queries honest; revisit if the deferred kinds
  land close together.

## Post-Review Divergences & Resolutions

Recorded per the "keep plans honest" rule. The MVP shipped (Tasks 1-10 + the
Open-Meteo half of Task 14) with a green `make test` (-race + WASM, 33 packages).
gismeteo (Tasks 11-13 + the gismeteo inspector half of Task 14) remains a separate
second increment. The following diverged from the plan or were added during the
five-lens review + targeted re-review:

1. **`RetainWeatherUserCity` uses `RETURNING id` (correctness).** The ON CONFLICT
   upsert preserves the original row id; without `RETURNING`, the Go `record.ID`
   held a freshly-minted phantom that the create handler would have returned in the
   201 body (a later DELETE by that id → 404). Now the actual stored id is scanned
   back.
2. **Sunrise/sunset stored as correct UTC (correctness).** Open-Meteo `timezone=auto`
   returns offset-less local ISO strings; `time.Parse` mislabeled them UTC. Decode
   now uses `time.ParseInLocation` with the provider's timezone (correct UTC instant
   stored); the renderer converts to the city's timezone for display. An unloadable
   timezone returns a clear error instead of silently storing a wrong instant.
3. **Provider tokens are exported domain constants.** `domain.ProviderOpenMeteo` /
   `domain.ProviderGismeteo` replace scattered string literals across the provider,
   the collection agent, the check agent, and the renderer — a bare literal in the
   check agent's observation lookup would have silently returned zero summaries on a
   token typo.
4. **Response reads bounded + logs scrubbed (security/privacy).** `io.LimitReader`
   (1 MB) caps Open-Meteo response reads (OOM defense). HTTP-error messages now log
   `host+path` only, never the query string — which carried lat/lng (forecast) and
   the search term (geocode). `WeatherAgent` logs per-location errors by location id
   only, never coordinates.
5. **`WeatherAgent` per-tick observability.** The agent now emits a
   `fetched/skipped/failed/total` proof-of-execution marker and per-location error
   lines, so a sustained Open-Meteo failure is visible in the collector log rather
   than only surfacing hours later as a missing summary. (Full `execution_history`-
   style queryable observability for weather is deferred; logging suffices for MVP.)
6. **Advisory health-inspector tier.** `inspector.NewAgentWithAdvisory` is additive
   (`NewAgent` unchanged); `OpenMeteoInspector` is registered advisory so a
   third-party outage is reported in `/health/check` but does not flip the aggregate
   to 503 / fail the deploy gate. Critical inspectors (SQLite, Telegram) are
   unchanged.

### Confirmed decisions

- "Local morning" = the subscribed **city's** timezone (stored from geocoding).
- `cmd/web` geocoding + health-probe egress is **direct** (it does not parse
  `BEACON_PROXY_URL`) — documented caveat; on a host that mandates proxy egress the
  city search would need `cmd/web` proxy wiring (deferred).
- Weather reuses the FX `rate_user_events` queue + `RateDispatchAgent` with an empty
  (NULL) `source_name`; no weather-specific transport.

### Explicitly deferred (rejected for the MVP, with rationale)

- **`WeatherGatewayDeps` struct** to collapse the 12-parameter `NewGateway` /
  `NewRouter` constructors (a reviewer Major) — deferred: the two trailing weather
  params are distinct interface types, so the param-swap risk a struct would prevent
  is not compiler-reachable; the wiring is correct and tested. Do the struct when the
  gismeteo increment adds the next weather dependency.
- **gismeteo provider, the shared chromedp fetch primitive, and cross-provider
  comparison** (Tasks 11-13) — the heavy/fragile second increment; the MVP never
  depends on it.
- **Rain / threshold / on-demand notification kinds** — always planned as deferred;
  the `WeatherNotifyKind` enum and the variadic renderer leave room for them.
