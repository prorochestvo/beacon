# Task Breakdown

## Overview

Smart-locate (`internal/application/rulegen/sanitizer.go::Locate`) is producing the
wrong window for pages where a bare currency code appears in `<head>` metadata long
before the actual rate table appears in `<body>`. The live BCC page
(https://www.bcc.kz/en/personal/currency-rates/) is the smoking gun: "USD" first
occurs at byte ~700 inside a `<meta name="description">` tag, while the real
`<div class="text-lg">USD</div>` rate row starts at byte ~43,000. The current
algorithm centres a ±40 KB window on byte 700 and ships [0, 40,700] to the LLM —
the rate table at byte 43,000 sits 2.3 KB past the window edge, so the model
never sees rate data and confabulates a plausible-shape but non-matching regex.
Three OpenAI models (nano, mini, full) all fail identically on this body, which
confirms the failure is upstream of model choice.

This plan delivers three targeted fixes to `internal/application/rulegen/`:

1. Strip `<head>...</head>` in `Sanitize` before `Locate` runs, so meta/title/link
   tags can never anchor the window.
2. Replace `Locate`'s flat-anchor scan with a two-tier priority: structural HTML
   anchors first; bare currency codes only as a fallback when no structural anchor
   matched.
3. Add a one-line `RE2 CONSTRAINTS` clause to `prompt.go` forbidding repeat counts
   over 1000 (Go's RE2 `maxRepeatCount`).

No new dependencies. No DOM parsing. No `<body>` extraction. No retry-budget or
model-selection changes. Smart-locate stays byte-level pattern search; the change
is purely how we choose the anchor.

## Assumptions

- The live BCC page structure observed during the smoke-test session
  (`<head>` ~0–5 KB, marketing/nav ~5–43 KB, rate table starts ~43 KB with
  `<div class="text-lg">USD</div>`) is representative. The fixture
  `tmp/testdata/sources/bcc.html` (already referenced by `TestSanitize`) is the
  authoritative regression artefact. If it is missing locally, that subtest skips;
  the new subtests added by this plan must be deterministic and synthetic so they
  never depend on a fixture being on disk.
- All seeded BCC `rate_sources` patterns in
  `migrations/202605.007.rate_sources.seed_initial.sql` use
  `<div class="text-lg[^"]*">\s*<CCY>\s*</div>` followed by
  `<div class="text-right">` — i.e. they all anchor on `<div class="text-lg`.
  Including `<div class="text-lg` in Tier 1 therefore both fixes BCC and matches
  the structural marker every production BCC rule already uses. Other seeded
  sources (`nationalbank.kz`, `gateway.prod.qazpost.kz`) return JSON, where
  `<head>` does not exist and Tier 1 will simply not match — the function falls
  through to Tier 2 (currency codes), which is the desired behaviour for JSON
  bodies.
- The `<head>` strip uses a non-greedy regex `(?is)<head\b[^>]*>.*?</head>`. On
  malformed HTML with no `</head>` close tag, the regex matches nothing and the
  pipeline is a safe no-op. On documents with multiple `<head>` tags
  (pathologically malformed), only the first is stripped — also acceptable
  because real rate pages are well-formed enough for browsers to render.
- Tier 1 vs Tier 2 is an internal implementation detail of `Locate`. Callers do
  not need to know which tier matched; `Locate` retains its `([]byte, bool)`
  return shape. The `bool` still means "any tier matched, window is valid."
- `BaseCurrency` and `QuoteCurrency` continue to come from `*domain.RateSource`
  (see `generator.go:121`). The call site changes from a single flat slice to
  two slices passed in via the new signature.

## Tasks

### Task 1: Strip `<head>...</head>` in Sanitize before Locate

- **Description:**
  Add a precompiled `headRe` regex at the top of
  `internal/application/rulegen/sanitizer.go`, alongside the existing `scriptRe`
  and `styleRe`. Pattern: `(?is)<head\b[^>]*>.*?</head>` — case-insensitive,
  dotall, non-greedy. In `Sanitize`, apply `headRe.ReplaceAll(out, nil)` as a
  third step, immediately **after** the `styleRe` strip and **before** the call
  to `Locate`. Order matters because `<head>` may legally contain
  `<script>` and `<style>` blocks; stripping scripts/styles first means we
  never have stray closing tags inside the eventual `<head>` match (the
  non-greedy `.*?` would otherwise still be correct, but pre-stripping makes
  the intent obvious and the input to the head match cleaner).
- **Acceptance Criteria:**
  - [ ] `sanitizer.go` declares one new package-level var `headRe` with the
        pattern above, sibling to `scriptRe` and `styleRe`.
  - [ ] `Sanitize` removes `<head>...</head>` from input before invoking
        `Locate`.
  - [ ] `TestSanitize` gains a subtest
        `"removes head section so locate skips meta tags"` that constructs a
        synthetic body shaped like
        `<html><head><meta content="USD here"/></head><body>` + padding +
        `<div>USD rate row</div></html>`, calls `Sanitize` with the new
        two-slice signature from Task 2, and asserts the returned window
        contains the body-side USD context (the rate-row marker) but **not**
        the meta-tag text.
  - [ ] `TestSanitize` gains a subtest
        `"no head tag is a safe no-op"` that feeds a body with no `<head>`
        (e.g. a JSON document) and asserts `Sanitize` returns the same bytes
        as it would without the head strip.
  - [ ] Existing `TestSanitize` subtests still pass without modification
        (other than signature plumbing required by Task 2; see Pitfalls).
  - [ ] `make test` is green.
- **Pitfalls:**
  - The head-strip regex must be non-greedy. A greedy `.*` would swallow the
    entire body if multiple `</head>`-like sequences exist anywhere later (they
    shouldn't, but defence in depth).
  - The fixture-based `"bcc fixture produces window containing USD when
    fixture present"` subtest currently asserts the output contains "USD".
    After the head strip + tier change, the located window should still
    contain "USD" — but it must now contain the body-side USD, not the meta
    USD. Adjust the assertion to also check that the window includes a
    structural marker (`<div class="text-lg`) so the test would fail under
    the old code path.
  - `(?is)` must include both case-insensitive (`i`) and dotall (`s`) flags;
    omitting `s` makes `.*?` stop at newlines and the strip would not match
    multi-line `<head>` blocks (which is every real-world case).
  - Do not add a `<body>` extractor. Stripping `<head>` is sufficient; adding
    a `<body>` step would be redundant work and could fail on malformed HTML.
- **Complexity:** Easy
- **Code Example:**
  ```go
  var (
      scriptRe = regexp.MustCompile(`(?is)<script\b[^>]*>.*?</script>`)
      styleRe  = regexp.MustCompile(`(?is)<style\b[^>]*>.*?</style>`)
      headRe   = regexp.MustCompile(`(?is)<head\b[^>]*>.*?</head>`)
  )

  // inside Sanitize, before the Locate call:
  out := scriptRe.ReplaceAll(body, nil)
  out = styleRe.ReplaceAll(out, nil)
  out = headRe.ReplaceAll(out, nil)
  ```

### Task 2: Promote Locate to tiered anchor priority

- **Description:**
  Change `Locate` in `internal/application/rulegen/sanitizer.go` from a flat
  anchor slice to a two-tier scan. Tier 1 (structural anchors) is searched
  first; the smallest non-negative offset within Tier 1 wins. Tier 2 (currency
  codes) is only consulted when Tier 1 produced no match. When neither tier
  matches, behaviour is unchanged: return `(body, false)`.

  New signature:
  ```go
  func Locate(body []byte, structural, currency []string, windowBytes int) ([]byte, bool)
  ```
  Rationale for two slices instead of an `LocateAnchors` struct: see
  Trade-offs section. The function reads more directly at the call site; the
  argument order (structural first, currency second) matches the tier order so
  the priority is encoded in the signature itself.

  `Sanitize` gets the matching signature change:
  ```go
  func Sanitize(body []byte, structural, currency []string) ([]byte, int, error)
  ```

  Tier 1 (structural) list — built in `generator.go` and passed in at the call
  site:
  ```
  "<table",
  "<tbody",
  "<tr ",
  "<div class=\"text-lg",
  "<div class=\"rate",
  "class=\"currency\"",
  "class=\"exchange\"",
  "data-currency=",
  ```
  These are validated against the seeded BCC rules in
  `migrations/202605.007.rate_sources.seed_initial.sql`: every BCC pattern in
  that file uses `<div class="text-lg`, which is therefore in the list. None of
  the Tier 1 strings can appear inside a `<head>` element (they require
  `<table>`, `<tbody>`, `<tr>`, or rate-table-specific class names),
  and `<head>` is stripped earlier by Task 1 anyway, so the tier 1 list is
  doubly safe against meta-tag false positives.

  Tier 2 (currency codes) — `[]string{src.BaseCurrency, src.QuoteCurrency}`.

  In `generator.go::Generate` (around the current line `anchors := []string{...}`)
  replace the single slice with the two new slices and pass them through:
  ```go
  structural := []string{
      "<table", "<tbody", "<tr ",
      `<div class="text-lg`,
      `<div class="rate`,
      `class="currency"`,
      `class="exchange"`,
      `data-currency=`,
  }
  currency := []string{src.BaseCurrency, src.QuoteCurrency}
  body, originalSize, err := Sanitize(rawBody, structural, currency)
  ```
- **Acceptance Criteria:**
  - [ ] `Locate` signature is
        `func Locate(body []byte, structural, currency []string, windowBytes int) ([]byte, bool)`.
  - [ ] When at least one Tier 1 anchor matches, the window is centred on the
        smallest Tier 1 offset, regardless of where Tier 2 anchors match.
  - [ ] When no Tier 1 anchor matches, the function falls back to Tier 2 with
        the same smallest-offset rule.
  - [ ] When neither tier matches, the function returns `(body, false)` —
        identical to current behaviour.
  - [ ] `Sanitize` signature is
        `func Sanitize(body []byte, structural, currency []string) ([]byte, int, error)`
        and forwards both slices into `Locate`.
  - [ ] `generator.go::Generate` builds the structural list as a local literal
        slice and the currency list as `{src.BaseCurrency, src.QuoteCurrency}`,
        then passes both to `Sanitize`.
  - [ ] `TestLocate` is rewritten so each scenario is a `t.Run(...)` subtest
        under one top-level `TestLocate`, covering:
        - tier-1 hit wins over an earlier tier-2 hit
        - tier-2 hit is used when tier-1 has no match
        - neither tier hits returns `(body, false)`
        - empty Tier 1 slice falls straight through to Tier 2
        - empty Tier 2 slice with no Tier 1 hits returns `(body, false)`
        - duplicate anchors within a single tier are deduplicated
        - empty-string anchors are skipped
        - window clamps correctly at both ends (existing edge cases preserved)
  - [ ] `TestSanitize` is updated to pass both slices to `Sanitize` and gains
        a subtest demonstrating tier-1 priority end-to-end: a body with
        currency code at offset 100 and a `<table` at offset 5000 should
        produce a window centred near 5000, not 100.
  - [ ] No call site of `Sanitize` outside `generator.go` remains. Verify with
        `grep -r 'Sanitize(' internal/`.
  - [ ] `make test` is green.
- **Pitfalls:**
  - Do not include `<div ` (bare `<div` followed by space or `>`) in Tier 1 —
    every page has thousands of divs and the first one is essentially always
    in nav/marketing, not in the rate table. The structural anchors must be
    selective.
  - The Tier 1 string `<tr ` includes a trailing space deliberately (matches
    `<tr class="...">` but not `<tr>` with no attrs). This is a regression
    risk if a real page uses bare `<tr>`. Keep both `<tr ` and `<table` so
    `<table` alone catches the bare-`<tr>` case via the enclosing table tag.
  - Backtick-delimited Go string literals (e.g. `` `<div class="text-lg` ``)
    are the natural choice — they let the embedded `"` characters appear
    without escaping. Use them.
  - Argument-order reversal at the call site is the most likely silent
    regression. Make the parameter names `structural` and `currency` (not
    `tier1`, `tier2`) so misuse is harder.
  - `TestLocate` currently lives as a single function with `t.Run` subtests
    (good; conforms to CLAUDE.md "one `Test*` per method"). Preserve that
    shape — do not split into multiple top-level functions.
- **Complexity:** Medium
- **Code Example:**
  ```go
  func Locate(body []byte, structural, currency []string, windowBytes int) ([]byte, bool) {
      if hit, ok := firstHit(body, structural); ok {
          return windowAround(body, hit, windowBytes), true
      }
      if hit, ok := firstHit(body, currency); ok {
          return windowAround(body, hit, windowBytes), true
      }
      return body, false
  }

  func firstHit(body []byte, anchors []string) (int, bool) {
      seen := make(map[string]bool, len(anchors))
      best := -1
      for _, a := range anchors {
          if a == "" || seen[a] {
              continue
          }
          seen[a] = true
          i := bytes.Index(body, []byte(a))
          if i < 0 {
              continue
          }
          if best < 0 || i < best {
              best = i
          }
      }
      if best < 0 {
          return 0, false
      }
      return best, true
  }

  func windowAround(body []byte, centre, windowBytes int) []byte {
      half := windowBytes / 2
      start := centre - half
      if start < 0 {
          start = 0
      }
      end := centre + half
      if end > len(body) {
          end = len(body)
      }
      return body[start:end]
  }
  ```

## Risks

- **Risk: A page exists where Tier 1 has a false-positive match earlier than
  the real rate table.**
  Example: a navigation `<table>` for layout (rare in modern HTML but legal),
  or a `<div class="text-lg">` used for headings.
  *Mitigation:* The seeded BCC patterns all use `<div class="text-lg`
  immediately followed by a currency code and `<div class="text-right">` — the
  pattern is rate-table-specific in the BCC layout. For unseen sources, the
  ±40 KB window centred on a wrong-table `<table` is still very likely to
  contain the real rate table somewhere in the same 40 KB region, given that
  rate-listing pages typically dedicate most of the body to the listing.
  Worst case: the LLM still misses, and the audit loop catches it — same
  outcome as today, no new failure mode.

- **Risk: A JSON source returns text that contains `<table` somewhere (e.g.
  in an error message field).**
  *Mitigation:* JSON bodies are small (the seeded JSON sources return < 10 KB);
  even an off-centre window will include the whole response. The smallest
  practical risk; verified by inspecting nationalbank.kz and qazpost.kz
  payloads — neither contains HTML markup.

- **Risk: `<head>` regex strips a `<header>` element by mistake.**
  *Mitigation:* The pattern uses `<head\b` — the `\b` word boundary requires
  the next character to be a non-word character (`>`, space, `/`). `<header>`
  has `e` after `head`, which is a word character, so the boundary fails and
  the strip skips it. Add a subtest asserting `<header>` is preserved.

- **Risk: Argument-order regression at the call site.**
  `Sanitize(body, structural, currency)` vs `Sanitize(body, currency, structural)`
  produces silently wrong behaviour. The compiler can't catch it because both
  slices have type `[]string`.
  *Mitigation:* Name parameters `structural` and `currency` (not `a`, `b`),
  call them by those names in the call-site literal where possible, and
  include a unit test where the two tiers have non-overlapping anchors so a
  swap would flip the test from green to red.

- **Risk: A `<head>`-less or malformed page degrades behaviour.**
  *Mitigation:* The head-strip regex is non-greedy and matches nothing on
  missing tags. The JSON-source subtest in Task 1 (`"no head tag is a safe
  no-op"`) covers this explicitly.

## Trade-offs

- **Two-slice signature vs `LocateAnchors{Structural, Currency []string}` struct.**
  Chose two slices.
  *Pros:* The argument order encodes the tier priority directly. Fewer types
  to declare and import. Reads naturally at the call site:
  `Sanitize(body, structural, currency)`.
  *Cons:* Two adjacent `[]string` parameters create a silent-swap hazard; a
  named struct would make `Sanitize(body, LocateAnchors{Structural: ..., Currency: ...})`
  self-documenting at the call site.
  *Mitigation for chosen path:* Use the parameter names `structural` and
  `currency` everywhere; add a unit test that would fail on a tier swap.
  *Reversibility:* Cheap. If a third tier is ever needed (e.g. JSON anchors),
  this becomes a struct in a one-commit refactor.

- **Tier 1 vs Tier 2 as an internal detail of `Locate` vs an exposed tri-state
  return.**
  Chose internal.
  *Pros:* Callers don't care which tier won — they only care that the window
  is valid. Keeping the `([]byte, bool)` shape avoids cascading changes in
  every test that asserts the boolean.
  *Cons:* Loses observability — we can't log "matched tier=structural" from
  the call site without re-deriving it.
  *Mitigation:* `Locate` can still emit a debug log internally if needed
  later, without changing the signature.

- **`<head>` strip vs full `<body>...</body>` extraction.**
  Chose minimal strip.
  *Pros:* Smaller change. No risk on malformed HTML where `</body>` is
  missing. Stripping `<head>` cleanly handles the only documented failure
  mode (meta-tag false positive).
  *Cons:* If a page someday tucks rate data in a comment or non-body element
  outside `<body>`, we wouldn't help. That hypothetical isn't worth designing
  for.

- **Hard-coded Tier 1 list vs configurable list per source.**
  Chose hard-coded.
  *Pros:* No new config surface; no per-source migration. The same eight
  structural anchors cover every HTML-based seeded source today.
  *Cons:* A future source with idiosyncratic markup would need its anchors
  added to the list (still a one-line patch — and any such addition would be
  a Tier 1 candidate for all sources, not just one).
  *Reversibility:* Trivial. The list lives in `generator.go::Generate`; making
  it configurable later is a small refactor.
