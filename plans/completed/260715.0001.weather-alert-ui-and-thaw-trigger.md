# Task Breakdown

## Overview

Two loosely-coupled deliverables against the existing weather-alert subsystem:

**ITEM 1 — Activate the dead alert form in the Mini App.** The backend
(`WeatherCityCreateRequest` DTO, `CreateMeWeatherCity` handler, `WeatherUserCity.Validate`)
and nearly the whole WASM frontend (UI renderers in `cmd/wasm/ui/me_weather_cities.go`,
page controller methods in `cmd/wasm/application/me_weather_cities.go`) already support
creating `alert_heat` / `alert_frost` / `alert_thunderstorm` / `rain_alert` subscriptions.
The **only** missing wire is DOM event binding: `bindWeatherCitiesHandlers` in
`cmd/wasm/main.go` never binds the "+ Add alert" button, the kind `<select>`, the threshold
input, or the Save/Cancel buttons. Today a user can only ever create a `morning_summary`.
This item binds those five controls to the already-existing controller methods.

**ITEM 2 — Add a new `alert_thaw` kind (minus→plus / thaw transition).** No thaw trigger
exists. `alert_heat` (TempMax ≥ threshold) is unsuitable: threshold 0 fires all summer and
carries no "was frozen, now warming" meaning. This item adds a new `WeatherNotifyKind`
following the existing stateless single-observation pattern, wired end-to-end
(domain evaluator → check-agent alert phase → render → handler normalization → Mini App
form), and covered by ITEM 1's new DOM wiring.

### Chosen thaw design (decision recorded — see Trade-offs for the rejected alternative)

**Design 1 — stateless single-observation daily-cross.** `alert_thaw` fires when the
forecast day itself crosses zero: `TempMin ≤ 0 AND TempMax > 0`. This is evaluated against
the single latest Open-Meteo daily observation the check agent already loads — **no previous
observation, no new repository query, no change to how the alert phase threads observations
into the pure evaluator.** It is structurally identical to the existing `alert_frost`/
`alert_heat`/`alert_thunderstorm` cases (compare today's forecast to a fixed rule; the 20 h
forecast cooldown prevents same-day re-spam). `condition_value` is unused (empty is
canonical, like `alert_thunderstorm`). Semantics delivered: "today the temperature climbs
from sub-zero overnight to above-freezing — thaw / ice-then-melt conditions."

Design 2 (stateful crossing using yesterday's stored observation) was evaluated and
**rejected** — full justification in the Trade-offs section.

## Assumptions

1. **The user considers no-rain→rain and plus→minus already covered** by the existing
   `rain_alert` and `alert_frost` kinds (frost = `TempMin ≤ threshold`; with threshold 0
   that is exactly "drops below freezing"). The prompt frames ITEM 2 as *only* the reverse
   (thaw). **If the user actually wants dedicated "transition since yesterday" semantics for
   rain or frost too, this plan does not deliver that — flag before implementing.**
2. `notify_kind` is a plain `TEXT NOT NULL DEFAULT 'morning_summary'` column with **no CHECK
   constraint** (verified: `migrations/202606.011.weather_user_cities.table_initiate.sql`
   line 13, and the `condition_value` add in 013 imposes none). Therefore **ITEM 2 needs no
   schema migration.** The `notify_kind` index (migration 015) already covers any new value.
   If a future migration adds a CHECK on `notify_kind`, this assumption breaks.
3. Thaw uses only forecast data (temp_max / temp_min) — **no new personal field**, so the
   PII policy is unaffected.
4. `dom.On(app, "change", ...)` is available and behaves like the existing `"click"` /
   `"input"` delegated bindings (verified: `dom.On` is the binding primitive already used in
   `bindWeatherCitiesHandlers`). Delegated `"change"` on the container captures `<select>`
   change events via event bubbling.
5. The plan file number `261` is taken as given by the request; the highest existing prefix
   across `plans/`, `completed/`, `history/` is lower, so `261` does not collide.

     the frost/heat nil guards). Reason string:
     `fmt.Sprintf("Thaw: %s → %s", formatAlertTemp(*obs.TempMin), formatAlertTemp(*obs.TempMax))`.
  5. Update the `WeatherUserCity` struct's `ConditionValue` field godoc (lines ~58-59) to

  - [ ] `TestWeatherUserCity_AlertThreshold` gains a subtest: `alert_thaw` → error (no

  - [ ] The reason string uses `formatAlertTemp` so negatives render with U+2212.
  - [ ] `CGO_ENABLED=0 go test ./internal/domain/` passes; `-race` variant per CLAUDE.md.

  - Use strict `>` for TempMax (an all-zero `min=0,max=0` day is not a thaw). Using `≥`

  - One `Test*` per method, scenarios as `t.Run` subtests (CLAUDE.md) — add to the existing

- **Complexity:** Easy
- **Code Example:**
  ```go
  case WeatherNotifyAlertThaw:
      if obs.TempMin == nil || obs.TempMax == nil {
          return false, "", nil // cannot evaluate a crossing without both bounds
      }
      if *obs.TempMin > 0 || *obs.TempMax <= 0 {
          return false, "", nil // never froze, or never rose above freezing
      }
      return true, fmt.Sprintf("Thaw: %s → %s",
          formatAlertTemp(*obs.TempMin), formatAlertTemp(*obs.TempMax)), nil
  ```

## Risks

- **WASM glue is not unit-tested.** Task 6's correctness rests on manual verification
  (`make build` + running the Mini App). A bug in the event binding (wrong event type for
  `<select>`, redraw-on-keystroke focus loss, missing `closest` walk) will not be caught by
  `make test`. Mitigation: the risky logic is thin and mirrors four working sibling
  bindings; the substantive logic is in the tested `application`/`ui` packages. Reviewer
  lens C (ops/UX) and a manual smoke test are the real gate.
- **Emoji / wording guess.** The thaw header emoji (`🫠`) and label ("Thaw alert") are
  choices, not requirements — confirm with the user; cheap to change, but changing after
  release alters delivered notification text.
- **Semantic mismatch on "thaw".** If the user's mental model of thaw is strictly
  "yesterday was frozen, today is not" (a day-over-day transition) rather than "today crosses
  zero," Design 1 will not match expectations. This is the single most important thing to
  confirm before implementing (see Assumption 1). Design 1 fires on any zero-crossing day,
  which in shoulder seasons is most days — that is intended, but must be the user's intent.
- **No migration = no rollback concern**, but also means a typo in the `alert_thaw` string
  literal would silently create rows that no evaluator matches. Mitigation: the string
  literal appears once (domain const) and is referenced by constant everywhere else; the
  DTO/UI use the same literal — a round-trip create-then-fire test (Tasks 2 + 4) catches a
  mismatch.

## Trade-offs

- **Stateless daily-cross (Design 1, chosen) over stateful yesterday-crossing (Design 2,
  rejected).**
  - *Design 2* would fire only when the **previous** stored observation's `TempMin ≤ 0` and
    today's `TempMax > 0` — a true day-over-day transition. It requires: (a) a **new
    repository query** to fetch the prior observation (non-trivial: the collector writes many
    rows per day, each scrape a new row ordered by `captured_at DESC`, so "yesterday" means
    "latest row with `forecast_date` strictly less than today's" — a new indexed query and
    its tests); (b) **threading the previous observation into the check-agent alert phase**,
    which today passes only the single latest observation to the pure `EvaluateAlert`
    signature — this breaks the uniform `EvaluateAlert(obs, now)` contract shared by all
    kinds, forcing either a signature change or a thaw-specific side channel; (c) more
    fixtures and more failure modes (what if there is no prior observation yet?).
  - *Design 1* delivers the same user-visible value ("heads up, it crosses zero today, ice
    then melt") using data already on the one observation the agent loads, with **zero**
    changes to the evaluator contract, repository layer, or agent observation-threading. It
    is structurally identical to the four existing stateless kinds, so it inherits their
    cooldown/anti-spam machinery unchanged. The cost is semantic: it keys on the forecast
    day's own range, not a literal frozen→thawed transition across days. Given every existing
    kind is stateless and the 20 h cooldown already caps it to once per day, the added
    complexity of Design 2 is not justified by the marginal semantic gain. **Chosen: Design
    1.** If the user later insists on strict day-over-day semantics, Design 2 is a clean
    follow-up (new query + agent change), not a rewrite.

- **One combined edit to `bindWeatherCitiesHandlers` (Task 6) over two edits.** ITEM 1 and
  ITEM 2 are independent, but both need the same WASM binding function. Editing it once for
  all five kinds avoids churn and a second review of the same glue. Cost: Task 6 depends on
  Task 5, coupling the two items' completion. Accepted — the coupling is one file and the
  items can still ship as separate PRs by cherry-picking the thaw option line.

- **Blank `condition_value` for thaw in the handler (Task 4) but leave thunderstorm's
  pre-existing non-blanking alone.** Fixing thunderstorm's asymmetry is out of scope;
  widening the change invites unrelated review. Thaw is normalized to match morning_summary;
  the thunderstorm wart is noted, not touched.

- **No schema migration.** Relying on the CHECK-less `notify_kind` column keeps the change
  code-only and instantly reversible. The alternative (adding a CHECK constraint enumerating
  valid kinds) would be safer against typo'd inserts but requires a SQLite table rebuild for
  every future kind and is a larger, riskier change than this feature warrants. Not done.
