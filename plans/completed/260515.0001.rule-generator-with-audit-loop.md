# Task Breakdown

## Overview

Plan 008 delivered the `AIClient` interface and three concrete drivers (OpenAI, Groq, OpenRouter), constructed at startup in `cmd/web/main.go` but not yet consumed. This plan turns those clients into a working rule-generator: an operator-invoked CLI binary (`cmd/rulegen`) that, given a `rate_sources.name`, fetches the source URL, asks the LLM for an extraction rule, validates the rule by executing it against the fetched body, and persists the rule into the existing `rate_sources.rules` column on success.

The service follows an **audit loop**: up to N=3 attempts on the primary client (cheap/bulk model), with each failure fed back into the next prompt as additional context. If primary cannot get there, escalate to the fallback client with the FULL transcript of failures — one attempt only. If fallback also fails, exit non-zero with a structured log line and persist nothing.

Key design decisions (resolved by reading the codebase rather than left as TBD — see the **Trade-offs** section for full justification):

1. **CLI binary** `cmd/rulegen <source-name>` for v1. Composes with the existing `cmd/sourceaudit` workflow; defers background-loop cost-risk to plan 010+.
2. **Storage** reuses the existing `rate_sources.rules` TEXT JSON column. No migration needed for the rule itself. A small `rate_sources.rule_metadata` column is added via migration `202605.009.rate_sources.add_rule_metadata.sql` to record `{provider, model, attempts_used, generated_at}`. Full transcripts go to logs, not the DB.
3. **Fetching** reuses `internal/application/sourceaudit/fetcher.go` (`Fetcher` interface + `NewHTTPFetcher`) verbatim. No new fetcher.
4. **Rule executor** reuses `internal/tools/rateextractor.ApplyRegex` and `ApplyJSONPath`. A new helper `ExecuteRule(rule domain.RateSourceRule, body []byte) (float64, error)` wraps them with the same post-processing (`,` → `.`, strip spaces, `ParseFloat`, NaN/Inf/range guard) that lives today in `RateExtractor.Run` and `sourceaudit.Auditor.probeSource`. Extracting this once means three call sites converge on one definition.
5. **Schema alignment.** The placeholder `RateExtractionRuleSchema()` (selector_type ∈ {css, xpath, regex, json_path} with `post_processing`) was aspirational. Every seeded source in production uses just `regex` or `json_path` (the two methods `RateExtractor.Run` actually implements). We replace the placeholder schema with one that matches `domain.RateSourceRule` directly: `{method: "regex"|"json", pattern: string}`. This keeps the LLM output trivially round-trippable into the `rate_sources.rules` JSON column with zero translation layer. CSS/XPath stay out of scope (no goquery/htmlquery dependency added).
6. **Sanity check**: positive finite float with `0.0001 < value < math.MaxInt32`, matching the existing guards in `RateExtractor.Run` and `Auditor.probeSource`. No relative-to-history check in v1 — brittle for new sources.
7. **Body cap**: strip `<script>` and `<style>` blocks via regex; truncate to 200 KB. Reject and abort if body exceeds 5 MB before stripping (caller is expected to narrow the URL).
8. **`--force-fallback`** flag added: skips primary entirely. Cheap to support; the operator often knows a source is hard.

## Assumptions

- **N=3 primary attempts, 1 fallback attempt.** Confirmed from the problem framing. Configurable via `--max-primary-attempts` (default 3) so the operator can experiment without recompiling.
- **The CLI runs locally / on the deploy host with `SQLITEDB_DSN`, `AI_PRIMARY_DSN`, and optionally `AI_FALLBACK_DSN` exported.** Same env-var contract as `cmd/web`; the operator runs it the same way they run `cmd/migrator`.
- **The source row already exists.** `cmd/rulegen` does not create new sources; it only generates rules for sources already present in `rate_sources` (seeded via migrations or inserted manually). Acceptance: if the source name is unknown, exit non-zero with a clear stderr message.
- **The current `rate_sources.rules` content is overwritten on success.** No "merge with existing rules" — generated rules replace existing rules atomically. Operators who want to preserve an existing rule simply don't run `rulegen` on that source. (Trade-off discussed below.)
- **Cost is bounded by N=3 + 1 LLM calls per invocation.** No retry-on-network-error in v1; if the AI call itself fails (HTTP 5xx, timeout), that attempt counts toward the budget. This keeps the cost model simple and visible.
- **Logs are the audit trail.** Full primary/fallback transcripts, including the system prompt, user message, raw response, and the rule execution outcome, are written to the `cmd/rulegen` logger (`internal.NewLogger(LogsDir, "rulegen", ...)`). The DB only stores the final accepted rule + 4 metadata fields.
- **Stub fallback is acceptable when `AI_FALLBACK_DSN` is absent.** The stub returns a deterministic canned response (`stubAIDefaultResponse`). If primary fails 3 times and the operator did not configure a real fallback, the stub's canned response is run through the executor exactly like any other model response — it will almost certainly fail validation. The CLI then exits non-zero. This matches plan 008's design and mirrors the existing single-Groq-key deployment shape.

### Task 2: Replace the placeholder schema with the domain-aligned shape

- **Description:** The placeholder `RateExtractionRuleSchema()` in `internal/infrastructure/artificialintelligence/schema.go` defines a shape (`selector_type` ∈ {css, xpath, regex, json_path}, `selector`, `post_processing`) that does NOT match `domain.RateSourceRule` (`method` ∈ {parse_float, regex, json, store_as_rate}, `pattern`, `options`). Replace it with one that matches the domain exactly. The LLM is asked to return JSON that can be deserialized directly into `[]domain.RateSourceRule`.

  Constraints baked into the schema:
  - The top-level response is an OBJECT (OpenAI strict mode requires the root to be an object, not an array) with a single `rules` field of type array.
  - Each rule object has `method` ∈ {`regex`, `json`} and a non-empty `pattern` string. `parse_float` and `store_as_rate` are EXCLUDED because (a) the existing seed corpus never uses them, and (b) the executor in Task 4 only implements the two we care about. We keep the door open by exposing `domain.Method` constants but the schema enum is narrow — the LLM cannot pick `store_as_rate` even if it wanted to.
  - `additionalProperties: false` on every object, both fields required, strict mode.

  New `RateExtractionRuleSchema()` body:
  ```go
  func RateExtractionRuleSchema() map[string]any {
      return map[string]any{
          "type":                 "object",
          "additionalProperties": false,
          "required":             []string{"rules"},
          "properties": map[string]any{
              "rules": map[string]any{
                  "type":     "array",
                  "minItems": 1,
                  "items": map[string]any{
                      "type":                 "object",
                      "additionalProperties": false,
                      "required":             []string{"method", "pattern"},
                      "properties": map[string]any{
                          "method": map[string]any{
                              "type": "string",
                              "enum": []string{"regex", "json"},
                          },
                          "pattern": map[string]any{
                              "type":      "string",
                              "minLength": 1,
                              "description": "expression evaluated against the response body; capture group #1 for regex, dotted path for json",
                          },
                      },
                  },
              },
          },
      }
  }
  ```

  Update `internal/infrastructure/artificialintelligence/schema_test.go` to assert the new shape (root object, `rules` array, both fields required, `method` enum is `[regex, json]`).

  Update `internal/infrastructure/artificialintelligence/stubai.go` `stubAIDefaultResponse` so the canned JSON matches the new schema:
  ```json
  {"rules":[{"method":"regex","pattern":"USD / KZT[\\s\\S]{1,500}?<div[^>]*>(\\d+\\.\\d+)<\\/div>"}]}
  ```

- **Acceptance Criteria:**
  - [ ] `RateExtractionRuleSchema()` returns the new shape. Tests in `schema_test.go` assert each property explicitly.
  - [ ] `stubAIDefaultResponse` is valid JSON conforming to the new schema (unmarshal it via the json decoder in a test).
  - [ ] All existing tests in `internal/infrastructure/artificialintelligence/` continue to pass (`make test`).
  - [ ] `RateExtractionRuleSchemaName` is unchanged.

- **Pitfalls:**
  - **OpenAI strict json_schema requires the root to be an object, not an array.** Wrapping with `{ "rules": [...] }` is mandatory. Returning a bare array will be silently coerced or rejected depending on provider — both are bad.
  - **`minItems: 1` may not be honored by all providers under strict mode.** Treat it as advisory; Task 4's executor will reject empty rule lists at validation time as a defense in depth.
  - **Schema drift between OpenAI strict and Groq's "json_object" mode.** The Groq driver in `internal/infrastructure/artificialintelligence/groq.go` may use a different response_format keyword than OpenAI's `json_schema`. Re-read the driver to confirm it passes `RateExtractionRuleSchema()` through to the API. If Groq only supports `{type: "json_object"}` without a schema, the prompt itself must carry the contract — see Task 3.
  - **`stubAIDefaultResponse` is referenced from integration tests in future plans.** Changing its content is fine; changing its function signature or its export status is not.

- **Complexity:** Easy

### Task 3: Implement the rule-generator service in `internal/application/rulegen`

- **Description:** Create a new package `internal/application/rulegen` with two files: `generator.go` (the orchestrator) and `prompt.go` (the system prompt as a package-level `const`). The orchestrator is the heart of this plan and owns the audit loop.

  Public API:
  ```go
  package rulegen

  type Result struct {
      Source       *domain.RateSource    // source as persisted (rules + metadata updated)
      Rules        []domain.RateSourceRule
      Metadata     domain.RateSourceRuleMetadata
      Value        float64               // value produced by the accepted rule
      AttemptsUsed int                   // 1..N+1; the +1 is fallback
      Escalated    bool                  // true if fallback produced the accepted rule
  }

  type Generator struct {
      primary       artificialintelligence.AIClient
      fallback      artificialintelligence.AIClient
      fetcher       Fetcher               // see Task 4
      executor      RuleExecutor          // see Task 4
      sourceRepo    rateSourceRepository  // interface, see below
      maxPrimary    int                   // default 3
      logger        io.Writer
  }

  func NewGenerator(
      primary, fallback artificialintelligence.AIClient,
      fetcher Fetcher,
      executor RuleExecutor,
      sourceRepo rateSourceRepository,
      maxPrimaryAttempts int,
      logger io.Writer,
  ) (*Generator, error)

  func (g *Generator) Generate(ctx context.Context, sourceName string, forceFallback bool) (*Result, error)
  ```

  Private interface:
  ```go
  type rateSourceRepository interface {
      ObtainRateSourceByName(ctx context.Context, name string) (*domain.RateSource, error)
      RetainRateSource(ctx context.Context, record *domain.RateSource) error
  }
  ```

  Flow inside `Generate`:
  1. Load the source via `sourceRepo.ObtainRateSourceByName`. If nil/not-found, return a plain `error` (CLI prints to stderr).
  2. Fetch body via `fetcher.Fetch(ctx, source.URL)`. If body length > 5 MB, return an error.
  3. Pre-process: strip `<script>...</script>` and `<style>...</style>` blocks (case-insensitive regex), then truncate to 200 KB. Track the original size for the log.
  4. Build the initial `userMessage` containing: source metadata (name, base/quote currency, URL, hint from `Options.Reserve` if present) and the (stripped, truncated) body.
  5. Loop `attempt = 1..maxPrimary`:
      - Skip primary if `forceFallback` is true.
      - Call `client.Complete(ctx, systemPrompt, userMessage)`.
      - Parse the response as `{rules: [...]}`. On JSON parse error, record the failure transcript and continue.
      - For each rule, run `executor.Execute(rule, body)` chained: the output of rule N becomes the input to rule N+1, mirroring `RateExtractor.Run`. The final output must parse as a positive finite float in `(0.0001, math.MaxInt32)`.
      - On success: build the `Result`, persist via `sourceRepo.RetainRateSource` (updating `source.Rules` and `source.RuleMetadata`), return.
      - On failure: append a transcript entry `{attempt, rule_json, error_or_value, reason}` to a slice, and rebuild `userMessage` to include the failure history (see prompt template in Task 3's `prompt.go`).
  6. After primary exhaustion: run one fallback attempt using the FULL transcript. Same success/failure handling.
  7. If fallback also fails: log a structured failure record (one line, structured fields; `internal.NewLogger` uses textual levels — write at LogLevelError) and return an error. DO NOT persist.

  The system prompt (full text, drop into `prompt.go` as a `const` — under 50 lines):
  ```text
  You generate JSON rules for extracting a single numeric exchange-rate value from an HTTP response body.

  CONTRACT
  - Return ONLY a JSON object matching the rate_extraction_rule schema. No prose, no markdown, no comments. The driver enforces this; do not fight it.
  - Top-level shape: {"rules":[{"method":"regex"|"json","pattern":"..."}]}.
  - Rules apply sequentially: the output of rule N (bytes) becomes the input to rule N+1. The final output must parse as a positive number.

  METHOD SELECTION
  - "regex" — apply when the body is HTML, JS, or any text format. The pattern is a Go regular expression (RE2). It MUST contain exactly one capture group ((...)) whose contents are the numeric value. Use [\\s\\S] not . when matching across lines.
  - "json"  — apply when the body is JSON and you want a dotted path to a value. Syntax: "key.subkey" or "key.arr[0].subkey". No JSONPath wildcards or filters; only literal keys and zero-based array indices.

  POST-PROCESSING
  - The runtime strips commas and spaces, then ParseFloats the final output, so you do NOT need to emit a numeric literal — a string like "1 234,56" is fine. If the rate is given in fractional units (kopeks, cents), prefer a "regex" rule that extracts the integer and then a second "regex" rule that re-parses with the decimal point inserted; or simpler: pick a different anchor in the page that already shows decimal form.

  FAILURE CONTEXT (only present on retries)
  - When the user message contains a "PREVIOUS ATTEMPTS" section, treat each entry as a hard signal that the proposed rule was wrong: it either did not match, matched the wrong number (the value will be shown), or produced a non-numeric result. Pick a DIFFERENT anchor in the body — do not return a near-duplicate of the previous regex.

  OUTPUT
  - Exactly one JSON object. No surrounding whitespace prose. If you cannot find any plausible anchor, still return a valid-shape object with your best guess; the audit loop will catch wrong answers.
  ```

  The user-message template (retry case includes the transcript):
  ```text
  SOURCE: {name}
  PAIR:   {base_currency}/{quote_currency}  ({kind})
  URL:    {url}
  HINT:   {options.reserve or "none"}

  BODY (truncated to 200 KB, scripts/styles stripped, original was {original_size} bytes):
  ----- BEGIN BODY -----
  {body}
  ----- END BODY -----

  PREVIOUS ATTEMPTS (only on retry):
  Attempt 1: rule={...}; outcome={"error: regex did not match" | "value 0 is not positive" | "matched 1234.56 but plausible range is..." | ...}
  Attempt 2: ...
  ```

### Task 4: Add `RuleExecutor` and a body sanitizer in `internal/application/rulegen`

- **Description:** Add `executor.go` and `sanitizer.go` to the new package. These are extracted from the duplicated logic in `internal/tools/rateextractor/rateextractor.go` (`Run` method, post-processing block lines 93-148) and `internal/application/sourceaudit/audit.go` (`probeSource` method, lines 92-150). Both today reimplement the same chain — regex/json_path → trim spaces → comma-to-dot → space-strip → ParseFloat → NaN/Inf/range guard.

  Note on refactoring scope: do NOT delete the duplicated logic from `RateExtractor.Run` or `Auditor.probeSource` in this plan. They have subtle differences (caching, logger writes, repo persistence in `RateExtractor`; status enums in `Auditor`) and unifying them is a separate refactor. Just add one more call site here that uses the same algorithm in a small, self-contained executor — and add a comment in `executor.go` pointing at the two existing implementations.

  ```go
  // executor.go
  package rulegen

  import (
      "bytes"
      "errors"
      "fmt"
      "math"
      "strconv"

      "github.com/seilbekskindirov/monitor/internal/domain"
      "github.com/seilbekskindirov/monitor/internal/tools/rateextractor"
  )

  // RuleExecutor applies a chain of RateSourceRules to a body and returns the
  // final numeric value. Implementations should be deterministic and pure.
  type RuleExecutor interface {
      Execute(rules []domain.RateSourceRule, body []byte) (float64, error)
  }

  type defaultRuleExecutor struct{}

  func NewRuleExecutor() RuleExecutor { return &defaultRuleExecutor{} }

  func (e *defaultRuleExecutor) Execute(rules []domain.RateSourceRule, body []byte) (float64, error) {
      if len(rules) == 0 {
          return 0, errors.New("rulegen: no rules")
      }
      payload := make([]byte, len(body))
      copy(payload, body)
      for i, rule := range rules {
          var (
              out []byte
              err error
          )
          switch rule.Method {
          case domain.MethodRegex:
              out, err = rateextractor.ApplyRegex(rule.Pattern, payload)
          case domain.MethodJSONPath:
              out, err = rateextractor.ApplyJSONPath(rule.Pattern, payload)
          default:
              return 0, fmt.Errorf("rulegen: unsupported method %q at rule %d", rule.Method, i)
          }
          if err != nil {
              return 0, fmt.Errorf("rulegen: rule %d (%s): %w", i, rule.Method, err)
          }
          payload = bytes.TrimSpace(out)
      }
      payload = bytes.ReplaceAll(payload, []byte(","), []byte("."))
      payload = bytes.ReplaceAll(payload, []byte(" "), []byte(""))
      v, err := strconv.ParseFloat(string(payload), 64)
      if err != nil {
          return 0, fmt.Errorf("rulegen: parse %q: %w", string(payload), err)
      }
      if math.IsNaN(v) || math.IsInf(v, 0) {
          return 0, fmt.Errorf("rulegen: NaN or Inf")
      }
      if v <= 0.0001 || v >= math.MaxInt32 {
          return 0, fmt.Errorf("rulegen: value %g outside plausible range (0.0001, %d)", v, math.MaxInt32)
      }
      return v, nil
  }
  ```

  ```go
      maxBodyBytesForLLM  = 200 * 1024      // 200 KB after stripping
  )

  var (
      scriptRe = regexp.MustCompile(`(?is)<script\b[^>]*>.*?</script>`)
      styleRe  = regexp.MustCompile(`(?is)<style\b[^>]*>.*?</style>`)
  )

  // Sanitize strips <script>/<style> blocks and truncates to maxBodyBytesForLLM.
  // Returns the trimmed bytes and the count BEFORE stripping (for logging).
  func Sanitize(body []byte) ([]byte, int) {
      original := len(body)
      out := scriptRe.ReplaceAll(body, nil)
      out = styleRe.ReplaceAll(out, nil)
      if len(out) > maxBodyBytesForLLM {
          out = out[:maxBodyBytesForLLM]
      }
      return out, original
  }
  ```

  And the `Fetcher` interface alias so the generator depends on a local shape, not the sourceaudit package directly:
  ```go
  // generator.go (top)
  type Fetcher interface {
      Fetch(ctx context.Context, url string) (*sourceaudit.FetchResult, error)
  }
  ```
  Or, to avoid the cross-package coupling, define a local `FetchResult` struct and pass an adapter from `cmd/rulegen`. Prefer the local-struct option — `internal/application/sourceaudit/FetchResult` is fine to import but creates a cross-application dependency that feels wrong. Concrete recommendation: declare:
  ```go
  type Fetcher interface {
      Fetch(ctx context.Context, url string) ([]byte, error)
  }
  ```
  and have the CLI wrap `sourceaudit.Fetcher` with a thin adapter that drops the content-type/status (the rule-generator doesn't use them).

- **Pitfalls:**
  - **Non-greedy regex `.*?` with `(?s)` mode is needed to avoid matching across multiple `<script>` tags as one giant span.** The pattern above uses `(?is)` (case-insensitive + single-line). Test with a body containing two scripts.

### Task 5: Build the `cmd/rulegen` CLI binary

- **Description:** New binary at `cmd/rulegen/main.go`. Mirrors the bootstrap shape of `cmd/web/main.go` (env-var parsing for `SQLITEDB_DSN`, `AI_PRIMARY_DSN`, `AI_FALLBACK_DSN`; logger init) but exits after a single source generation. Flags:

  ```
  rulegen <source-name> [--force-fallback] [--max-primary-attempts N] [--logs-dir DIR] [--verbosity LEVEL]
  ```

  Positional: exactly one source name. No source name → print usage to stderr, exit 2.

  Flow:
  1. Parse flags + positional arg.
  2. Init logger (same pattern as `cmd/web`: `internal.NewLogger(LogsDir, "rulegen", LogVerbosity)`).
  3. Open SQLite via `SQLITEDB_DSN`. Call `sqlitedb.RequireMigratedSchema` — fail fast if the operator hasn't migrated.
  4. Build AI primary + fallback the same way `cmd/web` does (copy the block, including the soft-fallback-to-stub when `AI_FALLBACK_DSN` is absent).
  5. Build the fetcher adapter (`sourceaudit.NewHTTPFetcher(time.Minute)` wrapped to satisfy `rulegen.Fetcher`).
  6. Build `rulegen.NewGenerator(primary, fallback, fetcher, rulegen.NewRuleExecutor(), rRateSource, *maxPrimary, logger)`.
  7. Call `Generate(ctx, sourceName, *forceFallback)`. On success print a one-line summary to stdout:
     ```
     OK source=<name> rules=<count> value=<value> attempts=<n> escalated=<true|false> provider=<name> model=<model>
     ```
     and exit 0.
  8. On failure print `FAIL source=<name> reason=<error>` to stderr and exit 1.

  Add a `make rulegen-help` target to the Makefile that runs the binary with `--help` to give operators a quick way to discover usage.

  - **Duplicating bootstrap code from `cmd/web/main.go` is acceptable in v1.** A shared `internal/bootstrap` package would be cleaner but is out of scope. State this in the trade-offs.
  - **The binary writes to the same logs directory as `cmd/web` by default.** Each `cmd/rulegen` invocation produces a log file `rulegen.YYYYMMDD.log` (depending on how `internal.NewLogger` names files). ## Risks

1. **Schema mismatch between domain and LLM output.** The schema in Task 2 closely mirrors `domain.RateSourceRule` but is not a direct serialization of it (we exclude `parse_float` and `store_as_rate` from the enum). If a future domain change adds a new `Method`, the schema must be updated in lockstep. Mitigation: add a comment in `domain/ratesource.go` next to the `Method` constants pointing at `RateExtractionRuleSchema()`, and a comment in `schema.go` pointing back. The compile-time link is the `domain.Method` constant references that Task 3's parser uses; if the JSON enum drifts from the constants, parsing will fail at runtime and a unit test in `schema_test.go` will catch it.

2. **Strict json_schema not honored by Groq.** Plan 008's review flagged that not every Groq model supports strict json_schema. If the configured default model (`llama-3.1-8b-instant`) rejects strict mode, every primary attempt returns HTTP 400 and the audit loop burns 3 attempts to discover this — then fallback also fails (stub or paid). Mitigation: the generator logs the provider error verbatim so the operator sees `400 Bad Request: schema not supported` and switches models. Documenting model compatibility belongs in the ops handoff (Task 7).

3. **Cost surprise on misconfigured fallback.** If `AI_FALLBACK_DSN` points at `anthropic/claude-3-opus` via OpenRouter, a single invocation can cost dollars. Mitigation: the README cost note in Task 7 calls this out; the generator logs the model name on the fallback call so the operator sees what they're paying for; the per-invocation cap of 4 LLM calls bounds the worst case.

4. **Body truncation drops the actual rate.** A 250 KB HTML page might have the rate at byte 220,000 — truncating at 200 KB loses it. Mitigation: in v1 the operator runs the CLI, sees `FAIL` with the transcript in logs, and either (a) narrows the URL (some banks support a per-currency JSON endpoint), or (b) bumps `maxBodyBytesForLLM` by patching the constant. The constant is exposed as a package-level `const`, not a flag — making it a flag in v1 is premature. Document the constant in the README.

5. **The schema replacement breaks the stub's canned response.** Plan 008's review explicitly said `stubAIDefaultResponse` should stay aligned with the schema. Task 2 updates both together; review carefully.

6. **Rule overwrite is destructive.** If an operator runs `rulegen <source>` on a source that already has a working hand-tuned rule, the existing rule is replaced. There's no `--dry-run`. Mitigation: add `--dry-run` as a Task 5 sub-deliverable if cheap (it should be — just skip the final `RetainRateSource` and print the would-be rule). I am explicitly leaving `--dry-run` OUT of v1 to keep the task minimal, but the architect's recommendation is to include it. If the engineer agrees, add it as a one-line addition to Task 5 and update the acceptance criteria. If it adds friction, defer.

7. **HTTP fetch from a script-heavy source returns mostly JS.** Some banks (especially React-based pages) serve a near-empty HTML shell hydrated client-side. The body the fetcher gets contains no usable text, just bootloader JS. The LLM cannot generate a rule for this. v1 surfaces this as a generation failure (operator's only option: find a non-React endpoint, e.g. an XHR target the page itself hits). Document in the README.

## Trade-offs

1. **CLI binary over background goroutine.** Chose CLI. Reasoning: operator-invoked aligns with the existing `cmd/sourceaudit` workflow and the existing `cmd/migrator` workflow (both also operator-invoked). Background goroutines that call paid APIs without a per-invocation operator confirmation are an operational hazard — they fire on the project's schedule, not the operator's, and a bug can rack up cost overnight. The cost of a CLI is "operator has to type a command"; the cost of a background loop is "cost spike on a Saturday night without anyone watching." For v1 the explicit choice wins. Plan 010+ can add a self-healing loop with a separate rate-limited code path.

2. **Reuse `rate_sources.rules` column, no new "extraction_rules" table.** Chose to reuse. Reasoning: the column already exists, already stores JSON, and the repository already round-trips it. Adding a sibling table would create a synchronization concern (which one wins on conflict?) and a SELECT join for every collector read. The metadata gets its own column (`rule_metadata`) only because it's conceptually distinct (provenance vs the rule itself), and not joining everything into one giant JSON keeps the metadata queryable for plan 010+'s "regenerate rules generated before date X" use case.

3. **Reuse `internal/tools/rateextractor` matchers, no new dependency.** Chose to reuse. Reasoning: every seeded source uses `regex` or `json_path`. Pulling in goquery/htmlquery to support CSS/XPath would add ~2 MB to the binary and a maintenance surface, with zero current users. If a future source needs CSS, add it then with one focused plan. Open-closed beats speculative breadth here.

4. **Schema enum narrowed to `regex`/`json`, dropping `parse_float`/`store_as_rate`.** Chose to drop. Reasoning: those two methods exist in `domain.Method` but `RateExtractor.Run` and `Auditor.probeSource` both treat them as either no-ops (`store_as_rate`) or implicit ParseFloat (`parse_float`) that the post-processing chain already does for free. They're vestigial. Letting the LLM emit them would just create dead branches in `executor.go`. We exclude them from the schema enum.

5. **Logs as the audit trail, DB as the accepted-rule store.** Chose this split. Reasoning: a full transcript table would grow unbounded with `rate_sources` × `attempts`, eat SQLite's I/O budget, and need its own pruning policy. Logs are already rotated by the existing `internal.NewLogger`. The information value of "what did attempt 2 try and why did it fail" decays fast — operators look at it within hours of an invocation, not weeks. If forensic audit is needed later, plan 010+ can add a `rule_generation_attempts` table.

6. **N=3 primary + 1 fallback, hardcoded with `--max-primary-attempts` override.** Chose the asymmetric 3+1. Reasoning: the primary is cheap (free Groq), so 3 attempts cost effectively nothing. The fallback is expensive, so 1 attempt is the sweet spot — if a high-quality model can't get it with full failure context, the source is probably structurally too hard (anti-scraping, JS hydration) and more attempts won't help. The flag override is there for operator experimentation, not because the default is wrong.

7. **No `--dry-run` in v1.** Chose to defer. Reasoning: increases task scope, and the operator can achieve the same effect by inspecting logs (rule + extracted value are logged on success before persistence). Acknowledged this is a judgment call; reviewer may push to include.

8. **CSS/XPath stay out of scope.** Chose to defer. Reasoning: see (3). Every existing seed uses regex.

9. **Body cap: 5 MB hard reject, 200 KB to LLM after stripping.** Chose these numbers. Reasoning: most bank pages are 50-300 KB; 200 KB after stripping covers the realistic working set. 5 MB hard reject catches misconfiguration (the operator pointed at a CDN-served binary or a giant SPA bundle) before burning LLM tokens. Both numbers are package-level `const` so they're greppable and tunable in a focused PR.

10. **Duplicate bootstrap code in `cmd/rulegen/main.go`.** Chose to duplicate from `cmd/web/main.go`. Reasoning: introducing an `internal/bootstrap` package now would be a refactor that touches `cmd/web`, `cmd/collector`, `cmd/notifier`, and `cmd/migrator` (they all share the same DSN-loading pattern). That belongs in its own plan, not piggybacked on this one. The duplication is ~30 lines and the wiring is stable enough that drift between binaries is unlikely.
