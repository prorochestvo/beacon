# Task Breakdown

## Overview

Make `alert_thaw` a **forced, system-managed** subscription: every tracked city must carry
exactly one thaw row, the user cannot turn it off, and it is back-filled onto all existing
cities. Thaw shipped in `r_0.0.3-alpha` (plans `260715.0001`, `260715.0002`) as an ordinary
opt-in kind; this plan converts it to automatic.

Two product decisions are **locked**: (1) forced always-on, non-removable per tracked city;
(2) back-filled to every existing city via a migration.

The UX fork ("how does a user still remove a whole city if thaw is undeletable?") is resolved
by coordinator decision in favour of an **explicit "Remove city" model** (a dedicated
location-scoped delete action), **not** cascade-on-last-non-thaw. This document is written to
that decision.

### The data model, and why "delete" is the crux

A "city" is **not** a single entity. It is N rows in `weather_user_cities`, one per
`notify_kind`, keyed `UNIQUE(user_type, user_id, location_id, notify_kind)` (migration 011).
The Mini App renders them grouped by `location_id` (`GroupWeatherCities`), and every kind row
gets its own ✕ delete button (`renderWeatherKindRow`,
`cmd/wasm/ui/me_weather_cities.go:234`). The delete path today is strictly **per-row by id**:

- WASM click reads `data-id` (the row `ID`) → `DeleteCity(id)` (`cmd/wasm/main.go:1718-1733`,
  `cmd/wasm/application/me_weather_cities.go:176`).
- `DELETE /api/me/weather/cities/{id}` loads the row, ownership-checks, deletes that one row
  (`DeleteMeWeatherCity` / `meWeatherCityOwnershipCheck`, `me_weather.go:287-343`).

There is no city-level ("remove the whole city") control anywhere. "Thaw cannot be deleted"
therefore collides with "let me remove a city entirely": if the thaw row is undeletable, a
city eventually reduces to a forced-thaw orphan the user could never clear.

### Chosen model — explicit "Remove city" (coordinator decision)

Two distinct delete surfaces, cleanly separated by intent:

1. **Individual row delete** — `DELETE /api/me/weather/cities/{id}` (existing endpoint).
   Allowed for **every kind except `alert_thaw`**. A direct thaw-row delete returns **409 +
   PublicError** ("Thaw alerts stay on for every tracked city; remove the city to turn it
   off."). Ownership check runs **first**, so a cross-user or missing id still returns **404**
   (no existence disclosure). **No cascade magic:** deleting the last non-thaw row deletes only
   that row and leaves a thaw-only city; the user clears it via action (2).

2. **Location-scoped "Remove city" delete** — NEW `DELETE /api/me/weather/locations/{location_id}`.
   Removes **all** rows for the caller's `(user, location)`, including the forced thaw row,
   atomically. Explicit "I want this whole city gone" intent. Ownership: only the caller's own
   rows; a location with no rows owned by the caller → **404**.

Why this over cascade-on-last:

- **Explicit intent, no surprise.** Removing "Morning summary" removes only that alert; the
  city stays (now thaw-only) until the user presses "Remove city". No single ✕ silently wipes a
  city.
- **Enforces "can't disable".** The thaw row has no ✕ (Task 7 hides it) and its direct API
  delete is refused (409). The only way thaw goes away is removing the whole city — exactly the
  product intent.
- **Ownership preserved.** Both surfaces run ownership scoping before acting; cross-user/missing
  → 404 on both. The 409 refusal on surface (1) only ever applies to the caller's own thaw row.

Cost vs cascade: one extra route + handler + DTO-free endpoint, one apiclient method, one page
method, one UI control. Accepted for the clearer UX (see Trade-offs).

### Chosen endpoint shape (surface 2)

`DELETE /api/me/weather/locations/{location_id}`, path-wildcard style.

Justification, against the existing route conventions (`internal/gateway/httpV1/routes/routes.go`,
`router.go`):

- Every existing single-resource mutation uses a **path wildcard**, never a query filter:
  `MeWeatherCityByID = "/api/me/weather/cities/{id}"`,
  `MeSubscriptionByID` (`DELETE`/`PATCH`). `DELETE /cities?location_id=…` would be the only
  query-filtered mutation in the router — inconsistent, and it overloads the `cities`
  collection with a filtering semantic on DELETE.
- A distinct sub-path `/weather/locations/{location_id}` reads as a **different resource**
  ("the whole location") than `/weather/cities/{id}` ("one subscription row"), which is exactly
  the intent split we want.
- Registered as `mux.HandleFunc("DELETE "+routes.MeWeatherLocationByID, h.DeleteMeWeatherLocation)`
  alongside the other weather routes; Go 1.22 `{location_id}` captures a single path segment.

**Request/response DTO:** none needed. Input is the `location_id` path segment; success is
**204 No Content** (mirrors the existing city delete). Errors are `PublicError` JSON bodies
(404 missing/cross-user, 500 persistence). `location_id` is an Open-Meteo geocoding id
(numeric-ish, single path segment); the client path-escapes it (`url.PathEscape`).

### Auto-create thaw on city add (unchanged from prior design)

`CreateMeWeatherCity` persists the requested row, then **ensures** a thaw row exists for the
same validated `(user, location)` via a second idempotent `RetainWeatherUserCity`
(`ON CONFLICT DO UPDATE`, confirmed at `weatherusercity.go:96-115`). Skipped when the requested
kind already **is** thaw. Thaw copies the same validated location fields, `condition_value=''`,
`notify_hour=7`.

### Back-fill migration `202607.021` (unchanged)

One idempotent `INSERT OR IGNORE ... SELECT ... GROUP BY` adds exactly one `alert_thaw` row per
distinct `(user_type, user_id, location_id)` that has any row but no thaw row.

### PII

No new personal field. The thaw row reuses coordinates the user already volunteered
(pre-approved per CLAUDE.md) plus forecast data; the back-fill copies existing columns; no
column is added to `weather_user_cities`. Confirmed.

## Assumptions

1. **Plan number 263 is correct.** Highest existing prefix is 262 (edge-trigger plan, shipped
   as `260715.0002`). Filename `263-thaw-alert-forced-always.md`.
2. **Next migration number is `202607.021`.** `202607.020` (`add_alert_latched`) is the current
   highest and already applied to prod. Filename immutable once applied (CLAUDE.md).
3. **All rows of a given `location_id` carry identical location fields.** True by construction
   (same geocoding result; `RetainWeatherUserCity` refreshes all to `excluded.*` on every
   re-subscribe). Makes the back-fill's `MIN(col)` aggregates safe. If they disagree, `MIN`
   picks the lexicographically smallest — harmless.
4. **The default kind for a UI-added city is `morning_summary`** (`SaveSelected` sends empty
   `NotifyKind`; `CreateMeWeatherCity` defaults empty → `morning_summary`,
   `me_weather.go:219-222`). Every UI-added city is `{morning_summary, thaw}` at minimum.
5. **`RetainWeatherUserCity` is idempotent and returns the stored id** via `RETURNING id`
   (`weatherusercity.go:112-115`). A second retain of an identical thaw row is a no-op
   `DO UPDATE`.
6. **No `WeatherCityRow`/create DTO change is needed.** `dto.WeatherCityRow` already carries
   `NotifyKind` (`me_weather.go:140`), so the client identifies thaw rows (hide ✕) without a
   wire change. The location-delete endpoint needs no request/response DTO (path segment +
   204).
7. **Thaw uses no `condition_value` and ignores `notify_hour`** (`Validate()` treats it like
   thunderstorm, `weatheruser.go:143`). Forced row stores `condition_value=''`, `notify_hour=7`.
8. **A partial auto-create failure (requested row saved, thaw retain errors) self-heals.** The
   requested row persists; the next successful add re-ensures thaw. Handler returns 500 so the
   failure is visible. Accepted (see Risks).
9. **`location_id` is a single URL-path-safe segment.** Open-Meteo geocoding ids are
   numeric-ish and contain no `/`; the client `url.PathEscape`s it. If a future provider yields
   a slash-bearing id, the `{location_id}` wildcard would need revisiting — flagged.

### Task 1: Back-fill migration `202607.021` — one forced thaw row per existing city

- **Description:**
  Create `migrations/202607.021.weather_user_cities.backfill_alert_thaw.sql`: a single
  idempotent `INSERT OR IGNORE ... SELECT ... GROUP BY` adding exactly one
  `notify_kind='alert_thaw'` row per distinct `(user_type, user_id, location_id)` that has ≥1
  row but no thaw row, copying location columns from that city's existing rows. Column list
  verified against migrations 011 (base) + 013 (`condition_value`) + 020 (`alert_latched`).

  ```sql
  INSERT OR IGNORE INTO weather_user_cities (
      id, user_type, user_id, location_id, display_name,
      latitude, longitude, timezone, country, admin1,
      gismeteo_city_id, notify_kind, notify_hour, last_notified_at,
      condition_value, alert_latched, updated_at, created_at
  )
  SELECT
      'WUC' || strftime('%Y%m%d%H%M%S','now') || 'Z0T' || upper(hex(randomblob(16))),
      src.user_type, src.user_id, src.location_id, MIN(src.display_name),
      MIN(src.latitude), MIN(src.longitude), MIN(src.timezone),
      MIN(src.country), MIN(src.admin1),
      NULL, 'alert_thaw', 7, NULL,
      '', 0,
      strftime('%Y-%m-%dT%H:%M:%SZ','now'),
      strftime('%Y-%m-%dT%H:%M:%SZ','now')
  FROM weather_user_cities AS src
  WHERE src.notify_kind <> 'alert_thaw'
    AND NOT EXISTS (
        SELECT 1 FROM weather_user_cities AS t
        WHERE t.user_type   = src.user_type
          AND t.user_id     = src.user_id
          AND t.location_id = src.location_id
          AND t.notify_kind = 'alert_thaw'
    )
  GROUP BY src.user_type, src.user_id, src.location_id;
  ```

  - `GROUP BY (user_type, user_id, location_id)` collapses a city's multiple non-thaw rows into
    **one** output → exactly one thaw row per city (kills "one thaw per source row").
  - `NOT EXISTS (...'alert_thaw')` makes rerun a no-op; `INSERT OR IGNORE` is a second guard
    against the UNIQUE constraint.
  - Generated `id` mirrors `identity.New(KindWeatherUserCity)` shape
    (`WUC<ts>Z0T<UUID-hex-upper>`); unique per row via `randomblob`; opaque TEXT PK.
  - Timestamps `strftime('%Y-%m-%dT%H:%M:%SZ','now')` match the app's `time.RFC3339`-UTC storage
    (`weatherusercity.go:133-134`). `last_notified_at=NULL` (armed), `alert_latched=0`,
    `gismeteo_city_id=NULL`.
- **Acceptance Criteria:**
  - [ ] File `migrations/202607.021.weather_user_cities.backfill_alert_thaw.sql` exists with the
        statement above.
  - [ ] Against a DB at 020, `make migrate` inserts exactly one `alert_thaw` row per distinct
        `(user_type, user_id, location_id)`; a 3-non-thaw-row city yields **1** thaw row.
  - [ ] Rerunning `make migrate` inserts nothing the second time (tracked in
        `__schema_migrations`).
  - [ ] A city that already has thaw is untouched (no duplicate, no field change).
  - [ ] Inserted rows read back `notify_kind='alert_thaw'`, `condition_value=''`,
        `alert_latched=0`, `last_notified_at IS NULL`, `notify_hour=7`, location columns equal
        to the source city's; `id` unique/non-empty.
- **Pitfalls:**
  - Do not omit `GROUP BY` (multi-kind city → duplicate thaw candidates, only first survives).
  - `randomblob`/`strftime` must be in the `SELECT` list (per-row evaluation) or the PK
    collides.
  - Filename immutable after any prod apply; never rename. No CHECK/index — pure data back-fill.
- **Complexity:** Medium

### Task 5: NEW endpoint — `DELETE /api/me/weather/locations/{location_id}` (Remove city)

- **Description:**
  Add the location-scoped "Remove city" endpoint.
  1. **Route const** in `internal/gateway/httpV1/routes/routes.go` (next to the weather block,
     ~line 102):
     ```go
     // MeWeatherLocationByID is the location-scoped delete endpoint (DELETE). Removes every
     // subscription row for the caller at this location, including the forced alert_thaw row.
     // Cross-user / no-rows access returns 404 (no existence disclosure).
     MeWeatherLocationByID = "/api/me/weather/locations/{location_id}"
     ```
  2. **Register** in `internal/gateway/httpV1/router.go` (with the weather routes, ~line 89):
     ```go

     ```
  3. **Handler** `DeleteMeWeatherLocation` in `me_weather.go`, mirroring `DeleteMeWeatherCity`'s
     auth + shape. No request/response DTO: path segment in, 204 out.
     ```go
     // DeleteMeWeatherLocation removes every city subscription (all notify kinds, including the
     // forced alert_thaw row) owned by the authenticated caller at the given location.
     //
     // DELETE /api/me/weather/locations/{location_id}
     // Auth: X-Telegram-Init-Data header only.
     //
     // 204 No Content on success.
     // 401 on auth failure.
     // 404 when the caller has no rows at this location (missing or cross-user — same response,
     //     no existence disclosure).
     // 503 when the weather service is not wired. 500 on persistence failure.
     func (h *Handler) DeleteMeWeatherLocation(w http.ResponseWriter, r *http.Request) {
         // ... nil-repo 503; validateInitData → tgUserID (as in the sibling handlers) ...
         locationID := r.PathValue("location_id")
         if strings.TrimSpace(locationID) == "" {
             http.Error(w, `{"error":"missing location id"}`, http.StatusBadRequest)
             return
         }

         // Ownership + existence: enumerate the caller's rows and confirm at least one at this
         // location. 404 (not 403) covers both "no such location for anyone" and "owned by
         // another user" — no existence disclosure, consistent with meWeatherCityOwnershipCheck.
         all, err := h.meWeatherCityRepo.ObtainWeatherUserCitiesByUserID(r.Context(), domain.UserTypeTelegram, tgUserID)
         if err != nil {
             h.internalError(w, fmt.Errorf("DeleteMeWeatherLocation list: %w", err))
             return
         }
         owned := false
         for i := range all {
             if all[i].LocationID == locationID {
                 owned = true
                 break
             }
         }
         if !owned {
             pub := internal.NewPublicError("city not found")
             http.Error(w, `{"error":"`+pub.Details()+`"}`, http.StatusNotFound)
             return
         }

         if err := h.meWeatherCityRepo.RemoveWeatherUserCitiesByLocation(r.Context(), domain.UserTypeTelegram, tgUserID, locationID); err != nil {
             h.internalError(w, fmt.Errorf("DeleteMeWeatherLocation remove: %w", err))
             return
         }
         w.WriteHeader(http.StatusNoContent)
     }
     ```
  4. **Interface** — add `RemoveWeatherUserCitiesByLocation(ctx, userType, userID, locationID) error`
     to the handler's `meWeatherCityRepository` interface (`me_weather.go:19-24`).
     `ObtainWeatherUserCitiesByUserID` is already present and reused for the ownership scan.
- **Acceptance Criteria:**
  - [ ] `TestHandler_DeleteMeWeatherLocation` (new; extend the mock with
        `RemoveWeatherUserCitiesByLocation` recording `(userType, userID, locationID)`):
        - caller owns rows at the location → `204`;
          `RemoveWeatherUserCitiesByLocation(user, location)` called once.
        - caller has no rows at the location (empty / other-user only) → `404`; body is
          `PublicError` "city not found"; **no** remove call.
        - empty/missing `location_id` path segment → `400`.
        - auth failure → `401`; nil repo → `503`.
  - [ ] `var _ meWeatherCityRepository = &mockMeWeatherCityRepo{}` stays valid with the new
        interface method; the real `WeatherUserCityRepository` satisfies it once Task 2 lands.
  - [ ] Route registered and reachable (a router-level test or the handler test exercising
        `r.PathValue("location_id")` via `httptest` with the `{location_id}` pattern).
  - [ ] `make test` green for `./internal/gateway/httpV1/...`.
- **Pitfalls:**
  - Ownership scan must be scoped to the caller (`tgUserID`); returning 404 for a location the
    caller doesn't own is the no-disclosure contract — do not 403.
  - The list-then-delete pair is two transactions; for this single-user Mini App the race is

  - `r.PathValue("location_id")` requires the Go 1.22 pattern name to match exactly

- **Complexity:** Medium

## Risks

- **Auto-create partial failure (Assumption 8).** Requested row saved, thaw retain fails → city
  temporarily without forced thaw, 500 returned. Self-heals on next add. Bounded, visible.
- **Back-fill id/format drift.** Migration mints ids in SQL (`WUC…T<hex-upper>`, nanos fixed
  `0`) rather than via `identity.New`. Format-compatible, unique, opaque PK used only for
  delete-by-id. Cosmetic — flagged so it's not mistaken for a bug.
- **`location_id` path-safety (Assumption 9).** Current Open-Meteo ids are single URL-safe
  segments; a future slash-bearing id would break the `{location_id}` wildcard. Client
  `url.PathEscape`s; flagged for revisit if the provider changes.
- **Thaw-only city is a normal, reachable state now.** With no cascade, deleting the last
  non-thaw row leaves a `{thaw}` city. Intended — the user clears it via "Remove city". The UI
  must make "Remove city" discoverable (Task 7) so a thaw-only city doesn't look stuck.
- **Create-vs-Delete cross-endpoint race (accepted).** Task 5's endpoint was revised during
  review to a single atomic check-and-delete (`RemoveWeatherUserCitiesByLocation` returns
  `internal.ErrNotFound` via `RowsAffected`), which removes the endpoint's own internal
  ownership-scan-then-delete race. A cross-*endpoint* race remains: a concurrent
  `CreateMeWeatherCity` (whose non-thaw path issues a second `RetainWeatherUserCity` to
  ensure the forced thaw row) racing a `DeleteMeWeatherLocation` on the same
  `(user, location)` can interleave so the ensure-thaw retain commits after the location
  delete's snapshot, leaving a thaw-only "ghost" row the delete never saw. Bounded: no data
  loss or leak, and the ghost self-heals on the next "Remove city" call (which deletes
  whatever rows exist at that instant, including the ghost). A full fix needs
  request-level serialization (e.g. per-location locking) that is disproportionate for a
  self-hosted, single-user-scale bot — accepted as a known risk, not mitigated by design.
- **Migration immutability.** `202607.021…sql` must never be renamed after any prod apply.

## Trade-offs

- **Explicit "Remove city" over cascade-on-last-non-thaw (coordinator decision).** Chosen for
  clear intent: no single ✕ silently wipes a city; whole-city removal is a deliberate, confirmed
  action. Cost: one extra route + handler, an apiclient method, a page method, and a UI control
  — more surface than reusing the per-row endpoint. **Rejected:** cascade-on-last, which is less
  code but surprises the user by removing a whole city when they delete its last visible alert.
  The explicit model matches the "forced, system-managed" framing: thaw goes away only when the
  city does, and that is an explicit choice.
- **`DELETE /api/me/weather/locations/{location_id}` over `DELETE /cities?location_id=…`.**
  Path-wildcard matches every existing single-resource mutation in the router
  (`cities/{id}`, `subscriptions/{id}`); a query-filtered DELETE would be the router's only one
  and would overload the `cities` collection. The distinct `locations/{id}` sub-path also reads
  as the right resource ("the whole location"). No request/response DTO — path segment + 204,
  same as the existing city delete.
- **Individual thaw delete → 409, not 403.** The row is the caller's own (ownership passed), so
  nothing to hide; 409 Conflict is the honest "exists but not removable", and it keeps 403 off
  the `/api/me/*` surface entirely.
- **Ownership scan reuses `ObtainWeatherUserCitiesByUserID` over a count-returning delete.**
  Reuses an existing method to get 404-on-none with no-disclosure, at the cost of one extra
  read. A `RowsAffected`-based approach would save the read but couldn't distinguish
  "cross-user" from "already gone" as cleanly. Chose clarity.
- **Auto-create via a second idempotent retain over a bespoke method.** Reuses the existing
  upsert, zero new SQL; cost is a second round-trip + the partial-failure window (both
  acceptable at this scale).
- **Back-fill in pure SQL over an app-level one-shot.** The migrator is the project's only
  schema/data-migration mechanism and the copy is a straight `INSERT … SELECT`. No app binary
  needed.
- **No `WeatherCityRow`/create DTO change.** `NotifyKind` already rides on the row, so the client
  derives "forced/undeletable" from the kind; the location endpoint needs no DTO. Rejected
  adding a `forced`/`deletable` boolean — it duplicates information the kind already carries.
- **`notify_hour=7`, `condition_value=''` for the forced thaw row.** Thaw ignores both
  (Assumption 7); fixing them to defaults keeps the row stable and matches the auto-create path.
