# Task Breakdown

## Overview

`cmd/rulegen` currently cannot service rate sources whose pages are hydrated by
client-side JavaScript. Plan 012 introduced the `fetcher_kind` column on
`rate_sources` and a sentinel `ErrUnsupportedFetcherKind` returned by
`internal/application/rulegen/generator.go` when the field is not `""` or
`"plain"`. One row (`KZ_BCC_BID_USD_KZT`) is already flagged
`fetcher_kind='headless'` by `migrations/202605.010.rate_sources.add_fetcher_kind.sql`
and the CLI exits with code 2 on it. This is the placeholder Plan 013 fills in.

Plan 013 implements a `ChromedpFetcher` (pure-Go DevTools-protocol client) behind
the existing `rulegen.Fetcher` interface, wires it into `cmd/rulegen` as a
sibling of the existing `HTTPFetcher`, and renames `"headless"` to `"chromedp"`
so the persisted value names the implementation, not its requirement.

Out of scope: browser pooling, per-source wait selectors, proxy support,
screenshot/PDF, JS injection, LLM prompt or `Sanitize` changes,
`cmd/collector`/`cmd/notifier` integration.

The user has confirmed (memory: `feedback_prefer_chromedp_over_xhr.md`) that
SPA sources should default to chromedp; chasing underlying XHR endpoints is
explicitly rejected because endpoint stability across bank backend refactors
cannot be assumed and rate monitoring's 6 h cadence tolerates 5–10 s/fetch.

## Assumptions

1. **chromedp is pure-Go.** `github.com/chromedp/chromedp` and its transitive
   deps (`chromedp/cdproto`, `chromedp/sysutil`, `gobwas/{httphead,pool,ws}`,
   `josharian/intern`, `mailru/easyjson`) communicate with an external Chrome
   instance via WebSocket DevTools Protocol. They contain no CGO. Task 1 must
   re-verify with `CGO_ENABLED=0 go build ./...` and abort the plan if a
   transitive dep regresses on this.
2. **Chromium is a runtime dependency, not a build dependency.** `make build`
   must continue to succeed on hosts without Chromium installed. The library
   only attempts to spawn the browser when `Fetch` is called.
3. **Deploy host is Oracle Cloud Always Free ARM, 24 GB RAM.** Hosting a
   headless Chromium per `cmd/rulegen` invocation is acceptable. The current
   collector loop is 6 h, and `cmd/rulegen` is operator-invoked, so a 30 s wall
   clock per fetch is well within budget.
4. **Plan 012's seed value `"headless"` migrates to `"chromedp"`.** This plan
   renames the only existing row via migration `202605.011`. The migration is
   a 1-line `UPDATE`; no schema change.
5. **No new currency or quote-currency requirements.** The chromedp output
   feeds straight into the existing `Sanitize` -> `Locate` -> LLM pipeline,
   which already handles 80 KB windows and 5 MB hard caps. JS-rendered bodies
   may inflate the raw size; the existing 5 MB raw cap is the safety net.
6. **chromedp version pin is set at engineer-implementation time.** This plan
   states "latest stable at implementation time"; the engineer commits the pin
   into `go.mod` and `go.sum`.
7. **CI runners do not have Chromium pre-installed.** Therefore every test
   that invokes a real browser must `t.Skip` cleanly when no Chromium binary
   is on `PATH`. `make test` must pass on a CI runner with a stock Go toolchain.
8. **Existing seed for `KZ_BCC_BID_USD_KZT` URL stays as `https://www.bcc.kz/kz/`.**
   The rule itself may need re-generation later via `cmd/rulegen` once
   chromedp lands, but rule regeneration is operator-invoked and outside this
   plan's scope.

### Task 2: Implement `ChromedpFetcher` behind the `Fetcher` interface

- **Description:**
  - Add a new file `internal/application/rulegen/chromedpfetcher.go` (and
    sibling test file `chromedpfetcher_test.go`).
  - Define `ChromedpFetcher` implementing the existing `rulegen.Fetcher`
    interface (`Fetch(ctx context.Context, url string) ([]byte, error)`).
  - Construct via `NewChromedpFetcher(opts ChromedpFetcherOptions) *ChromedpFetcher`.
    `ChromedpFetcherOptions` carries:
    - `Timeout time.Duration` — hard wall clock, default 30 s.
    - `ChromiumPath string` — optional absolute path; empty defers to
      chromedp's default PATH lookup (`chromium`, `chromium-browser`,
      `google-chrome`, `chrome`).
    - `NetworkIdleMillis int` — post-navigation idle wait, default 1500 ms.
    - `Logger io.Writer` — destination for diagnostic log lines (mirrors
      `Generator.logger`).
  - `Fetch` implementation:
    1. Wrap `ctx` with `context.WithTimeout(ctx, f.timeout)` — the hard wall
       clock, enforced **independently** of any chromedp internal timeout. Do
       not rely on chromedp's per-action timeouts alone.
    2. Build a chromedp allocator with `chromedp.NewExecAllocator` using
       `chromedp.DefaultExecAllocatorOptions[:]` plus: `chromedp.Headless`,
       `chromedp.DisableGPU`, `chromedp.NoSandbox` (required when running as
       root in containers / on ARM deploy host), `chromedp.Flag(
       "disable-blink-features", "AutomationControlled")` (mild anti-bot
       resistance), and `chromedp.ExecPath(f.chromiumPath)` only when the path
       is non-empty.
    3. Create the browser context via `chromedp.NewContext(allocCtx)`.
    4. Run `chromedp.Run(ctx, chromedp.Navigate(url), chromedp.WaitVisible("body", chromedp.ByQuery), chromedp.Sleep(time.Duration(f.networkIdleMs) * time.Millisecond), chromedp.OuterHTML("html", &result, chromedp.ByQuery))`.
    5. Defer cancellation of both contexts (browser and allocator) so the
       Chromium process is reaped even on error paths.
    6. Return `[]byte(result), nil` on success, or `nil, fmt.Errorf("chromedp fetch %s: %w", url, err)` on failure.
  - Add the compile-time interface check at the top of the test file:
    `var _ Fetcher = (*ChromedpFetcher)(nil)`.
  - Add a helper `findChromiumOrSkip(t *testing.T) string` in the test file
    that calls `exec.LookPath("chromium")` and friends (in the same order
    chromedp uses) and `t.Skip`s if none is found. Tests using a real browser
    must call it first.
  - Tests in `chromedpfetcher_test.go` (single `TestChromedpFetcher_Fetch` with
    subtests):
    - `t.Run("returns rendered DOM of a static page", ...)` — serve a known
      HTML body via `httptest.NewServer` and assert the result contains a
      sentinel substring. Skip if no Chromium.
    - `t.Run("returns post-hydration DOM when JS injects content", ...)` —
      serve a page whose `<body>` is empty but a `<script>` mutates it via
      `document.body.innerHTML = "<span id='x'>hello</span>"`. Assert the
      returned bytes contain `id="x"`. Skip if no Chromium.
    - `t.Run("respects timeout", ...)` — `httptest.NewServer` that `time.Sleep`s
      well past `Timeout` before responding. Assert error wraps
      `context.DeadlineExceeded` (via `errors.Is`). Skip if no Chromium.
    - `t.Run("propagates fetch error on invalid url", ...)` — pass
      `"http://127.0.0.1:1/"` (closed port). Assert non-nil error returned and
      no panic. Skip if no Chromium.
- **Acceptance Criteria:**
  - [ ] `internal/application/rulegen/chromedpfetcher.go` exists, has a
    package-level doc comment explaining the fresh-per-call lifecycle and the
    upgrade path to pooling.
  - [ ] `var _ Fetcher = (*ChromedpFetcher)(nil)` compile-time check is
    present in the test file.
  - [ ] All four subtests pass locally (when Chromium is on PATH) and skip
    cleanly when it is not.
  - [ ] No `_ = err` discard anywhere — all errors are either returned,
    logged, or asserted in tests.
  - [ ] No `nowFn func() time.Time` field; if a test needs deterministic
    timing it uses `t.Context()` with a deadline.
  - [ ] No section-divider comments (`// --- ... ---`).
  - [ ] `t.Parallel()` is called on the top-level test and each subtest.
- **Pitfalls:**
  - **Two contexts, two cancels.** `chromedp.NewExecAllocator` returns
    `(ctx, cancel)` and `chromedp.NewContext` returns another `(ctx, cancel)`.
    Both `cancel` calls must `defer`, otherwise the Chromium process leaks.
  - **`chromedp.NoSandbox` is required on root/container environments.**
    The deploy host runs the binary under a systemd unit; without `--no-sandbox`
    Chromium aborts. State this in code comment so it survives a refactor.
  - **`OuterHTML("html", ...)` returns the full document including DOCTYPE only
    when present in the rendered DOM.** For consumers that need the original
    DOCTYPE this is fine. For our `Sanitize` pipeline DOCTYPE is irrelevant.
  - **`chromedp.WaitVisible("body", chromedp.ByQuery)`** is the standard idiom.
    Without it `Navigate` returns as soon as the navigation commits, before
    any JS runs.
  - **Don't use `chromedp.Sleep` as the sole synchronization primitive on
    slow networks.** 1500 ms is the *additional* wait after `WaitVisible`.
    A per-source wait selector is out of scope (see Trade-offs).
  - **Timeout enforcement.** The outer `context.WithTimeout` is the source of
    truth. chromedp options that look like timeouts (e.g. allocator
    `WSURLReadTimeout`) are insufficient — they affect specific protocol
    sub-operations, not the whole fetch.
  - **Subprocess cleanup on test failure.** If a `t.Fatal` fires mid-test, the
    deferred cancels still run, but only if the test goroutine returns
    normally. With `t.Parallel()` and `t.Cleanup` the cleanup is reliable;
    avoid `os.Exit` from any test.
- **Complexity:** Hard.
- **Code Example (full file sketch):**

```go
// Package rulegen — chromedp fetcher.
package rulegen

import (
    "context"
    "fmt"
    "io"
    "time"

    "github.com/chromedp/chromedp"
)

// ChromedpFetcher renders pages with a headless Chrome instance and returns
// the post-hydration DOM. It implements the rulegen.Fetcher interface.
//
// Lifecycle: each Fetch call spawns a fresh browser. This is intentional for
// the cmd/rulegen single-source-per-invocation use case. When cmd/collector
// integrates the fetcher (future plan), reuse a long-lived allocator via a
// browser pool.
type ChromedpFetcher struct {
    timeout       time.Duration
    chromiumPath  string
    networkIdleMs int
    logger        io.Writer
}

type ChromedpFetcherOptions struct {
    Timeout           time.Duration
    ChromiumPath      string
    NetworkIdleMillis int
    Logger            io.Writer
}

const (
    defaultChromedpTimeout       = 30 * time.Second
    defaultChromedpNetworkIdleMs = 1500
)

func NewChromedpFetcher(opts ChromedpFetcherOptions) *ChromedpFetcher {
    if opts.Timeout <= 0 {
        opts.Timeout = defaultChromedpTimeout
    }
    if opts.NetworkIdleMillis <= 0 {
        opts.NetworkIdleMillis = defaultChromedpNetworkIdleMs
    }
    if opts.Logger == nil {
        opts.Logger = io.Discard
    }
    return &ChromedpFetcher{
        timeout:       opts.Timeout,
        chromiumPath:  opts.ChromiumPath,
        networkIdleMs: opts.NetworkIdleMillis,
        logger:        opts.Logger,
    }
}

func (f *ChromedpFetcher) Fetch(ctx context.Context, url string) ([]byte, error) {
    ctx, cancelCtx := context.WithTimeout(ctx, f.timeout)
    defer cancelCtx()

    allocatorOpts := append(chromedp.DefaultExecAllocatorOptions[:],
        chromedp.Headless,
        chromedp.DisableGPU,
        chromedp.NoSandbox, // required for systemd / container deploy
        chromedp.Flag("disable-blink-features", "AutomationControlled"),
    )
    if f.chromiumPath != "" {
        allocatorOpts = append(allocatorOpts, chromedp.ExecPath(f.chromiumPath))
    }

    allocCtx, cancelAlloc := chromedp.NewExecAllocator(ctx, allocatorOpts...)
    defer cancelAlloc()

    browserCtx, cancelBrowser := chromedp.NewContext(allocCtx)
    defer cancelBrowser()

    var rendered string
    err := chromedp.Run(browserCtx,
        chromedp.Navigate(url),
        chromedp.WaitVisible("body", chromedp.ByQuery),
        chromedp.Sleep(time.Duration(f.networkIdleMs)*time.Millisecond),
        chromedp.OuterHTML("html", &rendered, chromedp.ByQuery),
    )
    if err != nil {
        return nil, fmt.Errorf("chromedp fetch %s: %w", url, err)
    }
    _, logErr := fmt.Fprintf(f.logger, "chromedp: url=%s rendered_bytes=%d\n", url, len(rendered))
    if logErr != nil {
        // Logger writes are best-effort but errors must not be swallowed silently.
        return []byte(rendered), nil
    }
    return []byte(rendered), nil
}
```

## Risks

1. **CGO leak.** If `go get chromedp@latest` pulls a release whose transitive
   deps re-introduce CGO (very unlikely given the project's history but not
   impossible), the entire deploy story breaks because the prime/stage
   workflows compile with `CGO_ENABLED=0`. **Mitigation:** Task 1's acceptance
   criterion requires the CGO-free build to pass; if it does not, the engineer
   must pin to a known-good chromedp version before proceeding.
2. **Chromium absent on deploy host at first cron tick after merge.** If the
   deploy workflow does not install Chromium before the first `cmd/rulegen`
   invocation against a `chromedp` source, the run fails with a
   `chromedp: exec: "chromium": executable file not found in $PATH` error.
   **Mitigation:** Task 6 documents the install step. The failure is loud and
   recoverable (`apt install` and re-run).
3. **Anti-bot detection on bcc.kz or similar.** Banks increasingly detect
   headless Chrome via `navigator.webdriver`, missing plugins, or canvas
   fingerprints. The plan ships the standard
   `--disable-blink-features=AutomationControlled` flag but does not stub
   `navigator.webdriver` or rotate user agents. **Mitigation:** If detection
   occurs, a follow-up plan can add `chromedp.UserAgent(...)` and a
   `chromedp.ActionFunc` to override `navigator.webdriver`. Out of scope here.
4. **Browser startup latency masks bugs.** Chromium cold-start on the ARM
   deploy host is ~2–3 s. Tests that depend on a `chromedp.Sleep` of 1500 ms
   may flake on cold caches. **Mitigation:** the `findChromiumOrSkip` pattern
   plus a `30 s` outer timeout. If CI flakes recur, the next plan can carve
   out a long-lived browser pool.
5. **Halykbank fixture is a negative example, not a positive one.**
   `tmp/testdata/sources/README.md` explicitly says halykbank's anti-headless
   logic loads rates via XHR after hydration, leaving the rendered DOM
   without values. **Decision:** do NOT add halykbank as a new seed row in
   this plan. The user's memory entry `feedback_prefer_chromedp_over_xhr.md`
   accepts that chromedp can fail for some sources — that is acceptable; the
   alternative (chasing XHR) is rejected. State this as an out-of-scope item
   below.
6. **Bankffin fixture is positive.** `bankffin.html` plus
   `bankffin_expected.json` exist in `tmp/testdata/sources/`. Adding a
   `KZ_BANKFFIN_*` seed row is technically possible here but expands scope.
   **Decision:** do NOT add it in this plan. The fixtures stand as future
   targets for a dedicated "add new sources" plan once chromedp is proven in
   production on the existing BCC homepage row.
7. **Co-location guard tuning.** The `Sanitize` co-location radius is
   currently 5 KB (`defaultCoLocationBytes` in `sanitizer.go`). JS-rendered
   pages may have larger gaps between structural anchors and rate values
   because client-side frameworks inject wrapper divs. If rule generation
   against the BCC homepage post-chromedp consistently fails the co-location
   guard, a follow-up plan tunes the radius. Out of scope.

## Trade-offs

1. **Generic `body visible + 1500 ms` vs per-source wait selectors.** Generic
   wins for v1 because the only initially supported chromedp source is the
   BCC homepage, where the static skeleton is fully laid out within ~1 s of
   navigation. A per-source `wait_selector` belongs in `rate_sources.options`
   JSON, but introducing it now would couple the schema change to a feature
   that no current row needs. **Decision:** generic. **Upgrade path:** add
   `Options.WaitSelector` when the second or third chromedp source needs it,
   and read it via the existing `domain.RateSourceOptions` struct without a
   new column.
2. **Fresh browser per call vs long-lived browser.** Fresh wins for
   `cmd/rulegen` because invocations are operator-scheduled, single-source,
   and tolerant of a 3 s cold-start. A long-lived browser would matter for
   `cmd/collector`'s periodic many-source loop, but plan 013 does not wire
   the fetcher into the collector. **Decision:** fresh-per-call. **Upgrade
   path:** plan 014+ extracts a `BrowserPool` type that the fetcher consumes
   instead of building its own allocator.
3. **`CHROMIUM_PATH` env var vs CLI flag.** Env wins because `cmd/rulegen`
   already reads its production config from env (`SQLITEDB_DSN`,
   `AI_PRIMARY_DSN`), and adding a flag would create two ways to express the
   same thing. **Decision:** env var. **Upgrade path:** add a `--chromium-path`
   CLI flag later if operator ergonomics demand it.
4. **Rename `"headless"` to `"chromedp"` vs alias.** Rename wins because
   "headless" describes the *requirement* while "chromedp" names the
   *implementation* — and the project tracks the latter in
   `__schema_migrations` and seed rows. Aliasing would mean every future
   validator switch carries both. **Decision:** rename via migration 011.
   **Upgrade path:** if a different headless engine (e.g. Playwright) is
   later integrated, the new kind would be `"playwright"`, not a re-aliased
   `"headless"`.
5. **Two `Fetcher` fields on Generator vs a factory.** Two fields wins
   because there are exactly two kinds today and the routing is a 4-line
   switch. A `FetcherFactory` interface would add indirection for no
   payoff. **Decision:** two named fields. **Upgrade path:** if a third
   kind appears, refactor to a `map[string]Fetcher` keyed by kind name in a
   focused PR.
6. **Adding new seed rows for halykbank/bankffin in this plan.** Out — the
   risks section spells out the reasoning. The halykbank case is genuinely
   adversarial (negative fixture), and bankffin is an unrelated source
   expansion that deserves its own plan once chromedp is proven in prod on
   the BCC homepage.
