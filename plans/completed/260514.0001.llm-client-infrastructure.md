# Task Breakdown

## Overview

Introduce a new infrastructure package `internal/infrastructure/artificialintelligence/` that exposes a typed `AIClient` interface and three concrete drivers — OpenAI, Groq, OpenRouter — plus a deterministic stub. The package is adapted from the reference implementation at `/Users/seilbekskindirov/Projects/developments/neuro_compass/internal/infrastructure/artificialintelligence/` and trimmed to match `fx_rate_monitor`'s needs.

This plan covers only the LLM client layer. A later plan will build the rule generator service that calls `AIClient.Complete()` with prompts derived from rate-source HTML, parses the JSON response into a rule, and validates the rule via `cmd/sourceaudit`. The schema delivered here (`rate_extraction_rule`) is the minimal contract the rule generator will consume; refinements live with that follow-up plan.

Two DSN env vars are wired into `cmd/web/main.go`:
- `AI_PRIMARY_DSN` — mandatory, intended for Groq (cheap/bulk worker on a free key).
- `AI_FALLBACK_DSN` — optional, intended for OpenAI or `anthropic/claude-*` via OpenRouter (higher-quality escalation). When absent or undecodable the bootstrap silently substitutes the stub so the project can run on a single Groq key.

`cmd/collector`, `cmd/notifier`, `cmd/migrator`, `cmd/sourceaudit`, and `cmd/wasm` are NOT touched. They have no current need for LLM access.

## Assumptions

1. The reference at `neuro_compass/.../artificialintelligence/` is the canonical source for the interface, DSN parsing helpers, OpenAI Responses-API integration, OpenRouter chat-completions integration, and the stub. We copy the structure and adapt to `fx_rate_monitor` conventions (package path, `internal.PublicError` / `internal.NewTraceError`, no `analysisIDContextKey`, no mandatory stored-prompt).
2. Groq's REST API is OpenAI-compatible at base URL `https://api.groq.com/openai/v1`. Using the official `github.com/openai/openai-go/v3` SDK with `option.WithBaseURL` would work, but pulling in a new heavy SDK for one provider is overkill; a hand-rolled chat-completions client adapted from `openrouter.go` is simpler and consistent with the existing house style (the project already speaks raw `net/http` for telegram, sqlite is hand-written too). See Trade-offs.
3. The user has API keys for OpenAI, Groq, OpenRouter but plans to run the service on a tiny host. Cost minimisation drives the "primary = Groq, fallback = optional" default.
4. The schema delivered now (`rate_extraction_rule`) is a placeholder shape — minimal but plausible. It is wired so the OpenAI/Groq/OpenRouter clients can attach a `json_schema` response format today. The rule-generator plan may iterate on field names; clients only care that `schema.go` exposes `RateExtractionRuleSchemaName` and `RateExtractionRuleSchema() map[string]any`.
5. `dsninjector.Unmarshal(envName)` parses an env-var-backed DSN. Existing code already uses it for `SQLITEDB_DSN` and `TELEGRAMBOT_DSN`. The driver portion of the DSN (`openai`, `groq`, `openrouterai`) selects which concrete client `NewClient(...)` constructs.
6. Logger handling matches the existing convention in `cmd/web/main.go`: pass `l.WriterAs(internal.LogLevelInfo)` (an `io.Writer`) into the constructor, which wraps it with a `log.Logger`.
7. `internal.PublicError` is for user-facing messages from the HTTP gateway. LLM failures are never shown verbatim to end users — they are internal. The package returns plain `error` values wrapped with `errors.Join(err, internal.NewTraceError())`, matching how neuro_compass handles errors and how `internal/repository/*.go` and `internal/infrastructure/sqlitedb/*.go` handle internal failures.
8. `system_prompt_for_analyze` from neuro_compass is dropped. fx_rate_monitor does not have stored prompts in any provider dashboard; the rule-generator will pass an inline system prompt to `Complete(ctx, systemPrompt, userMessage)`. This simplifies the bootstrap (no extra mandatory DSN option) and the openai branch (no Prompt.ID handling).
9. `analysisIDContextKey` / `WithAnalysisID` is dropped. fx_rate_monitor doesn't have an "analysis ID" concept in domain code, and the rule-generator can either log call sites itself or be added later via a generic `context.Value` key inside the rule-generator package. Adding correlation now is speculative abstraction.
10. The `_ =` discards on `defer resp.Body.Close()` are still allowed (matches `CLAUDE.md`'s "Close() in defer" exception). Other `_ =` patterns from neuro_compass tests must be replaced with explicit `require.NoError(t, err)` per the no-swallowed-errors rule.

### Task 2: Implement `schema.go` and `stubai.go`

- **Description:**
  Add `schema.go` exporting:
  - `RateExtractionRuleSchemaName = "rate_extraction_rule"` (matches the regex `^[a-zA-Z0-9_-]{1,64}$` required by `response_format.json_schema.name`).
  - `RateExtractionRuleSchema() map[string]any` — returns a freshly-built map per call so callers cannot mutate a shared instance.

  The schema represents a single rule for extracting a numeric rate value out of an HTTP response body. Proposed shape (placeholder; final naming will be fixed in the rule-generator plan):

        "type": "object",

  ```go
  // internal/infrastructure/artificialintelligence/groq.go
  //
  // Groq client. The Groq REST API is OpenAI-compatible at the chat-completions
  // path, so this driver is a thin wrapper over the shared openAICompatibleClient.
  //
  // DSN: groq://_:<base64url(KEY)>@api.groq.com/openai/v1?model=<model>&timeout=<dur>
  // Default model: llama-3.1-8b-instant (cheap, supports response_format=json_schema).
  package artificialintelligence

  func newGroqClient(dns dsninjector.DataSource, logger io.Writer) (*groqClient, error) {
      apiKey, err := parseDSNKey(dns)
      if err != nil { return nil, errors.Join(err, internal.NewTraceError()) }
      baseURL, err := url.JoinPath("https://"+dns.Addr(), dns.Database())
      if err != nil { return nil, errors.Join(err, internal.NewTraceError()) }
      model := "llama-3.1-8b-instant"
      if v := dns.Option("model"); v != "" { model = v }
      timeout, err := parseDSNTimeout(dns)
      if err != nil { return nil, errors.Join(err, internal.NewTraceError()) }
      return &groqClient{
          inner: openAICompatibleClient{
              baseURL: baseURL, model: model, apiKey: apiKey,
              httpClient: &http.Client{Timeout: timeout},
              logger: log.New(logger, "groq ", log.LstdFlags),
              providerName: "groq",
          },
      }, nil
  }

  type groqClient struct{ inner openAICompatibleClient }
  func (c *groqClient) Name() string { return "Groq[" + c.inner.model + "]" }
  func (c *groqClient) CheckUP(ctx context.Context) error {
      return c.inner.ping(ctx, "llama-3.1-8b-instant", chatPingPrompt, chatPingExpectedToken, chatPingMaxTokens)
  }
  func (c *groqClient) Complete(ctx context.Context, systemPrompt, userMessage string) (string, error) {
      return c.inner.complete(ctx, systemPrompt, userMessage, true)
  }
  ```

## Risks

1. **Schema drift between this plan and the rule-generator plan.** We commit to a placeholder `rate_extraction_rule` shape now. The rule-generator plan will discover real-world rules that don't fit (e.g. multi-step extraction: fetch sub-resource, then extract). _Mitigation:_ keep `RateExtractionRuleSchema()` in one place and make it a pure function so the rule-generator plan can refactor it without changing every client. The contract this plan commits to is the function signature, not the inner shape.
2. **OpenAI SDK transitive CGO.** Adding `github.com/openai/openai-go/v3` could in principle pull in CGO. _Mitigation:_ run `CGO_ENABLED=0 go build ./...` after `go mod tidy` and reject the dependency if the build fails. If the build fails, fall back to a hand-rolled Responses-API client modelled on `openai_compatible.go` (more work, but no surprise dependency).
3. **Groq model availability.** `llama-3.1-8b-instant` may be deprecated or renamed by Groq before this plan ships. _Mitigation:_ document the chosen default at the top of `groq.go` with a rationale; check Groq's model card before merging Task 4.
4. **Test flakiness on `t.Context()`.** The `t.Context()` helper used in the reference tests is Go 1.24+. fx_rate_monitor's `go.mod` declares `go 1.26`, so it is available, but verify before copying tests verbatim. If a CI image disagrees, fall back to `context.Background()` plus `t.Cleanup(cancel)`.
5. **Operational risk: secrets in env files.** The DSN format encodes the API key (base64url) in the password field of the env var. A leaked env file leaks the key. _Mitigation:_ document in the PR that `AI_PRIMARY_DSN` and `AI_FALLBACK_DSN` must be added to the systemd `EnvironmentFile` with mode `0600`, owned by the service user. This matches the existing policy for `TELEGRAMBOT_DSN`.
6. **Cost surprise.** Misconfiguring `AI_FALLBACK_DSN` to a paid frontier model (e.g. `anthropic/claude-3-opus`) could rack up cost quickly once the rule-generator starts calling it. _Mitigation:_ this plan only constructs the clients; no calls are issued. Add the cost warning to the rule-generator plan's risks, not here.

## Trade-offs

1. **Shared `openAICompatibleClient` vs two separate hand-rolled clients (Task 3 + 4).** Chose to share because Groq and OpenRouter both speak the same wire format and the duplication would be substantial (~150 LOC). Trade-off: introduces an unexported type with two parametric methods, slightly less local readability per file. Net win is clear: one place to fix bugs, one place to add retries later, one place to extend with new OpenAI-compatible providers (Anthropic offers an OpenAI-compatible endpoint too).

2. **Hand-rolled OpenAI compat client vs official SDK with `WithBaseURL` for Groq.** Chose hand-rolled. Reasoning: the official OpenAI Go SDK is only needed for the Responses API + structured-output strict mode, which is OpenAI-only. Pointing it at Groq's base URL works for chat-completions but adds a heavy dependency for a single second use. Hand-rolled chat-completions is ~80 LOC including tests and matches the project's existing house style (sqlite, telegrambot, dsninjector are all hand-rolled rather than wrapping a fat SDK).

3. **Drop `analysisIDContextKey` from the reference.** Chose to drop. Reasoning: fx_rate_monitor has no "analysis ID" domain concept; the rule-generator might want a `rule_generation_id` instead, but inventing a generic context key for "the thing this plan doesn't have yet" is speculative abstraction. Trade-off: when the rule-generator wants log correlation, it will need to add the feature back. That is acceptable; the cost of "add a context key in the next plan" is small, and the cost of carrying a misleadingly-named one now is wrong-by-default.

4. **Drop `system_prompt_for_analyze`.** Chose to drop entirely rather than make optional. Reasoning: making it optional spreads three code paths across each provider (`storedPrompt+systemPrompt`, `storedPrompt-only`, `systemPrompt-only`), as the reference openai.go shows. fx_rate_monitor's rule-generator will pass an inline system prompt; that's it. If we later want stored-prompt support, adding it as a new DSN option is a focused change. Optional-but-never-used is worse than absent.

5. **Primary mandatory, fallback optional.** Chose this asymmetry. Reasoning: the user runs on a tiny host with a free Groq key; mandatory primary mirrors the existing "all infra configs required" policy; optional fallback mirrors "fall back gracefully when premium escalation isn't configured." Alternative — both optional — was rejected because it makes the rule-generator's escalation logic dependent on runtime nil-checks instead of a stable contract.

6. **Log AI clients at startup via `Name()` instead of marking them as TODO discards.** Chose this so `make lint` does not flag unused values and so ops can see which providers were resolved at startup. Trade-off: a slightly noisier startup log. Worth it for the operational signal.

7. **No `internal.PublicError` wrapping in the AI package.** Chose plain `error` for all returns. Reasoning: LLM call failures (rate limit, 401, schema validation) are not safe to show to end users — they may contain API key fragments, request IDs, or provider-specific shape leaks. The HTTP gateway's fallback message ("Something went wrong. Try again later.") is the right surface for these. Trade-off: if a rule-generator failure ever does need a user-facing branch (e.g. "we cannot generate a rule for this source yet"), that branch lives in the rule-generator service layer, not here.
