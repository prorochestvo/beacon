# ADR-001: Adopt Go-WASM as the sole frontend

**Date:** 2026-05-05
**Status:** Accepted
**Supersedes:** —
**Superseded by:** —
**Related:** [`docs/reviews/260505.0001.project-review-wasm-focus.md`](../reviews/260505.0001.project-review-wasm-focus.md)

## Context

Today the project ships two parallel frontends:

- A vanilla-JS dashboard (`cmd/web/static/index.html`, ~16 KB) that handles
  every screen and interaction served by the web binary.
- A Go-WASM module (`cmd/wasm/main.go` → `cmd/web/static/app.wasm`, 3.3 MB)
  that re-implements a small subset of the same UI but is never loaded by any
  served page (no `<script src="wasm_exec.js">` tag exists in the HTML).

The review surfaced this as the dominant smell: the WASM artifact is built,
embedded, and shipped to clients on every request, yet does nothing. The
review proposed two options: **A.** delete the WASM target; **B.** commit to
WASM as the only frontend.

The owner is using this project as a personal learning environment for
WASM and has stated a clear preference for option B.

A common misconception was clarified before this decision: Go-WASM still runs
in the browser sandbox and cannot reach the server's database directly. All
data still flows through HTTP endpoints. WASM does not eliminate REST
endpoints; it changes the client that consumes them.

## Decision

Commit to **option B**. Go-WASM becomes the only frontend for the web binary.
The vanilla-JS implementation in `index.html` will be removed once the WASM
port reaches feature parity. The Telegram Mini App page
(`cmd/web/static/app/subscriptions.html`) is in scope as well — there will be
no JS-only screens left in the project.

The migration is staged through:

- A separate ADR or an addendum to this one if a frontend framework
  (`go-app`, `vecty`, `vugu`, `gomponents`, …) is adopted. The default choice
  is **no framework** — raw `syscall/js` plus a thin in-repo helper package.
  The owner's stated preference is to avoid extra dependencies, and this ADR
  records that constraint as a tie-breaker for any future framework debate.
- A migration plan in `plans/001-migrate-frontend-to-wasm.md` (architect-owned,
  per `CLAUDE.md`'s pipeline).
- An optional later plan to relocate web/static assets into a top-level
  `./web/` directory — tracked separately, not coupled to this decision.

## Consequences

### Positive

- A single language across the stack. Validation, formatting, and small
  business-logic helpers can be shared between server and client without
  rewriting them in JS.
- A shared `pkg/api` (or similar) for typed request/response DTOs eliminates
  the schema drift risk called out in the review (review §3).
- Dead code is gone: either WASM does the work, or it is removed — no third
  state where it ships but is unused.

### Negative / accepted trade-offs

- **Bundle size.** ~3 MB+ on cold load with the standard Go toolchain. TinyGo
  can shrink this to ~200–500 KB but with caveats (reflection limits,
  `encoding/json` quirks, sparser tooling). Compiler choice is deferred to
  the migration plan.
- **Build-tag isolation.** `//go:build js && wasm` keeps the frontend out of
  the regular `go vet`/`go test` runs. The migration plan must define a way
  to run WASM tests in CI (likely `wasm_exec` in a headless browser, or
  pure-logic tests behind a `!js` shim).
- **Empty-screen-until-loaded.** The first paint waits for WASM download +
  init. Mitigated by either an HTML skeleton, an SSR data-bootstrap block
  (`<script>window.__BOOT = {...}</script>` read by Go on startup), or both.
- **DOM access is verbose.** `syscall/js` plus `innerHTML` strings is fine for
  this project's scale but requires disciplined HTML escaping (review §5
  documents the current XSS asymmetry — it must not be reintroduced in the
  port).

### Out of scope for this ADR

- Choice of WASM framework or compiler (TinyGo vs. standard `go`) — to be
  resolved in conversation with the owner before the architect drafts the
  plan, and recorded either as an addendum here or as ADR-002.
- Relocation of static assets to `./web/` — independent task, separate plan.

## Alternatives considered

- **Option A: delete the WASM target.** Lowest-effort, removes 3.3 MB of
  shipped dead bytes and the build-tag isolation issue. Rejected because the
  owner wants to learn WASM on this project and the JS dashboard is small
  enough that re-implementing it in Go is a reasonable exercise rather than a
  rewrite.
- **Status quo (both frontends).** Rejected: the review documents this as the
  smell that prompted the conversation. There is no scenario in which
  shipping WASM-but-not-loading-it is the right outcome.

## Addendum (2026-05-05): framework, compiler, and package layout

The first open question from the original ADR ("choose framework / compiler")
is resolved here. Recorded inline because the decisions are tightly coupled
to ADR-001 itself; a separate ADR-002 would have added navigation overhead
without new substance.

### Framework: none (raw `syscall/js`)

Rejected: `maxence-charriere/go-app` (PWA bundle ships features the project
does not need; tightest mismatch with the "minimize dependencies" constraint),
`hexops/vecty` (vDOM is overkill for the screen count and hides the WASM
mechanics the owner is here to learn), `vugu` (custom file format and
codegen step complicate the build), `maragudk/gomponents` (deferred — may
be adopted later if HTML-string concatenation becomes painful; that change
will get its own ADR).

Chosen: **raw `syscall/js`** plus a thin in-repo helper package. Reasons:
zero new entries in `go.mod`, full visibility of the Go↔JS boundary (the
explicit learning goal), no build-pipeline changes, easy to evolve.

### Compiler: standard Go (TinyGo deferred)

Bundle size will be ~3 MB on cold load. TinyGo can compress to ~200–500 KB
but introduces caveats (reflection limits, `encoding/json` quirks, sparser
tooling). For a learning/admin project the size is acceptable. Switching to
TinyGo, if ever, will be its own ADR triggered by an explicit size or
perf requirement.

### Package layout

WASM gets its own layered structure mirroring the server's mental model.
It is **not** nested under `internal/gateway/` — the server's `gateway/` is
the *inbound HTTP boundary on the server*, while WASM lives at the
*opposite end of the wire*. The two are sibling concepts, not parent/child.

```
pkg/api/                       # shared DTOs, no build tag — imported by
                               # both the server (gateway handlers) and the
                               # WASM client. Eliminates the DTO drift
                               # called out in review §3.

internal/wasm/                 # WASM-only code. Every file:
                               #   //go:build js && wasm
│
├── application/               # Page controllers / state machines, one
│   │                          # file per screen. No DOM access here —
│   │                          # this layer can be unit-tested as plain Go.
│   ├── sources.go
│   ├── source_detail.go
│   └── errors.go
│
├── ui/                        # Rendering: pure State→HTML functions, plus
│   │                          # the apply-to-DOM step. This is the WASM
│   │                          # binary's "gateway" — its boundary to the
│   │                          # outside (DOM), the way internal/gateway/
│   │                          # is the server binary's boundary to HTTP.
│   ├── components.go
│   └── render.go
│
├── apiclient/                 # Typed HTTP client for /api/... — wrappers
│   └── client.go              # like client.ListSources(ctx) ([]api.Source, error).
│                              # Counterpart to internal/infrastructure/
│                              # on the server side.
│
└── dom/                       # Cross-cutting helpers: esc(), fetchJSON[T],
    └── helpers.go             # on(el, event, handler) with proper
                               # js.Func.Release() semantics. Counterpart
                               # to internal/tools/.

cmd/wasm/main.go               # Thin composition root: wires apiclient →
                               # application → ui → dom and starts.
                               # Counterpart to cmd/web/main.go.
```

#### Server↔WASM symmetry (cheat sheet)

| Server | WASM | Role |
|---|---|---|
| `cmd/web/main.go` | `cmd/wasm/main.go` | Composition root |
| `internal/gateway/` (HTTP-in) | `internal/wasm/ui/` (DOM-out) | Outer boundary |
| `internal/application/service/` | `internal/wasm/application/` | Orchestration |
| `internal/infrastructure/` | `internal/wasm/apiclient/` | External-system clients |
| `internal/tools/` | `internal/wasm/dom/` | Cross-cutting utilities |
| `internal/domain/` | `pkg/api/` (shared) | Types |
