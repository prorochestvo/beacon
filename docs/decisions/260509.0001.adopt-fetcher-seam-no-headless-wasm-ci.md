# ADR-002: Skip headless WASM in CI; rely on the Fetcher seam plus pure-Go controllers

**Date:** 2026-05-09
**Status:** Accepted
**Supersedes:** —
**Superseded by:** —
**Related:** [ADR-001](260505.0001.adopt-wasm-frontend.md), [Task 11 of the migration plan](../../plans/001-migrate-frontend-to-wasm.md)

## Context

After the WASM migration (Tasks 1–10), the only code that genuinely requires a
WASM-runtime test environment is the small `dom` glue layer
(`internal/wasm/dom/event.go`, `internal/wasm/dom/fetch.go`) and the thin
bridge `internal/wasm/apiclient/fetcher_wasm.go`. Every other package in the
WASM tree — `internal/wasm/application/`, `internal/wasm/ui/`,
`internal/wasm/apiclient/client.go`, `apiclient/urls.go`,
`apiclient/fetcher_http.go` — carries no `syscall/js` import and is exercised
by the regular `make test` run under the host OS. `pkg/api/` (shared DTOs) is
plain Go and always was.

The project currently has no CI pipeline: there is no `.github/workflows/`
directory and no `.gitlab-ci.yml`. A choice was therefore needed before the
migration plan could be called complete: introduce Node and the `wasm_exec.js`
headless runner as a first-class dev/CI dependency now, or accept that the
small `dom` glue layer is exercised only by manual browser smoke and opt-in
local runs. The architect evaluated both options before Task 11 was assigned
and recorded a recommendation; the owner confirmed option B.

## Decision

Option B is chosen. The `Fetcher` seam introduced in Task 3 and exercised
end-to-end in Task 10 (`internal/wasm/apiclient/client_integration_test.go`)
makes every `Client` method testable via `httptest.NewServer` under the host
toolchain — no WASM runtime required. Controller and UI logic was deliberately
structured to be pure Go and to return strings (`Render(state) string`), so
that 90%+ of the new WASM surface area is covered by `make test`. A new
`make test-wasm` target is added for developers who want to exercise the
remaining `dom` glue locally under the WASM runtime when Node.js is available.

## Consequences

- Positive: no Node dependency is introduced into the build or any future CI
  configuration. `make test` remains the single build-gate command.
- Positive: `make test` runs in seconds with no headless-browser setup or
  flake risk.
- Negative: `dom/event.go` (the `addEventListener`/`js.Func.Release` glue)
  and `dom/fetch.go` (the `js.FuncOf`/`window.fetch` bridge) are not exercised
  by `make test`. A regression in `js.Func` lifecycle discipline would only
  surface in browser smoke or when a developer explicitly runs `make test-wasm`.
- Negative: `cmd/wasm/main.go` (the composition root, wiring all layers
  together) is not covered by any automated test. Mitigation: manual browser
  smoke is required after each ship gate.
- Re-evaluate this decision when: a CI pipeline is added to the repository, or
  when a `dom`-layer regression reaches production and the cost of the gap
  becomes concrete.

## Alternatives considered

**Option A: headless `wasm_exec` in Node.** Run
`GOOS=js GOARCH=wasm go test -exec "$(go env GOROOT)/lib/wasm/go_js_wasm_exec" ./internal/wasm/...`
inside a Node.js environment in CI. Rejected for three reasons:

1. No CI pipeline exists yet; adding Node as a required system dependency is a
   non-trivial setup cost with no immediate return.
2. `wasm_exec.js` is tightly coupled to the Go toolchain version. The ABI has
   changed between major Go releases, and the path (`lib/wasm/` vs.
   `misc/wasm/`) shifted at Go 1.24. On Go upgrades the CI node job can silently
   break in ways unrelated to the application.
3. The layered architecture (Fetcher seam, pure-Go controllers, string-returning
   renderers) was designed specifically to minimise the surface area that
   requires a WASM runtime — Option A's benefit is therefore narrow.
