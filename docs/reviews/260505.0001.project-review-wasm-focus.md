# Project Review — WASM Focus

**Date:** 2026-05-05
**Author:** Claude (on user request)
**Branch state at review:** `main` @ `bbce1eb` (clean)

## Summary

User request: a pros/cons review of the project with focus on the WASM
integration — it intuitively feels wrong, but the reason is hard to articulate.

Main conclusion: **the WASM target is decorative**. The artifact is built,
embedded into the binary, and served by the file server, but no page ever
loads it. That is the source of the "something feels off" feeling — it is
dead weight that additionally duplicates JS functionality in a stripped-down
form and bypasses the project's shared architectural conventions.

## WASM cons (in decreasing order of severity)

### 1. Artifact is built, embedded, and used by nobody
`cmd/web/static/index.html` does not include `<script src="wasm_exec.js">`
and does not call `WebAssembly.instantiateStreaming`.
`cmd/web/static/app/subscriptions.html` doesn't either. So `Makefile:87-88`
builds a 3.3 MB `app.wasm` plus a 17 KB `wasm_exec.js`, `cmd/web/main.go:201`
embeds them into the binary, the file server hands them out — but the browser
never loads them.

### 2. WASM is a stripped-down duplicate of `index.html`
`index.html` already provides: source list, filters, sorting, source detail,
rate history, subscriptions, daily events, errors, pagination, and
`PATCH /active`. `cmd/wasm/main.go` (152 lines) only does the source list and
rate history. If WASM were ever loaded, it would be a feature regression.

### 3. Duplicated DTOs
`sourceInfo` (`cmd/wasm/main.go:11`) is a hand-rewritten JSON contract for
`/api/sources`. It is missing the `active` and `title` fields that the API
actually returns and that the JS consumes (`index.html:120-127`). Any schema
drift in `internal/gateway/...` will silently break WASM — its build is
separate, and it has no tests.

### 4. Build tag hides the code from tooling
`//go:build js && wasm` excludes `cmd/wasm/` from the regular `go vet ./...`
and `go test ./...` runs. The pipeline mandated by `CLAUDE.md`
(architect → engineer → reviewer, mandatory `make test`) physically does not
apply to this package.

### 5. XSS asymmetry
In `cmd/wasm/main.go:80-83` source names and currency codes are spliced into
`innerHTML` via `fmt.Sprintf` with no escaping, including inside
`onclick="window._loadRates('%s')"`. The JS version uses `esc()` everywhere
(`index.html:42-48`). If a source name ever contains `'`, `<`, or `&`, the
WASM branch is broken; the JS branch isn't.

### 6. Architectural mismatch
The global `js.Func` `_loadRates` lives forever, because it is invoked from
inline `onclick="..."` attributes (`cmd/wasm/main.go:36-44`). The Go↔JS
boundary is therefore still a string of HTML. Go's type safety is lost across
that boundary, and the simplicity of plain JS is lost too — the result is
~150 lines of Go rendering what ~30 lines of JS already render better.

### 7. Size
3.4 MB on cold load to render two tables, versus ~16 KB of vanilla JS already
embedded in `index.html`. Roughly 200× the bytes for less functionality.

## Recommendations (input for a future ADR)

Two honest paths, no middle ground:

**A. Delete it.** `rm -rf cmd/wasm`, drop the `Makefile:87-88` and
`Makefile:120` lines that build/clean `app.wasm` and `wasm_exec.js`.
The simplest and most coherent step — the current WASM does nothing.

**B. Commit to WASM.** Wire `wasm_exec.js` into `index.html`, port *all* of
the JS functionality (filters, sorting, pagination, PATCH) to Go, extract DTOs
into a shared `pkg/api` so frontend and backend share types, and add
`js && wasm` unit tests via `wasm_exec` in CI. This is significant work and
only justified by a concrete goal (typed contract, sharing business logic
between frontend and backend).

Without strong motivation — go with A. The decision should be captured in a
dedicated ADR under `docs/decisions/`.

## Project pros (brief)

- Clear layered architecture and the `PublicError` contract are described in
  `CLAUDE.md` and actually followed.
- `cmd/web/main.go` reads well: composition root, fail-fast on missing DSNs,
  clean `embed.FS` with a `--static-dir` override for development.
- Telegram WebApp initData HMAC auth is isolated in
  `internal/tools/tgwebapp/initdata.go` and injected as a function — testable.
- The `index.html` frontend has no bundler and no dependencies; it is readable
  in ten minutes. For an admin UI, that is a feature.
- Pure-Go SQLite (`modernc.org/sqlite`) plus `CGO_ENABLED=0` makes builds and
  cross-compilation trivial.

## Side notes (not WASM)

- `cmd/web/main.go:188` — the `time.Sleep(10 * time.Millisecond)` before the
  "initialization completed" log looks like a workaround. If the goal is to
  wait until `srv.ListenAndServe` is up, a listener probe is more correct;
  otherwise the sleep can simply be removed.
- `cmd/web/main.go:81` — the double `TrimPrefix("https://")` after
  `TrimPrefix("http://")` is correct but fragile. `dsninjector.Parse` already
  yields a parsed DSN — building the URL through `net/url` would be cleaner.
